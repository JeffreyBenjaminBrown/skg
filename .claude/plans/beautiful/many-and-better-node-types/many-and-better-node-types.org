* Shared context

The original trigger was a specific slow view ('to watch'): see [[../../speed.org][speed.org]]. Investigation showed the latency isn't fixable from skg's side — it lives in TypeDB's query scheduling. That prompted Jeff to ask a bigger question: what if we stopped depending on TypeDB for the common, cheap queries and kept an in-process graph that Rust owns?

Through dialog the correctness angle emerged: today the render pipeline consults three sources interchangeably — disk (via 'SkgNodeMap' and 'skgnode_from_map_or_disk'), TypeDB (for graph relations), and the per-connection pool ('SkgnodesInMemory' in [[../../server/types/memory.rs][server/types/memory.rs]]). Rules about which to consult are scattered. The scattered-ness is the real problem; perf would be a bonus.

The plan splits into three concerns, each independently valuable:

- *Plan A (detailed below)*: homogenize the SkgNode-ish types. Today 'SkgNode' is used everywhere and each destination silently ignores the fields it doesn't care about. Replace with a family of destination-specific structs so the type system enforces "TypeDB doesn't have bodies", "Tantivy doesn't have relations", etc.
- *Plan B (sketch below)*: unify merges through 'Vec<DefineNode>'. Today merges have per-store implementations (fs, TypeDB). Make merges produce 'DefineNode' instructions that all sinks apply uniformly, eliminating duplicate logic and pre-empting the "third parallel merge implementation" tax that Plan C would otherwise introduce.
- *Plan C (sketch below)*: introduce the in-Rust partial projection. Server-wide 'ArcSwap<Arc<Graph>>' holding 'im::HashMap<ID, NodeRust>'. Reads that fit the projection go here; advanced / recursive queries stay on TypeDB. Plan C becomes a small, clean plan once A+B have landed.

Decisions already locked through dialog:

- Projection tier: *A* — as wide as currently needed by in-memory consumers so no regression. Concretely: everything 'NodeComplete' has /except/ 'misc' (which is only read by Tantivy indexing and the org-roam importer, not by render/save/merge). We can always narrow later once a specific consumer is proven unused.
- Concurrency: *ArcSwap + im::HashMap*.
- Recursive queries stay on TypeDB indefinitely ('full_containerward_ancestry', 'containerward_path_stats_bulk', 'path_to_first_nonlinearity').

Ground truth (reused across plans):

- 'SkgNode' current fields ([[../../server/types/skgnode.rs][server/types/skgnode.rs]]): 'pid', 'source' (in-memory only; '#[serde(skip_...)]'), 'extra_ids', 'title', 'aliases: MSV<String>', 'body: Option<String>', 'contains: Vec<ID>', 'subscribes_to: MSV<ID>', 'hides_from_its_subscriptions: MSV<ID>', 'overrides_view_of: MSV<ID>', 'misc: Vec<FileProperty>'. 'source' is in memory but not on disk (inferred from file location). 'textlinks_to' is derived from body parsing, not stored on disk.
- Save pipeline ([[../../server/save.rs][server/save.rs]]): fs → TypeDB → Tantivy (latter two parallel), all consume 'Vec<DefineNode>'. Pool merged reactively afterward.
- Merge pipeline ([[../../server/merge.rs][server/merge.rs]], [[../../server/merge/typedb.rs][server/merge/typedb.rs]], [[../../server/merge/validate_merge.rs][server/merge/validate_merge.rs]]): separate path; reads and writes TypeDB directly without going through 'DefineNode'.
- Corpus: 29,050 .skg files, ~6.6 MB. In-RAM footprint < 50 MB.

* DONE Plan A (detailed): homogenize SkgNode-ish types

** Goal

Replace the single 'SkgNode' type, currently used by every store, with a small family of destination-specific structs. Each struct contains exactly the fields its destination needs and no others. Conversions between them are mechanical 'From' impls. No behavior change; pure refactor for type safety.

** Why now

Today, the save-to-Tantivy path takes a full 'SkgNode' and ignores 'contains', 'subscribes_to', 'overrides_view_of', 'hides_from_its_subscriptions'. The TypeDB path takes a full 'SkgNode' and ignores 'title', 'body', 'aliases'. Both /could/ accidentally read those fields and silently rely on them — the type system gives no protection. Future refactors compound this risk. Plan C in particular depends on a distinct 'NodeRust' type to enforce "this is the projection, the other fields aren't here."

'subscribes_to', 'hides_from_its_subscriptions', and
'overrides_view_of' use 'MSV<T>' (Maybe-Specified Vector; see
[[../../server/types/misc.rs][server/types/misc.rs]]), not 'Option<Vec<T>>'. The distinction between
"user didn't mention this field" and "user wants it empty" is real
and load-bearing.

***** Walkthrough: two options

You're right that 'NodeComplete' shouldn't serialize to YAML — the
only thing disk should see is the on-disk shape. And even
*de*-serialization is ambiguous: serde doesn't know the 'source',
so it has to default-construct it or leave it unset, and then
something outside serde attaches the real source.

So the question is: do we need 'NodeFS' as a distinct type, or can
we get the same safety with 'NodeComplete' and '#[skip_serializing,
skip_deserializing]' on 'source' (which is what we already do for
'SkgNode' today)?

****** Option X (no split): one type, 'NodeComplete'

Looks like today's 'SkgNode', renamed:

#+begin_src rust

      #[serde (skip_serializing, skip_deserializing)]
      pub source : SourceName,
      pub pid    : ID,

Reading YAML: serde produces 'NodeComplete' with 'source' at its
default (empty / "main"). Caller sets the real source based on
file location.

Writing YAML: serde skips 'source'. YAML on disk is bit-exact with
today.

- /Pros/: Simpler. One type. Matches today almost literally (just
  a rename). One fewer 'From' impl. No 'NodeFS' file to maintain.
- /Cons/: 'NodeComplete' can exist in a "source wasn't set" state
  immediately after deserialization. The type system doesn't force
  you to attach one — you just have to remember. That's today's
  footgun.
- /Is the footgun real?/: it /could/ bite if someone deserialized a
  node and passed it somewhere before source-attachment. In
  practice it hasn't: the filesystem read functions are the only
  deserialization site, and they attach source immediately. One
  location, one convention, no bugs observed.

* Problems, out of scope

Things noticed during exploration that aren't Plan A/B/C material but seem worth logging somewhere. Most could move into [[../../TODO/problems.org][TODO/problems.org]] at your discretion.

** Inconsistent concurrency discipline across TypeDB read wrappers

In [[../../server/dbs/typedb/][server/dbs/typedb/]] many functions that issue one TypeDB query per ID use 'stream::iter(...).buffer_unordered(TYPEDB_CONCURRENT_TRANSACTIONS)' — e.g. 'contains_from_pids', 'pids_from_ids', 'which_ids_exist'. But 'fetch_all_graphnodestats_from_typedb' uses unbounded 'join_all' instead (see [[../../server/dbs/typedb/search/all_graphnodestats.rs][server/dbs/typedb/search/all_graphnodestats.rs]]). No explanation in a comment; looks like an oversight. Consistent application of the cap is cheap to add and would make behavior predictable under heavy load.

** Doc drift on 'initial_node_limit' default

[[../../api-and-formats.md][api-and-formats.md]] says 'initial_node_limit' defaults to 100. [[../../server/consts.rs][server/consts.rs]] defines 'DEFAULT_INITIAL_NODE_LIMIT = 1000'. The hardcoded test default in [[../../server/types/misc.rs][server/types/misc.rs]] is 100. Pick one and update the other.

** Per-connection pool is populated as a reactive side-effect

Today the pool in 'SkgnodesInMemory' is populated after a render completes, by merging the transient 'SkgNodeMap' back in ([[../../server/update_buffer.rs][server/update_buffer.rs]] line ~107). This makes the pool implicitly stale relative to disk / TypeDB for the duration of a render. Plan C replaces this with an always-current server-wide graph, which is the clean fix. Noting here in case Plan C slips.

** 'pid_and_source_from_id' is a TypeDB round-trip for data already in the pool

'pid_and_source_from_id' resolves a node's primary id and source via TypeDB ([[../../server/dbs/typedb/search.rs][server/dbs/typedb/search.rs]] line ~127), but if the node is already in the pool its 'SkgNode' has everything we need (once 'source' moves onto 'NodeComplete' per Plan A's Problems section). Today this is a free TypeDB call per view render, per node. Plan C eliminates the call; Plan A eliminates the awkwardness of the source being external to the node. Worth noting as a latent inefficiency even without Plan C.

** 'textlinks_to' is the only derived-not-stored relation

Four of the five graph relations ('contains', 'subscribes_to', 'hides_from_its_subscriptions', 'overrides_view_of') live on 'SkgNode' on disk. 'textlinks_to' does not — it's derived from body parsing at init/save time. This asymmetry is invisible today because TypeDB materializes it and callers never know; but it becomes visible in Plan A ('NodeTypedb' needs an extra argument) and would become visible again if Tier B of Plan C ever tried to put 'textlinks_to' in the in-memory graph. Consider whether the asymmetry is worth preserving or whether 'textlinks_to' should become a real field on 'NodeComplete' and get written into the YAML on save. Pros of making it real: symmetry, easier reasoning. Cons: body-parse results become part of on-disk format, subtle versioning concern.

** 'SkgNodeMap' is a per-render transactional layer that's hard to reason about

The 'SkgNodeMap' threaded through every render function is a staging area between the pool and disk, with custom "check map first, then pool, then disk, then writeback to map" semantics ('skgnode_from_map_or_disk'). It's a small module but the mental model leaks into every caller. Plan C collapses it into a single 'graph.load_full()' call at function entry, after which reads are plain HashMap gets. Worth calling out as a meaningful simplification win for Plan C beyond what the sketch already says.

** 'merge_one_node_in_typedb' builds Cypher-ish queries by string concat

In [[../../server/merge/typedb.rs][server/merge/typedb.rs]] the merge logic generates TypeQL by format-stringing IDs directly into queries. Not an injection risk in this trust model (we control the IDs), but parameterized queries or a small query-builder abstraction would be cleaner. Plan B may naturally obsolete most of this file; if not, worth tidying separately.

** Does 'ViewNode' also need narrowing?

Narrowing in the sense that
NodeTypedb narrows from NodeComplete.

*** Answer: yes in principle, but the payoff is much smaller

'ViewNode' already narrows in one way — via its 'ViewNodeKind' enum
('True' / 'Scaffold' / 'Deleted' / 'DeletedScaff'). That sum-type
split handles the biggest "which data is relevant" distinction.

What's left is narrowing /within/ 'TrueNode': it carries 'title',
'body', 'source', 'birth', 'id', 'indef_or_def', 'graphStats',
'viewStats', 'diff', 'view_requests', 'parent_relation' — all on
every node, even though different render passes touch different
subsets. A 'TrueNodeBare' vs 'TrueNodeEnriched' split would mirror
the NodeComplete → NodeTypedb pattern.

Payoff is smaller than the 'SkgNode' split for two reasons:

1. 'ViewNode' is internal to the rendering pipeline. Bugs from
   reading-an-empty-field manifest as a bad render, not as a wire
   error or a silent disk divergence. So the cost of such bugs is
   lower.
2. Rendering is a pipeline of enrichment passes — BFS, 'graphStats'
   enrichment, diff application. Each pass /adds/ fields. A
   "progressive enrichment" via several narrow types + 'From's
   would need six-ish intermediate states instead of two-ish. That
   might be worth it for clarity, but it's a deeper restructuring
   than Plan A's "split by destination."

Verdict: not blocking Plan A. If a specific pass-confusion bug
appears after Plan A lands, revisit then.

'ViewNode' in [[../../server/types/viewnode.rs][server/types/viewnode.rs]] already has distinct kinds ('TrueNode', 'Scaffold', 'DeletedNode', 'DeletedScaff') and holds rendered fields like 'title', 'body', 'graphStats'. It is already fairly well-typed. But there are still places where a full 'TrueNode' is carried around when only a subset is used. Out of scope for Plan A, but worth a second-pass audit in the future.
