#+TITLE: Plan C: route reads through the in-Rust memory

* Context

"Plan C" in the parent plan ([[../../.claude/plans/many-and-better-node-types.org][many-and-better-node-types.org]]) introduces a server-wide, ArcSwap-fronted 'im::HashMap<ID, NodeRust>' — the "in-Rust memory" — so reads that fit the projection can bypass TypeDB. Phases 1, 2, and 4 are already on this branch: the memory is scaffolded ([[../../server/dbs/memory.rs][server/dbs/memory.rs]]), maintained on save and merge ([[../../server/save.rs][server/save.rs]], [[../../server/merge.rs][server/merge.rs]]), and rebuilt on rebuild-dbs ([[../../server/serve/handlers/rebuild_dbs.rs][rebuild_dbs.rs]]).

What this plan covers: phases 3 and 5 — /route reads through the in-Rust memory/ and /consistency-audit it/. For the /why/ (original latency trigger on the "to watch" view, correctness argument about scattered read sources, the ArcSwap + 'im::HashMap' concurrency choice), see the parent plan.

Terminology: "in-Rust memory" (or "memory") is the concept. The Rust type is 'Graph' (slated for rename to 'InRustMemory' — see Miscellaneous fixes). When talking about the role it plays in the system we say "in-Rust memory"; when talking about the data structure today we say 'Graph'.

* Current state

- 'Graph' in [[../../server/dbs/memory.rs][server/dbs/memory.rs]] holds 'im::HashMap<ID, NodeRust>' and nothing else. No inverse indexes.
- Every read still hits TypeDB or disk:
  - 'pid_and_source_from_id' ([[../../server/dbs/typedb/search.rs][dbs/typedb/search.rs]]) — TypeDB point query, called once per node in render paths.
  - 'skgnode_from_map_or_disk' ([[../../server/types/memory.rs][types/memory.rs]]) — disk fallback behind the per-request 'SkgNodeMap' shadow.
  - 'find_related_nodes' — per-relation TypeDB lookup (contains, subscribes, hides, overrides, textlinks_to).
  - 'fetch_all_graphnodestats' ([[../../server/dbs/typedb/search/all_graphnodestats.rs][all_graphnodestats.rs]]) — eight-field aggregate per node. Identified as the dominant latency contributor on "to watch".
  - Recursive containerward queries ([[../../server/dbs/typedb/ancestry.rs][ancestry.rs]], [[../../server/dbs/typedb/paths.rs][paths.rs]]).
- 'SkgnodesInMemory.pool' exists per-connection but is lazy-populated at view-open and largely redundant with the in-Rust memory (see [[../../one-memory-or-two.org][one-memory-or-two.org]]).

* Target state

- Read paths that fit the NodeRust projection are served from memory — no TypeDB hit.
- Full inverse-index coverage closes the "reverse-lookup" gap for all five outbound relations (contains, subscribes, hides_from_its_subscriptions, overrides_view_of, textlinks_to).
- Containerward recursive queries walk the in-Rust memory instead of TypeDB.
- 'fetch_all_graphnodestats' runs entirely in memory.
- 'SkgnodesInMemory.pool' and 'SkgnodesInMemory.id_resolver' are retired; the per-connection structure shrinks to view bookkeeping.

* Approach

** Step 1: widen NodeRust and add inverse indexes

Widen NodeRust (in [[../../server/types/nodes/rust.rs][server/types/nodes/rust.rs]]):

  - Add 'textlinks_to: Vec<ID>'. Derived at load / save / merge time by parsing body text — mirrors the existing derivation in 'NodeTypedb::from_complete_parsing_textlinks' ([[../../server/types/nodes/typedb.rs][nodes/typedb.rs]]). This breaks Plan A's earlier invariant "NodeRust = NodeComplete minus misc"; the new invariant is "NodeRust = NodeTypedb plus title + aliases + body" (i.e. what a render-time consumer might need, derivations included).

Add six fields on 'Graph':

  - 'contained_by            : im::HashMap<ID, im::HashSet<ID>>' — reverse of 'contains'.
  - 'subscribers_of          : im::HashMap<ID, im::HashSet<ID>>' — reverse of 'subscribes_to'.
  - 'hiders_of               : im::HashMap<ID, im::HashSet<ID>>' — reverse of 'hides_from_its_subscriptions'.
  - 'replacements_of         : im::HashMap<ID, im::HashSet<ID>>' — reverse of 'overrides_view_of'.
  - 'textlinks_in            : im::HashMap<ID, im::HashSet<ID>>' — reverse of 'textlinks_to'.
  - 'extra_id_to_pid         : im::HashMap<ID, ID>' — maps any of a node's extra_ids to that node's pid.

Maintain all six (plus the six forward fields on NodeRust) in 'apply_definenodes' and 'Graph::from_nodecompletes'. On a Save, diff the old node (if any) against the new: remove old inverse entries, add new. On a Delete, purge all inverse entries the deleted node created.

PITFALL: 'apply_definenodes' already reads the pre-save NodeRust before writing the post-save one (for the forward-edge replacement). Extend that read to cover /all/ outbound fields so the inverse-index diffs are correct. During a merge batch the COW + atomic publish boundary means no reader sees a half-applied state.

** Step 2: route 'pid_and_source_from_id' through memory

[[../../server/dbs/typedb/search.rs][search.rs]]'s 'pid_and_source_from_id' becomes a memory lookup: try 'Graph::get(id)' → 'NodeRust.source'; else try 'extra_id_to_pid.get(id)' → 'Graph::get(pid)' → 'NodeRust.source'. Returns None if both miss.

Signature change: takes '&Graph' (a snapshot taken by the caller via 'handle.load_full()') instead of '&TypeDBDriver'. Call sites: [[../../server/to_org/util.rs][to_org/util.rs]] ('skgnode_and_viewnode_from_id'), the initial-BFS path, and any other render-time callers.

Eliminates one TypeDB call per node in the render path.

** Step 3: route 'skgnode_from_map_or_disk' through memory

'skgnode_from_map_or_disk' in [[../../server/types/memory.rs][types/memory.rs]] currently returns '&NodeComplete' from the per-request 'SkgNodeMap' (with disk fallback). Change the fallback order to 'SkgNodeMap' → memory → disk.

Plan A established that no in-memory consumer reads 'NodeComplete.misc' (only Tantivy indexing and the org-roam importer do). With Step 1's widening, NodeRust also carries 'textlinks_to'. So a NodeRust looked up in memory has everything render / save / merge need. The caller receives a NodeRust (not NodeComplete) — this is a type change that ripples through the callers in [[../../server/to_org/][to_org/]] and [[../../server/update_buffer/][update_buffer/]], but they don't use the missing field.

Save-pipeline exception: during the in-flight portion of a save, 'SkgNodeMap' holds the /in-progress/ NodeCompletes (the save that's being built). The in-Rust memory doesn't have those yet (memory is updated only /after/ the FS write). Keep the 'SkgNodeMap' fallback ordering as written: SkgNodeMap first (captures in-progress edits), memory second (captures everything else), disk last (final fallback, which should rarely fire once memory is always-current).

** Step 4: migrate containerward recursive queries

- 'full_containerward_ancestry' in [[../../server/dbs/typedb/ancestry.rs][ancestry.rs]] → BFS over 'contained_by' in memory.
- 'containerward_path_stats_bulk' in [[../../server/dbs/typedb/paths.rs][paths.rs]] → BFS over 'contained_by' with fork/cycle detection in memory.
- 'path_containerward_to_first_nonlinearity' → linear walk over 'contained_by' in memory.
- 'path_sourceward_to_first_nonlinearity' → linear walk over 'textlinks_in' in memory (newly possible after Step 1's widening). Before Step 1, this stayed on TypeDB; after Step 1, it can move too.

Each migration replaces the function body; call sites are unchanged. Bonus: call-site TypeDBDriver parameters can be dropped where the function no longer needs one.

** Step 5: migrate 'fetch_all_graphnodestats' entirely

With Step 1's wide inverse indexes plus 'textlinks_in', every field in [[../../server/dbs/typedb/search/all_graphnodestats.rs][all_graphnodestats.rs]]'s output can be computed from memory:

  - 'num_contents'                     — 'NodeRust.contains.len()'
  - 'num_containers'                   — 'contained_by.get(pid).map_or(0, |s| s.len())'
  - 'num_links_in_from_containers'     — iterate 'textlinks_in.get(pid)', count those whose NodeRust has non-empty 'contains'.
  - 'num_links_in_from_leaves'         — same iteration, count those with empty 'contains'.
  - 'has_subscribes'                   — 'NodeRust.subscribes_to' non-empty OR 'subscribers_of.get(pid)' non-empty (see clarification on directionality in Problems).
  - 'has_overrides'                    — 'NodeRust.overrides_view_of' non-empty OR 'replacements_of.get(pid)' non-empty.
  - 'container_to_contents'            — iterate pids, read 'NodeRust.contains'.
  - 'content_to_containers'            — 'contained_by' lookup per pid.

Nothing stays on TypeDB for this function. The entire 'fetch_all_graphnodestats' becomes an in-memory pass: microseconds per node, not milliseconds.

** Step 6: shrink 'SkgnodesInMemory'

Once Steps 2-3 land, 'SkgnodesInMemory.pool' is strictly redundant with memory. Remove:

  - 'pool: HashMap<ID, NodeComplete>'
  - 'id_resolver: HashMap<ID, (ID, SourceName)>' — its role is now covered by 'extra_id_to_pid' plus 'NodeRust.source'.

Remaining fields — 'views', 'root_ids' — are genuinely per-connection view state. Keep the struct, consider renaming to reflect its new scope ('ViewRegistry' or similar). Out of scope for this plan; flag as follow-up.

Also remove the post-render pool-merge side-effect in [[../../server/update_buffer.rs][update_buffer.rs]] (~line 107). The pool no longer exists; nothing to merge into.

Commits can be fine-grained: land Step 1's data-structure additions, then each read-path migration, then the pool removal. The atomic unit is "build compiles + tests pass", not "one PR contains all of Plan C".

** Step 7: consistency-audit helper

A test-time function 'audit_memory_against_typedb' (new module [[../../server/dbs/memory/audit.rs][server/dbs/memory/audit.rs]]) that compares the in-Rust memory to TypeDB and returns a list of mismatches. Its action, per node:

  1. /Forward pass/: for each of the five outbound relations (contains, subscribes_to, hides_from_its_subscriptions, overrides_view_of, textlinks_to), read NodeRust's field in memory, query TypeDB for the same relation's peers, and diff the two sets.
  2. /Inverse pass/: for each of the five inverse indexes ('contained_by', 'subscribers_of', 'hiders_of', 'replacements_of', 'textlinks_in'), read the index entry in memory, query TypeDB for the inverse-role peers, and diff.

Any difference is a 'Mismatch' record carrying the node id, the relation, and the in-memory vs TypeDB sets. An empty 'Vec<Mismatch>' means memory and TypeDB agree for every node in the fixture.

Callable from tests: add one call after each save/merge in the existing merge tests and in [[../../tests/move_source.rs][move_source.rs]]. Cost scales with the test fixture's node count; running it against a small fixture (< 100 nodes) takes < 1 s. Document in the helper's doc-comment that it is intended for /test fixtures/ and is not suitable for running over the full 29k-node production corpus (which would take minutes).

The purpose is to catch silent drift: if 'apply_definenodes' has a subtle bug — e.g. an inverse-index update missed on one specific relation — the audit makes it visible immediately after the save/merge that introduced it, rather than letting drift accumulate.

** Step 8: opt-in scheduled audit

Beyond the test-time use in Step 7, expose the audit as an opt-in scheduled-at-startup check for live servers.

*Config*: add 'auto_audit_daily: bool' to the config struct in [[../../server/types/misc.rs][server/types/misc.rs]] (the 'SkgConfig' type) plus its TOML parser. Default false. Set it to true in [[../../data/skgconfig.toml][data/skgconfig.toml]] (Jeff's own config) as part of this step.

*Staleness check at startup*: in [[../../server/main.rs][server/main.rs]]'s post-init path, if 'auto_audit_daily' is true, compare today's date (local date, via 'chrono' or manual date formatting from 'SystemTime') against the contents of '{data_root}/last-audit.time'. If the file is missing or its date is earlier than today, schedule an audit.

*Backgrounded-at-lowest-priority execution*: spawn the audit via 'std::thread::spawn' and drop the thread's scheduling priority before running. On Linux the cheapest way is 'libc::nice(19)' (nicest; runs only when CPU is idle); a heavier but more thorough option is 'SCHED_IDLE' via 'libc::sched_setscheduler'. Either way, the audit must never block the main connection loop. When it finishes, write today's date to 'last-audit.time' regardless of result.

*Error reporting*:

  - If the audit returns an empty 'Vec<Mismatch>': silent success. Update 'last-audit.time' and exit the thread.
  - If non-empty: append a dated headline to '{data_root}/audits.org' with one sub-headline per 'Mismatch'. Append, don't overwrite — the file is a running log.
  - Notify the client by /piggy-backing on the next outbound buffer/ (view render or save response). The audit thread sets a "pending audit warning" slot on 'ConnectionState' (or a server-wide slot if reached before any connection). The next time the server is about to send a buffer to a client — whatever the triggering request was — it prepends a warning line like "N audit errors detected during the daily audit — see 'audits.org'." and clears the slot. No new message type in the TCP protocol; no client polling.

*Scope of the audit*: full corpus. At 29k nodes × 10 TypeDB queries the audit takes minutes; acceptable because it runs at nice(19) / SCHED_IDLE so the user never feels it. If that proves painful, sharding (audit 1/Nth of nodes per day) is a low-effort follow-up.

*audits.org format*:

  * 2026-04-21 (42 errors)
  ** Node 'abc-def-...': relation 'contains' mismatch
  Memory has: [x, y, z]
  TypeDB has: [x, y]
  ** Node ...

The audit is a check for silent drift between the in-Rust memory and TypeDB. It does this, for every node in the in-Rust memory:

The three-tier fallback of 'skgnode_from_map_or_disk' (SkgNodeMap → memory → disk) matters because the in-Rust memory is updated only /after/ the FS write, while 'SkgNodeMap' carries in-progress edits for the current save. Keep the fallback order. If a reader somehow runs /during/ a save (shouldn't happen in single-client, but a background enrichment thread could), the memory snapshot it holds is the pre-save one — consistent, not stale-half-done. The consistency story is: "every reader sees a consistent snapshot; writers publish atomically."

** DONE Deciding when to take the memory snapshot
Rendering a view typically runs many lookups. Today each 'skgnode_from_map_or_disk' call may re-consult the per-request map. After Step 3, the pattern becomes: take a single 'Arc<Graph>' via 'handle.load_full()' at request start; thread it through. If the render pipeline is long-running and a save commits mid-render, the render still sees the pre-save memory. That's fine for correctness (consistent snapshot) but means a freshly-saved node won't appear in an in-progress render. Current behavior is similar (pool is also a snapshot). No regression; worth documenting.

Org-mode's emphasis parser scans for paired apostrophes as part of various emphasis / smart-quote / quotation rules; a sequence of 40+ unclosed-looking ''...'' markers across a long file can trigger pathological regex backtracking in 'org-do-emphasis-faces' or related font-lock regexes, especially inside links and property blocks.
