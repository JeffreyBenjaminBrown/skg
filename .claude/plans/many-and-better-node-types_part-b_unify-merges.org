#+TITLE: Plan B: unify merges through Vec<DefineNode>

* Context

Today, [[../../server/merge.rs][server/merge.rs]] dispatches to four per-sink merge modules. Three of them ([[../../server/merge/fs.rs][fs.rs]], [[../../server/merge/tantivy.rs][tantivy.rs]], and the in-memory graph block inline in 'merge_nodes') already do the same thing: flatten each Merge to three 'DefineNode's via ''Merge::to_vec'' and pipe those through the ordinary save-pipeline functions. Only [[../../server/merge/typedb.rs][merge/typedb.rs]] still has per-relation rewiring code that bypasses the save pipeline.

The goal: replace [[../../server/merge/typedb.rs][merge/typedb.rs]] with the same "flatten to 'Vec<DefineNode>', hand to the save pipeline" shape the other three sinks use. When done, 'merge_nodes' becomes a thin wrapper that computes one 'Vec<DefineNode>' and fans it out through the existing ''update_*_from_saveinstructions'' functions. One code path for merges; the instruction-list vocabulary is shared with normal saves.

For the big-picture motivation — why we wanted a uniform 'Vec<DefineNode>' across sinks in the first place, and how this interacts with the in-memory graph (Plan C) — see [[../../.claude/plans/many-and-better-node-types.org][many-and-better-node-types.org]]. This plan only describes the /what and how/ of Plan B.

* Current state

Entry point: ''merge_nodes'' in [[../../server/merge.rs][server/merge.rs]]. For each sink, the status today (on branch 'many-and-better-node-types'):

- *Filesystem*: [[../../server/merge/fs.rs][merge/fs.rs]] calls ''update_fs_from_saveinstructions'' on the flattened 'DefineNode's. Uniform. Good.
- *In-memory graph*: the inline block in 'merge_nodes' flattens Merges via ''Merge::to_vec'' and calls ''apply_definenodes''. Uniform. Good.
- *TypeDB*: [[../../server/merge/typedb.rs][merge/typedb.rs]] performs ~15 distinct TypeQL mutations per merge (see ''merge_one_node_in_typedb''). Calls ''create_node'' for the preserver, then eight ''reroute_relationships_for_merge'' / ''drop_relationships_for_merge'' calls for the five relation types and their two roles, plus ''reroute_what_acquiree_hides'' (which itself runs a TypeDB query), plus ''reroute_extra_ids_for_merge'' and ''create_and_connect_extra_id'', plus the acquiree delete, plus the final ''insert_relationship_from_list'' to add the preserver to the acquirer's contents. Not uniform. Plan B replaces this.
- *Tantivy*: [[../../server/merge/tantivy.rs][merge/tantivy.rs]] calls ''update_tantivy_from_saveinstructions'' on the flattened 'DefineNode's. Uniform. Good.

The 'Vec<DefineNode>' that ''Merge::to_vec'' produces ([[../../server/types/save.rs][server/types/save.rs]] line 185) is three instructions per Merge:

  - 'SaveNode(acquiree_text_preserver)' — a brand-new node carrying the acquiree's body.
  - 'SaveNode(updated_acquirer)' — the acquirer with its new 'contains', 'extra_ids' (includes acquiree's pid and extra_ids), 'subscribes_to', 'hides_from_its_subscriptions', and 'overrides_view_of' already baked in by ''three_merged_skgnodes'' in [[../../server/merge/mergeInstructionTriple.rs][mergeInstructionTriple.rs]].
  - 'DeleteNode(acquiree_to_delete)'.

Those three 'DefineNode's fully describe what the merge does /from the acquirer's and acquiree's perspective/. What they don't describe are *inbound* edges: relations in other nodes that pointed at the acquiree.

* The simplifying insight

Every inbound reference to the acquiree, whether it lives in another node's 'contains' / 'subscribes_to' / 'hides_from_its_subscriptions' / 'overrides_view_of' field or is derived from another node's body text (a textlink), stays valid /without being rewritten/. The acquiree's pid becomes an 'extra_id' on the acquirer (that's already in 'updated_acquirer.extra_ids'). TypeDB's ''insert_relationship_from_list'' ([[../../server/dbs/typedb/relationships.rs][server/dbs/typedb/relationships.rs]] line ~150) already resolves a reference either directly or through 'has_extra_id'. So a neighbor node whose 'contains' vec still holds 'acquiree_id' will, on any save, have a TypeDB 'contains' relation created pointing at the acquirer.

That's also the answer to "does N still hide or override the merged entity post-merge?" — yes, because the user intent was expressed toward one of the two merged nodes and those are now one thing. Extra_id resolution propagates that intent automatically.

Two consequences:

1. *No neighbor .skg file edits are required.* Neighbor 'contains' / 'subscribes_to' / 'hides_from_its_subscriptions' / 'overrides_view_of' vectors stay as they are, with 'acquiree_id' in place; resolution to acquirer happens at read time.

2. *Neighbor SaveNodes are still needed — but only to close the temporal gap in TypeDB.* TypeDB cascade-deletes every relation involving the acquiree when the acquiree is deleted. Without something re-creating those relations immediately, TypeDB is inconsistent with FS until each affected neighbor happens to be saved again for unrelated reasons. Re-saving those neighbors /now/ (with their 'NodeComplete's unchanged) is enough: the save pipeline cascade-re-creates their outbound relations, extra_id resolution points them at the acquirer. FS and TypeDB end up identical post-merge.

* Target state

''merge_nodes'' in [[../../server/merge.rs][server/merge.rs]] shrinks to roughly:

  1. For each Merge, compute the three primary DefineNodes ('Merge::to_vec()') plus a set of "affected neighbor" SaveNodes (defined below).
  2. Concatenate into one 'Vec<DefineNode>'.
  3. Call ''update_fs_from_saveinstructions'', ''apply_definenodes'' on the graph, ''update_typedb_from_saveinstructions'', ''update_tantivy_from_saveinstructions'' in the same order as today's [[../../server/save.rs][save.rs]].

[[../../server/merge/typedb.rs][server/merge/typedb.rs]] goes away, along with [[../../server/merge/fs.rs][server/merge/fs.rs]] and [[../../server/merge/tantivy.rs][server/merge/tantivy.rs]] (they become trivial wrappers at that point, so fold them into 'merge_nodes' directly).

* Approach

** Step 1: discover affected neighbors

A neighbor N is "affected" if any of the following is true in TypeDB — in each case, the acquiree plays the object (second relationship member) and N plays the subject (first relationship member) — i.e. the outbound edge lives on N, pointing at acquiree:

  - N plays 'container' in a 'contains' relation (N contains acquiree).
  - N plays 'subscriber' in a 'subscribes' relation (N subscribes to acquiree).
  - N plays 'source' in a 'textlinks_to' relation (N's body has a textlink to acquiree).
  - N plays 'hider' in a 'hides_from_its_subscriptions' relation (N hides acquiree).
  - N plays 'replacement' in an 'overrides_view_of' relation (N replaces acquiree's view).

Everything where acquiree played the /subject/ (first relationship member) — acquiree contains X, acquiree subscribes to Y, acquiree's body has textlinks out to Z, acquiree hides W, acquiree replaces V — is already covered by the primary 3×N DefineNodes: updated_acquirer's fields inherit those, and the preserver re-derives its outbound textlinks from the preserved body text when saved. No neighbor save needed for those.

Add a function ''affected_neighbors_of_merge'' (probably living in [[../../server/merge/mergeInstructionTriple.rs][mergeInstructionTriple.rs]] alongside ''three_merged_skgnodes'') that:

  - Issues the five TypeDB queries above for a given acquiree_id, returning a deduplicated 'Vec<ID>' of PIDs.
  - Filters out any PIDs that are themselves acquiree_ids or acquirer_ids in /any/ merge in the current batch — those are already handled by the primary 3×N DefineNodes, and re-saving them would double-process.

For each affected neighbor PID, load its current 'NodeComplete' from disk (via ''skgnode_from_id'' in [[../../server/dbs/filesystem/one_node.rs][server/dbs/filesystem/one_node.rs]]) and wrap it in 'SaveNode'. The NodeComplete's fields are /unchanged/ — acquiree_id stays in whatever vector it's in. Extra_id resolution does the work.

Concatenate: '[primary 3×N DefineNodes] ++ [neighbor SaveNodes]' → one 'Vec<DefineNode>'.

** Step 2: fix latent save-pipeline gaps

Two things that ''update_typedb_from_saveinstructions'' in [[../../server/save.rs][server/save.rs]] does not do today, which Plan B needs:

*** Extra_id updates for existing nodes

''create_only_nodes_with_no_ids_present'' in [[../../server/dbs/typedb/nodes.rs][dbs/typedb/nodes.rs]] calls ''insert_extra_ids'' only for /new/ nodes (ones whose pid isn't already in TypeDB). For existing nodes being re-saved — which includes every merge's 'updated_acquirer' — the new extra_ids never get written.

Fix: before ''create_all_relationships'', for every 'to_write_pid' that already exists, delete all 'has_extra_id' relations with that node as 'node', then call ''insert_extra_ids'' with the node's current extra_id list. Idempotent; same pattern as today's "delete outbound contains then recreate" dance.

This fix benefits /all/ saves, not just merges. Today, a user editing a node's extra_ids via the save pipeline would silently drop the change in TypeDB — that's a latent bug that Plan B happens to expose.

*** Outbound-relation cleanup for all relation types

''update_typedb_from_saveinstructions'' currently calls ''delete_out_links'' only for ('contains', 'container'). It then runs ''create_all_relationships'' which inserts relations for all five types. For an existing node being re-saved, outbound textlinks_to / subscribes / hides / overrides get /duplicated/ because their old copies weren't deleted.

Fix: loop over ''OUTBOUND_RELATIONSHIP_TYPES'' in [[../../server/dbs/typedb/relationships.rs][dbs/typedb/relationships.rs]] and call ''delete_out_links'' for each '(relation_name, role_a)'. Then ''create_all_relationships'' runs against a clean slate.

Also a latent bug Plan B exposes: today, re-saving an existing node with (say) edited textlinks would leave the old textlinks in place alongside the new ones.

** Step 3: rewrite 'merge_nodes'

The new ''merge_nodes'' body:

  1. If 'merge_instructions' is empty, return Ok.
  2. For each Merge, call ''affected_neighbors_of_merge'' to find neighbor PIDs.
  3. Dedup neighbor PIDs across the batch; subtract all acquirer_ids and acquiree_ids.
  4. Load neighbor NodeCompletes from disk; wrap in SaveNode.
  5. Concatenate primary DefineNodes + neighbor SaveNodes.
  6. Call ''update_fs_from_saveinstructions'' with empty source_moves. On error, return.
  7. Call ''apply_definenodes'' on the graph handle.
  8. Call ''update_typedb_from_saveinstructions''. On error, ''rebuild_typedb_from_disk'' (same recovery as today's 'merge_nodes').
  9. Call ''update_tantivy_from_saveinstructions''. On error, ''rebuild_tantivy_from_disk'' and return the new index.

Structurally identical to ''update_graph_minus_merges'' in [[../../server/save.rs][save.rs]]. In fact after Plan B the two functions are different only in /what they put into the Vec<DefineNode>/; the fan-out is the same. A single shared helper that takes a 'Vec<DefineNode>' and applies it to all four sinks with the usual recovery would finish the unification — noted as a follow-on.

* Problems

** DONE How many TypeDB queries per merge?

Step 1 issues five queries per acquiree (one per relation-role pair). On a batch of N merges, that's 5N queries. Each returns at most thousands of neighbors in an adversarial case, typically O(10). Not obviously a problem — merges are rare and user-triggered — but flagging it because Plan C has established that TypeDB queries are the /latency/ bottleneck we set out to address. Worth measuring before concluding it's fine.

Possible mitigation: if Plan C phase 3 (route reads through graph) has landed, three of the five queries ('contains', 'subscribes', 'hides', 'overrides' inbound) could come from the in-memory graph /if/ the graph maintains inverse indexes. It does not today. Building those indexes is Plan C's concern; Plan B should not block on them.

Textlinks inbound cannot come from the graph — the graph stores NodeRust, which omits 'textlinks_to'. That query stays on TypeDB regardless.

** DONE Neighbor NodeComplete loads hit the disk

Step 1 loads N neighbor NodeCompletes from .skg files after discovering their PIDs. For typical merges (few neighbors), negligible. For pathological merges (thousands of neighbors), multi-second. Flagging because it scales with neighbor count.

** DONE Tantivy over-indexing

** Ordering within the DefineNode batch

''update_typedb_from_saveinstructions'' partitions into 'to_delete' and 'to_save' and processes all deletes before all saves. That's fine for us: the acquiree delete cascades first, then the preserver save creates the preserver, then the updated_acquirer and neighbor saves re-establish relations with extra_id resolution pointing at the acquirer.

One subtle ordering requirement: 'has_extra_id' for the acquirer (the new acquiree-pid-as-extra-id) must exist /before/ any neighbor save runs, because the neighbor saves resolve acquiree_id through has_extra_id. Current ''create_all_relationships'' does not create has_extra_id; ''create_node'' does via ''insert_extra_ids'' /but only for new nodes/. So the Step 2 fix (extra_ids update for existing nodes) must happen strictly before the per-neighbor ''create_all_relationships''.

Concretely: the Step 2 fix's "delete all has_extra_id then re-insert" pass should happen once, for all saves, before the ''create_all_relationships'' loop. Not per-node-interleaved. That's a schedule constraint worth writing into the fix's implementation.

The implementation must carry a PITFALL comment at the extra_ids-update call site in ''update_typedb_from_saveinstructions'' explaining the ordering requirement — specifically: "This pass must complete for every to_write_pid /before/ any ''create_all_relationships'' call runs, because those calls resolve target IDs through 'has_extra_id', and a neighbor save whose target is the acquirer needs the freshly-added has_extra_id (acquiree_pid → acquirer) to exist at lookup time."

** The three-DefineNode abstraction is output-oriented; merge is input-oriented

The real architectural tension behind Plan B: 'DefineNode' describes /what a node should look like after this operation/. A merge, in its natural formulation, describes /what was pointing at the acquiree that must now point at someone else/. The simplifying insight (extra_id resolution does the rewiring) bridges the gap without changing the abstraction — but only because extra_id resolution happens to cover all five relation types' inbound cases. If a future relation type is added that extra_id resolution /doesn't/ handle (e.g., a relation that must distinguish the merged nodes rather than treat them as equivalent), Plan B's approach would need revisiting. Not a concern today; worth remembering.
