* Context

'SkgNodeMap = HashMap<ID, NodeComplete>' is threaded as '&mut SkgNodeMap' through the render, complete-viewtree, and save pipelines as a per-request transactional shadow. Pre-Plan-C (from .claude/plans/beautiful/many-and-better-node-types/many-and-better-node-types_part-c_use-in-rust-memory.org) it had two jobs:

  1. /Read cache/: accumulate NodeComplete lookups so a pipeline stage doesn't re-read the same node twice (from disk, TypeDB, or pool).
  2. /In-progress save state/: built from 'Vec<DefineNode>' via 'skgnode_map_from_save_instructions', passed into rerender so the post-save view reflects the new state /before/ any store other than FS committed.

After Plan C, both are redundant. The in-Rust memory ([[../../server/dbs/memory.rs][server/dbs/memory.rs]]) is updated by 'apply_definenodes' during the save pipeline ([[../../server/save.rs][save.rs]]:63, [[../../server/merge.rs][merge.rs]]:63) /before/ any rerender runs, so memory carries the post-save state by the time rendering happens. Memory reads are effectively free: 'snapshot_global' is an 'ArcSwap' load (one atomic), and 'im::HashMap::get' is O(1). The SkgNodeMap just duplicates a snapshot memory already holds.

Removing it collapses ~60 call-site signatures and drops one conceptual layer from the "where does this NodeComplete live?" model. Most of the diff is mechanical (deleting '&mut map' parameters and arguments); the interesting work is choosing the right /replacement helper/ at each read site — and in particular, avoiding an async-contamination cascade through sync tree walkers.

* Design

** Two replacement helpers, not one

The sketch's plan routes every read through the async 'skgnode_from_memory_or_disk'. That's wrong for the ~8 sync call chains: it forces async infection through recursive sync tree walkers that have no other reason to be async. Concretely:

  - 'set_metadata_relationships_in_node_recursive' at [[../../server/update_buffer/graphnodestats.rs][graphnodestats.rs]]:48 is sync recursive. Making it async forces 'Pin<Box<dyn Future>>' around the recursive call, or a CPS-style rewrite.
  - 'clobberIndefinitiveViewnode' at [[../../server/to_org/complete/contents.rs][to_org/complete/contents.rs]]:16 is sync and called from sync contexts under 'complete_truenode_preorder' at [[../../server/update_buffer/complete_parent_first/truenode.rs][complete_parent_first/truenode.rs]]:63 — which has no other async work, and would become async purely to satisfy the lookup.
  - 'skgnode_and_viewnode_from_pid_and_source' at [[../../server/to_org/util.rs][to_org/util.rs]]:60 is sync for the same reason.

The fix: expose a /sync/ memory-or-disk helper in [[../../server/types/memory.rs][server/types/memory.rs]] that requires '(pid, source)' (so no id→source resolution is needed) and never reaches TypeDB. The sync helper takes the short name 'skgnode_from_memory_or_disk'; the existing async helper (which takes an id and does id→source resolution) is renamed to 'skgnode_from_memory_or_disk_async':

#+begin_src rust
/// Memory-first NodeComplete read, disk fallback, given an
/// already-resolved '(pid, source)'. Sync — never consults TypeDB,
/// so callers in sync contexts don't have to become async. For the
/// id-only path (id→(pid,source) resolution plus lookup) use the
/// async 'skgnode_from_memory_or_disk_async'.
pub fn skgnode_from_memory_or_disk (
  config : &SkgConfig,
  pid    : &ID,
  source : &SourceName,
) -> Result<NodeComplete, Box<dyn Error>> {
  if let Some (n) = nodecomplete_from_memory (pid)
    { return Ok (n); }
  Ok ( skgnode_from_pid_and_source (
         config, pid . clone (), source ) ? ) }
#+end_src

Every current sync call site already has '(pid, source)' in scope (the source is on the ViewNode), so this is a drop-in replacement for those.

Async sites ([[../../server/to_org/expand/backpath.rs][backpath.rs]], parts of [[../../server/to_org/util.rs][util.rs]], 'attach_containerward_ancestries_to_removedhere_phantoms' in [[../../server/update_buffer.rs][update_buffer.rs]], 'skgnode_and_viewnode_from_id' in [[../../server/to_org/util.rs][util.rs]]) route through the renamed async 'skgnode_from_memory_or_disk_async' — which handles id→(pid,source) resolution itself.

Agreed — the name I picked parses ambiguously. Your suggestion is right: keep the short name 'skgnode_from_memory_or_disk' for the function whose name tells the whole story ("try memory, else disk"), and rename the existing one to 'skgnode_from_memory_or_disk_async' to mark that it's the async variant.

The two variants also differ in their /input/, not just sync-vs-async:
