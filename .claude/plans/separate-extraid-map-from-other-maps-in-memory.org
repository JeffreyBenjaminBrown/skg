#+TITLE: Canonical-keyed maps (forward and inverse) for the in-Rust memory

* Context

The in-Rust memory ([[../../server/dbs/memory.rs][server/dbs/memory.rs]]) keeps five inverse indexes ('contained_by', 'subscribers_of', 'hiders_of', 'replacements_of', 'textlinks_in') plus 'extra_id_to_pid'. Today, all five inverse maps are keyed by the /raw/ ID that appears in the source field — whatever the .skg file literally has in 'contains: [X]', 'subscribes_to: [X]', etc. If that raw X later becomes an extra_id of some other pid P (via a merge), the inverse index keeps the entry under X; nothing migrates it to P.

That's wrong under "who points at pid P?" queries. The audit caught this when I wired it into the merge tests: 'pid=2 relation=subscribes direction=inverse memory={} typedb={ID("subscribes-to-1")}' — memory had the entry under "1" (the pre-merge pid), TypeDB resolves "1" → "2" via 'has_extra_id' and returned the right answer. I patched the audit to union inverse-index entries under pid + its extra_ids so the comparison would be apples-to-apples ('inverse_peers_resolved' in [[../../server/dbs/memory/audit.rs][audit.rs]]). But the underlying inconsistency stays — every reader of the inverse indexes has to know to do the union, and most don't.

This plan switches the inverse indexes to /canonical-pid keys/: when an edge points at a raw ID, resolve it through 'extra_id_to_pid' before indexing, and on merges migrate any pre-existing entries keyed under a now-extra_id to the new pid. Forward fields on 'NodeRust' stay raw (matching what's on disk); the inverse indexes become the place where "what points at pid P" has a single, authoritative answer.

The outcome: every reader becomes a one-line 'inverse_X.get(&pid)' — no union dance, no correctness caveats. One pre-existing under-reporting bug in 'find_related_nodes_from_memory' (see "Problems, out of scope") gets fixed for free.

* Design

** Forward fields: unchanged

'NodeRust.contains', 'NodeRust.subscribes_to', etc., keep raw IDs as they appear on disk. This preserves the "memory mirrors disk" invariant and avoids having to rewrite every neighbor's outbound fields when a merge happens.

The audit's outbound resolution (map memory's raw peers through 'extra_id_to_pid' before comparing to TypeDB) stays as-is; it's compensating for this asymmetry and that's fine.

** Inverse indexes: canonical-pid keyed

Every 'add_to_inverse_map' call resolves the peer ID through 'extra_id_to_pid' first. If the peer isn't a known extra_id, it's used as-is (matches today for non-aliased targets). Concretely: 'id_to_pid(g, id) = g.extra_id_to_pid.get(id).cloned().unwrap_or_else(|| id.clone())'. The helper accepts either an extra_id (mapped via the extra_id_to_pid table) or a primary pid (passed through unchanged); "assuming the ID exists at all" means: if neither, we fall through to the ID unchanged rather than failing — inverse maps already tolerate keys for unknown targets (see "corpus load" below).

** Migration at merge time

When a 'Save' in a batch introduces new extra_ids — i.e., the new 'NodeRust.extra_ids' has IDs that weren't on the old 'NodeRust.extra_ids' — each such /new/ extra_id E might have inverse-index entries keyed under E from before it was an extra_id. For each of the five inverse maps, we move 'inverse[E]' into 'inverse[pid]' and remove 'inverse[E]'. 'im::HashMap' makes this O(1) per map per new extra_id (structural sharing on the union; remove is free).

** Ordering within 'apply_definenodes'

Critical: extra_id_to_pid updates must happen before the inverse-index adds, so peer resolution for this save and for subsequent saves in the same batch sees the just-acquired aliases.

For each 'DefineNode::Save(new)':

  1. Read old 'NodeRust' for this pid (if any).
  2. 'remove_from_inverse_indexes(g, old)' — resolves old peers through the /current/ 'extra_id_to_pid' (which may already reflect prior saves in this batch). Old peers that are aliases resolve to the pid they currently alias; matches the key the old add landed under.
  3. Migrate: for each extra_id in 'new.extra_ids', move 'inverse_X[extra_id]' into 'inverse_X[new.pid]' (all five maps). Idempotent if 'inverse_X[extra_id]' is empty.
  4. Update 'g.extra_id_to_pid' with 'new.extra_ids'.
  5. Insert 'new' into 'g.nodes'.
  6. 'add_to_inverse_indexes(g, new)' — each peer goes through 'id_to_pid' before keying.

Steps 4 and 5 can stay inside 'add_to_inverse_indexes' (as today) if we call migrate BEFORE add. Ordering in the step list above makes the dependency explicit.

For each 'DefineNode::Delete(node)':

  1. Read 'NodeRust' of node.
  2. 'remove_from_inverse_indexes(g, node)' — each peer resolves through current 'extra_id_to_pid', which finds the canonical key it was added under.
  3. Remove the deleted node's extra_ids from 'extra_id_to_pid' (today's code already does this defensively).
  4. Remove from 'g.nodes'.

No migration needed on Delete — we're not promoting anything to an alias.

** Invariants worth documenting in comments

Three invariants we're relying on; the first two go into PITFALL comments near 'apply_definenodes', the third near 'find_related_nodes_from_memory' and 'fetch_all_graphnodestats_in_rust':

  1. /Monotonic alias acquisition/: within a batch, 'extra_id_to_pid' only grows and never retracts. Peer-resolution during 'remove_from_inverse_indexes' of an old NodeRust uses the current 'extra_id_to_pid', which finds the canonical key the old add originally landed under — correct because nothing revoked that mapping between then and now. If revocation is ever added as a real operation, this analysis breaks.

  2. /Extra_id revocation is unsupported/: the in-Rust memory does not gracefully handle a save whose new NodeRust drops an extra_id that the old NodeRust had. Neighbor raw references to the dropped extra_id would fall through 'id_to_pid' unresolved and land under the raw id in the inverse indexes — inconsistent with TypeDB's view. If a user ever needs this, a dedicated migration (symmetric to the acquire path) would be required.

  3. /All peer reads from memory are canonicalized/: both inverse queries ('inverse_X[pid]' direct lookup) and forward queries (reading 'NodeRust.contains' etc.) return canonical pids. Inverse is canonicalized at index time; forward is canonicalized at read time via 'id_to_pid' (Step 9). The forward fields on 'NodeRust' themselves remain raw — they mirror disk — but no public memory read exposes raw peer IDs.

** Step 3 — Add 'migrate_inverse_entries_for_new_extraids'

New free function in [[../../server/dbs/memory.rs][memory.rs]]:

#+begin_src rust
  fn migrate_inverse_entries_for_new_extraids (
    g            : &mut InRustGraph,
    target_pid   : &ID,
    new_rust     : &NodeRust,
    old_rust_opt : Option<&NodeRust>,
  ) {
    fn migrate_one (
      map  : &mut im::HashMap<ID, im::HashSet<ID>>,
      from : &ID,
      to   : &ID,
    ) {
      if let Some (entries) = map . remove (from) {
        let mut merged : im::HashSet<ID> =
          map . get (to) . cloned () . unwrap_or_default ();
        merged . extend (entries);
        map . insert ( to . clone (), merged ); } }
    let old_extraids : HashSet<&ID> = match old_rust_opt {
      Some (old) => old . extra_ids . iter () . collect (),
      None       => HashSet::new (), };
    for extraid in &new_rust . extra_ids {
      if old_extraids . contains (extraid) { continue; }
      if extraid == target_pid { continue; }
      migrate_one (&mut g . contained_by, extraid, target_pid);
      migrate_one (&mut g . subscribers_of, extraid, target_pid);
      migrate_one (&mut g . hiders_of, extraid, target_pid);
      migrate_one (&mut g . replacements_of, extraid, target_pid);
      migrate_one (&mut g . textlinks_in, extraid, target_pid); } }
#+end_src

** Step 7 — Two-pass corpus load in 'from_nodecompletes'

'InRustGraph::from_nodecompletes' in [[../../server/dbs/memory.rs][memory.rs]] currently iterates NodeCompletes in whatever order the caller supplies. Under canonical-keyed inverse, that order matters — a node whose outbound peer is an extra_id of a not-yet-loaded node would land under the raw ID.

Change it to two-pass:

  1. First pass: iterate every NodeComplete, populate 'g.extra_id_to_pid' for each node's extra_ids. No inverse index work.
  2. Second pass: iterate again, this time calling 'add_to_inverse_indexes' (which now sees a fully-populated 'extra_id_to_pid' and resolves every peer correctly).

The 'g.nodes' insertion can happen in either pass; cleanest is the second pass alongside 'add_to_inverse_indexes'.

* Problems

** Peer resolution in 'remove_from_inverse_indexes' uses the /current/ 'extra_id_to_pid'
When removing old peer entries, we resolve through the current map. If an extraid was added earlier in the same batch, the old peer resolves to the post-merge pid — which is exactly where the old add originally landed (because the /previous/ save already saw the alias). So removal hits the right key. This works /only/ because 'apply_definenodes' processes saves sequentially with monotonic updates to 'extra_id_to_pid' — no save ever /removes/ an alias that a later save depends on. That invariant holds for merges (aliases are acquired, not revoked). If we ever add "un-merge" or "alias revocation" as an operation, this assumption needs revisiting.
