** Summary of decisions baked in
   - Escape-by-default for literal search.
   - Fourth axis =operators=true|false= (default false). When
     true, heuristically escape only intra-word operator chars so
     =AND= / =OR= / phrases / grouping still work.
   - Retire =C-c t t= / =C-c t e=.
   - =C-c s= is a prefix; =C-c s RET= is the default, =C-c s i= is
     interactive.
   - Ship regex on tokenized text (don't add a raw_title field yet).
   - Body search is additive: titles are always searched; =body=
     means /also/ body. Interactive prompt wording reflects this.
   - Body indexer commits per chunk; a marker file signals
     completion for crash recovery.
   - Incremental init re-indexes modified nodes with bodies inline
     (no background needed for the incremental path).
   - Surface body-index readiness in both server logs /and/ the
     search buffer header.
   - No back-compat shim for the request rename.

* DONE Goal 1: escape-character round-trip integration test

** Approach
   A round-trip test against today's code would fail: titles like
   =C++ tips= or =foo:bar= aren't findable by typing the literal
   text, because 'QueryParser' interprets =+=, =:=, etc. as
   operators. The "big lacuna to fill" is /literal search of
   titles containing special characters/. The test must exercise
   that, so we need both an escape layer and the test together.

   Escape layer: pre-process the query in the default (regex-off)
   mode to backslash-escape Tantivy operator characters. The test
   creates nodes with specials in their titles and asserts the
   literal query finds them.

** Implementation sketch
   Two escape modes in [[../../server/dbs/tantivy.rs][tantivy.rs]], selected by =operators=:
   - 'escape_tantivy_literal(query)' — operators=false (default).
     Backslash-escapes every occurrence of =+ - & | ! ( ) { } [ ]
     ^ " ~ * ? : \ /= and wraps bare =AND= / =OR= / =NOT= in
     single-quote phrase escapes. Every operator char is treated
     as literal.
   - 'escape_tantivy_intra_word(query)' — operators=true. Walks
     the query and escapes an operator char /only/ when both its
     neighbors are non-whitespace non-operator characters (i.e.
     the char is glued to a word on both sides). =C++= becomes
     =C\+\+= but =AND= is untouched; =foo:bar= becomes
     =foo\:bar= but =title_or_alias:dog= is untouched. Quotes
     anchored to whitespace stay (phrase syntax); quotes mid-word
     get escaped. Heuristic — document the edge cases in
     [[../../docs/COMMANDS.org][COMMANDS.org]].

   In 'search_index' (or its caller), apply whichever escape
   mode matches =operators= when regex=false. In regex mode,
   leave the pattern alone.

   Integration test at
   [[../../tests/integration/search-escape-chars/][tests/integration/search-escape-chars/]] (parallel to
   [[../../tests/integration/search-for-title-and-visit-link/][search-for-title-and-visit-link/]]):
   - =data/skg-data/= with three nodes whose titles exercise
     different specials: e.g. ="c_plus" title "C++ tips"=,
     ="brackets" title "[draft] plan"=,
     ="colons" title "cat:dog"=.
   - =test-emacs.el= calls 'skg-search' three times, once per
     literal title, asserts the matching =(id ...)= appears in
     the search buffer. Then runs a fourth case with
     operators=true and a phrase query ="fox in socks"= against
     a matching node, proving the operator-preserving path.
   - =run-test.sh= copied from the existing search test.

   Small unit tests alongside in [[../../tests/tantivy.rs][tests/tantivy.rs]] covering
   both 'escape_tantivy_literal' and
   'escape_tantivy_intra_word' directly (the integration path
   is slow).

** Background path (only if measurement warrants)
   The shape, modeled on the existing enrichment thread at
   [[../../server/serve/handlers/title_matches.rs][title_matches.rs:187]]:

   1. /Init/: 'wipe_then_init_tantivy_db' indexes
      title+alias only (skip body in
      'create_documents_from_node' when a new flag
      'SkipBody=true' is passed).
   2. /After init, in =main()/=: before 'serve' is called, spawn
      a thread that walks all SkgNodes again and, for each,
      deletes the existing id-keyed docs and re-adds with body
      populated. Commit in chunks of e.g. 1000 nodes.
   3. /Readiness flag/: 'Arc<AtomicBool> body_index_ready',
      cloned into the thread. Exposed via 'TantivyIndex' (or a
      new 'SearchReadiness' sibling struct passed to 'serve' and
      each handler invocation).
   4. /Handler behavior/: in 'handle_text_search_request', when
      =body=true= and =body_index_ready= is false, respond with a
      banner prepended to the rendered search buffer — e.g.
      =# Body index still building; titles only.= — and proceed
      with body=false. Same spirit as the phase-1/phase-2
      enrichment pattern already in place.
   5. /Logging/: 'tracing::info!' at build start ("Body indexer:
      started, N nodes") and end ("Body indexer: done in Ts,
      committed K chunks"). Logs land in
      =<data_root>/logs/server-to-user.log= via the existing
      subscriber in 'init_tracing' ([[../../server/main.rs][main.rs:250]]).
   6. /Commit cadence/: per chunk of ~1000 nodes. See also the
      crash-recovery problem below.

   Nodes re-added during a save
   ('update_tantivy_from_saveinstructions' in
   [[../../server/save.rs][save.rs:228]]) will race with the background indexer. Two
   safe patterns:
   - Serialize with an 'Arc<Mutex<()>>' writer lock; saves and
     the background thread both acquire before creating an
     'IndexWriter'. Tantivy allows only one writer anyway, so
     this merely formalizes it.
   - Or: the background thread re-reads the node from disk
     before each re-add, so a concurrent save wins last-write.

   Prefer the writer-lock pattern — simpler.

** Crash recovery: =.body_index_complete= marker
   File: =<tantivy_folder>/.body_index_complete=. Absent by
   default; written (empty file, fsync'd) only after the final
   commit of a successful body build.

   Init order on startup:
   1. Commit title indexing.
   2. Check for =.body_index_complete=:
      - Present → body index is complete; set
        =body_index_ready=true= and skip the background indexer.
      - Absent → spawn background indexer; indexer removes the
        marker file at start (redundant but cheap), builds
        bodies in chunks, commits final chunk, fsyncs the
        index, then writes and fsyncs the marker, then flips
        =body_index_ready= to true.
   3. When 'full_init' wipes the index directory it also wipes
      the marker (it's inside the directory). A schema change
      triggering full rebuild therefore resets body-index
      status. No separate schema-version gate needed for the
      marker.

   PITFALL comment to add on the marker-write call:
   #+begin_src rust
     // PITFALL: must fsync the Tantivy commit BEFORE writing this
     // marker, and fsync the marker before flipping ready=true.
     // Otherwise a crash between steps leaves: (a) marker present
     // but some bodies missing, which subsequent startups would
     // trust; (b) ready=true broadcast to handlers before the
     // marker is durable, causing the next startup to redo work.
     // Edge case still possible: disk-level reordering if the
     // filesystem doesn't honor fsync (NFS, some tmpfs).
   #+end_src

** Incremental init interaction
   'incremental_init_of_dbs' ([[../../server/dbs/init.rs][init.rs:139]]) re-indexes only the
   .skg files modified since the last marker. Body-indexing
   policy there:
   - Modified nodes get their bodies indexed /inline/ as part of
     the delete-then-readd step — no skip, no background
     thread. The incremental set is small (typically a handful
     of saves); there's no startup-latency pressure to defer.
   - Unmodified nodes retain their bodies from the previous
     run, which is why the =.body_index_complete= marker must
     have been present before we took the incremental path.
   - Precondition for taking 'incremental_init_of_dbs' must therefore
     include =.body_index_complete= existing. Add it as a
     fifth precondition at [[../../server/dbs/init.rs][init.rs:95]]. If absent, fall
     through to 'full_init' (which will spawn the background
     body indexer).
   - On successful incremental init: re-fsync the marker
     (touching it is enough; content is empty) and set
     =body_index_ready=true= immediately.

** Architectural choices the goals don't address

*** DONE Regex on tokenized text
    'RegexQuery' matches tokens, not raw strings. A regex with
    whitespace can't match — =foo bar= will never match the
    title =foo bar= because the tokenizer split it. Fixes
    involve adding a =raw_title= STRING field, doubling index

** Pushback on complexity

*** Do we need regex at all in v1?
    Tantivy's QueryParser already supports =+foo -bar=, phrases,
    grouping — most of what users reach for when they say
    "regex." Real regex buys: leading-wildcard, char classes,
    backreferences. Of those, only leading-wildcard is commonly
    asked for. The priority list in
    [[../../text-search-options.org][text-search-options.org]] §"Recommended priority order" puts
    regex at #4, behind input sanitization. Worth
    considering implementing sanitization (goal 1) and body
    search (goal 3) first, and regex second. I folded all three
    into one plan because the wire protocol is easier to change
    once.

*** Three-boolean prompt is clunky
    =C-c s i= asking three y-or-n questions in sequence is
    clicky. Alternatives: (a) one 'completing-read' with
    preset combinations like ="rooty | regex | body+rooty"=; (b)
    prefix args (=C-1 C-c s=, =C-2 C-c s=, ...). I kept the
    spec's three-prompt form — but (a) scales better if axes
    grow.
