** Bug 1: opening the public "cooking" comes up indefinitive
*** What actually happens
You did not open a content view at all. You opened the *override-choice
menu*. That menu is, by construction, "an ordinary Skg buffer of
indefinitive nodes" -- the requested node as root, drawn indefinitively
(server/to_org/render/override_menu.rs:51, ~mk_indefinitive_viewnode~),
with each overrider as an indefinitive Independent child. So "comes up
indefinitive" is exactly the menu's signature, not a broken content view.
The Cheese node opens normally because *nothing overrides it*, so no menu
is generated and the normal render path runs.
*** Why the public node is overridden
The menu fires whenever the opened node is the *target* of an
~overrides_view_of~ edge and at least one overrider is visible
(server/serve/handlers/single_root_view.rs:113-162 -> ~override_menu_view~).
Your public "cooking" (95fc4b53) is such a target: Cheese Man's "cooking"
(056a7de8) both ~subscribes_to~ it and ~overrides_view_of~ it. That pair
of edges is the fork shape (see glossary.md, "clone, and the fork
operation") -- Cheese Man forked your note into his own source and stayed
subscribed. The root's own metadata corroborates it: ~(rels "... 1S 1O")~
means it participates in both a subscribe edge and an override edge.
*** Why a *foreign* override still hijacks the open
This is the part worth your judgment. Override *substitution* ignores
foreign edges -- "Foreign override edges are display-only facts and never
substitute" (schema.tql:73). But the *menu* deliberately does not: it
lists "every edge, user-owned and foreign alike"
(server/to_org/render/override_menu.rs:106-123, ~visible_overriders~;
also documented in api-and-formats.md under "The override-choice menu:
... user-owned or foreign"). Because your search returned a Cheese result,
the Cheese source is active (an inactive-source node would have been
refused outright, single_root_view.rs:42-58), so the foreign overrider is
visible and the menu triggers.
*** Is it a bug?
Per the current spec, no -- it is documented, intended behavior, and the
menu is meant as navigation ("Choose a destination"). But I think it is a
defensible design wart, and probably the thing that surprised you:
opening *your own* node gets diverted into a chooser purely because some
foreigner forked it, even though that foreign fork can never actually be
substituted in for your node anywhere. If a foreign override is
display-only for substitution, a case can be made that it should not, by
itself, suppress the plain content view of the overridden node -- perhaps
foreign-only overriders belong in a herald or a soft prompt rather than a
menu that replaces the view. I have left this alone pending your call; it
is a policy choice, not a code fault.

** Bug 2: the save that "clearly has a source" is refused
*** The message is misleading; your instinct is correct
The config is fine and Cheese-Man does exist in it. The error is not
about the config at all. The offending node is the pasted subscribee
reference:
: *** (skg (node (id 056a7de8-...) indef)) cooking
It carries *no* ~(source ...)~ atom. The check that fails is
~has_valid_source~ (server/from_text/buffer_to_viewnodes/local.rs:461-466):
: t . source . as_ref() . is_some_and( |s| config.sources.contains_key(s) )
With ~t.source == None~, ~is_some_and~ short-circuits to ~false~ *before the
config is ever consulted*. So the message
"ActiveNode must have a source that exists in the config"
(local.rs:420-422) is doubly wrong as a diagnosis: the source is absent,
not present-but-unknown, and the config was never checked. A truer message
would be "this node has no source and I could not infer one."
*** Why the source is absent -- enrichment can't reach it
The save pipeline resolves missing sources *before* validating
(server/from_text.rs:97-114; the comment at :100 says validation "expect[s]
sources to be inherited/resolved"). But the only source-filling rule is
~inherit_parent_source_if_possible~
(server/from_text/buffer_to_viewnodes/add_missing_info.rs:131-151), which
copies a source *only from an ActiveNode org-parent*. Your pasted node's
org-parent is the ~subscribeeCol~ scaffold -- a PartnerCol, not an
ActiveNode -- so ~parent_source~ comes back ~None~ and nothing is inherited.
The node stays sourceless, and validation rejects it.
*** The real gap
A subscribeeCol member is a *reference* to an existing node, not a node
being written. Its source is knowable from the graph purely from its ID:
the in-Rust graph already exposes ~pid_and_source~
(server/dbs/in_rust_graph.rs:150-155), and enrichment already does an
ID->PID lookup a few lines earlier (~replace_ids_with_pids~,
add_missing_info.rs:49). It simply never does the analogous ID->source
lookup. When you *paste from the link stack* you get only an ID (Emacs has
no source to attach), so the col member arrives bare -- whereas a
server-rendered col member always carries its source (see the sibling
subscriber on line 25 of the buffer above, which has ~(source Cheese-Man)~).
The same node therefore appears twice in your buffer, sourced once and
sourceless once.
*** Cycles are a red herring
You wondered whether the reciprocal subscribe was being rejected as a
cycle. It is not, on two counts. First, the save aborts in the structural
validation stage (from_text.rs:112) -- long before any subscribe/override
cycle reasoning runs -- so it never even gets the chance to object.
Second, there is no forbidden cycle here anyway: only *user-owned*
~overrides_view_of~ cycles are prohibited (schema.tql:69-74), the override
in play is foreign, and mutual *subscriptions* are allowed and expected.
So if the source were resolved, the reciprocal subscribe would go through.
