* Why: the sticky rule, and its deliberate asymmetry

'apply_sticky_levels' ('server/from_text/supplement_from_disk.rs')
resolves every edge's privacy level at save:

- EXPLICIT: a '(relSource NAME)' atom wins, floored at the edge's
  default (the more private of the two endpoints' homes).
- STICKY: absent an atom, an edge already on disk KEEPS its disk
  level. "Renormalization never lowers an edge's privacy silently;
  removing the atom means 'no opinion', not 'reset to default'."
- DEFAULT: only a NEW edge gets the default.

The clamp in 'sticky_level_for' is intentionally one-directional:
when a home moves MORE PRIVATE, its edges rise with it (a section
may never be more public than its home); but, quoting the comment,
"the converse move leaves old, more-private levels in place:
publicizing memberships takes the explicit gesture."

After your move, every in-subtree edge's DEFAULT became pers-i (both
endpoints' homes are now pers-i), but sticky kept the recorded level
at pers-p. Nothing was wrong by the rule's lights; the rule simply
had no wholesale way to say "and publicize the edges too."

Note that 'skg-migrate-to-telescopes' cannot fix this: it only
RAISES edges to at least their default. These edges sit ABOVE their
default; lowering takes the explicit '(relSource ...)' path.

* Should the software change? (options -- now largely implemented)

The new recursive gesture is the manual fix. Beyond it, as decided
2026-07-22, 'skg-set-source' (single and recursive) now:

- ANALYZES the requested move first: it computes which affected
  content edges the move would leave stuck at a level more private
  than their new default (edges already carrying an explicit
  '(relSource ...)' atom count as deliberately leveled and are
  excluded). If nothing would be left behind, the user is not
  bothered.
- OFFERS to fix it in one go: writing the same '(relSource ...)'
  atoms C-c s R would, at each edge's new default. Declining leaves
  the edges alone and the summary message mentions that C-c s R can
  publicize them later.
- WARNS LOUDLY about indefinitive instances: they are no longer
  edited at all (the save would silently drop the edit); their IDs
  go to *Messages* and the summary carries "WARNING: N indefinitive
  nodes NOT changed ... see *Messages* for the ID list."

Not implemented (deliberately): server-side auto-lowering on
publicizing moves -- it would erase the distinction between "edge at
old default" and "edge deliberately held private that happened to
equal the old home", weakening the never-lower-silently principle.

Still open, in 'TODO/problems.org': the server-side warning for a
source edit that reaches the save on an indefinitive instance by
some path other than skg-set-source.
