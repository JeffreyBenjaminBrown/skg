** Glossary (so this section stands alone)
- *relationship heralds / "numeric heralds"*: the compact tokens
  assembled in =server/herald_tokens.rs=, one per graph relation, of the
  shape =[inNum][inLetters] X [outNum][outLetters]= where X is the
  relation letter -- (C)ontains, text(L)inks_to, (H)ides,
  (S)ubscribes, (O)verrides -- plus the two count tokens =Ak= (aliases)
  and =Ik= (extra IDs). Example: =2aC3= means "2 nodes contain it (one
  is its visible parent, the 'a'); it contains 3."
- *ancestor letters a/b/c*: lowercase flags inside a token marking that
  a tracked ANCESTOR is a member on that side. =a= = visible parent
  (generation 1), =b= = grandparent, =c= = great-grandparent, ... So
  =aC= = "my parent contains me"; =Ca= = "I contain my parent."
- *reason for being* (code term: *birth* relation; see =Birth= and
  =birth_relations()= in =server/update_buffer/viewnodestats.rs=): the
  one relation a viewnode was drawn for -- usually Contains (it's
  content of its parent), but for a col member it's the col's relation
  (e.g. a subscribeeCol member's birth is Subscribes). Today this token
  is pulled to the front and colored orange.
- *multi-contained*: =containers > 1= (the "number before C"), i.e. the
  =RelationCounts.containers= field.
- The color of a herald today is decided by a Rust rule table
  (=server/heralds.rs=) that the Emacs "lens engine"
  (=elisp/skg-sexpr/skg-lens.el=) applies; faces (fg/bg pairs) live in

*** Your birthHerald question, answered from the code
Good instinct -- but "birth" names TWO different things, and only one is
display-only. I audited every consumer:

1. =(birthHerald "aC")= -- the assembled DISPLAY STRING.
   - the herald renderer shows it (display);
   - =skg-sexpr-search.el= STRIPS it (and =rels=) as display noise before

   - the server DISCARDS it on save (=parse_metadata_sexp.rs=:

   computation. Folding it into the styled spans loses NOTHING functional
   -- the "reason for being" survives as the black-on-white token,
   decided by the same =birth_relations()=.

   GRAFT playing an ancestry role. Used by the backpath engine (view
   generation), stale-claim cleanup (=to_org/util.rs=), search
   enrichment, and it GATES the load-bearing =rel_source= computation (a
   graft is not a genuine member, so it earns no edge-level herald). It
   ROUND-TRIPS through the wire. *Untouched by anything here* -- different
   atom, different field.

So your instinct is right that "birth" has load-bearing uses -- they live
