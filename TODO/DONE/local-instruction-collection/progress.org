* changes to the plan's envisioned order
- The downstream-stage rewrites (text-claim validation, visibility
  resolution) were written before the parallel entry point, because
  the entry point composes them. The plan listed the entry point
  first.
- The nodeMerge-pairs feed was implemented before the differential
  test rather than after, because the differential test compares
  nodeMerge output too.
- Differential-to-regression conversion: the differential cases now
  live as tests/new/local_instruction_collection/pipeline.rs,
  asserting concrete new-path outputs (pid sequences plus the
  distinguishing fields) rather than exhaustive plan equality.
* what's imperfect about what's done
- Two order vectors. The differential test caught that a bare
  first-emission order saves an ID at its first appearance even when
  that appearance is signals-only (e.g. subscribee-as-such first,
  definitive instance later), where the old path saved it at its
  definitive instance's position. 'CollectedIntents' therefore
  carries 'order' (first emission, used by claim validation and
  visibility extraction) and 'lowerable_order' (first save-or-delete
  emission, used by lowering). Faithful, but two orders is less
  elegant than one.
- Two emission variants carry a source. The plan's NodeIntent_Local
  shapes had no source, but lowering to DeleteNode/NodeSaveIntent
  needs one, and only the self-emissions know it; so SetTitleAndBody
  and Delete carry 'source', and IntentsForOneId has a source slot.
- 'visit_active_vognode' centralizes all vognode emission. It is the
  longest function in the new module (~70 lines), though each arm is
  small.
- The pipeline regression tests assert key fields, not whole plans;
  the deleted differential harness checked more exhaustively (it
  compared entire DefineNode values). The extraction-behavior suite
  (tests/new/local_instruction_collection/extraction.rs, the
  re-pointed descendant of tests/new/viewnodes_to_instructions.rs)
  covers the per-field semantics.
- The error for two same-kind defining cols under one node changed
  from the collector's "Expected at most one ..." to the
  instructionMerge conflict error ("Conflicting aliases intents for
  ID ..."). Validation ('siblings_cannot_include') still rejects the
  shape first in production, so users should never see either.
* choices determining the future
** the new recursion surface, as implemented
The plan's allowlist (definitive members of read-only cols and their
subtrees, descendants of Inactive vognodes, descendants of
Phantom::Diff now save their own edits) is implemented, and the
plan's two user-visible changes (self-edits honored there; defining
cols never squawk about duplicates) are tested. Two further surfaces
fell out of recurse-into-everything; both seem right to me, and both
are pinned by tests, but you should know about them:
*** TODO Delete and nodeMerge requests are honored in newly recursed positions
A definitive member of (say) a subscriberCol carrying
'(editRequest delete)' now deletes that node -- a self-edit like any
other. Previously the request was silently discarded. Same for
nodeMerge requests. (In subscribee-as-such position both requests
are still ignored, as before.)
*** TODO an independent child of a SubscribeeCol is a self-writer
Subscribee-as-such position is defined as: Active, parentIs=Affected,
direct child of a SubscribeeCol. A parentIs=independent child parked
under a SubscribeeCol is therefore an ordinary self-writer (its own
edits save; it joins nothing). The old path made such nodes entirely
inert -- not even recursed into. This matches how classify assigned
roles (affected children only) and parallels the read-only-set rule
in the glossary.
** TODO subscribee-as-such under a diff phantom no longer breaks the save
Validation permits a SubscribeeCol under a diff phantom
('is_active_or_diff_phantom'), but the old classify hard-errored on
any affected TrueNode child of such a col, killing the whole save.
Now the phantom passes its identity down (indefinitive, not
save-eligible), so those children stay in subscribee-as-such
position: their title/body edits bounce via text claims, unedited
they contribute nothing, and no visibility signal is emitted (the
subscriber instance is indefinitive). The plan's literal rule
("phantom children get UnderVognode{None}") would instead have made
them self-writers, letting a title edit in subscribee-as-such
display silently become a real edit; I judged that dangerous and
deviated.
** TODO 'nonmerge' compounds kept
'NonmergeSavePlan', 'extract_nonmergeSavePlan_locally',
'nonmerge_plan', 'nonmerge_defineNodes' survive. No bare 'merge'
token appears in them, and "non-nodeMerge" spellings read terribly,
so I left them. Say the word if you want a rename (and a preferred
spelling).
