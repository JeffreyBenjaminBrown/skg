* 3. Elegance
** 3.1 THE big one: the source_diffs / sharing_diffs two-field seam  [verified]
After #3 collapsed the overlay into per-node inline diff, =CompletionContext=
(=complete.rs:33-45=) still carries *two* diff fields, and the result is a
genuine comprehension hazard:
- =source_diffs= is *always =None=* in production. Both entry points bind it to a
  local =None= (=update_buffer.rs:288=, =:324=). Its own doc admits "Always None
  ... kept ... because several content helpers still take a =source_diffs=
  argument." It is then threaded through ~10 helpers across 5 files
  (=expand_true_content_at_truenode= -> =reconcile_content_children= ->
  =complete_content_children= -> =build_child_creation_data=, plus
  =execute_truenode_view_requests= and =extendDefinitiveSubtree_fromGit=). With it
  =None=, the entire phantom branch it feeds (=truenode.rs:614-632=) is
  unreachable. It is pure dead weight.
- =sharing_diffs= is the *real* diffs handle now -- it drives =process_truenode_diff=
  (content axes, the phantom flip, TextChanged, IDCol, AliasCol) *and* the QualCol
  reconcilers *and* the sharing cols. Its name (and its doc's "Named =sharing_diffs=
  historically...") is a self-admitted misnomer.
- The boundary is *inverted*: =complete.rs= passes =context.sharing_diffs= (real)
  into reconciler params literally named =source_diffs= (=relation_col.rs:29= etc.),
  while =context.source_diffs= (=None=) goes into a param *also* named
  =source_diffs= (=truenode.rs:67=). So one identifier, =source_diffs=, carries
  the real diffs at one call site and =None= at another. And the construction-site
  local is =real_diffs=, re-destructured as =real_diffs= at =complete.rs:302= --
  three names for one value.

This is the cleanest high-value cleanup on the branch. *Recommend*: delete the
=source_diffs= field and its parameter from the content path entirely
(=process_truenode_diff= owns content phantoms now); rename =sharing_diffs= to one
honest name in your compound style (e.g. =gitDiffs= or =realDiffs=) and propagate
that one name to the reconciler parameters; drop the history clause. Low risk
(=source_diffs= is provably =None= on every reachable path).

* 5. Correctness (beyond §1)
These are real but low-impact; I list them because you asked for any I found.
** 5.1 A flipped phantom prepends IDCol/AliasCol that are then deadened  [verified]
In =process_truenode_diff= a node can, in one call, both flip to a DiffPhantom
(=normal_to_phantom=, =diff.rs:74=) *and* get IDCol/AliasCol diff children
prepended (=:104-114=) -- reachable when its worktree file is Deleted
(=existence.unstaged=Minus= -> flip) while its staged stage is Modified with a
non-empty =ids_diff=/=aliases_diff=. Those new cols are enqueued; at their own BFS
visit the generalized-orphan check sees a =DiffPhantom= parent (cols require a
=Normal= ancestor), so =deaden_generalized_orphan_col= converts them to
DeadScaffold and disposes the diff children. The id/alias diff detail is emitted
and then destroyed before render. No fixture exercises this (progress.org:774-775
admits it), so it is untested rather than known-good. *Decide* whether a phantom
should show id/alias sub-diffs: either guard the prepends on "will not flip", or
exempt a phantom's own diff cols from the deaden -- and add a fixture
(removed-here node + =ids_diff=) to lock it in.

** 5.2 Relation-col truncation drops a nondeterministic subset  [verified*]
=inbound_pids_for_relation= returns members in =HashSet= iteration order
(=relation_accessors.rs:131-138,205=), fed straight into =cap_goal_list_to_budget=
which truncates the tail. For a famous node (thousands of inbound subscribers/
hiders/overriders) that exceeds the budget, *which* members survive varies
run-to-run. Order is user-irrelevant (§6.8) so this is not a spec violation, but
the plan's own "sort by ID if a test needs stability" applies under truncation
too. *Recommend* sorting the relation goal list by ID before the budget cap so a
truncated col is reproducible.

** 5.3 extendDefinitiveSubtree_fromGit: two take(limit) on differently-filtered lists  [verified]
=definitive.rs:397-408= builds the existence-query input as
=contents.filter(not hidden).take(limit)= but the append loop is
=contents.take(limit)= then =if hidden continue= -- the two windows can diverge
when hidden ids occupy raw slots. Today it is not a bug (every appended child
still falls inside the query window) and it sits on the dead/defensive path
(§2.3), but the invariant is non-obvious and fragile. Drive both from one
filtered+capped list.

** 5.4 SaveRelaxLock is skipped on an unparseable view-uri  [verified* -> partial]
=save_buffer.rs= always sends the early =SaveLock=, but in =update_views_after_save=
the =SaveRelaxLock= send sits inside =if let Ok(uri) = ...= (=update_buffer.rs:141=),
so an =Err= view-uri never relaxes. The verifier downgraded this to minor: in the
=Err= branch the early lock-set is *empty* (=uris_of_views_to_lock= returns
=Vec::new()=), and the client guard refuses a nil view-uri, so no real locked
state lingers; =save-result= unlocks regardless. Worth a one-line defensive
