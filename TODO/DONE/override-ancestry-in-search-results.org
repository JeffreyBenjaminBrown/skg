** The goal, in one paragraph
When you text-search, an overridden node and the node(s) that override
it should be reachable from one another *within the search results*,
the same way containerward ancestry already hangs beneath each result.
Then, when one of the user's OWN nodes overrides a result, that result
need not also appear at top level -- it will already be visible as a
descendant of the user's node. The payoff: for text search you rarely
need the separate override-choice menu; you navigate overrides straight
from the results. (The override-choice menu stays as-is for the
open-a-node path and any non-search context -- Jeff, Q1.)
** What already exists (so we build with the grain)
*** Search enrichment is a bespoke, container-only grafter
Phase-2 enrichment grafts each result's *containerward* ancestry as
inverted read-only org-descendants. Entry point
'insert_containerward_ancestries_into_search_view'
(server/serve/handlers/text_search/render_enriched_search_buffer.rs:17);
each graft is hardcoded 'Birth::Backpath (RelationRole::CONTAINER)'
(render_enriched_search_buffer.rs:105,110). It does NOT use the generic
role-aware backpath engine -- it is its own containerward-only path,
fed by a TypeDB pre-fetch ('ancestry_by_id_from_ids_async', computed in
the detached enrichment thread, text_search.rs:266).
*** The generic backpath engine already knows all nine roles
'build_and_integrate_path_with_source_set'
(server/to_org/expand/backpath.rs:61) is fully role-generic and already
handles 'overrider'/'overridden' (and re-attaches containerward
ancestry to each grafted partner, backpath.rs:77-79). But it is driven
by '(viewRequests (path ROLENAME))' and reads TypeDB, and is not wired
into search enrichment. So we can either extend the bespoke grafter or
borrow this engine -- see open corners.
*** The graph already exposes both override directions
- nodes X overrides (X's outbound 'overrides_view_of'):
  'graph.outbound_pids_for_relation (X, OverridesViewOf)'.
- nodes that override X (inbound): the dedicated inverse index
  'graph.overriders_of' (server/dbs/in_rust_graph.rs:94), also
  'inbound_pids_for_relation (X, OverridesViewOf)', also
  'visible_overriders' (override_menu.rs:106).
- ownership test for the suppression anchor:
  'SkgConfig::user_owns_source (&node.source)' (server/types/misc.rs:499).
*** There is no result suppression today
Nothing currently drops a top-level result because it appears elsewhere
in the buffer. The top-level set is decided in 'build_search_viewforest'
(text_search.rs:445; ranks, then '.take (SEARCH_DISPLAY_LIMIT)',
~lines 458-468), which produces the canonical 'search_results : Vec<ID>'.
That is the hook point for suppression.
** Decisions already made (from Jeff's answers)
- It's a search-results feature; the open-a-node override menu stays for
  other contexts. (Q1)
- Override relatives graft as org-DESCENDANTS, exactly like containerward
  ancestry does today -- NOT as ancestors. (Q2)
- Graft BOTH directions: overriderward (who overrides me) and
  overriddenward (what I override). (original Aha text)
- Suppress a result from top level ONLY if it will actually appear
  deeper under a shown result. (Q3)
- Suppression is anchored at a USER-OWNED overrider; a foreign overrider
  never suppresses the node it overrides. (Q4 + the "boring author"
  rationale)
** The design
*** Grafting (phase 2, the enrichment step)
For each shown top-level result X, graft as inverted read-only
indefinitive org-descendants, recursively and cycle-guarded:
- its OVERRIDDENWARD relatives -- the nodes X overrides -- each marked
  'Birth::Backpath (RelationRole::OVERRIDDEN)'. Walk
  'outbound overrides_view_of'.
- its OVERRIDERWARD relatives -- the nodes that override X -- each
  marked 'Birth::Backpath (RelationRole::OVERRIDER)'. Walk
  'overriders_of'.
Cycle guard: stop a branch at the first repeated id (the stats pass then
marks it 'cycle'), exactly as the override menu and the generic backpath
engine already do. Both foreign and user-owned edges are followed for
DISPLAY (navigation shows raw graph facts, as the menu does today).
**** DONE herald note -- Jeff was right, "pO"/"Op" was stale; corrected here and in api-and-formats.md
Jeff: "those 'parent' heralds are now rendered as 'a' (grandparent 'b',
etc.)." Correct. The per-relation flag atoms ('overridesParent'/'pO',
'parentOverrides'/'Op', 'grandparentOverrides'/'gO', and the subscribe/
hide analogues) were removed in the uniform-herald refactor; the struct
'ViewNodeStats' no longer holds them (server/types/viewnode.rs) and
'org_to_text.rs' no longer emits them. They are now folded into the
assembled 'birth_herald'/'rels_herald' token strings
(server/herald_tokens.rs): each relation renders
'[inNum][inLetters] X [outNum][outLetters]', where a lowercase letter
flags a tracked ANCESTOR that is a member on that side --
'a' = visible parent, 'b' = grandparent, ... So for these grafts, using
the "1C8a" containerward form as ground truth (the 'a' sits on the
OUTBOUND side of C = "this graft contains its org-parent"):
- an OVERRIDERWARD graft (it overrides the hit above it) renders 'O'
  with the ancestor letter on the OUTBOUND side -- e.g. birth herald
  "Oa" for a graft whose only override target is its org-parent;
- an OVERRIDDENWARD graft (overridden BY the hit above it) renders 'O'
  with the letter on the INBOUND side -- e.g. "aO".
Deeper links in a chain render against their IMMEDIATE org-parent (a
gnode child tracks only its visible parent, generation 'a';
'viewnodestats.rs' 'tracked_ancestors'), so each link is "Oa"/"aO"
relative to the link above it, not "b"/"c". (Generations past 'a' show
up mainly for col members, where a col scaffold sits between.) The
stale PARTNER_ROLE_VOCAB glyph column and the api-and-formats.md
"Stats metadata" section are addressed in TODO/problems.org.
*** Suppression (phase 1, in build_search_viewforest)
Over the ranked result set R, mark result X SUPPRESSED iff there exists
another result Y in R such that:
1. Y is user-owned ('user_owns_source (Y.source)'), and
2. Y recursively overrides X -- X is reachable from Y by following
   'outbound overrides_view_of' edges, cycle-guarded. (The intermediate
   edges may be foreign; only the ANCHOR Y must be user-owned. This is
   what makes "my U overrides foreign F overrides foreign G" suppress
   both F and G -- Jeff, Q4.)
Suppressed results are removed from the top-level list; they still
appear as overriddenward descendants of Y (grafting guarantees it, since
Y is shown and grafts its overriddenward chain). Foreign-only override
clusters -- including a foreign F<->G mutual override with no user-owned
overrider -- suppress nothing, so both show at top level (Jeff, Q4).
*** Why this satisfies each of Jeff's cases
- "U overrides F overrides G", U user-owned & a hit -> F, G suppressed,
  shown under U. U stays (nothing user-owned overrides U).
- Foreign F<->G mutual, user owns neither -> neither suppressed; both
  top-level; each shows the other as an overrider/overridden descendant.
- Boring foreign B overrides exciting E -> E NOT suppressed (B is
  foreign), so the user who skips B still sees E and E's content.
** Code hook points (concrete)
- Suppression: 'build_search_viewforest' (text_search.rs:~445-468).
  Compute suppression over the ranked set BEFORE '.take
  (SEARCH_DISPLAY_LIMIT)', using the in-Rust graph, so suppressed slots
  are backfilled up to the display limit (see open corner O2).
- Grafting: alongside 'insert_containerward_ancestries_into_search_view'
  (render_enriched_search_buffer.rs:17), a sibling pass that grafts the
  two override roles. Source of the relatives: the in-Rust graph
  ('overriders_of' + outbound list), which is cheap and available at
  graft time -- no new TypeDB pre-fetch needed (see open corner O1).
- Heralds/roles already exist: the roles 'overrider'/'overridden' are in
  PARTNER_ROLE_VOCAB (relation_accessors.rs:187-198) and the birth herald
  for a 'Birth::Backpath (role)' graft is assembled by the uniform-herald
  token grammar (server/herald_tokens.rs), not the vocab's (now vestigial)
  glyph column. Rendered forms: overriderward "Oa", overriddenward "aO"
  (see the herald note under Grafting). No new wire vocabulary.

- A foreign F<->G mutual override, user owns neither; assert both top-level.
