* RESOLVED Heralds: the rule table was volatile session state with no self-heal (2026-06-19)
  The herald rule table ('heralds--transform-rules') is fetched once
  per connect by 'skg-request-herald-rules' (called from
  'skg-client-init') and has NO on-disk source. If it became nil,
  'heralds-minor-mode' just disabled itself with "Heralds disabled: no
  herald rule table from the server" -- even when the server was
  perfectly healthy and would answer a re-fetch instantly. The feature
  looked broken intermittently: the table was lost, then silently
  restored the next time a skg buffer triggered 'skg-client-init' (the
  "it broke, then fixed itself without my changing anything" report on
  2026-06-19). NB commit 8cd3bf00 (the git-add 'org-map-entries'
  rewrite) was wrongly suspected; it only touches skg-git-add.el and
  cannot affect the herald table -- it was the *reload while testing
  it* that tripped the loss path below.

  Two fixes landed:
  - Loss path closed. 'skg-reload' captured the table, unloaded
    'heralds-minor-mode' (wiping it), reloaded, then re-installed the
    captured copy as a plain sequential step -- so any error in a
    reloaded file aborted the reload before the re-install and lost the
    table for the session. The re-install is now wrapped in
    'unwind-protect' (the reload's destructive half split into
    'skg--reload-modules'), so the table survives a botched reload while
    the error still propagates. Regression test:
    tests/elisp/test-skg-reload.el.
  - Self-heal added. Enabling 'heralds-minor-mode' with a nil table now
    calls 'skg-herald-rules-ensure', which re-requests the table from
    the server up to 'skg-herald-rules-max-attempts' (3) times, each
    with a bounded wait, then gives up cleanly (bounded so a silent or
    wedged server cannot freeze Emacs). On give-up or hard failure the
    user gets an informative message and heralds disable. This also
    backstops a dropped/raced connect-time fetch. The manual fetch
    command was removed ('skg-request-herald-rules' is no longer
    'interactive'), since recovery is now automatic. Regression tests
    in tests/elisp/test-heralds-minor-mode.el
    (test-heralds-self-heals-missing-rule-table,
    test-heralds-gives-up-after-bounded-fetch-attempts).

** DONE 'compute_interleaved_diff' can collide for large unique-item lists
   WARNING: TODO.org refers to the headline title verbatim.
   (Kept verbatim: TODO.org's copy is also prefixed "DONE ".)

   Fixed 2026-06-12: rewrote 'compute_interleaved_diff'
   (server/types/list.rs) to diff the items directly via
   'similar::capture_diff_slices' instead of mapping each item to a
   Unicode scalar. There is no scalar encoding left to collide, so
   distinct items can never be conflated regardless of list length.
   Added 'Ord' to the bound (and to 'net_diff_from_per_stage', the one
   generic caller). Regression test:
   'git_ops::test_interleaved_diff_many_distinct_items_do_not_collide'
   swaps two items deep in the old collision band and asserts the swap
   is seen.

   server/types/list.rs maps each unique item to a Unicode scalar and
   feeds the resulting strings to 'similar::TextDiff::from_chars'.

   The mapping starts at 0x100 and uses
   'char::from_u32(char_counter).unwrap_or('?')'. Once the counter
   reaches surrogate code points, or eventually passes Unicode's max
   scalar value, multiple distinct items map to '?'. That can make the
   diff incorrect.

   It probably does not matter for small relationship lists, but a
   generic list diff helper should either avoid scalar encoding or
   return an error instead of silently colliding.

** DONE Retained inactive nodes emit positional save intentions for their container
   Verified fixed 2026-06-12 by the strip-inactive-node-fields work:
   'content_members', 'subscribeeCol_members' and 'partnerCol_members'
   (traverse.rs:331-407) now all collect only 'Vognode::Active' children;

   server/from_text/local_instruction_collection/traverse.rs.
   The metaplan-2 model for source-set safety (this file, "Sharing-col
   members from inactive sources make a buffer unsavable", and
   TODO/full-schema/metaplan_2.org) is: omit inactive members from
   display, and reconstruct their membership at save by MERGING the
   visible-subset buffer list with disk's invisible members
   ('preserve_invisible_members' -> weave / set_difference_merge in
   from_text/weave.rs). Under that model an inactive node contributes
   NOTHING to any ancestor's saved list; the disk merge is the sole
   authority for invisible members.

   The order-meaningless list follows the model: 'partnerCol_members'
   (overriddenCol) collects only Active children (traverse.rs:336-345),
   and 'set_difference_merge' restores the inactive overrides from disk.

   The two ORDER-MEANINGFUL lists do not. 'content_members'
   (traverse.rs:373-389) pushes each non-phantom InactiveNode's
   'collected_id' into the parent's 'SetContains'; 'subscribeeCol_members'
   (traverse.rs:353-366) does the same into the owner's 'SetSubscribesTo'.
   So a RETAINED inactive node (one kept in the view because it has
   visible descendants) becomes a positional member of its
   container's/owner's saved list, drawn from the buffer. The comments
   justify this ("subscribee order matters, so a buffer-present
   InactiveNode counts as a positional member"), and weave.rs's
   "POSITIONED ... includes retained InactiveNodes" was co-designed with
   it -- so this is deliberate drift from the metaplan model, not an
   oversight. But it is both unnecessary and unsafe:

   - Unnecessary: the weave already restores an omitted invisible
     member to its correct position by anchoring it to its nearest
     preceding positioned disk member. With disk [A, X, B] (X inactive)
     and buffer [A, B], the weave yields [A, X, B] whether or not X is
     in the buffer list. So excluding X loses no position in the common
     case.

   - Unsafe: because the buffer list is fed to the weave as authority,
     including X lets a stale or concurrently-edited buffer RESURRECT an
     invisible member that was authoritatively removed. Concretely: disk
     P.contains = [A, B] (another save removed X); a stale buffer still
     shows [A, X, B] with X retained inactive; saving P (no user edit)
     extracts SetContains [A, X, B]; the weave pushes every buffer
     member, so X reappears in P.contains on disk. The noop filter does
     not catch it (buffer != disk), and write-suppression does not catch
     it (P's source is active, so P's instruction stands). Symmetrically,
     reordering a read-only inactive placeholder among visible siblings
     persists as a real write to the container -- editing the position of
     something the user cannot edit.

   This is the inbound twin of the original "silent deletion" bug: that
   one dropped invisible members by omission; this one re-adds them by
   inclusion. The fix is to make 'content_members' and
   'subscribeeCol_members' collect only Active members (mirroring
   'partnerCol_members'), and let the weave own invisible members'
   presence AND position. Tradeoff to confirm with Jeff: a retained
   inactive node then cannot be repositioned by the user (its post-save
   position is weave-determined), which is the correct read-only
   semantics but a visible behavior change. 'inactiveNode_is_phantom'
   gating and the InactiveNode 'collected_id' accessor become dead on the
   save path if this lands.
