* What remains after 2p5

This report supersedes the earlier =3_what-remains.org= text, which
was written before the 2p5 cleanup completed.

The subscribee-as-such hide/unhide feature now has the important
save-path shape:

- subscribee-as-such child-list removals no longer become
  subscribee graph =contains= edits;
- subscribee-as-such visible children are extracted as
  =SubscribeeHiderelIntent= values;
- visibility folding happens before =DefineNode= construction;
- disk supplementation is policy-neutral and no longer has a
  subscribee-as-such =contains= restoration guardrail;
- post-=reconcile_nodeIntents= lookup is keyed by
  =SameIdReconciledNodeIntents=;
- validation and rerender/completion boundaries are documented in
  [[./2p5_simplify-save.org][2p5_simplify-save.org]].

The items below are the remaining work that still looks real.

* Ranked opportunities

** DONE a. Replace stale sharing-completion comments

The comments in
[[../../server/to_org/complete/sharing/mod.rs][server/to_org/complete/sharing/mod.rs]]
still describe older worries about save-buffer integration:

- =maybe_add_subscribeeCol_branch= says an existing =SubscribeeCol=
  would need to be integrated if this code ran during save-buffer;
- =maybe_add_hiddenInSubscribeeCol_branch= says an existing
  =HiddenInSubscribeeCol= should be integrated.

The current story is narrower:

- these helpers are initial-render helpers, not the saved-view
  rerender reconciliation path;
- saved-view/collateral rerender uses the preorder/postorder
  completion path and =reconcile_sharing_scaffold_children=;
- the =SubscribeeCol= guard is structurally defensive because current
  call paths append a fresh branch before reaching it;
- the =HiddenInSubscribeeCol= guard can be reached after postorder
  sharing-scaffold reconciliation, so an existing collection has already
  been handled.

No behavior change is implied. This is a comment cleanup to keep
future refactors pointed at the right code path.

Status: complete. The two guard comments now point to rerender
sharing-scaffold completion as the path that reconciles existing
generated sharing branches.

This is still a real boundary question. Normal saved views must

extraction to interpret them as visibility edits. Diff views also
contain generated phantom/removed children, and those display nodes
must not infer hides or unhides.

Suggested next step: add a diff-mode regression proving phantom or
removed children do not infer hide/unhide intent, then decide whether
=source_diffs.is_none()= remains the right guard or should be replaced
by a more direct distinction.

Status: complete. A focused git-diff saved-view regression now proves
that a subscribee-as-such branch regenerates phantom/removed
children from =source_diffs= instead of preserving a stale direct
child list. The misleading TODO was replaced with that rationale.

** DONE c. Define policy for extra view-children under =AsSubscribee=

The current save path satisfies the crucial invariant: extra
view-children under an =AsSubscribee= branch are never saved as the
subscribee's graph =contains=.

What is not explicitly settled is the user policy for a view-child
that is not pre-save graph-content of the subscribee:

- reject the save;
- ignore it as visibility signal while allowing its own save semantics
  where applicable;
- normalize it into a different edit surface.

Suggested next step: add tests for inserting a new view-child under a
foreign =AsSubscribee= branch and under an owned =AsSubscribee=
branch, then document or enforce the chosen policy.

Execution plan: [[./3pC_DONE.org][3pC_DONE.org]].

Status: complete. The policy is documented in
=apply_hiderels_from_intents=. Saved =AsSubscribee=
completion now normalizes view-children that are not graph-content of
the subscribee to =Birth=Independent= and orders graph-content
view-children by the subscribee's graph-content order. Foreign and
owned regression tests cover the behavior.

** DONE d. Decide owned subscribee-as-such title/body policy

Foreign title/body edits in subscribee-as-such position currently
reach foreign-node validation, which is covered by tests.

The intended product policy may be stricter for owned nodes: real
title/body edits to a subscribee should probably happen when viewing
that node directly, not when it is shown as someone else's subscribee.

Suggested next step: add a focused test for owned direct
=AsSubscribee= title/body edits and decide whether to allow, reject,
or normalize them.

Execution plan: [[./3pD_DONE.org][3pD_DONE.org]].

Status: complete. Owned title/body edits in subscribee-as-such
position are rejected with a focused validation error. The existing
foreign-node validation behavior is preserved.

** DONE e. Document direct-only visibility semantics for users

The implemented visibility intent reads only direct children under a
definitive =AsSubscribee= node:

- removing a direct child can hide that content from the subscriber;
- restoring a direct hidden child can unhide it;
- editing deeper descendants remains ordinary graph editing for those
  descendant nodes, not subscriber visibility editing.

This should be written in user-facing terms before the feature is
presented as complete.

Status: complete. =docs/sharing-model.md= now documents that immediate
view-children of subscribed content control subscriber visibility,
that deeper descendants keep their own edit semantics, that extra
children become independent rather than graph-content of the
subscribed node, and that title/body edits belong to the node's own
view.

** DONE f. Decide whether HiddenIn/HiddenOutside collections are editable

The current hide/unhide edit surface is subscribee-as-such visible
content. =HiddenInSubscribeeCol= and
=HiddenOutsideOfSubscribeeCol= are display-derived scaffolds, not
general editable relation collections.

Open product questions:

- Should deleting a row from =HiddenInSubscribeeCol= mean unhide?
- Should adding a row to =HiddenOutsideOfSubscribeeCol= mean hide
  from all subscriptions?
- Should these scaffolds visibly communicate that they are generated
  display, not normal editable relation collections?

This is product behavior, not save-path cleanup.

Execution plan: [[./3pF_DONE.org][3pF_DONE.org]].

Status: complete. These collections are documented as generated
read-only displays. Regressions prove deleting from
=HiddenInSubscribeeCol= does not unhide, and adding to
=HiddenOutsideOfSubscribeeCol= does not hide.

** DEFERRING
*** i. Add UI support for starting a subscription

If a node has no subscriptions, no =SubscribeeCol= is automatically
present. The user can type the scaffold by hand, but this is tedious
and error-prone.

A command should insert a =SubscribeeCol= and probably help choose or
insert the first subscribee.

*** j. Build an overrides edit/view surface

=overrides_view_of= is still in the data model but has no real
view/edit surface. The herald flag is not enough to explain which
side of the relation a node is on or to edit it.

This is outside subscribee-as-such hide/unhide, but it remains a
sharing-model gap.

*** k. Generalize relationship collections

=vision.org= still points toward collections for all relation directions.
The current work improves one relation surface: subscriber visibility
over subscribee-as-such content. It does not generalize relation
display or editing.

*** l. Settle user-facing language

The code and planning docs use several implementation terms:

- subscribee
- shown as a subscribee;
- subscribee-as-such branch;
- visibility intent;
- hiderel;
- =HiddenInSubscribeeCol=.

User-facing commands and docs should probably say something closer to
"hide this subscribed item from this view" and "show this subscribed
item again", rather than "edit subscribee-as-such child lists."
