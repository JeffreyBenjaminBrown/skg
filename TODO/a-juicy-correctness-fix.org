* Recommended fix: 'compute_interleaved_diff' silent mis-diff
** What it is
   server/types/list.rs. The generic list-diff helper maps each
   distinct item to a 'char' to reuse 'similar::TextDiff::from_chars':

   #+begin_src rust
   let mut char_counter : u32 = 0x100;
   for item in old.iter().chain(new.iter()) {
     if !item_to_char.contains_key(item) {
       item_to_char.insert(item, char::from_u32(char_counter).unwrap_or('?'));
       char_counter += 1; }}
   #+end_src

** The defect
   'char::from_u32' returns None for the 2048 surrogate code points
   (U+D800..=U+DFFF) and for everything past U+10FFFF, and the code
   folds every such case to the same '?'. With the 0x100 start offset,
   the (55_040)th distinct item is the first to land on a surrogate, so
   from there on many distinct items share the char '?'. 'similar' then
   treats those distinct items as EQUAL.

   Because the result is reconstructed from 'old[old_idx]'/'new[new_idx]'
   by position (lines 60-77), the emitted *values* stay real -- but
   their *classification* is wrong: a genuine change among collided
   items is reported 'Unchanged', and reorders among them vanish. It is
   a silent wrong answer: no panic, no error, no log.

** Reachability (honest)
   Every caller diffs a single node's relation / alias / id list:
   server/git_ops/diff.rs (x6), server/diff_analysis/diff.rs:331,
   server/to_org/complete/partner_col/goal_list.rs (x2),
   server/types/git.rs:357. None of those reaches ~55k distinct
   entries in one list, so the bug cannot fire on real data today.

** Why recommend it anyway
   - It is a *generic* helper in 'types/list.rs', not a one-off; the
     entry author flagged exactly this ("a generic list diff helper
     should... return an error instead of silently colliding"). The
     next reuse on a large sequence inherits the landmine.
   - It is the only open correctness item where the failure is silent
     corruption rather than a guarded panic or a documented trade-off.
   - The fix is provable and zero-risk: it changes no output for any
     input that does not already collide, and there are no colliding
     inputs in production.

** Proposed fix
   Drop the char encoding entirely and diff the slices directly.
   'similar' 2.7 (already a dependency) offers 'capture_diff_slices',
   which diffs '&[T]' for 'T: Eq + Hash' -- exactly the existing bound
   on 'compute_interleaved_diff<T: Clone + Eq + Hash>'. Sketch:

   #+begin_src rust
   use similar::{capture_diff_slices, Algorithm, ChangeTag};

   let ops = capture_diff_slices (Algorithm::Myers, old, new);
   let mut result : Vec<Diff_Item<T>> = Vec::new();
   for op in &ops {
     for change in op.iter_changes (old, new) {
       match change.tag() {
         ChangeTag::Equal  => result.push (Diff_Item::Unchanged (change.value().clone())),
         ChangeTag::Delete => result.push (Diff_Item::Removed   (change.value().clone())),
         ChangeTag::Insert => result.push (Diff_Item::New       (change.value().clone())), }}}
   result
   #+end_src

   This removes the HashMap and the two String allocations as a bonus,
   and makes collision structurally impossible (no scalar encoding to
   collide). The exact 'iter_changes'/'DiffOp' surface should be
   confirmed against similar 2.7's docs while wiring it; the algorithm
   choice (Myers) matches similar's text default, so existing diffs are
   unchanged for non-colliding inputs.

   If for some reason the direct-slice path is undesirable, the minimal
   alternative is to keep the encoding but return 'Result' and error
   when 'char_counter' would enter the surrogate range -- i.e. make the
   silent corruption a loud failure. The direct-slice rewrite is
   strictly better; I mention this only as the "smallest possible
   change" option.

** Test plan
   Add a unit test that today fails and post-fix passes:
   - Build 'old : Vec<u32>' of ~56_000 distinct values so that a band
     of items lands in the surrogate-collision zone, and 'new' = 'old'
     with two items *inside that zone* swapped (a pure reorder of
     distinct items).
   - Assert the diff reflects the reorder (the swapped pair shows
     Removed/New, not all-Unchanged).
   Under the current code both swapped items encode to '?', so the
   reorder is invisible and every item reports Unchanged -- the
   assertion fails, pinning the bug. Under the slice-diff fix it
   passes. Keep it small enough to stay fast (tens of thousands of
   u32s is microseconds).

** Risk
   None for production inputs (no behavior change unless an input
   already collides, which none does). The change is local to one
   function and one test.

* Alternatives I considered (and why not first)
** elisp: 'skg-strip-heralds-from-sexp' fabricates a '(node ...)' child
   server/elisp/skg-sexpr/skg-sexpr-search.el. The herald stripper
   calls 'skg-edit-nested-sexp' with merge instructions
   '(skg (node (graphStats (DELETE ...)) (viewStats (DELETE ...))))';
   the DSL's merge semantics append a '(node ...)' to any sexp lacking
   one, so a stripped phantom/scaffold/search sexp comes out carrying a
   fabricated '(node ...)'. That would make 'skg-sexp-subtree-p ...
   (skg (node))' match non-truenodes.

   Why not first: it is latent today. The callers that test
   '(skg (node))' (skg-metadata.el:42,237; skg-org-fold.el:212) read
   the metadata *directly* from the headline text, never through the
   stripper; the stripper's callers (skg-first-sexpr-on-line ->
   skg-request-file-path) gate on a more specific '(skg (node (id)))'
   and so dodge it. So no present misclassification -- but it is a real
   trap for the next caller that strips-then-tests the bare pattern.
   Clean fix: have the stripper use DELETE-only instructions, or
   short-circuit when the target has no '(node ...)'. Good follow-up;
   harder to unit-test than the Rust fix, hence second.

** Staged-diff test coverage gap (the "guarantee correctness" angle)
   problems.org notes that the git-diff-view integration tests almost
   all exercise *unstaged* changes; the staged paths are thinly
   covered, which is "how phantom_axes, TextChanged, and the
   node_changes_for_truenode merge bugs all slipped through." Several
   of the now-fixed items in the audit above were staged-path bugs.
   Adding staged-variant tests (and mixed staged+unstaged on one file)
   is the highest-leverage way to *guarantee* diff-mode correctness
   going forward. I did not lead with it because it is a test campaign,
   not a single fix, and the request was for a candidate fix. Flagging
   it as the better choice if the goal is assurance over a point fix.
