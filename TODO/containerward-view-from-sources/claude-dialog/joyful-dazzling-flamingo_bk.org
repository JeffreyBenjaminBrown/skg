* context
When a user requests a sourceward view ('skg-local-sourceward'),
the link sources appear without context.
If multiple nodes link to the same target
they look literally identical -- just link labels with no hierarchy.

The goal: after each source node is inserted into the sourceward view,
also insert its containerward ancestry as children beneath it,
so the user can see where each link source lives.

The desired output example is in
[[../../TODO/containerward-view-from-sources/vision.org][the vision document]].

* approach overview
After the sourceward backpaths are integrated
(the existing 'build_and_integrate_backpaths' call),
a new function walks the subtree to find every Birth::LinksTo node,
computes their containerward ancestries in parallel,
and inserts those ancestries beneath each source node.

No new files. All new code goes in
[[../../server/to_org/expand/backpath.rs][backpath.rs]].
The integration test at
[[../../tests/integration/sourceward-view-request/][sourceward-view-request/]]
gets extended to cover the new behavior.

* the stats pipeline needs no changes
'rerender_view' (in [[../../server/update_buffer.rs][update_buffer.rs]] line 207)
calls 'complete_viewtree' first
(which executes view requests, including sourceward expansion),
then 'set_graphnodestats_in_forest', then 'set_viewnodestats_in_forest'.
Any ancestry nodes we insert during the sourceward expansion
will automatically receive their stats in those later passes.

** new function: 'add_containerward_ancestry_to_source_nodes'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

Three phases, cleanly separating borrow constraints:

*** phase 1 (collect, immutable borrow of tree)
Walk the subtree under node_id.
Collect every (NodeId, PID) pair where birth == Birth::LinksTo.
Return early if empty.

*** phase 2 (compute, no tree borrow)
For each collected PID, call 'full_containerward_ancestry'
(from [[../../server/dbs/typedb/ancestry.rs][ancestry.rs]], line 44).
Run them all in parallel via futures::future::join_all.
Collect results into a HashMap<ID, AncestryTree>.
Log and skip any failures.

This is the same pattern used by 'ancestry_by_id_from_ids'
in [[../../server/serve/handlers/title_matches.rs][title_matches.rs]] line 231.
It parallelizes safely because 'full_containerward_ancestry'
only needs &TypeDBDriver (shared reference);
each call creates its own read transactions internally.

**** TODO (Jeff)
Could we not simply use ancestry_by_id_from_ids here,
instead of writing a separate, somewhat-duplicative function?

*** phase 3 (insert, mutable borrow of tree)
For each source node whose ancestry is an AncestryTree::Inner,
iterate its children in reverse
(reverse because 'prepend_indefinitive_child' prepends)
and call 'insert_ancestry_tree_recursive' for each.

If the ancestry is Root / Repeated / DepthTruncated
(i.e. the source node itself is a root, or the BFS hit a cycle/limit
before finding any containers), nothing is inserted.
That's correct -- there's no ancestry to show.

* problems, out of scope
** search enrichment duplicates the ancestry insertion pattern
'insert_containerward_ancestry_tree' in
[[../../server/serve/handlers/title_matches/render_enriched_search_buffer.rs][render_enriched_search_buffer.rs]]
does essentially the same thing as the new
'insert_ancestry_tree_recursive' in backpath.rs,
but synchronously, using Tantivy for title lookups
and Birth::Independent instead of Birth::ContainerOf.

A future refactor could unify these two functions.
The differences (sync vs async, Tantivy vs TypeDB, Birth value)
could be parameterized.
Not worth doing now -- the duplication is small (< 15 lines each),
and the contexts are different enough that forcing them together
would obscure both.
