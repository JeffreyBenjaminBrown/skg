* context
When a user requests a sourceward view ('skg-local-sourceward'),
the link sources appear without context.
If multiple nodes link to the same target
they look literally identical -- just link labels with no hierarchy.

The goal: after each source node is inserted into the sourceward view,
also insert its containerward ancestry as children beneath it,
so the user can see where each link source lives.

The desired output example is in
[[../../TODO/containerward-view-from-sources/vision.org][the vision document]].

* approach overview
'build_and_integrate_sourceward_path' stops delegating
to the generic 'build_and_integrate_backpaths'.
Instead, it does three things in sequence:
1. Query sourceward paths and extract source PIDs (data only).
2. Compute containerward ancestry for every source PID,
   in parallel, before touching the viewtree.
3. Integrate everything into the viewtree:
   first the sourceward paths, then ancestry beneath each source.

'build_and_integrate_containerward_path' keeps calling
'build_and_integrate_backpaths' unchanged.

No new files. All new code goes in
[[../../server/to_org/expand/backpath.rs][backpath.rs]].
The integration test at
[[../../tests/integration/sourceward-view-request/][sourceward-view-request/]]
gets extended to cover the new behavior.

* the stats pipeline needs no changes
'rerender_view' (in [[../../server/update_buffer.rs][update_buffer.rs]] line 207)
calls 'complete_viewtree' first
(which executes view requests, including sourceward expansion),
then 'set_graphnodestats_in_forest', then 'set_viewnodestats_in_forest'.
Any ancestry nodes we insert during the sourceward expansion
will automatically receive their stats in those later passes.

* implementation
** rewrite 'build_and_integrate_sourceward_path'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]], line 82.

Currently a one-liner that delegates to 'build_and_integrate_backpaths'.
Rewrite it to inline the relevant logic and add ancestry:

#+begin_example
  pub async fn build_and_integrate_sourceward_path (
    tree, map, node_id, config, driver,
  ) -> Result<(), Box<dyn Error>> {

    // --- data preparation (no tree mutation) ---

    // 1. Query sourceward paths.
    let terminus_pid = get_id_from_treenode(tree, node_id)?;
    let paths = paths_to_first_nonlinearities(
      &config.db_name, driver, &terminus_pid,
      "textlinks_to", "dest", "source"
    ).await?;

    // 2. Extract every source PID from the paths.
    let source_pids = extract_pids_from_paths(&paths);

    // 3. Compute containerward ancestry for each, in parallel.
    let ancestry_map = compute_ancestry_map_async(
      &source_pids, config, driver
    ).await;

    // --- tree mutation ---

    // 4. Integrate the sourceward paths.
    for p in paths {
      integrate_path_that_might_fork_or_cycle(
        tree, map, node_id,
        p.path, p.branches, p.cycle_nodes,
        config, driver, Birth::LinksTo
      ).await?;
    }

    // 5. Attach ancestry beneath each Birth::LinksTo node.
    attach_ancestry_to_links_to_nodes(
      tree, map, node_id, &ancestry_map, config, driver
    ).await
  }
#+end_example

Steps 1-3 are pure data preparation: queries to TypeDB,
no tree mutation. Steps 4-5 are tree mutation.
This addresses the concern about computing ancestry
before touching the viewtree.

** new function: 'compute_ancestry_map_async'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

#+begin_example
  async fn compute_ancestry_map_async (
    pids      : &[ID],
    config    : &SkgConfig,
    driver    : &TypeDBDriver,
  ) -> HashMap<ID, AncestryTree>
#+end_example

For each PID, calls 'full_containerward_ancestry'
(from [[../../server/dbs/typedb/ancestry.rs][ancestry.rs]] line 44).
Runs all calls in parallel via futures::future::join_all,
collects results into a HashMap.

This parallelizes safely because 'full_containerward_ancestry'
only needs &TypeDBDriver (shared reference);
each call creates its own read transactions internally.

[[../../server/serve/handlers/title_matches.rs][title_matches.rs]] line 231,
and as 'query_containers_for_frontier' in
[[../../server/dbs/typedb/ancestry.rs][ancestry.rs]] line 101.

** new function: 'attach_ancestry_to_links_to_nodes'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

Walks the subtree under node_id to find every
TrueNode with birth == Birth::LinksTo.
For each, looks up its PID in the ancestry_map.
If the ancestry is an AncestryTree::Inner,
inserts the node's containers as view-children
(i.e. org-subheadlines in the view)
beneath the source node, with Birth::ContainerOf.

Terminology note: AncestryTree::Inner(id, vec) means
"the node with `id` has containers listed in `vec`."
Those containers become org-subheadlines (view-children)
of the source node, each marked Birth::ContainerOf
to indicate "this node is a container of its view-parent."

If the ancestry is Root (no containers),
Repeated (cycle), or DepthTruncated (BFS limit hit),
there are no containers to insert -- the source node
is already a leaf in the containment hierarchy.

** new function: 'insert_ancestry_tree_recursive'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

Called by 'attach_ancestry_to_links_to_nodes' for each container.
Async recursive function (Pin<Box<dyn Future>> pattern,
same as 'integrate_linear_portion_of_path' at line 156).

For each AncestryTree node:
- call the existing 'prepend_indefinitive_child' with Birth::ContainerOf
- if Inner (i.e. this container itself has containers),
  recurse for each of its containers

Reuses 'prepend_indefinitive_child' (backpath.rs line 238),
which already handles fetching the SkgNode via TypeDB and
creating an indefinitive ViewNode with the given Birth.

The containers are iterated in reverse order.
Since 'prepend_indefinitive_child' prepends
(inserts at the front of the sibling list),
reversing the iteration preserves the original order.
