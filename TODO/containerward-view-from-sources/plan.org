* context
When a user requests a sourceward view ('skg-local-sourceward'),
the link sources appear without context.
If multiple nodes link to the same target
they look literally identical -- just link labels with no hierarchy.

The goal: after each source node is inserted into the sourceward view,
also insert its containerward ancestry as org-subheadlines beneath it,
so the user can see where each link source lives.

The desired output example is in
[[../../TODO/containerward-view-from-sources/vision.org][the vision document]].

* approach overview
Three changes, in order:

1. Fix a pre-existing bug: search enrichment uses Birth::Independent
   for ancestry nodes that are actually containers.
   Change to Birth::ContainerOf.

2. Extract 'ancestry_by_id_from_ids_async' into ancestry.rs
   so that both the search enrichment (sync, via block_on)
   and the new sourceward ancestry (async, via .await)
   can share the same core logic.
   (Cannot reuse 'ancestry_by_id_from_ids' directly because
   it uses futures::executor::block_on, which deadlocks
   when called from an async context.)

3. Add containerward ancestry to the sourceward view:
   - Extract 'integrate_backpaths' from 'build_and_integrate_backpaths'
     so the sourceward path can insert ancestry between query and integration.
   - Rewrite 'build_and_integrate_sourceward_path' to query paths,
     compute ancestry in parallel, integrate paths, then attach ancestry.
   - All data preparation (TypeDB queries, ancestry computation) happens
     before any viewtree mutation.

** step 3: rewrite 'build_and_integrate_sourceward_path'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]], line 82.

#+begin_example
  pub async fn build_and_integrate_sourceward_path (
    tree, map, node_id, config, driver,
  ) -> Result<(), Box<dyn Error>> {

    // --- data preparation (no tree mutation) ---

    // 1. Query sourceward paths.
    let terminus_pid = get_id_from_treenode(tree, node_id)?;
    let paths = paths_to_first_nonlinearities(
      &config.db_name, driver, &terminus_pid,
      "textlinks_to", "dest", "source"
    ).await?;

    // 2. Extract source PIDs and compute ancestry in parallel.
    let source_pids = extract_pids_from_paths(&paths);
    let ancestry_map = ancestry_by_id_from_ids_async(
      &source_pids, &config.db_name, driver,
      config.max_ancestry_depth
    ).await;

    // --- tree mutation ---

    // 3. Integrate the sourceward paths.
    integrate_backpaths(
      tree, map, node_id, paths, config, driver, Birth::LinksTo
    ).await?;

    // 4. Attach ancestry beneath each Birth::LinksTo node.
    attach_ancestry_to_links_to_nodes(
      tree, map, node_id, &ancestry_map, config, driver
    ).await
  }
#+end_example

Steps 1-2 are pure data preparation (queries to TypeDB,
no tree mutation). Steps 3-4 are tree mutation.

** step 3a: new function 'extract_pids_from_paths'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

Extracts every PID from a Vec<PathToFirstNonlinearity>.
Each path has a 'path' field (Vec<ID>),
a 'branches' field (HashSet<ID>),
and a 'cycle_nodes' field (HashSet<ID>).
All of those IDs are potential source nodes
that need containerward ancestry.

Returns a deduplicated Vec<ID>.

** step 3b: new function 'attach_ancestry_to_links_to_nodes'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

Walks the subtree under node_id to find every
TrueNode with birth == Birth::LinksTo.
For each, looks up its PID in the ancestry_map.
If the ancestry is an AncestryTree::Inner,
inserts the node's containers as org-subheadlines
beneath the source node, with Birth::ContainerOf.

The AncestryTree::Inner(id, vec) means
"the node with `id` has containers listed in `vec`."
Those containers become org-subheadlines
of the source node, each marked Birth::ContainerOf
to indicate "this node is a container of its view-parent."

If the ancestry is Root (no containers),
Repeated (cycle), or DepthTruncated (BFS limit hit),
there are no containers to insert -- the source node
has no containment hierarchy to show.

** step 3c: new function 'insert_ancestry_tree_recursive'
File: [[../../server/to_org/expand/backpath.rs][backpath.rs]].

Called by 'attach_ancestry_to_links_to_nodes' for each container.
Async recursive function (Pin<Box<dyn Future>> pattern,
same as 'integrate_linear_portion_of_path' at line 156).

For each AncestryTree node:
- call the existing 'prepend_indefinitive_child' with Birth::ContainerOf
- if Inner (i.e. this container itself has containers),
  recurse for each of its containers

Reuses 'prepend_indefinitive_child' (backpath.rs line 238),
which already handles fetching the SkgNode via TypeDB and
creating an indefinitive ViewNode with the given Birth.

The containers are iterated in reverse order.
Since 'prepend_indefinitive_child' prepends
(inserts at the front of the sibling list),
reversing the iteration preserves the original order.
