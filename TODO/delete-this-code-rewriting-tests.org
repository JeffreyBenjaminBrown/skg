* How this was found / how to verify (reproduce before trusting)
  1. Reachability, Rust:
       cargo run -q -p call-graph        # regenerates tools/introspect/call-graph/db/{functions,calls}.csv
     then BFS from =main= over calls.csv; anything not reached is a
     candidate. NOTE the call-graph is NAME-BASED and cannot see:
       - trait-impl methods (Display/Deref/Drop/serde derive ...),
       - serde-attribute string refs (=#[serde(deserialize_with="...")]=),
       - deref-coercion, fn-pointers, macro-generated calls.
     So "unreachable" OVER-reports. Always confirm with the compiler.
  2. The ORACLE for Rust is the compiler:
       cargo check --all-targets         # lib + every bin + every wired test
     If a deletion still compiles, nothing references it anywhere that
     compiles (serde/trait/tests/tools included). This is ground truth.
  3. Private items: =cargo check --lib= reports ZERO dead_code warnings,
     so every PRIVATE Rust item is production-live. Only fully-=pub=
     items can be dead-but-unwarned. Don't bother hunting private dead
     code; there is none.
  4. Elisp has no static unused-detection. Signal = textual symbol refs
     across all production =elisp/*.el= + ALL of =tests/= (this MUST
     include =tests/integration/*/test-emacs.el=, the end-to-end driver
     scripts — a fn used there is a real feature, NOT dead). Verify with
     =check-parens= per file + =bash bash/emacs-tests.sh= (239 tests) +,
     for anything integration-referenced, =bash bash/integration-tests.sh=.
  5. Transitive/aggressive: after each deletion, regenerate the
     call-graph and recompute — deleting a leaf exposes new leaves.
     Iterate to a fixpoint. (Both DONE batches were taken to fixpoint.)
