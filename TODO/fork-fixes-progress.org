   that is disk-content of a subscribee becomes a hide by F; re-adding

* What's done and went well
** Hide-on-removal (request item "Editing a forked node is bad")
- 'infer_hides_from_contains_removals'
  (server/from_text/local_instruction_collection/
  resolve_visibility.rs), a new pass inside visibility resolution,
  running before the subscribee-as-such signal loop so explicit
  subscription-view gestures win: for each owned Save intent with
  Specified contains, ids removed relative to disk that some (post-
  save) subscribee still contains become hides; ids added back that
  were hidden become unhides. Members invisible under a restricted
  source-set never count as removals (they were omitted, not
  removed). 'resolve_visibility' gained the restricted-source-set
  parameter; 'LoweredIntents' gained the accessor
  'save_intents_with_specified_contains'.
- Fork-creation analogue: 'build_fork_clone' now receives N's disk
  contains and sets the clone's hides to the children the forking
  edit deleted (previously always empty, so deleting a child WHILE
  forking would have left it to reappear as unintegrated subscribed
  content). 'ForeignPolicyOutcome::ForkCandidate' carries N's whole
  disk node now, not just its title.
- Tests: two new driven-save tests in tests/hidden_from_subscriptions.rs
  (fixtures-clone-edit): removal hides only subscribee-contained
  children; re-adding unhides. Verified to FAIL with the new pass
  neutered. Unit test fork_clone_hides_children_the_edit_deleted.
  Full Rust suite: 476/476.
