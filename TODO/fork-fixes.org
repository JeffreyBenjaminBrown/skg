*** Claude: neither, quite -- there is no operator grouping, and the regex flavor is Rust's, not Emacs's.
How regex search works ('build_regex_query' in
server/dbs/tantivy/search.rs): the query is split on WHITESPACE
into pieces; each piece becomes its own per-token RegexQuery
(indexed tokens never contain spaces, so a regex spanning a space
could never match anything); the keywords AND / OR / NOT between
pieces only set each piece's occurrence (must / must-not / should)
at the document level, with no nesting. Parentheses are never
operator grouping -- they reach the regex engine inside whichever
piece they sit in. Hence your three cases:
- 'cat.* AND dog.*' is two valid regex pieces, both MUST. Works.
- '\(cat.* AND dog.\)*' fails because Tantivy uses Rust regex
  syntax, where '\(' is a LITERAL paren -- the inverse of the Emacs
  convention. Piece 1 matches only tokens beginning with the
  literal text '(cat', which exist nowhere, and an unmatched MUST
  piece makes the whole query find nothing.
- '(cat.* AND dog.*)' errors because whitespace-splitting yields
  the piece '(cat.*', an unclosed group -- an invalid regex, hence
  the RegexQueryError you saw.
Grouping DOES work at the regex level within one whitespace-free
piece: '(cat|dog).*' is a single valid piece and works today.
