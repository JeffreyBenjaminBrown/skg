** in more detail
Burying fork content should not
resurface it in the subscribee-as-such.
Suppose F forks original O and both contain C.
C is hidden from O when O is viewed as a subscribee-as-such.
Suppose C is now removed from F's children.
It should not, as a result of that removal,
 reappear among O's children in a subscribee-as-such view of O.
That is, a hide relationship should be made from F to C.

** Claude: done 2026-07-08.
- 'skg-strip-trailing-whitespace-from-bodies' (elisp; the nvim
  command is StripBodyWhitespace) sends the new request "strip body
  whitespace"; the server handler is
  server/serve/handlers/strip_body_whitespace.rs, documented in
  api-and-formats.md. After the strip, the client echoes the server's
  per-source report plus your requested suggestion to review with
  'git diff --ignore-all-space'.
- It processes every source in the config, foreign ones included,
  per your spec.
- Each line's tail is stripped, and so are the body's trailing blank
  lines, because the canonical YAML writer (strip-chomping block
  scalars) cannot represent them -- keeping one would make the
  refreshed caches disagree with the file just written.
  A body that strips to nothing is dropped from its file.
- Only files whose body changes are rewritten; a clean file stays
  byte-identical. A rewritten file is re-serialized canonically, so a
  hand-formatted file can see one-time style churn beyond whitespace
  (the same churn any save of it would cause).
- Caches: the in-Rust graph and Tantivy are refreshed; TypeDB is
  untouched (it stores no body text, and textlink extraction cannot
  see trailing whitespace).
- Verified end to end against a throwaway server on fixture data;
  unit tests in tests/unit/strip_body_whitespace.rs; full Rust suite
  484/484 and emacs suite 261/261 pass.
