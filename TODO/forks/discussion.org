* TL;DR
The forks vision is *almost entirely already built*. The relations
('subscribes', 'hides_from_its_subscriptions', 'overrides_view_of'),
all eight PartnerCols, override substitution, the override-choice
menu, and the save-time hide/unhide interpretation of subscribee-as-
such edits all exist and are tested (the whole 'TODO/DONE/sharing/'
arc). The single missing piece -- the lone unresolved item in
'TODO/DONE/sharing/vision.org' ("fork a foreign node by editing it")
-- is the *fork/clone operation itself*: turning a foreign node N into
an owned clone C that copies N's text and child-list, subscribes to N,
and overrides N.

There is also one *prerequisite bug* that the fork depends on and that
your "nonobvious point" is exactly about: the subscribee-as-such
content display subtracts only the subscriber's hides, not the
subscriber's own integrated content. Until that is fixed, a fresh fork
would redundantly re-show everything it just integrated as
"unintegrated". See [[*Prerequisite: the subscribee-as-such display must subtract the subscriber's own contains][the prerequisite section]].

* The gap: the fork/clone operation
Per 'docs/forks.md' and 'docs/sharing-model.md', to "edit" a foreign
node N you instead create a *clone* C in a source you own, where:

- C.title and C.body copy N's (then become editable);
- C.contains copies N's 'contains' list -- *the child IDs only,
  shallow*, not N's descendants (this is what makes forking cheap:
  'docs/sharing-model.md' para 5);
- C 'subscribes' to N (so N's later additions surface to C);
- C 'overrides_view_of' N (so override substitution draws C wherever N
  would appear as content).

Today, editing a foreign node instead *rejects the whole save*:
'apply_foreign_policy' returns 'ModifiedForeignNode' (changed
title/body/contains) or 'CreatedForeignNode' (new node in a foreign
source), collected into a fatal error vector
('server/from_text/validate.rs:21-95'). The fork is the affirmative
alternative to that rejection.

* Reconciling docs/forks.md with the implemented model
'docs/forks.md' depicts the unintegrated subscribed content as a
single auto-generated branch titled
'<skg< unintegrated subscribed content from ... >>' hung under the
clone. *That branch should not be built.* As you noted: the existing
'subscribeeCol' under C already serves this exact role -- it shows N
as a subscribee-as-such, and the content under it is precisely what C
has not yet integrated (assuming the prerequisite fix below). So this
plan treats forks.md's '<skg<...>>' branch as obsolete narrative;
['docs/forks.md' must be rewritten] to describe the subscribeeCol
mechanism instead. (forks.md itself flags its examples as provisional:
"TODO: These examples are goofy.")

* tentative plan
** Prerequisite: the subscribee-as-such display must subtract the subscriber's own contains
This is the heart of your "nonobvious point", and it is a blocker for
forking, so it comes first.
*** DONE DECIDED (Jeff): Option B. A subscribee-as-such shows only what
the subscriber neither HIDES nor CONTAINS. So 'content_goal_list' must
subtract the subscriber's own 'contains' as well as its hides
(generalizing the formula below). The fork then needs NO explicit hides:
C's copied contains are excluded from its own subscribee-as-such view by
this rule, leaving only N's later additions. (This reverses an earlier
"Option A" detour where the fork would have materialized hides; Jeff:
"Container R should not both hide and contain content T.") The code
analysis below -- the formula, the test gap -- is still accurate; only
the prescribed fix is settled: change the display, not the fork.

*** What the code does today
The content shown under a subscribee-as-such is computed by
'content_goal_list' ('server/update_buffer/reconcile/content.rs:398-419'):

#+begin_src text
  subscribee.contains  MINUS  subscriber.hides_from_its_subscriptions
#+end_src

It subtracts only the subscriber's *hides*. It does not subtract the
subscriber's own 'contains'.

*** Why that is wrong for integration-by-moving
'resolve_visibility.rs:61-66' deliberately does NOT create a hide when
the user moves subscribee content into the subscriber's own graph
("hiding one's own content would be silly"). So after such a move the
content is in 'R.contains' but not in 'R.hides'. The display formula
above therefore still shows it under the subscribee-as-such -- so it
appears *twice*: once as R's integrated content and once as
"unintegrated" subscribed content. This contradicts
'docs/data-model.org'/'vision.org' example 2 (which expects it to drop
out of the subscribee view) and contradicts your statement that "the
subscribeeCol displays precisely its unintegrated content".

Evidence it is genuinely unhandled:
'tests/hidden_from_subscriptions.rs::test_moving_foreign_subscribee_content_to_subscriber_does_not_hide'
moves 'e1' into the subscriber and asserts only (a) no hide is created
and (b) 'e1' appears as content of 'r'. It never asserts 'e1'
disappears from the subscribee-as-such branch. Its existing-content
assertion 'contains("** (skg (node (id e1) ...")' would even match a
'**** ' (deeper) occurrence, so a duplicate would pass unnoticed.

*** Why it is a hard blocker for forks
A fresh fork has 'C.contains = N.contains' and 'C.hides = {}'. The
subscribee-as-such view of N under C would be
'N.contains - {} = N.contains' -- i.e. every child C just integrated
shown again as unintegrated. The fork would be unusable on day one.
'docs/sharing-model.md' para 2 already names the intended rule: there
is "an implicit 'hides_from_its_subscriptions' relationship from the
clone to all of its normal content ... any branch that shows up as
ordinary integrated content does not need to also show up in the
subscriptions." The display simply does not implement that implicit
rule yet.

*** The fix (two candidate shapes; recommend B)
- (A) Materialize the implicit hides: at fork time write
  'C.hides_from_its_subscriptions = C.contains'. Cheap to add, but
  bloats the file with redundant data and must be kept in sync with
  'contains' on every later edit -- fragile.
- (B) *Change the display to subtract the subscriber's own contains.*
  Make 'content_goal_list' for a subscribee-as-such return
  'subscribee.contains - subscriber.hides - subscriber.contains'. This
  implements 'sharing-model.md's "implicit hides to all normal
  content" without writing redundant data, keeps 'hides' meaning only
  "explicitly hidden", fixes the pre-existing move-into-graph bug
  (example 2) at the same time, and makes a fresh fork show empty
  unintegrated content automatically.

Recommend B. It is a localized change (one function reads the
grandparent/subscriber's 'contains' alongside its 'hides' and subtracts
both). It must be mirrored anywhere else the subscribee-as-such content
membership is derived; audit the initial-render path
('server/to_org/complete/partner_col/') for a sibling of
'content_goal_list', and the HiddenIn/HiddenOutside goal lists
('goal_list.rs') so the hidden buckets stay consistent (a node that is
both integrated and explicitly hidden should not double-count).

*** Order
Land B with its own regression *before* the fork work, so the fork's
"unintegrated content is empty initially" test has correct ground
truth. The regression: open a subscriber whose own 'contains' includes
a node M that a subscribee also contains; assert M does not appear
under the subscribee-as-such. (This is independent of forking and is
worth fixing regardless.)

** Design of the fork operation

*** Trigger and transport
**** DONE SUPERSEDED by the gesture veto (see Problems -> "The trigger
deviates from the long-term vision"). This section recommends the
EditRequest::Fork command, which Jeff vetoed in favor of the implicit
gesture (editing a foreign node forks it, with a save-time y/n
confirmation buffer). Rewrite this section to the gesture model once the
gesture sub-questions in that discussion are answered. Kept below only
for the reasoning it records.
The original N is shown as read-only/indefinitive content (under a
subscribeeCol, via override substitution, or as a foreign view root).
'EditRequestOnIndefinitive' ('parse_metadata_sexp.rs:188-200') forbids
attaching an edit request to an indefinitive node, and structurally an
'IndefOrDef::Indefinitive' has nowhere to store one. So the request
cannot ride on N. Two viable shapes:

- *Recommended -- EditRequest::Fork(target) on a NEW owned headline.*
  Mirrors merge exactly: merge marks the *acquirer* (an owned node)
  with '(editRequest (merge ACQUIREE))'; fork marks the *clone-to-be*
  (a new owned node, ID minted in 'add_missing_info') with
  '(editRequest (fork ORIGINAL))'. The new node is Definitive, so the
  indefinitive guard never fires. On save, a fork-extraction phase --
  modeled on 'nodeMerge_instructions_from_pairs'
  ('server/nodeMerge/nodeMergeInstructionTriple.rs:101-140'), which
  already does an async disk fetch of the referenced node -- fetches N
  and fills C's save intent (title, body, contains copied;
  subscribes_to=[N]; overrides_view_of=[N]). Stays entirely within the
  save pipeline (the sole graph-mutation path), reuses the writable-col
  persistence for the two relations, and matches a known precedent.

- *Alternative -- a dedicated "fork node" request that returns a
  pre-populated buffer.* Like single-root-view/override-menu: the
  server (no mutation) builds a buffer for a new C with title/body/
  contains copied and a subscribeeCol+overriddenCol already holding N;
  the user reviews/edits and saves through the normal pipeline. More
  "preview", but it needs speculative col construction for an unsaved
  node, which is new and fiddly.

Both keep the copy server-side (the client must never reconstruct N's
data). Recommend the EditRequest shape; see [[*Open questions][Open questions]] Q1.

*** What the clone C gets
- title := the title the fork gesture produced -- the user's edited
  title if they changed it while editing N, else N.title verbatim
  (decided with Jeff);
- body := N.body;
- contains := N.contains (shallow; the child IDs stored RAW -- decided
  with Jeff -- with override substitution applied only at render time,
  not at copy time);
- subscribes_to := [N];
- overrides_view_of := [N];
- source := a user-owned source (see next);
- hides_from_its_subscriptions := none. Per Jeff's Option-B decision
  (Problems -> "unintegrated content"), the subscribee-as-such display
  excludes what the subscriber both hides AND contains, so C's copied
  contains are already filtered out -- the fork records no hides, and the
  subscribee-as-such under C shows only N's later additions. (The
  prerequisite display fix carries this.)

*** Source selection for C
C must live in a source the user owns; N's is foreign/read-only, and
'add_missing_info' only inherits a source from a buffer *parent*
('add_missing_info.rs:102-121'), with a sourceless node hard-failing at
lowering ('lower.rs:250-252'). Resolution (decided with Jeff, under the
gesture model where the user edits N in place):
- infer C's source from N's nearest OWNED vognode ancestor in the view;
- if there is none (the nearest ancestor is also foreign), do NOT die:
  the fork-confirmation buffer (see the gesture discussion) is WRITEABLE
  for this -- the user sets C's source there with the metadata
  source-rotation command (on 'C-c s s' / 'C-c v m'), and the buffer
  explains that edits to anything else in it are ignored;
- if the resolved owned source is INACTIVE under the active source-set,
  FORBID the fork (Jeff: in a restricted set you are not meant to touch
  those sources) -- abort with an explanatory error, never create an
  invisible clone.

*** Monogamy: forking a node you have already forked
If N already has a user-owned overrider C1, minting C2 over N fails
'MultipleUserOwnedOverriders'
('override_invariants.rs:65-73,127-156'). The fork must *pre-check*:
look up user-owned overriders of N (the resolver already provides this,
'override_resolution.rs') and, if one exists, *redirect* the user to
edit the existing clone C1 -- never mint a second, and never
auto-delete C1 (that would silently discard C1's accumulated edits).
MVP minimum: reject with a message naming C1 ("you already forked this;
your clone is <C1>") and let the client offer to open it. (Decided with
Jeff: this reject-with-error is also how the save-time fork-confirmation
handles a monogamy-blocked node. There is NO analogous constraint on
subscriptions -- only on user-owned overrides, and a node may have at
most one user-owned overrider.)

*** One save, with correct ordering
Create + subscribe + override in a single save. Unlike move+merge
(genuinely conflicting; 'CannotMoveAndMergeSimultaneously',
'validate.rs:182-207'), these are additive on a fresh node, so no split
is needed. The constraint is ordering inside the save: the touched-
override invariant check reads the *post-save simulated graph*
('override_invariants.rs' header comment), so C and its
'overrides_view_of' edge must be in 'touched' and in the simulated
graph before validation -- exactly as nodeMerge simulates its three
nodes.
