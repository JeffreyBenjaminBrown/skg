#+title: Forks: implementation plan
#+startup: overview

A declarative plan for the fork/clone feature ("fork a foreign node by
editing it"). The decisions here were settled in 'discussion.org'; this
document states them, not the dialog. It points back to 'discussion.org'
only where the rationale is non-obvious.

Glossary: a *foreign* node lives in a source the user does NOT own
(read-only); an *owned* node lives in a source they do. *N* is a foreign
node the user edits; *C* is the *clone* a fork creates in an owned
source. C copies N's title, body, and child-list, *subscribes* to N (so
N's later additions surface to C as unintegrated content), and
*overrides* N (so override substitution draws C wherever N would
otherwise appear). A *source-set* is the subset of configured sources
active in a view.

* What already exists (do not rebuild)
The whole sharing substrate is built and tested (the 'TODO/DONE/sharing/'
arc):

- the relations 'subscribes', 'hides_from_its_subscriptions',
  'overrides_view_of' ('schema.tql'), persisted across FS/Rust/TypeDB;
- all eight PartnerCols, including the writable 'subscribeeCol' (writes
  'subscribes_to' via 'subscribeeCol_members') and 'overriddenCol'
  (writes 'overrides_view_of' via the 'SetOverrides' intent,
  'local_instruction_collection/traverse.rs:307-323');
- override substitution with the round-trip 'overridesHere' marker
  ('update_buffer/reconcile/content.rs'; validated in
  'buffer_to_viewnodes/validate_tree.rs'), visibility-gated so an
  inactive overrider falls back to drawing the original;
- the override invariants -- monogamy (<=1 user-owned overrider per
  node) and acyclicity -- checked at save and rebuild
  ('dbs/in_rust_graph/override_invariants.rs');
- the override-choice menu;
- save-time interpretation of subscribee-as-such child edits into
  hides/unhides ('local_instruction_collection/resolve_visibility.rs').

The gap this plan fills: there is no way to fork. Editing a foreign node
today is rejected outright ('from_text/validate.rs', 'apply_foreign_policy'
-> 'ModifiedForeignNode'). The fork is the affirmative alternative to
that rejection.

* The user experience
The user makes a foreign node N editable (the existing definitive-view
request, 'C-c s d'), edits it however they like -- title, body, add /
remove / reorder children, at any depth -- and saves. Any real change to
a foreign node is read as a request to fork it. The save does not commit
blindly: the server first shows a confirmation buffer listing every node
about to be forked and asks the user to approve. On approval, each edited
foreign node N becomes an owned clone C that overrides and subscribes to
N; from then on the user sees and edits C wherever N appeared, and N's
later additions show up under C as unintegrated subscribed content.

forks.md's auto-generated "<skg< unintegrated subscribed content >>"
branch is NOT built: the 'subscribeeCol' already shows exactly that, once
the prerequisite below lands.

* Prerequisite: a subscribee-as-such shows only the unintegrated remainder
Land this first; it is independent of the rest and fixes a pre-existing
bug on its own.

A subscribee-as-such must show only what the subscriber neither HIDES nor
CONTAINS. Today 'content_goal_list'
('update_buffer/reconcile/content.rs:398-419') computes
'subscribee.contains - subscriber.hides' -- it subtracts hides but not
the subscriber's own contains. Consequence (independent of forks): when a
user moves subscribed content into their own graph, 'resolve_visibility'
deliberately records no hide ("hiding your own content is silly"), so the
moved item still shows under the subscribee-as-such -- it appears twice.

Fix: make 'content_goal_list' for a subscribee-as-such subtract the
subscriber's own 'contains' as well as its hides. Mirror the change in
any sibling that derives subscribee-as-such membership, and confirm the
HiddenIn / HiddenOutside goal lists ('to_org/complete/partner_col/goal_list.rs')
stay consistent (a node both integrated and explicitly hidden must not
double-count).

Why this matters for forks: a fresh clone has 'C.contains = N.contains'.
With this rule, C's own subscribee-as-such view of N starts empty (every
copied child is excluded as contained) and fills only as N gains new
children. So the clone needs NO explicit hides. (This is "Option B" in
discussion.org; the rejected "Option A" would have had the fork
materialize hides, which double-books a node as both hidden and
contained.)

Regression: a subscriber whose own 'contains' includes a node M that a
subscribee also contains; assert M does not appear under the
subscribee-as-such.

* The fork

** Trigger and detection
Editing a foreign node forks it. Only a *definitive* (editable) foreign
node can be forked: an *indefinitive* one is read-only, so the user makes
it definitive first (the definitive-view request, 'C-c s d'). Definitiveness
is merely the prerequisite for editing; the trigger is the edit itself.
Concretely, remove the rejection of foreign edits in
'apply_foreign_policy': a 'SaveNode' for a foreign node whose buffer
content differs from disk ('buffernode_differs_from_disknode') becomes a
fork candidate instead of a 'ModifiedForeignNode' error -- so an unchanged
definitive foreign node forks nothing. (Creating a new node in a foreign
source -- 'CreatedForeignNode' -- is still an error; fork is about editing
an existing foreign node.) Any real change counts -- title, body, or
child-list, at any depth.

** The confirmation stage
A save carrying fork candidates does not commit silently. The server
collects every candidate, runs the monogamy and source checks below, and
returns a *fork-confirmation* response instead of a save result: a buffer
listing each node to be forked as a flat forest (one headline per
candidate under an invisible buffer-root), with explanatory text before
the first. Modeled on the override-choice menu (a server-built buffer the
client adopts) plus a follow-up request carrying the decision.

The buffer is read-only EXCEPT that the user may set each clone's source
(the metadata source-rotation command, 'C-c s s' / 'C-c v m'); edits to
anything else in it are ignored, and the buffer says so. It stays usable
for navigation: the user can run the usual ID-stack-push / search command
from it to look up relevant IDs.

- Approve (y): the server performs the forks and completes the save.
- Decline (n): the server aborts the WHOLE save (no node is written) and
  the client leaves the confirmation buffer open (the user still needs it
  to search for IDs).

** What the clone C gets
- title := the title the gesture produced -- the user's edited title if
  they changed it, else N.title verbatim;
- body := N.body (or the user's edit);
- contains := N.contains, shallow -- the child IDs only, stored RAW (not
  resolved); override substitution applies at render time;
- subscribes_to := [N];
- overrides_view_of := [N];
- hides_from_its_subscriptions := none (the prerequisite display rule
  excludes C's contains from its own subscribee-as-such view);
- source := chosen as below.

** Source selection
C must live in an owned source. Resolve it:
- infer from N's nearest OWNED vognode ancestor in the view;
- if there is none (the nearest ancestor is also foreign), the user sets
  it in the confirmation buffer (the one writeable field);
- if the resolved source is INACTIVE under the active source-set, FORBID
  the fork with an explanatory error -- under a restricted set the user
  is not meant to touch those sources, and an invisible clone is never
  created silently.

'add_missing_info' only inherits a source from a buffer parent
('buffer_to_viewnodes/add_missing_info.rs:102-121') and a sourceless node
hard-fails at lowering ('lower.rs:250-252'), so the inference and the
buffer-set path both feed a concrete owned source before lowering.

** Monogamy
A node may have at most one user-owned overrider; there is no analogous
constraint on subscriptions. So forking a node the user has already
forked would violate monogamy
('override_invariants.rs:65-73,127-156'). The confirmation stage detects
this with a new helper 'existing_user_owned_overrider_of(N)' (over the
resolver's inverse lookup) and rejects that candidate with an
explanatory error naming the existing clone; the client offers to open
it. Never mint a second clone, never auto-delete the first.

** Shallow copy and override resolution
C stores N's child IDs raw. At render each child runs through
'resolve_override', so:
- if a copied child is itself a node the user has forked, C shows that
  child's clone -- desirable, and visible via heralds;
- if N contains itself (or a cycle through the copied IDs), the
  resolver's cycle guard draws the repeat as an indefinitive cycle node
  -- normal cycle display, nothing special.

** One save: ordering and round-trip
Creation + subscribe + override happen in one save (they are additive on
a fresh node, unlike move+merge). The touched-override invariant check
reads the post-save simulated graph, so C and its 'overrides_view_of'
edge must be in 'touched' and in that simulated graph before validation
-- as nodeMerge already simulates its nodes.

The fork is the first code path that ever writes 'overrides_view_of'
from a save, so the 'overridesHere' round-trip (render -> save ->
re-render, keeping the container's stored 'contains' pointed at N, never
rewritten to C) needs dedicated coverage. The inactive-source fallback
(an inactive clone does not substitute; the original is drawn raw) is
already coded; it needs a test, not new logic.

* Collateral-rerender staleness fix (in scope)
A definitive subscriber's 'subscribeeCol' is not refreshed from the graph
during a collateral rerender. 'reconcile_subscribee_col_children'
('update_buffer/reconcile/subscribee_col.rs') computes the fresh goal
list but only applies it under 'if parent_indefinitive ||
source_diffs.is_some()' (line ~74), so a definitive subscriber in a
non-diff collateral view keeps stale children. The guard is correct for
the SAVED view (a definitive subscriber there is the source of truth for
that save) but wrong for a collateral rerender (which should refresh from
the graph). Fix: thread a "this is a collateral rerender" signal and
reconcile children regardless of definitiveness in that case. Forking
creates subscribers, which makes this easy to hit; pre-existing, but in
scope here. Test: open two views of a definitive subscriber, change its
subscriptions in one, save, assert the other refreshes.

**** Glossary (the terms in that paragraph)
- *view* / *buffer* :: one window onto the graph, shown as an org
  buffer (a tree of headlines). You can have several open at once,
  and two of them can show the same graph node at the same time.
- *subscriber* / *subscribee* :: a node that "subscribes" to another.
  The subscribee's content surfaces under the subscriber as
  "unintegrated subscribed content". (Forking uses this: a clone
  subscribes to its original. So once forks exist, subscribers become
  common.)
- *subscribeeCol* :: an auto-generated scaffold listed under a
  subscriber, holding the nodes it subscribes to. skg GENERATES it
  from the graph; the user does not type its membership. Its
  "children" are those subscribed nodes.
- *definitive* vs *indefinitive* :: definitive = shown editable
  (read-write); indefinitive = shown read-only. The SAME graph node
  can be definitive in one view and indefinitive in another.
- *the saved view* vs *a collateral view* :: when you save a buffer,
  that buffer is "the saved view". Any OTHER open buffer that shares a
  node with it is "collateral". After a save, skg re-renders the saved
  view AND every collateral view, so two windows onto the same node
  agree.
- *reconcile* against a *goal list* :: to refresh a generated
  collection, skg computes the "goal list" (the members it SHOULD have
  per the current graph) and reconciles the on-screen collection
  against it -- adding missing members, dropping stale ones.
- *stale* :: showing what the graph USED to say, not what it says now.
- *the guard* :: the `if parent_indefinitive || source_diffs.is_some()`
  condition that decides whether to actually apply the fresh goal list.
- *source_diffs.is_some()* :: shorthand for "we are in git-diff mode".
**** Plain English
Picture the same subscriber node S open in two windows, A and B.
In window A you change S's subscriptions and save. skg updates the
graph and re-renders A correctly. Window B is collateral -- it should
also refresh so ITS subscribeeCol shows the new subscription list. But
it doesn't: B keeps the OLD list on screen. That stale list is the bug.

Why: the function that refreshes a subscribeeCol
('reconcile_subscribee_col_children') always computes the fresh list
from the graph, but only APPLIES it when the guard holds -- i.e. when
the subscriber is read-only (indefinitive) OR we're in git-diff mode.
If the subscriber is editable (definitive) and we're not in diff mode,
it skips applying and leaves whatever was already drawn.

That skip is RIGHT for the saved view: in the window you just saved,
your buffer text is the authority on the subscriptions (you may have
just typed them), so skg must not overwrite it mid-save with a
graph-derived list. But the skip is WRONG for a collateral view, which
you did NOT edit -- it should always defer to the just-saved graph.
The code can't yet tell those two apart: "definitive + no diff" lumps
"the view you just saved" (don't refresh) together with "a collateral
view of the same node" (do refresh).

The fix: thread one extra boolean down the render path meaning "this
is a COLLATERAL re-render, not the view that was just saved." When it's
true, refresh the subscribeeCol from the graph regardless of
definitiveness. The saved view stays byte-stable; collateral views
catch up. (Forks make this matter because forking mints subscribers,
so you'll often have a subscriber open in two windows -- exactly when
this latent bug shows.)

* Implementation order
1. Prerequisite display fix (subscribee-as-such subtracts the
   subscriber's contains) + its regression. Independent; land first.
2. Collateral-rerender staleness fix + its regression. Independent.
3. Fork detection: in 'apply_foreign_policy', turn a modified-foreign

4. Fork extraction: a 'fork_instructions_from_pairs'-style async function

5. Monogamy pre-check ('existing_user_owned_overrider_of') and

6. The confirmation stage: the fork-confirmation response-type, the

   follow-up request that approves (commit) or declines (abort whole

7. One-save ordering + the override-write round-trip, with tests.
