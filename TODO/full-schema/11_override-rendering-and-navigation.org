* plan : override rendering and navigation
This plan covers the metaplan_2.org step "define override rendering
and navigation" -- the largest step, and the bridge to
[[../../docs/forks.md]].  Today the 'overrides_view_of' relation is
stored, validated (monogamy and no-owned-chains, at save and
rebuild) and displayed in cols, but never changes what is drawn.
After this plan: recursive content draws the user-owned overrider in
place of what it overrides; saving such a view round-trips to the
original IDs (the hardest correctness obligation here); visiting an
overridden node offers a choice instead of choosing silently; and
each navigation surface is classified, with a bypass for the
surfaces that must see the raw node.

* scope
** covered
*** The override resolver
one cycle-guarded function, one result

*** Substitution inside unified completion,

*** The save-side round-trip: a drawn overrider stands for the

*** The 'overridesHere' marker and its red "Oh" herald.
*** The overridden-as-such exception (draw and expand the original,

*** The override-choice buffer (the "menu") for new single-root views
  of overridden nodes, foreign overriders included.
*** Navigation surface classification, including the bypass request
  parameter and its Emacs surfaces.
*** api-and-formats.md and glossary.md touch-ups where behavior

*** The forks UX (the clone gesture, the social half) -- the

* desired behavior
** the resolver, one type
'server/dbs/in_rust_graph/override_resolution.rs', beside the
invariants it complements:
#+begin_example
resolve_override (config, graph, active, id) -> OverrideResolution
  { effective       : ID,      // the PID to draw
    path            : Vec<ID>, // overriders traversed; empty = none
    cycle_detected  : bool }
#+end_example
*** The input resolves through 'pid_of' first (extra-ID safety).
*** Walk: from the current PID, find its user-owned overriders
  ('user_owned_overriders_of' is the model).  An edge is followed
  iff its overrider is user-owned (OWNERSHIP is set-independent:
  'user_owns_it', never "currently active") AND its overrider's
  source is active (VISIBILITY gates substitution, per edge: an
  inactive overrider does not substitute, so the walk stops and the
  last visible node is the effective one).  Foreign edges are never
  followed; they are display-only facts.
*** Total on any data: a seen-set guards cycles ('cycle_detected',
  effective = the input, no substitution); more than one user-owned
  overrider at a hop (invalid data) stops the walk at that hop.
  Both are unreachable in a validated graph (see current behavior)
  but must not hang or panic on imports or hand-edited files.
*** 'path.len () > 1' signals a legacy compound chain; consumers
  surface the vision.org notice "Compound overrides relationship
  traversed. Such chains are legal but potentially confusing."
** substitution inside unified completion
When completion would CREATE a viewnode for node N as recursive
CONTENT -- de novo recursive content, definitive expansion, new
children introduced during post-save or collateral reconciliation --
it draws 'resolve_override (N).effective' = R instead, marked
'(overridesHere N)' whenever R differs from N.  Precisely:
*** CREATION-ONLY.  Existing viewnodes are never rewritten: a buffer
  already showing the original keeps showing, and editing, the
  original.  Closing and reopening normalizes.
*** CONTENT-ONLY.  PartnerCol members (all eight kinds), QualCol/Qual
  scaffolds, view roots, search-result roots, ancestry/backpath
  insertions, and diff phantoms always draw the raw node.
*** GATES, in order:
  1. omission first: if N's source is inactive, stage 9's omission
     removes it from the goal list before substitution is consulted.
     (Drawing R there would require a marker naming the inactive N
     -- a leak.  So an inactive original draws nothing, even when
     its overrider is active.)
  2. diff off: no substitution when
     'CompletionContext.source_diffs' is Some.
  3. the resolver's own per-edge ownership and visibility gates.
*** MATCHING.  The orderkey becomes the collected ID.
Background: the reconciler
('complete_relevant_children_in_viewnodetree') makes a parent's
actual children agree with a GOAL LIST -- the members the graph
says belong there, in order.  To do that it must decide, for each
existing child viewnode, which goal member it corresponds to.
That correspondence is the 'view_child_orderkey' closure: it
extracts an identity (the "orderkey") from each child; a child
whose orderkey appears in the goal list is kept (and reordered to
goal position), a goal member with no matching child is created
via 'create_child', and a child whose orderkey is not in the goal
list gets the stale treatment (deleted if leaf, demoted if
branch).

The problem substitution creates: P's goal list says N (contains
lists always hold original IDs), but the child drawn for N
carries R's own ID.  If the orderkey stayed "the child's own ID"
(today's definition), every rerender of P would misfire twice:
goal member N looks MISSING (so a second R is created for it),
and the existing R looks STALE (so it is deleted or demoted).
The buffer would churn on every save.

The fix is to define the orderkey as the COLLECTED ID -- the
'(overridesHere N)' original when the marker is present, else the
child's own ID.  Then the existing R-marked-N has orderkey N,
matches goal member N, and is simply kept.  Concretely, for P
contains [N], R overrides N:
- fresh expansion: no child has orderkey N, so 'create_child (N)'
  runs and (under the gates above) creates R marked
  '(overridesHere N)';
- post-save rerender: the marked child's orderkey is N, so it
  matches and nothing is recreated -- this is what makes saves
  idempotent;
- legacy buffer drawing raw N (no marker): its orderkey is its
  own ID, N, so it also matches and is kept as-is -- existing
  viewnodes are never force-upgraded to the overrider, which is
  the "creation-only" rule falling out of the same mechanism.
*** The created R is an ordinary content child: definitive-eligible
  (R is user-owned by construction, since only user-owned edges
  substitute), deduped buffer-wide by the existing defmap draw
  rule, budgeted like any new node.
*** Two goal members resolving to the same R: both are drawn, each
  marked with its own original.  The buffer shows the same drawn ID
  twice under one parent; collected-ID distinctness still holds,
  and the defmap keeps at most one occurrence definitive.
*** Compound chains emit the resolver's notice through the warning
  channel of whatever response carries the render (save responses
  via the existing sinks; de novo content-view responses need a
  small warnings-plumbing addition in 'multi_root_view_via_env').
** the round-trip through the original ID
The buffer draws R where the contains list says N; saving must not
rewrite N to R.
*** 'ViewNodeStats.overridesHere : Option<ID>' -- Some(N) means "this
  viewnode was drawn here in place of N, which it (transitively)
  overrides".  Serialized as the keyed form '(overridesHere N)'
  inside '(viewStats ...)'; parsed back by a new arm in
  'parse_viewstats_sexp'.  (Decided 2026-06-11: the marker carries
  the original ID, superseding the recorded bare-bool intent, and
  lives in viewStats as proposed.)
*** COLLECTED-ID RULE: 'content_members', 'visible_content_members'
  and the content distinctness validators all read the marker --
  for a marked child they collect N, not R.  Everything downstream
  (lowering, the weave, supplementation, the noop filter) already
  operates on collected IDs and needs no change.
*** The marker does NOT change self-writing: edits to the drawn R's
  title, body or cols save to R, keyed by R's own ID, like any
  definitive node.  Moving the marked child to another parent moves
  N (the lens travels with the viewnode); deleting it deletes N
  from the parent's list; reordering it positions N.
*** TAMPER VALIDATION.  A marker the server would not have drawn aborts the save.
Background: the marker is load-bearing text.  It travels to Emacs
as ordinary buffer characters, and the save pipeline trusts the
parsed metadata -- so anything a render can write there, a hand
edit, a killed-and-yanked headline, or a bug can write too.  And
wherever a marker appears, save extraction collects the marker's
ID instead of the child's own.  That is exactly the power that
must not be extended to arbitrary text.

The problem if unvalidated: put '(overridesHere M)' on any
visible child X of P, where M is ANY ID whatsoever.  On save, X
contributes M instead of X to P's collected contains, so P
silently loses X and gains M -- while the buffer still LOOKS like
it shows X under P.  No render produced that marker, so the save
would rewrite a graph fact the user never saw change: the
ID-aliasing hazard this whole section exists to kill, re-entering
through the metadata door.  Malice is not required; a yanked
headline carries its old metadata along.

The check, at buffer validation: every node carrying
'(overridesHere N)' must have its own ID equal
'resolve_override (N).effective' computed with active = None --
ownership-gated but visibility-UNGATED.  Both halves of that form
matter:
- "effective", not "a direct edge exists": resolution is
  transitive, so on a legacy chain Z over Y over X the drawn node
  is X marked '(overridesHere Z)' with no direct X-to-Z edge.  A
  direct-edge check would reject markers the server itself wrote.
- visibility-ungated: a marker that was honest when rendered must
  not start failing because the source-set switched afterward.
  The visibility gate governs what NEW renders draw; it says
  nothing about what an old buffer may truthfully claim.

On failure the save aborts with an explanatory error, like other
buffer validation failures.  Abort is the only safe response:
ignoring the marker would collect the carrier's own ID, honoring
it would collect N -- each a silent contains rewrite in one
direction or the other -- and the server cannot know which the
user meant.
*** Saves in diff mode still honor markers already present in the
  buffer (a stale marked buffer must not corrupt on save); the
  diff-mode gate governs creation only.
*** SOURCE-SET SWITCHES: the marker survives InactiveNode conversion.
(Found 2026-06-11 while specifying the tamper check; see the TODO
problem below, which awaits your blessing.)  Stage 9's
switch-rerender converts a now-inactive Active viewnode into an
InactiveNode, whose representation today carries only id and
source.  If a retained marked child -- R marked '(overridesHere
N)', kept because it has active view-children -- lost its marker
in that conversion, R would thereafter act as an ordinary
positional content member, so the next save would collect R where
P's disk says N and the weave would insert R into P's contains: a
graph edit the user never made.  So InactiveNode gains an
optional drawn-in-place-of field; conversion copies the marker;
the collected-ID and orderkey rules read it exactly as they read
the Active marker.  (The childless case needs nothing:
convert-and-prune removes InactiveNode leaves, and completion
redraws the position from the goal list.)
** the overridden-as-such exception
An Affected child of an OverriddenCol always draws the original
(cols never substitute, so this is automatic), and definitive
expansion requested from it expands the original: full contains, no
hides applied (the owner's hides are scoped to subscriptions;
'content_goal_list' already behaves this way because the node is
not a subscribee-as-such).  Its immediate children are created
WITHOUT substitution -- the user asked to see the original, and the
overrider is already its grandparent.  Deeper levels follow the
general rule (decided 2026-06-11: no cascade).
vision.org's example is the fixture: R subscribes to and overrides
E, R hides H, E contains G and H; with both cols expanded:
#+begin_example
  * R
  ** subscribeeCol
  *** E
  **** G
  ** overriddenCol
  *** E
  **** G
  **** H
#+end_example

** the override-choice buffer
When a "single root content view" request names a node Z that is
overridden by ANYONE -- any edge in 'overriders_of', user-owned or
foreign, with at least one overrider visible under the active
source-set -- the handler returns, after the inactive-root refusal
and the switch-to-view short-circuit but before rendering, an
ordinary Skg buffer of indefinitive nodes instead of a content
view.  (Not read-only as a buffer, decided 2026-06-11: like a
search buffer, its generated lines define nothing, but the user
can edit and save it, e.g. parking their own content under a
line.)
*** Root: Z, indefinitive, ParentIs::Absent.
*** Children: each overrider drawn as an indefinitive Independent
  child of what it overrides, following 'overrides_view_of' from
  overridden to overrider recursively, ALL edges including foreign,
  each branch stopping with the 'cycle' viewstat at the first
  repeated ID.  Overriders from inactive sources are omitted, with
  their branches.
*** Built from the in-Rust graph ('overriders_of') by a small
  seen-set BFS (model: the containerward ancestry traversal),
  reusing backpath.rs's indefinitive-Independent child-insertion
  helpers where they fit; run the standard finishing pass so the
  source heralds and "Op" (this node overrides its parent) appear
  -- both land for free.
*** If NO overrider is visible, fall through to a normal render: a
  menu with no choices would be noise.
*** The menu appears in diff mode too (decided 2026-06-11): it is
  navigation, not decoration, and it presents raw facts.  The
  diff-policy step will assert this.
*** Registered as a view under a dedicated URI kind, deduped by root,
  so it can be locked and rerendered without hijacking
  'content_view_uri_for_root_id'.  Precedence, decided 2026-06-11:
  an open raw content view of Z beats the menu (the switch-to-view
  check runs first); a second menu request for Z switches to the
  open menu; a raw view opened via bypass coexists with Z's menu.
*** The explanation goes to the minibuffer, never buffer text:
  "The requested node is overridden. Choose a destination."
  Delivered (decided 2026-06-11) via a new optional
  '(to-minibuffer "...")' field on the content-view response,
  which Emacs shows with 'message' alone -- no popped messages
  buffer.  The rare compound-chain notice keeps the warnings
  channel.
From inside the menu the normal gestures work: 'skg-goto' on an
overrider opens it; requesting a definitive view of any line --
including the root -- edits that node in place; and the bypass
command below opens the original as its own buffer.

** navigation surface classification
The "single root content view" request gains an optional field,
'(override-choice . "menu" | "bypass")', defaulting to menu.
Bypass skips the menu and renders the requested root raw; recursive
content beneath it still follows the substitution rules.
*** following a textlink to an overridden node: the menu (no Emacs
  change; it funnels through the one request).
*** 'skg-goto' and friends, including visiting a search result: the
  menu (ditto).
*** readable-ID jumps from magit: bypass -- 'skg-goto' sends bypass
  when invoked in a magit buffer.
*** requesting a definitive view inside an existing buffer: no
  request involved.  The node itself is an existing viewnode (never
  rewritten); its new children substitute, except under an
  overridden-as-such, whose immediate children draw raw (above).
*** opening (new buffer) from an overriddenCol child: the menu (falls
  out: the child is overridden by the col's owner at least).
*** collateral rerender after save: existing viewnodes unchanged; new
  content substituted (falls out of creation-only).
*** a new command, 'skg-goto-bypassOverride', sends bypass explicitly: the
  menu's escape hatch, usable anywhere.

** substitution rendering
*** De novo: P contains N; user-owned active R overrides N -> P's
  child is drawn R with '(overridesHere N)'.
*** Foreign overrider -> raw N.  Inactive overrider -> raw N.  Diff
  mode -> raw N.  Inactive N, active R -> neither drawn.
*** Chain X over Y over Z (hand-built graph) -> drawing Z draws X,
  path length 2, the compound notice in the response warnings.
*** Subscribee-as-such expansion: hides filter first, substitution
  after, on the survivors.
*** Two members of one parent resolving to one R: both drawn, marked
  distinctly.
** save round-trips (the hardest obligation)
*** Save the de novo buffer above: P's contains on disk still names
  N; R's file untouched by P's save; a second save is a noop
  (idempotence through the orderkey matching).
*** Edit the drawn R's title -> R's file changes, N's does not.
*** Delete the drawn child -> P loses N.  Move it under P2 -> N moves.
  Reorder it -> contains order reflects N's new position.
*** Subscribee-as-such: 'visible_content_members' reports N for a
  marked child, so no phantom hide of N is inferred.
*** Tamper: a hand-written marker failing the resolve check aborts
  the save with an explanation.
*** A legacy buffer drawing N raw still saves identically.
*** A marked buffer saved while diff mode is on still collects N.
*** A switch deactivating the drawn R: childless marked R is pruned
  and the position redraws raw N; a marked R with active children
  is retained as an InactiveNode that still collects N, so P's
  contains is unchanged by the next save.
*** Tamper-check staleness: a marked buffer still validates after a
  switch deactivates R's source (the check is visibility-ungated).
** overridden-as-such (R/E/G/H)
*** subscribeeCol expansion of E shows G only; overriddenCol

** glossary for this section
*** "Substitution"
Drawing the user-owned overrider R in place of
node N wherever completion would create N as recursive content..
*** "The resolver"
The one function computing overrider R from overridden N
(transitive, cycle-guarded, ownership- and visibility-gated).
*** "Transitive"
The resolver follows chains: if N's user-owned overrider Y is
itself overridden by user-owned X, the walk continues, and X is
the effective node.  Valid data cannot contain such chains (the
no-owned-chains invariant), so a walk longer than one edge signals
legacy data and surfaces the "Compound overrides relationship
traversed..." notice.
*** "Cycle-guarded"
The walk keeps a seen-set of visited PIDs.  On a repeated PID it
stops, reports 'cycle_detected', and substitutes nothing (effective
= the input).  Unreachable in a validated graph; the guard exists
so one bad import or hand-edited file cannot hang the server.
*** "Ownership-gated"
An edge is followed only if its overrider's source has
'user_owns_it = true'.  Foreign override edges never substitute;
they are display-only facts (cols, heralds, the menu).  Ownership
is read from the config alone, independent of the active
source-set.
*** "Visibility-gated"
An edge is followed only if its overrider's source is active in
the connection's source-set.  An inactive overrider cannot be
drawn, so it does not substitute: the walk stops there and the
last visible node is drawn instead.  Switching source-sets can
therefore change which node a view draws.
*** "The marker" / '(overridesHere N)':
metadata on a drawn viewnode
recording which original it stands for; what save extraction
consults to round-trip the parent's contains list.
*** "Collected ID"
the ID a child contributes to its parent's saved
lists -- the marker's original if present, else the child's own.
*** "The menu" / override-choice buffer
the buffer of indefinitive nodes offered instead of a content view
when the requested root is overridden.  An ordinary editable,
savable buffer, like a search buffer: only its generated lines
define nothing.

** DONE the marker must carry the original ID, contradicting a recorded decision
The 2026-06-10 decision (recorded in TODO/problems.org,
"Architecture note: nothing can represent 'drawn in place of
another node'") was a bare bool plus the convention that save
extraction recovers the original.  I believe the bool is unsafe.

disk contains ("which disk member resolves to this drawn R?"), and

*** glossary for this question
**** "Navigation surface"
Any gesture that asks to visit a node: following a textlink,
'skg-goto' and friends, visiting a search result, a readable-ID
jump from magit, opening a node from inside an existing view.  The
plan classifies each surface as a menu surface or a bypass surface.
**** "The menu"
The override-choice buffer: when a visited node is overridden, the
server offers the chain of overriders instead of silently picking
one.
**** "Bypass"
The other classification: skip the menu and open the requested
node itself as the buffer root.
**** "Raw"
Drawn as itself -- no overrider substituted for it.
**** "Root selection vs recursive content"
A bypass decides only WHICH node becomes the buffer's root.  The
content drawn beneath that root still substitutes per the normal
rules.  ("Fully raw" -- no substitution anywhere in the buffer --
exists today only in diff mode.)

The proposed optional field on the "single root content view"
request, defaulting to menu, by which Emacs tells the server which
classification this gesture wants.

'skg-readable-ids-mode' annotates UUIDs in magit buffers with
their titles; the "jump" is 'skg-goto' invoked with point on such
a UUID.  The metaplan classifies this surface as raw: it lands on
the original node, no menu.
**** 'skg-goto'
The existing Emacs command that finds an ID near point (metadata,
link, bare UUID, or .skg filename) and requests a single-root
content view of it.
**** 'skg-goto-bypassOverride'
The proposed new command that sends bypass explicitly, from
anywhere -- in particular the menu's escape hatch for opening the
overridden original as its own buffer.

** DONE a switch-converted substitute would corrupt its parent's contains
The scenario: under set 'all', P's content draws R marked
'(overridesHere N)' (P's disk contains says N); the user expands
R, giving it view-children; a source-set switch then deactivates
R's source.  Stage 9's convert-and-prune RETAINS R (it has active
children) as an InactiveNode -- whose representation today has no
marker field, so the marker would silently vanish.  From there,
rules that are each correct on their own compose into corruption:
stage 9 deliberately counts a retained InactiveNode as an ordinary
positional content member (right, for genuine members), so the
next save collects R where P's disk says N, and the weave inserts
R into P's contains -- a graph edit the user never made, of
exactly the kind the round-trip machinery exists to prevent.

Proposed fix, now in the body ("SOURCE-SET SWITCHES: the marker
survives InactiveNode conversion"): give InactiveNode an optional
drawn-in-place-of field, serialized only when present; the
conversion copies the marker; the collected-ID and orderkey rules
read it like the Active marker; the tamper check (deliberately
visibility-ungated) accepts it.  The childless case needs nothing,
since InactiveNode leaves are pruned and the position redraws from
the goal list.  Alternatives, if you dislike growing InactiveNode:
prune retained substitutes despite their children (discards the
user's subtree -- seems worse), or forbid saving such a buffer
(loud, but punishes an innocent user mid-edit).  Bless the field
or pick another treatment.

*** the herald payload-swallowing question is presentational
The situation: a substituted node's metadata will contain
'(overridesHere N)', where N is a UUID.  With heralds on, the user
should see a red "Oh" there -- and should NOT see the UUID.
"Swallowing the payload" means the herald rule matches the keyed
form, displays its fixed label, and discards the value.

Verified 2026-06-11 that the lens engine supports exactly this, so
the question is settled, not just presentational: in
'skg-lens.el', a rule's ANY directive matches whatever follows a
label, and its output is ONLY the fixed tokens listed after ANY --
the matched values are echoed only where the IT directive appears
among those tokens.  (The file's own example: rule
'(a (b (ANY x)))' outputs 'x' for object '(a (b c))' -- the 'c' is
discarded.)  The existing table uses both modes: the
'(sourceHerald V)' and '(merge ID)' rules include IT, so their
values show; the viewRequests rules list only fixed strings, so
the matched atom is replaced wholesale.  The "Oh" rule is
therefore one line in the existing style -- match 'overridesHere',
ANY with the single fixed token "Oh", no IT -- and the UUID never
reaches the screen.

The fallback mentioned in "the Oh herald" stays documented only as
a contingency in case some untested subtlety bites: the serializer
would additionally emit a derived display-only bare atom whenever
the marker is present, the "Oh" rule would match that atom, and
the keyed form would get a value-discarding rule.  That bare atom
would also need an accept-and-discard arm in
'parse_viewstats_sexp' (which errors on unknown bare atoms) so
buffers round-trip.  Cost: one redundant atom in the metadata of
substituted nodes.

Either way nothing load-bearing is at stake -- hence
"presentational": save extraction and the tamper check read only
the keyed form, the conformance test forces a rule to exist in
either shape, and the two branches differ only in what the
metadata's displayed form looks like.
