* plan : test the relationship matrix
This plan covers the metaplan_2.org step "test the relationship
matrix".  The matrix comes from metaplan 1 ("Test the relationship
matrix"): for every collection type, cover initial rendering,
definitive expansion, save after reorder, insertion of independent
children, deletion or mutation of generated members, source-set
restricted rendering, and collateral rerender after save; plus an
override-specific list and a warnings list.

Stages 8 through 12 each shipped with tests, so much of the matrix
is already covered.  This plan is therefore mostly an AUDIT (which
cells are covered, and by which test) plus a bounded list of new
tests that fill the real gaps.  Writing tests for already-covered
cells would only slow the suite; the audit names the covering test
instead, so a future reader can check the claim.

* glossary
Plain-language definitions of every term of art this plan uses.
- "relationship matrix" :: the grid of test obligations: one row per
  collection type, one column per behavior (the seven listed above).
- "relation" :: one of the five connection types in schema.tql:
  'contains', 'textlinks_to', 'subscribes',
  'hides_from_its_subscriptions', 'overrides_view_of'.
- "PartnerCol" :: a generated headline in a view that groups the
  nodes related to its owner by one relation, in one direction.
  Example: under node N, a "subscriberCol" lists the nodes that
  subscribe to N.  There are eight kinds.
- "outbound / inbound" :: direction of the relation relative to the
  col's owner.  Outbound: the owner points at the members (the
  owner subscribes to them).  Inbound: the members point at the
  owner (they subscribe to the owner).
- "writable col" :: a PartnerCol whose membership the user can edit
  by editing the buffer: subscribeeCol and overriddenCol.  Adding,
  removing or reordering members there changes the graph on save.
- "read-only set" :: a PartnerCol whose membership is computed from
  the graph and cannot be edited through the col: subscriberCol,
  overriderCol, hiderCol, hiddenCol.  Edits to it are repaired on
  save (members restored, intruders relabeled) with a warning.
- "read-only filter col" :: hiddenInSubscribeeCol and
  hiddenOutsideOfSubscribeeCol.  Their membership is derived from
  hide state rather than read directly off one relation.
- "QualCol" :: the alias and ID collections (AliasCol, IDCol).
  Their children are strings about the owner, not graph nodes.
- "member" :: a node listed inside a col because the graph really
  relates it to the col's owner.
- "independent child" :: a headline the user parked somewhere (for
  example inside a col) that is NOT a member; it carries
  '(parentIs independent)' and does not affect the owner on save.
- "definitive / indefinitive" :: a definitive viewnode is editable
  and shows its full content; an indefinitive one is a read-only
  mention (herald "☮").  Col members render indefinitive.
- "definitive expansion" :: asking the server to turn an
  indefinitive viewnode into a definitive one, in place, showing
  its content.
- "subscribee-as-such / overridden-as-such" :: a member sitting
  under a subscribeeCol / overriddenCol, shown in that role.
  Expanding a subscribee-as-such applies the subscriber's hides;
  expanding an overridden-as-such shows the raw original.
- "substitution" :: when rendering would create a viewnode for node
  N as ordinary content but N has a visible user-owned overrider R,
  the view draws R instead, marked '(overridesHere N)'.
- "monogamy" :: the validation rule that no node may have more than
  one user-owned overrider.
- "chain" :: X overrides Y and Y overrides Z with X and Y both
  user-owned; rejected at save and rebuild.
- "the menu" :: the override-choice buffer offered when the user
  asks to open a node that something overrides.
- "source set" :: the set of sources currently visible.  A node
  whose source is outside it is "inactive": omitted from cols,
  retained read-only in already-open views if it has visible
  children, never written by a save.
- "the weave / the set-difference merge" :: the save-time
  operations that put invisible (inactive-source) members back
  into a saved list so a restricted view cannot delete what it
  cannot show.  The weave preserves order ('contains',
  'subscribes_to'); the set-difference merge handles the
  order-free 'overrides_view_of'.
- "goal list" :: the member list reconciliation steers a col
  toward: graph truth, in some order.
- "demotion" :: reconciliation's treatment of a stale member that
  has children: it is kept but relabeled independent.  A stale
  member with no children is deleted.
- "collateral rerender" :: after a save, every other open view
  sharing a node with the saved one is re-rendered and pushed to
  Emacs.
- "phantom" :: a diff-mode-only line showing something that existed
  at an earlier git stage but not now.  Phantoms define nothing on
  save.
- "seam" :: a function boundary at which a behavior can be tested
  in isolation.  This plan uses three:
  - the "extraction seam": 'collect_instructions_locally' and its
    pure lowering, reachable without TypeDB via
    'definenodes_from_tree' (tests/new/local_instruction_collection/
    extraction.rs).  Tests what a buffer's text means for the graph.
  - the "reconciliation seam": the pure helpers under
    'server/update_buffer/reconcile/' (for example
    'view_order_preserving_goal_list'), tested in tests/unit/.
    Tests how a view is repaired toward graph truth.
  - the "pipeline seam" (end-to-end): a real save through
    'update_from_and_rerender_buffer' against a real TypeDB
    database, usually via 'server/test_utils.rs'
    ('run_with_test_db', 'update_from_and_rerender_buffer_test').
    Slow; reserved for wiring that the pure seams cannot reach.
- "the global graph handle" :: a process-wide snapshot of the
  in-Rust graph ('try_init_global_handle' / 'snapshot_global').
  Some features read it (substitution, gO/gS stats, marker
  validation).  A test that installs it should live in its own
  test target, because under plain 'cargo test' (one process per
  target) it leaks into sibling tests; 'cargo nextest run' (one
  process per test) is immune.
- "test target" :: one 'tests/NAME.rs' file, possibly pulling in a
  'tests/NAME/' directory of submodules and fixtures.  Cargo
  compiles each target as its own binary.
- "scenario" :: one unit of the batched matrix test: a before-file
  (the buffer as the user saves it, real org text) and an
  after-file (the expected rerendered buffer), plus any in-code
  assertions org text cannot express (disk contents, warnings).
  Scenarios run serially inside one test function, against one
  shared database of mutually independent subgraphs.

* scope
** covered
- The audit: every matrix cell mapped to a covering test or marked
  as a gap.
- New tests for the gaps, written at the cheapest seam that
  genuinely exercises the behavior.
- The override-specific checklist and the warnings checklist from
  metaplan 1, audited and filled the same way.
- The four named regressions from metaplan_2's step text, verified
  present (three already exist) or added.
** not covered
- Production code changes, with two decided exceptions (2026-06-12):
  - a failed save's response must carry warnings alongside errors
    (decision (b) under problems); if the pipeline does not already
    do that, the small change to make it so is in scope;
  - if the ONGOING expansion discussion (under problems) settles on
    generalizing the raw-immediate-children rule, that change is
    executed only once settled -- the rest of this plan does not
    wait for it.
  Otherwise: trivial test hooks only.  If a test exposes a bug,
  fixing it is in scope only if small; otherwise it is recorded in
  TODO/problems.org and the test pins current behavior with a
  comment.
- No new Emacs-side (elisp or integration) tests (decided
  2026-06-12).  The behaviors in this matrix are server-side; the
  Emacs wiring they ride (save responses, rerender streams, warning
  display) already has tests.
- The documentation step (next metaplan step) stays separate.

* the warnings checklist, audited
- successful save with warnings:
  covered.  tests/partner_col_warnings.rs
  ('readonly_col_repairs_warn': restored member, demoted intruder,
  and the discarded col-headline-text warning, all in one save).
- failed save with both warnings and errors:
  GAP.  Parse-time warnings (for example col headline text) are
  produced before validation can fail, but nothing said whether a
  failed save's response carries them alongside the errors.
  Decided 2026-06-12 (option (b) under problems): warnings always
  accompany errors.  The new test asserts it; if the pipeline does
  not already comply, the small change to make it so rides along.
- single-root content view warning delivery:
  covered.  tests/override_chain_notice.rs (a fresh view's
  compound-chain notice arrives in the content-view response).
- rerender-all warning delivery:
  reinterpreted, then GAP.  Stage 8 deliberately made collateral
  rerenders and rerender-all repair SILENTLY (a repair there is not
  the user's edit), so the original matrix cell ("rerender-all
  delivers warnings") is obsolete.  The cell that should exist
  instead: a save whose COLLATERAL view needs repair must not
  blame the saver -- the save response's warnings stay scoped to
  the saved view.  Stage 8 enforced this by construction but
  skipped the test as too heavy; the collateral harness in
  tests/hidden_from_subscriptions.rs has since made it cheap.  One
  new test.

* new tests to write, in order
Grouped by seam, cheapest first.  Names are suggestions; fixture
sketches use the title-equals-ID convention the suite already uses.

** at the extraction seam (pure, fast; extend tests/new/local_instruction_collection/extraction.rs)
- 'reordering_subscribees_reorders_subscribes_to': owner with
  subscribeeCol members [a, b, c] reordered to [c, a, b]; the
  emitted SetSubscribesTo (visible in the lowered DefineNode)
  lists [c, a, b].
- 'removing_one_subscribee_keeps_the_rest': members [a, b, c] with
  b deleted; emitted list is [a, c] (not empty, not unspecified).
- 'reordering_overridden_col_is_harmless': overriddenCol members
  [a, b] reordered to [b, a]; the emitted override set is equal as
  a set and nothing else about the owner changed.  (Pins the
  order-free property the set-difference merge depends on.)
- 'deleting_from_hiddenCol_emits_no_hide_change': owner with a
  hiddenCol member deleted; no intent touches the owner's
  'hides_from_its_subscriptions'.  (The filter cols have this
  guarantee tested; the plain hiddenCol does not.)

** at the save-simulation seam (pure-ish, fast; extend tests/save/override_invariants.rs)
- 'merge_that_collides_two_overriders_is_rejected': disk-shaped
  graph where R1 overrides N1 and R2 overrides N2, all user-owned;
  a save whose nodeMerge list merges N2 into N1; expect the
  monogamy violation.  Exercises the 'nodeMerge_definenodes' arm of
  'validate_override_invariants_after_save', currently untested.

Design:
- ONE test function builds ONE database (the database build --
  fixture copy, TypeDB create-and-load, in-Rust graph, Tantivy --
  is indeed the dominant per-test cost) and then runs scenarios in
  serial.  It must be one function, not one per scenario: nextest
  gives every test FUNCTION its own process, so splitting the
  scenarios across functions would rebuild the database per
  scenario.
- The fixture holds mutually independent subgraphs, one per
  scenario, IDs prefixed by scenario name (for example
  'subRm-owner', 'subRm-a'), so a save in one scenario cannot
  perturb another and the single database stays sound across the
  serial run.
- A scenario is a directory holding 'before.org' (the buffer as
  the user saves it -- real org text, easy to eyeball and easy to
  edit if a col's intended behavior ever changes) and 'after.org'
  (the expected rerendered saved view).  Assertions org text
  cannot express (disk lists for writable cols, expected warnings)
  live in a per-scenario code entry beside the runner.
- Scenario failures ACCUMULATE: the runner collects every mismatch
  and fails once at the end, reporting all of them, so one broken
  col does not mask the rest of the matrix.
Coverage, one scenario per cell where the cell applies:
- de novo render: a root that owns members in all four read-only
  relation cols (subscriberCol, overriderCol, hiderCol, hiddenCol),
  members indefinitive, sorted-ID order.  (This scenario compares a
  fresh render to an expected file rather than saving.)  Closes the
  plain-mode "initial rendering" gaps for hiderCol and hiddenCol.
- save after reorder: each read-only set (order preserved,
  warning-free); subscribeeCol (new order persisted to disk);
  overriddenCol (unperturbed, since the relation is order-free).
- insertion: a non-member into each read-only set (demoted to
  independent, with warning); a member into each writable col (the
  subscription / override edge appears on disk).
- deletion: a member from each read-only set (restored, with
  warning); ONE member from each writable col (exactly it removed
  from disk, the others kept); a member from hiddenCol (no unhide
  -- the extraction twin lives in the extraction seam above).
- omission under a restricted source-set: cols holding an
  inactive-source member beside an active one, rendered with a
  restricted set passed per call; the inactive member is OMITTED
  (no placeholder line), the active one shows.  Closes the
  restricted-rendering cells for the read-only and filter cols.
A second, separate test function in the same target (its own
database, since it must observe a REJECTED save) covers
buffer-level monogamy: a save adds a second user-owned overrider
for an already-overridden target via an overriddenCol; the save
response carries the monogamy error and disk is unchanged.  This
is the buffer-level half of the rejected-at-save checklist item.
The target should not need the global graph handle; keeping
overriders foreign-or-unique in the fixtures avoids tripping
substitution.

* suite-cost accounting
New pure tests: about seven (extraction, simulation).  New
TypeDB-backed test functions: about seven -- but the batched matrix
function carries roughly twenty file-pair scenarios on a single
database build, so exhaustive per-col coverage costs about what
three ordinary TypeDB tests cost.  New test targets: the matrix
target, plus possibly one for init refusal.  No new integration
(bash/Emacs) tests (decided 2026-06-12).  Ride-along: rename
'test_every_kind_of_col' to 'test_subscribee_and_filter_cols'
(tests/hidden_from_subscriptions.rs), retiring a name that
overstates its coverage.  This respects the metaplan's instruction
to prefer the pure seams; every end-to-end test above exists
because the behavior it pins lives in wiring (disk writes,
response contents, the collateral stream) that the pure seams
cannot see.

* test-suite hygiene rules for this stage
- A test that installs the global graph handle goes in its own
  target (precedent: tests/override_chain_notice.rs).  None of the
  planned tests should need it; if one turns out to (substitution
  sneaking into a fixture), isolate it rather than share.
- Every test that saves and then reads Tantivy must call
  'wait_for_tantivy_writes_idle' first (common-gotchas.md).  The
  planned tests read disk and rendered views, not Tantivy, so this
  should not arise.
- Unique database names per test ('skg-test-...'), since tests run
  in parallel under nextest.
- Known pre-existing flakes (the 'ids::save' group under plain
  'cargo test'; the once-seen 'test_title_diff_staged...' failure)
  are recorded in TODO/problems.org; do not chase them here, and
  run the suite via 'cargo nextest run' as the stages before did.
