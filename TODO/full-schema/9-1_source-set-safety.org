* plan : source-set safety for sharing relations
This plan covers the metaplan_2.org step
"preserve source-set safety for sharing relations", as revised by
two of Jeff's decisions (2026-06-11):
- Inactive SUBSCRIBEES-as-such must be shown (as placeholders), not
  omitted, because subscribee order matters, like content order.
  Only cols whose order does not matter omit inactive members.
- The write-to-inactive-node protection must not abort the save.  A
  source-set change can leave a whole subtree inactive; the save
  rightly reads those now-inactive children as belonging to their
  parents, and aborting would force the user to delete them from
  view -- destructive, since inactive nodes can affect their active
  parents.  Instead, save intentions that would modify an inactive
  node are suppressed, with the warning "Inactive nodes present in
  saved buffer remain unchanged in graph."

The organizing principle that fell out: a col's treatment of
inactive members follows whether its order is semantically
meaningful.
- Order matters (content; subscribeeCol): inactive members render
  as placeholders, count positionally at save, and round-trip.
  (UNDER RECONSIDERATION -- see the ONGOING open question: Jeff has
  proposed hiding inactive children everywhere and weaving them
  back in at save time.)
- Order does not matter (the six read-only cols; overriddenCol,
  decided 2026-06-11): inactive members are omitted from
  display, and saving preserves them via the set-difference merge
  (where the col is writable) or trivially (where membership is
  never read).

* scope
** covered
- Accepting InactiveNode placeholders under cols at validation, so
  stale buffers survive a source-set change.
- SubscribeeCol: inactive subscribees render as placeholders and are
  saved positionally, mirroring content placeholders.
- Order-meaningless cols: render-side omission of inactive members;
  for the writable 'overriddenCol', the set-difference merge so
  saving preserves invisible 'overrides_view_of' targets.
- Inactive-node rewrite suppression: save intentions that would
  modify an inactive node are dropped, not executed and not fatal,
  with a warning.
- Small doc updates where behavior changes (api-and-formats.md's
  source-set passages); the full documentation pass remains the
  metaplan's last step.
** not covered
- Diff-mode/source-set refusals.  A later metaplan step forbids the
  combination; until then this plan only keeps the combination from
  leaking (see open questions).
- Heralds, override rendering, and any change to content placeholder
  behavior: inactive CONTENT already round-trips correctly.
- Changing how source-set switches close buffers.  The
  render-then-switch scenario is served by making the save
  tolerant; whether switches should keep buffers open is separate.

* desired behavior
** inactive members under cols do not block saving
'validate_inactive_node' accepts an InactiveNode whose parent is a
col (QualCol or PartnerCol), not only a gnode.  What then happens to
the placeholder depends on the col:
- under a subscribeeCol it is a real, positional member (next
  subsection);
- under an order-meaningless col it is tolerated at save and removed
  by the rerender (the omission subsection).
** subscribeeCol: inactive subscribees are placeholders, like content
Because subscribee order matters, inactive subscribees follow the
content-placeholder model rather than being omitted:
- Rendering keeps them: the subscribeeCol's goal list does NOT
  filter by source activity, and 'apply_source_set_to_viewforest'
  placeholder-izes them as it already does.
- Saving counts them: subscribeeCol member extraction includes
  non-phantom Inactive children at their positions, the way
  'content_members' already does for content.  'SetSubscribesTo'
  therefore carries the full list, and NO merge is needed for
  'subscribes_to'.
- The placeholder itself stays read-only (editing its title/body/
  metadata remains a validation error), but moving it within the col
  changes the saved order, and -- pending an open question --
  deleting it removes the subscription, exactly parallel to content
  placeholders, whose moves and deletions are already honored.
- A placeholder subscribee-as-such shows no content and produces no
  visibility (hide/unhide) signals; the subscriber's hide list is
  untouched by its presence.
** order-meaningless cols omit inactive members
For the six read-only cols ('Subscriber', 'Overrider', 'Hider',
'Hidden', and the two hidden-subscription filter cols) and the
writable 'overriddenCol' (its order does not matter; decided
2026-06-11), members from
inactive sources are omitted from display entirely.  Their order
carries no meaning, so omission loses nothing, and "a private node
subscribes to public node N" is itself information a placeholder
would leak.  Three faces:
- goal lists exclude inactive members (fresh renders and post-save
  reconciliation never create them);
- an InactiveNode parsed from a saved buffer as a child of one of
  these cols is removed during reconciliation.  Decided 2026-06-10:
  delete it if it is a view-leaf (with the focus safeguard the
  reconciler already uses); convert one with view-children to a
  DeadScaffold, so the existing orphan handling
  ('mark_orphans_under_dead_parents_independent') preserves the
  user's subtree as independent nodes.  No warning either way.
- de novo col creation ('maybe_add_partnerCol_branches') filters
  before deciding the member set is non-empty, so a col whose
  members are all inactive is not created at all.
** saving an overriddenCol under a restricted set preserves invisible members
(Applies to 'overrides_view_of' only; 'subscribes_to' no longer
needs it, per the subscribeeCol decision.)  Semantics decided
2026-06-11, replacing an earlier buffer-order-first formulation that
was too diff-noisy: the saved list is computed as a SET DIFFERENCE
against disk, preserving disk order.
- removed = the VISIBLE disk members absent from the saved buffer
  (only visible members can have been deliberately deleted);
- added = buffer members not on disk;
- saved list = the disk list minus removed, in disk order, with
  added appended at the end.
Reordering visible members of an overriddenCol therefore has no
effect on disk, consistent with its order carrying no meaning -- and
the diff stays quiet.  A disk member that is invisible (inactive
source) or unresolvable (dangling reference) can never be in
'removed', so preservation of both falls out automatically.
Under 'all' (or with no set provided), every member is visible, so
this still honors any deletion; one named, unit-tested function.  It
must not depend on the client sending anything about invisible
members.
** inactive-node rewrite suppression: no instruction modifies an inactive node, without aborting
Decided 2026-06-11, replacing the earlier abort-style guard, which
was infeasible: a source-set change can leave a multi-node subtree
inactive in a stale buffer; the save rightly reads those
now-inactive children as belonging to their parents (preserving the
parents' contains lists), and aborting would force the user to
delete the subtree from view -- destructive.

Instead, after extraction, any instruction that would modify an
inactive node is SUPPRESSED:
- a 'DefineNode' (Save or Delete) whose source is inactive is
  dropped;
- a 'SourceMove' with an inactive endpoint is dropped (the node
  stays put);
- a nodeMerge whose acquirer or acquiree is inactive is dropped.
Whenever something was suppressed, the save succeeds and carries
the warning:
  "Inactive nodes present in saved buffer remain unchanged in graph."
Placement: after the noop filter, so an untouched stale node (whose
instruction the noop filter already discarded) does not trigger the
warning; only edits that were actually discarded do.  See open
questions.

This suppression also covers synthesized intents, e.g.
'apply_hiderel_delta_to_subscriber' creating a save intent for a
subscriber read from disk whose source is inactive.

*** motivation and wondering
It's a giant pain, both to code and (I imagine)
for a user who could find themselves
staring at a screen mostly full of empties.

Suppose view-node Nv, corresponding to graph node Ng,
has view-children Cv_1 ... Cv_k corresponding to graph nodes
Cg_1 ... Cg_k.
Suppose a source change renders some of the Cv_i inactive.
Could the user still be free to manipulate the visible Cv_i,
without clobbering the invisible ones?

Inserting new nodes should be no problem.
(It's a little ambiguous where they should go,
but a deterministic algorithm seems easy to write.)
Deleting nodes should be even easier.
Moving nodes around will be the tricky part.

When git sees that a passage moved,
somehow it finds a way to present the diff
to minimize the amount of red.
That would be ideal, but I think we can forego it
for a simpler greedy algorithm on this first pass.
*** a proposed algorithm
- The graph knows about gnodes, and the view, vnodes.
- When first displaying Nv's children, partition them
  (without reordering them) into 'visibility groups',
  each of type <(visible : bool, nodes : <NodeId>)>.
  Display only the visible groups, in order.
  (The total visible node order is defined by
  the order of the groups and the order within each group.)
- To apply a source change to an already-drawn buffer,
  first convert every inactive node to an InactiveNode.
  Then prune the tree, via DFS postorder, of every:
  - Inactive leaf
  - Qual leaf whose owning gnode (grandparent) is inactive
  - indefinitive leaf partner (child of PartnerCol)
  - empty QualCol or PartnerCol
  - DeadScaffold leaf (because why not)
- There might still be visible inactive nodes after the prune.
  Recurse into them, but otherwise ignore them;
  in particular, don't count them as children of their parents.
- When the buffer is saved, recompute visibility groups from disk.
- The server processes the view-children it receives
  from the client in order, to generate a list of gnodes
  to use as the intended saved content.
  - For the first view-child:
    - If it is new, prepend it to the first visible group.
    - If it is not new:
      - Read through the visible visibility groups for it
        (since it can't be in an invisible visibility group).
        If it is found, place it there.
        If it is not found, we have an error,
        but I believe this is unreachable
        (we just stipulated it's visible and not new,
        so it must be in one of the visibile groups).
  - For every viewchild Cv after the first:
    - Read through the visible visibility groups (gnodes)
      starting from where the last-processed vchild was placed.
      This is the greedy part.
      Read forward until finding the gnode corresponding to Cv.
      Put Cv there.
      If Cv is not in the list, then just place Cv
      right after wherever the last viewchild was placed.
- Now we have a list of visibility groups,
  in which some of the visible gnodes represent spots
  where saved vnodes were placed.
  Remove the other gnodes from those visible groups.
  Leave the invisible groups of gnodes as-is.
- Now we have the order of gnodes to save as content children.
  It has all the same invisible gnodes as before,
  and the visible gnodes appear
  in the order in which they were saved.
  It's possible that the visible and invisible ones
  are more shuffled than necessary, but whatever, good enough.

**** the anchored weave
Classify the disk children into visible and invisible.  Each
maximal invisible RUN is anchored to the visible item immediately
before it in disk order (a leading run is anchored to START).  The
user edits only the visible sequence, producing V'.  The saved list
is rebuilt in one pass: START's run first, then for each item of
V', in the user's order: the item, then the run anchored to it (if
any).  A run whose anchor was DELETED reattaches to the anchor's
nearest preceding visible sibling that survived, else to START.
Properties:
- One pure function, 'weave (disk_list, is_visible, buffer_list)
  -> merged_list'.  O(n), no scanning state, exhaustively
  unit-testable.
- MOVES behave well: an invisible run follows its anchor wherever
  the user drags it.  Your greedy scan re-places visible nodes
  against a fixed group skeleton, so dragging a visible node far
  from home can strand the invisible nodes that followed it.
- Inserts and deletes are trivial; when everything is visible, or
  the user changed nothing, it degenerates to the identity.
- The invariant is one sentence: "invisible children cling to the
  visible sibling they followed."
