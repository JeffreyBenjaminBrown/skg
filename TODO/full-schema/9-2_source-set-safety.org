* plan : source-set safety for sharing relations, take 2
This supersedes [[./9-1_source-set-safety.org]], which became a
dialog; the decisions that emerged from it (all Jeff's, 2026-06-11)
are baked in here:
- The ANCHORED WEAVE is blessed.  Inactive children are omitted from
  display everywhere -- content, subscribees, every col -- and the
  save weaves the invisible members back into the saved lists, so
  nothing is clobbered.  No more screens full of empty placeholders.
- EXCEPTION: an inactive node that has active view-children stays on
  screen (as an InactiveNode).  It cannot affect its parent, and its
  children cannot affect it, but its children have lives of their
  own.
- Source-set switches now RE-RENDER open views in place instead of
  closing them ("closing them all was a wart").  This is new scope.
- The switch-rerender prunes, among other things, ALL indefinitive
  leaf partners (children of PartnerCols), active or inactive: an
  indefinitive partner defines nothing, and completion regenerates
  the cols that belong under the new active set anyway.
- Deliberately removing an INVISIBLE relationship is impossible, and
  that is good ("That was dangerous").
- 'overrides_view_of' saves via the set-difference merge (its order
  does not matter); 'contains' and 'subscribes_to' save via the
  weave (their order matters).
- The write-protection is "inactive-node rewrite suppression":
  instructions that would modify an inactive node are dropped, not
  fatal, with the warning "Inactive nodes present in saved buffer
  remain unchanged in graph."

* scope
** covered
- The anchored weave for 'contains' and 'subscribes_to'; the
  set-difference merge for 'overrides_view_of'; plain omission for
  the read-only cols.
- Render-side omission of inactive children everywhere, with the
  retained-inactive exception (active view-children).
- Source-set switches re-rendering open views in place (server and
  Emacs), including the convert-and-prune step.
- Inactive-node rewrite suppression, with its warning.
- Validation updates so buffers containing InactiveNodes (under cols,
  or with children) parse and save.
- Doc updates where behavior changes (api-and-formats.md source-set
  passages); the full documentation pass remains the metaplan's
  last step.
** not covered
- Diff-mode/source-set refusals (a later metaplan step).  For the
  interim, decided 2026-06-11: restricted diff views quietly omit
  inactive members' removed-member phantoms along with everything
  else inactive -- leak-free, merely lossy, and short-lived.
- Heralds and override rendering.
- 'textlinks_to': derived from the user's own body text, no col, no
  source-set interaction to manage.

* current behavior, verified 2026-06-10/11
** the bug this plan fixes
Recorded in TODO/problems.org ("Sharing-col members from inactive
sources make a buffer unsavable").  Under a restricted source-set,
'apply_source_set_to_viewforest' ('server/source_sets.rs') converts
EVERY Active viewnode from an inactive source into an InactiveNode
placeholder, including members of PartnerCols.  Buffer validation
('validate_inactive_node',
'server/from_text/buffer_to_viewnodes/local.rs' near line 343)
requires an inactive placeholder's parent to be a gnode, so such a
buffer cannot be saved.  And if that validation were relaxed alone,
the failure mode would be silent relationship deletion:
'partnerCol_members' (traverse.rs) collects only Active children,
the resulting 'SetSubscribesTo' / 'SetOverrides' lowers to
'MSV::Specified', and 'supplement_unspecified_fields_from_disk'
fills only Unspecified fields -- the inactive member vanishes from
disk.
** facts the design leans on
- Content placeholders currently round-trip positionally
  ('content_members' includes non-phantom Inactive children); the
  weave REPLACES this placeholder model, so several behaviors in
  'tests/source_sets.rs' (placeholder rendering, placeholder
  moves/deletes editing contains) will be retired -- the main test
  churn.
- Hides need no merge: 'apply_hiderel_delta' (lower.rs) edits the
  subscriber's hide list as a DELTA against disk, so unmentioned
  targets are preserved by construction, and omitted (invisible)
  subscribees produce no visibility signals.
- Aliases carry no sources; AliasCol is unaffected.
- The save pipeline never learns the active source-set today:
  'handle_save_buffer_request' has it, but
  'buffer_to_validated_saveplan' ('server/from_text.rs') takes only
  (buffer, config, driver).  'CompletionContext' (render side)
  already carries 'active_source_set'.
- Pipeline order in 'extract_nonmergeSavePlan_locally'
  ('local_instruction_collection/mod.rs'): collect, text claims,
  lower, resolve visibility, supplement from disk, noop filter.
- Switching the active source-set today closes every open Skg view:
  'set_active_source_set' ('server/serve/handlers/source_sets.rs')
  clears the server's view registry, and Emacs closes the buffers
  (api-and-formats.md).  The streaming machinery the replacement
  needs already exists: rerender-all's lock / per-view / done
  message flow ('server/serve/handlers/rerender_all_views.rs').
- Direct view requests for inactive-source nodes are already
  refused, so an inactive view ROOT cannot arise from a fresh
  render.
- 'ActiveSourceSet' offers 'is_all' and 'contains_source'; it is per
  connection.  'env.find_source (pid, deleted_map)' resolves a
  member's source, in-Rust graph first.

* desired behavior
** rendering: inactive children are omitted
Under a restricted set, a fresh render never draws a node from an
inactive source: content goal lists and every col's goal list
exclude them, and de novo col creation skips a col whose members
are all inactive.  No placeholders.  Note the recursive consequence:
omitting an inactive container omits its whole branch, even active
descendants (they remain reachable by direct visit or search); the
retained-inactive exception below arises only in views that were
ALREADY OPEN when a switch occurred.  Confirmed 2026-06-11,
including the asymmetry that a fresh render of the same root shows
less than a switched-in-place view does.
** the retained inactive node
When a source-set switch is applied to an open view (below), an
inactive node whose view-children include active nodes stays on
screen, converted to an InactiveNode.  It is read-only; its
relationship lists cannot be affected from this view (rewrite
suppression below); its active children remain editable self-writers.
Because the buffer then CONTAINS that InactiveNode, it acts as an
ordinary positional item in its parent's saved list (the weave sees
it in the buffer), so MOVING it within its parent is honored.
DELETING it (with its subtree) is honored as a VIEW edit only:
omission keeps inactive nodes out of the content goal lists, so
nothing recreates it in this view -- it stays gone from the screen.
The GRAPH relationship survives regardless: the weave preserves
invisible disk members, so under 'all' the parent still contains
it.  (2026-06-11: corrected from an earlier draft that wrongly
claimed it would respawn in the view; Jeff caught this.)
** saving: the anchored weave for contains and subscribes_to
One pure function,
'weave (disk_list, is_visible, buffer_list) -> merged_list', applied
at supplementation (disk in hand) to 'contains' and 'subscribes_to'
when the buffer specified them and the active set is restricted:
- Classify disk members: visible iff their source is active.
  (Unresolvable sources count as invisible, hence preserved.)
- Each maximal run of invisible disk members is ANCHORED to the
  visible member immediately before it in disk order; a leading run
  anchors to START.
- Output: START's run, then each buffer member in buffer order
  followed by its anchored run (if that member survived as an
  anchor).  A run whose anchor is gone from the buffer reattaches
  to the anchor's nearest PRECEDING visible disk-sibling that
  survived, else to START.
Invariant: "invisible children cling to the visible sibling they
followed."  Moves of visible members carry their invisible
followers; inserts and deletes are trivial; with everything visible
(set 'all', or no set) it degenerates to the identity, i.e. today's
behavior.  A buffer-present InactiveNode (retained case) is just a
buffer member like any other.
Consequences worth stating:
- The user cannot delete what they cannot see; an explicitly emptied
  col or parent still keeps its invisible members.
- 'subscribes_to' and 'contains' get identical treatment; no
  positional placeholder machinery remains.
** saving: the set-difference merge for overrides_view_of
Order does not matter, so simpler than the weave: the disk list
keeps its order; members the user visibly deleted (visible on disk,
absent from buffer) are subtracted; new buffer members are appended;
nothing else changes.  Reordering an overriddenCol is a no-op and
the diff stays quiet.  Invisible and unresolvable members can never
be subtracted, so they survive automatically.
** read-only cols
Membership is never read from the buffer, so omission alone
suffices.  Goal lists exclude inactive members; a stale InactiveNode
under such a col (from an old buffer) is deleted if a view-leaf and
deadened (DeadScaffold; existing orphan handling preserves the
subtree as independent) if it has view-children.  No warnings: an
omitted member is not a repair, and warning would partially leak.
** source-set switches re-render open views in place
Replacing today's close-all behavior, on both sides:
- Server ('set_active_source_set'): instead of clearing the view
  registry, for each open view: apply the CONVERT-AND-PRUNE step
  below, then run completion with col creation enabled (a switch
  can also ACTIVATE sources, so newly visible members and cols must
  appear), then the usual finishing pass.  Stream the results with
  the existing rerender-all message flow (lock, per-view, done).
- Emacs: stop closing buffers on switch; reuse the rerender-all
  handlers to lock, replace and unlock them.  The existing
  confirmation prompt and the cancellation of in-flight search
  enrichment stay.
- View-local niceties that pruning destroys (e.g. a read-only col's
  user-chosen member order) normalize, exactly as closing and
  reopening used to normalize them.
** convert-and-prune (applying a switch to a drawn view)
1. Convert every Active viewnode from a now-inactive source to an
   InactiveNode.
2. Prune by DFS postorder (so emptied parents prune in the same
   pass), removing every:
   - InactiveNode leaf;
   - Qual leaf whose owning gnode is inactive;
   - indefinitive leaf partner (child of a PartnerCol), ACTIVE OR
     INACTIVE -- an indefinitive partner defines nothing, and
     completion regenerates current membership;
   - empty QualCol or PartnerCol;
   - DeadScaffold leaf.
3. What survives includes: active nodes; inactive nodes with
   surviving children (the retained case); definitive partners (the
   user may be mid-edit inside them).
Completion then rebuilds cols and members for the new active set.
** inactive-node rewrite suppression
Unchanged from 9-1 (decided 2026-06-11): after the noop filter, any
instruction that would modify an inactive node is dropped --
'DefineNode' Save/Delete on an inactive source, 'SourceMove' with an
inactive endpoint, nodeMerge touching an inactive node, and
synthesized intents (e.g. 'apply_hiderel_delta_to_subscriber' for an
inactive subscriber).  If anything was suppressed, the save succeeds
and carries the warning
"Inactive nodes present in saved buffer remain unchanged in graph."
Timing decided 2026-06-11: edits-only -- the warning fires only when
an instruction was actually suppressed.  An untouched stale node
saves silently, because the noop filter discards its
identical-to-disk instruction before suppression looks.
** validation updates
- 'validate_inactive_node' accepts an InactiveNode under a col
  parent (stale buffers) and WITH children (the retained case);
  editing the placeholder's own title/body/metadata stays an error.
- Extraction continues to count buffer-present InactiveNodes as
  positional members of content and subscribeeCols (they are
  anchors), and to ignore them under read-only cols.

* open questions -- all closed 2026-06-11, answers folded into the body
** glossary for the questions below
- "Inactive source": a source excluded by the connection's active
  source-set; its nodes are hidden from view.  ("Active" = in the
  set.)
- "Omission": this plan's rule that rendering simply skips inactive
  nodes (no placeholder).
- "Retained inactive node": the exception -- an inactive node that
  stays on screen (read-only) because it has active children, which
  can only happen in a view that was already open when the
  source-set switched.
- "The weave": the save-time function that merges the buffer's
  visible list with the disk list, so invisible members survive a
  save, anchored near their old neighbors.
- "Inactive-node rewrite suppression": the save stage that silently
  drops any instruction which would modify an inactive node,
  warning "Inactive nodes present in saved buffer remain unchanged
  in graph."
- "The noop filter": an existing save stage that drops instructions
  whose content is identical to what is already on disk.
- "Diff mode": the git diff view -- open views re-rendered with
  decorations showing changes since HEAD, including phantom entries
  for members REMOVED since HEAD.
** DONE when should the suppression warning fire?
CLOSED: 2026-06-11
(Carried over from 9-1.)  Suppose a buffer contains a
now-inactive node.  If the user EDITED it, the suppression drops
the write and should clearly warn.  But if the user left it
untouched, its write instruction is already identical to disk and
the noop filter discards it before suppression looks: under the
plan's current choice the save is then SILENT.  The alternative is
presence-based warning: warn whenever inactive nodes were in the
saved buffer at all, even untouched -- closer to your original
wording, but it would warn on every save of such a buffer, which
risks training the user to ignore warnings.  Choose: warn only when
an edit was discarded (plan's choice), or warn on presence.

** DONE deleting a retained inactive node silently fails -- acceptable?
CLOSED: 2026-06-11
The weave cannot distinguish "this invisible disk member was
omitted at render" (must preserve) from "this retained inactive
node was visible and the user deleted it" (deletion intended) --
in both cases the member is on disk and absent from the buffer.
The plan preserves in both cases, so deleting a retained inactive
node from the buffer silently undoes itself: it respawns on the
next render.  Moving it, by contrast, IS honored.  Options:
- Accept the asymmetry (plan's choice): movable but undeletable,
  consistent with "inactive data is untouchable".
- Add a warning when the rerender restores one (the stage-8 repair
  warning machinery could carry it).
- Make deletion work by consulting the server's stored copy of the
  view (it knows the node was displayed); costs coupling the weave
  to per-view state.
