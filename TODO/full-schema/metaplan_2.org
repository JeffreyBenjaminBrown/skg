* metaplan 2 : full schema UI
** purpose of this document
This supersedes [[./metaplan_1.org]].  Write sub-plans from the
"steps, in order" section below, one file per step, as was done for
metaplan 1.  Move finished sub-plans to [[./DONE/]] and log progress
in [[./progress.org]].

The goal is unchanged: make every relationship in 'schema.tql'
('contains', 'textlinks_to', 'subscribes',
'hides_from_its_subscriptions', 'overrides_view_of') inspectable,
editable where owned, safe under source-set filtering, clear in
heralds, and coherent under save, rerender and git diff views.

The destination this work aims to unlock is [[../../docs/forks.md]]:
a user clones a branch of someone else's (foreign, read-only)
document into their own document; the clone subscribes to the
original (so upstream additions appear, foldably, as unintegrated
content); the clone overrides the view of the original (so the user
sees their own version wherever the original would have been drawn);
and read-only inbound collections let the original's owner discover
who has forked their data.  Subscription machinery mostly exists.
The largest missing piece on the path to forks is override
rendering and navigation.
** what changed while full-schema was paused
The code was overhauled after metaplan 1 stalled.  Sub-plans drawn
from this metaplan should trust the descriptions below over anything
in metaplan 1, vision.org, or 7_saving-readonly-cols.org.
*** save extraction was replaced by local instruction collection
'server/from_text/local_instruction_collection/' (see glossary.md
and TODO/DONE/local-instruction-collection/).  One pure DFS preorder
('collect_instructions_locally' in 'traverse.rs') emits per-field
intents into a map keyed by PID; downstream stages consume the map in
order: text-claim validation, lowering to 'NodeIntent's,
visibility resolution, disk supplementation
('build_diskSupplemented_defineNodes'), the noop filter.

Facts that matter for the remaining full-schema work:
- The writable cols are AliasCol, SubscribeeCol and OverriddenCol.
  Each emits a Set* intent ('SetAliases', 'SetSubscribesTo',
  'SetOverrides') only when its owner is save-eligible.  An absent
  col lowers to 'MSV::Unspecified' (no opinion, filled from disk by
  'supplement_unspecified_fields_from_disk'); a present-but-empty col
  lowers to 'MSV::Specified(vec![])' (explicit empty set).
- The six read-only RoleCols, the IDCol, and Qual leaves give their
  vognode descendants 'LocalContext::UnderReadOnlyCol': those nodes
  are self-writers; their membership in the col is never read.
- 'relation_col_members' (traverse.rs) collects only
  'Vognode::Active' children with 'parentIs=Affected'.
  Inactive placeholders are counted as content
  ('content_members') but NOT as relation-col members.  See the
  source-set step below; this asymmetry is load-bearing.
*** rendering and post-save updating are one unified process
'complete_viewforest' in 'server/update_buffer/complete.rs' (see
TODO/DONE/local-view-update/plan_v2.org) is a single level-order BFS
used both for de novo views ('render_initial_view') and post-save
rerendering ('rerender_view', driven by 'update_views_after_save').
Per-collection reconcilers live in 'server/update_buffer/reconcile/';
the sharing-col reconcilers share
'reconcile_sharing_col_children' and 'build_child_data'
('server/to_org/complete/sharing/child_data.rs'), which delegate to
'complete_relevant_children_in_viewnodetree'
('server/update_buffer/util.rs').

The uniform stale-member rule (plan_v2.org §6.0): a stale
parentIs=Affected member that is a leaf is deleted; one with children
is demoted to 'parentIs=Independent'; duplicates are deleted.
Missing goal members are created.  Children are reordered to
goal-list order, irrelevant (e.g. independent) children first.
*** the viewnode vocabulary changed
- 'parentIs=collector' no longer exists.  'ParentIs' is now
  '{Affected, Independent, Absent}'.  Collection membership is
  marked 'Affected' (the default); demotion targets 'Independent'.
  Metaplan 1's stage 6 introduced 'collector', and
  7_saving-readonly-cols.org speaks 'collector' throughout; both are
  stale on this point.
- 'ViewNodeKind' is now: 'Vognode(Active(TrueNode) | Inactive)',
  'Phantom(Diff | Deleted | Unknown)', 'QualCol(ID | Alias)',
  'Qual', 'PartnerCol(RoleCol)', 'BufferRoot', 'DeadScaffold'.
- 'RoleCol' has eight variants: writable 'Subscribee' and
  'Overridden'; read-only 'Subscriber', 'Overrider', 'Hider',
  'Hidden', 'HiddenInSubscribee', 'HiddenOutsideOfSubscribee'.
*** inbound relation order is deterministic but not user-controlled
'inbound_pids_for_relation'
('server/dbs/in_rust_graph/relation_accessors.rs') sorts inbound
members by ID so renders are stable.  Reconciliation then reimposes
that sorted order, so a user's reordering of a read-only col is
reset on every save.  Outbound (writable) col order is the disk
Vec order, which the user's save defines, so it is preserved.

** DONE make read-only sets ergonomic under save and rerender
Supersedes 7_saving-readonly-cols.org.  The reconciliation machinery
already restores missing members, deletes stale leaves, demotes
stale branches and removes duplicates.  Three behaviors are missing.
*** one typed col policy
Add a policy method on 'RoleCol' (beside 'relation_member_role' and
'caller_label', consolidating them where natural) returning an enum,
something like:
- 'WritableSet' for 'Subscribee' and 'Overridden': membership edits
  are graph edits; absent col means no opinion; empty col means
  explicit empty set.
- 'ReadOnlySet' for 'Subscriber', 'Overrider', 'Hider', 'Hidden':
  membership is generated; user order is respected view-locally;
  repairs warn.
- 'ReadOnlyFilter' for 'HiddenInSubscribee' and
  'HiddenOutsideOfSubscribee': membership is derived (these cols
  have no direct relation role; their goal lists are computed from
  hide state, not from 'relation_member_role').
Reconciliation dispatch, save extraction's read-only treatment, and
the herald metadata should all consult this one method, so the next
relation we add cannot have inconsistent policies in different
layers.

Stale-member policy, decided 2026-06-10: demote, not discard.  The
current reconciler's uniform rule (delete a stale leaf, demote a
stale branch to independent) is confirmed for ALL col kinds,
including the filter cols, superseding the earlier discard decision
from 7_saving-readonly-cols.org.  The user keeps any subtree they
built under a stale hidden entry.  Record this in the enum's
documentation so the policy cannot drift per-file again.
*** preserve user-visible order in read-only sets
For 'ReadOnlySet' cols, compute the goal list as:
1. the col's existing Active parentIs=Affected children, in current
   buffer order, filtered to graph-real members;
2. then any graph members not present, appended in sorted-ID order
   (determinism for the appended tail only).
This is a change to how 'reconcile_relation_col_children' computes
'goal_list' before calling 'reconcile_sharing_col_children'; the
reconciler itself already applies goal-list order.  The order is
view-local: it lives in the Emacs buffer and in the 'ViewState'
viewforest, never on disk.  Two buffers showing the same read-only
set may disagree on order.  Closing and reopening a view may
normalize to sorted order.
*** warn when reconciliation repairs user edits
Add a warning sink to the completion pass (e.g. a field on
'CompletionContext' in 'server/update_buffer/complete.rs'), drained
into 'SaveResponse.warnings' via the existing
'RerenderAfterSaveContext::warnings'.  Emit warnings only when
completing the view the user saved; de novo rendering, collateral
rerenders and rerender-all repair silently (those repairs do not
correspond to edits the user just made in that buffer).

Make the warning a typed enum (col kind, owner PID, affected child
PIDs, repair kind), rendered to strings late.  Batch per col owner:
"Repaired subscriberCol under <owner>: restored 2 members, demoted 1
non-member to independent."  Useful categories:
- non-member relabeled independent,
- stale leaf member removed,
- missing member restored,
- duplicate member removed,
- filter-col stale member discarded.
*** col headline text edits
Col scaffolds are titleless server-side; Emacs heralds supply their
labels.  The parser currently extracts each headline's title and
discards it for scaffolds ('viewnode_from_metadata' has no title
field for them).  At parse time, if a scaffold headline carries
nonempty title text, emit a warning that the text was discarded.
This needs no new state in 'ViewNode'; it is a check at the
'MpViewnode' construction site.  If inspection shows the title is
unavailable even there, document the impossibility in the sub-plan
and drop the warning rather than adding parser state.
*** IDCol edits abort the save
Decision from vision.org stands: editing IDs through the UI is
dangerous and pointless; direct file editing remains the escape
hatch.  Current behavior: structurally invalid IDCol shapes already
error ('server/from_text/buffer_to_viewnodes/local.rs'), but
deleting or reordering ID scaffolds is silently ignored and
regenerated.  Add validation that the multiset of ID scaffolds under
each IDCol equals the node's disk/graph ID list, erroring with an
explanation if not.  This is a buffer-validation-stage check; the
in-Rust graph already knows each node's ID list, so no extra disk
read is needed.

** DONE preserve source-set safety for sharing relations
Source-set filtering must neither leak inactive-source data nor
destroy invisible relationship targets.  Currently
'apply_source_set_to_viewforest' ('server/source_sets.rs') converts
EVERY Active node from an inactive source into an
'InactiveNode' placeholder, including members of relation cols.
Buffer validation then requires an inactive placeholder's parent to
be a gnode ('validate_inactive_node'), so a buffer whose
subscribeeCol contains an inactive-source member cannot be saved at
all.  (Recorded in TODO/problems.org.)  And if that validation were
relaxed without further work, 'relation_col_members' would drop the
inactive member from the 'Specified' list and the save would
silently delete the relationship: 'supplement_from_disk' fills only
'Unspecified' fields.
*** an inactive member under a col must not block saving
Decided 2026-06-10: a buffer containing an inactive-source member
under a sharing col must still be savable.  (A user might render
subscribees and then change the active source-set -- say, because
someone walked into the room -- and they should still be able to
save.)  So 'validate_inactive_node' must accept an InactiveNode
whose parent is a col, rather than erroring.  Extraction already
ignores such children ('relation_col_members' collects only Active
members), and the visible-subset merge below preserves their
memberships from disk; tolerating them at validation is the missing
piece.  The save's rerender then omits them per the next
subsection.  Note the consequence: the user cannot remove an
invisible member's relationship while it is invisible, by design.
*** render: omit, do not placeholder-ize, col members
For all eight RoleCols, members from inactive sources should be
omitted from the display entirely.  Order is not semantically
meaningful for sharing relations, so omission loses nothing, and "a
private node subscribes to public node N" is itself information the
placeholder would leak.  Content placeholders are unchanged: content
order matters, so inactive content stays placeholder-represented.
*** save: writable col lists are visible-subset edits
When the active source-set is restricted, a 'SetSubscribesTo' or
'SetOverrides' list extracted from the buffer describes only the
visible subset.  Define one named merge operation (tested once, used
for both relations): the saved list is the buffer's visible list
(user order preserved), followed by the disk list's members whose
sources are inactive (order meaningless, so append).  Visible
members absent from the buffer are removed; invisible members are
always preserved.  When the active source-set is 'all', the
operation degenerates to the current overwrite.

This probably belongs at the lowering or supplementation stage,
where disk state is already in hand; it must NOT depend on the
client sending anything about invisible members.
*** guard: no instruction may touch an inactive source
Add an explicit save-stage check, after supplementation, that no
'DefineNode' (Save or Delete) and no 'SourceMove' targets an
inactive source; reject with an explanatory error.  This protects
against present and future leak paths, e.g. visibility resolution's
'apply_hiderel_delta_to_subscriber', which can synthesize a save
intent for a subscriber read from disk; that subscriber's source
might be inactive in the current view.

*** the resolution rule, as one type
Define a resolver in one place (probably beside
'override_invariants.rs' in 'server/dbs/in_rust_graph/'):
given a PID, follow user-owned 'overrides_view_of' edges from
overridden to overrider, transitively, with a seen-set cycle guard.
Return a type, something like:
- 'effective' : the PID to draw (equal to the input when no
  user-owned overrider exists),
- 'path' : the chain traversed (empty when no substitution),
- 'cycle_detected' : bool.
Foreign override edges never participate in substitution; they are
display-only facts.  Valid saved data cannot contain user-owned
chains, but the resolver must still terminate on invalid or foreign
data (imports, hand-edited files), hence the guard, not an assert.
A path longer than one signals legacy data; surface the minibuffer
notice "Compound overrides relationship traversed..." from
vision.org when it happens.
*** substitution inside unified completion
When completion would CREATE a viewnode for node N as recursive
content (de novo views, definitive expansion, and new children
introduced during post-save reconciliation), it draws
resolve(N).effective instead.  Existing viewnodes are never
rewritten: a buffer already showing the original keeps showing, and
editing, the original; closing and reopening normalizes.  This
matches the collateral-rerender decision in vision.org.

Substitution must be OFF when diff mode is enabled (diff views show
raw graph facts).  Source-sets interact in two distinct ways
(corrected 2026-06-11, see the discussion below): OWNERSHIP is
set-independent ("user-owned" means 'user_owns_it', never "currently
active"), but VISIBILITY gates substitution -- an overrider from an
inactive source cannot be drawn, so it does not substitute, and the
original is drawn instead.  Switching source-sets can therefore
change which node a view draws, necessarily.

Two interactions need explicit treatment in the sub-plan:
- the contains list still names the original N; the drawn child is
  resolve(N).effective.  Save extraction reads the buffer, so the
  drawn overrider's presence as content must round-trip to the
  ORIGINAL ID in the container's contains list, or saving a
  container would silently rewrite its contains.  This likely needs
  the substitution recorded on the viewnode (see 'overridesHere'
  below), consulted by 'content_members'.
- the overridden-as-such exception: an Affected child of an
  OverriddenCol always draws the original, and definitive expansion
  requested from it expands the original (the overrider is already
  its grandparent).  Mirror of the existing subscribee-as-such
  pattern, including: expansion from an overriddenCol does NOT apply
  the owner's hides (those are scoped to subscriptions), whereas
  expansion from a subscribeeCol does.  vision.org's R/E/G/H example
  is the test fixture for this.

** (incorrect) the hidden-subscription stale-member policy drifted
From the decided spec (discard stale members) to the uniform rule
(demote stale branches, delete stale leaves).  Maybe an improvement,
but it happened by unification rather than by decision.  Solved by

** (dangerous) inactive-source members of sharing cols break saving
Restricted source-set plus a sharing col with an inactive-source
member makes the buffer unsavable today, and the failure mode if
that validation were relaxed is silent relationship deletion
('MSV::Specified' overwrite semantics).  Both recorded in
TODO/problems.org.  The source-set step's visible-subset merge is
the fix.  Until it lands, restricted-set users can hit a save error
with no workaround inside the buffer.
** (dangerous) override traversals can hit cycles in unvalidated data
Override substitution is a rendering-time graph traversal over data
that validation does not fully control (foreign sources, imports,
hand-edited files).  An assert or an unguarded loop is a server hang
one bad import away.  Solved by spec: the resolution-rule step
requires the resolver (and the override-choice traversal) to be
cycle-guarded and total.  The sub-plan must keep that requirement.
** (dangerous) substitution creates an ID-aliasing hazard at save
The buffer draws overrider R where the contains list says N.  If
save extraction naively read children, every save of such a
container would rewrite its contains from N to R -- a silent graph
edit the user never made.  Solved by spec: the substitution step
requires the round-trip through the original ID, marked by
'overridesHere'.  The sub-plan should treat this as its hardest
correctness obligation, with tests.
** (dangerous) repair warnings could misattribute collateral repairs
The same reconcilers run for the saved view, collateral views and
rerender-all; warning from the latter two would blame the user for
edits they never made, training them to ignore warnings.  Solved by
spec: the warnings step scopes the sink to the saved view.

** (other) herald rules and server stats are string-coupled
The rules in 'elisp/heralds-minor-mode.el' form a parallel table to
the server's metadata vocabulary, coupled only by strings, so adding
a stat (like 'hiding') has been a scavenger hunt.  Jeff asked
(2026-06-10) whether both sides could refer to one data file, JSON
perhaps.  Solved: almost exactly that, with one refinement -- the
heralds step now includes "one vocabulary, machine-checkable": Rust
GENERATES a JSON enumeration of the metadata vocabulary from its
existing enums, a conformance test checks the elisp herald table
against it, and elisp may additionally load its rule table from the
JSON.  Rust generates rather than refers because making JSON the
source of truth for Rust would cost exhaustive-match checking (or
require build-step codegen to win it back), while Emacs, having no
compile-time checks to lose, can refer freely.
