** stage 3 : TypeDB knows source ownership
*** planned
Wrote 'TODO/full-schema/3_typedb-knows-ownership.org',
a focused implementation plan for representing configured source
ownership in TypeDB.

The first draft kept the existing node 'source' attribute for
compatibility, but that was rejected as insufficiently clean.
Rewrote the plan so TypeDB sources are first-class entities and nodes
belong to sources only through a 'has_source' relationship.

The plan now includes the wider TypeDB query migration, source-move
updates, direct test-helper setup changes, config-change rebuild
guard, and tests proving that TypeDB nodes no longer own the old
string 'source' attribute.
*** implemented
Changed the TypeDB schema so configured sources are first-class
'source' entities with:
- 'source_name',
- 'user_owns_it'.

Removed the node-owned TypeDB string 'source' attribute.  Nodes now
belong to sources through 'has_source', with the node role constrained
to exactly one source:
#+begin_example
plays has_source:node @card(1)
#+end_example

Added TypeDB source helpers for creating source entities, replacing a
node's source relation, querying node source names and ownership, and
counting node-source relations for tests.

Updated TypeDB initialization and direct test setup paths to create
source entities before creating nodes.  Node insertion now creates the
node and its 'has_source' relation in the same transaction.

Rewrote 'pid_and_source_from_id' to recover source names through
'has_source'.  Source moves now update the 'has_source' relation
instead of a node attribute.

Updated the TypeDB/in-Rust graph audit to compare source membership
through 'has_source'.

Added the startup incremental-init guard for config changes: a running
server does not poll 'skgconfig.toml', but on restart a config mtime
newer than '.skg_init_marker' prevents incremental init.

Collapsed the source-move TypeDB helper surface back to one public
function, 'update_node_source'.  The implementation still updates the
'has_source' relation, but the temporary
'replace_node_source_relation' name is gone.

** global string replacement would be unsafe
The words "replacement" and "replaced" appear in many unrelated
contexts, including buffer/string replacement and content-link
replacement.  The implementation should classify matches before
editing rather than applying a mechanical global rename.

Stage 1 was implemented with targeted edits only.
Searches for old override role strings confirmed that remaining
"replacement" / "replaced" language is unrelated to the
'overrides_view_of' TypeDB roles.

** TypeQL match-insert can silently insert nothing
When node insertion was rewritten to match a source entity and then
insert the node plus 'has_source' relation, missing source entities did
not produce a TypeDB error.  The match simply had no rows, so the
insert did nothing.  This showed up as source-move tests finding no
nodes in TypeDB.
*** resolution
Node creation now explicitly checks that the source entity exists
before issuing the insert.  Setup paths were also updated to create
source entities before nodes.

'cargo test --test move_source -- --nocapture' passes after the fix.

*** resolution
Updated that helper to call 'create_all_sources' before

'cargo nextest run --test collateral_source_move' passes after the
