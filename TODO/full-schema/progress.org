** stage 4 : validate override invariants
*** planned
Wrote 'TODO/full-schema/4_validate-override-invariants.org',
a focused implementation plan for rejecting ambiguous or recursive
user-owned override states at rebuild/init and save time.

The plan centers the stage on one shared graph-shaped validator,
save-time post-apply simulation before persistence, full-graph
startup validation even after incremental database updates, extra-ID
resolution before comparison, and client-facing validation errors.
*** implemented
Added a shared override-invariant validator for the in-Rust graph.
It rejects:
- more than one user-owned overrider for the same resolved overridden
  PID,
- user-owned override chains of the form X overrides Y overrides Z
  where X and Y are user-owned,
- nodes whose source is absent from the loaded config.

The validator resolves override targets through the graph's primary-ID
mapping before checking collisions.

Factored the in-Rust graph DefineNode application logic so the save
path and validation simulation share one mutation semantics.

Save-time validation now simulates the real post-save graph before any
store is mutated.  The simulation applies the same delete-propagation
cleanup as the save path, then applies merge DefineNodes, and returns
a structured buffer validation error if override invariants would be
violated.

Full TypeDB rebuild/init now validates the full node set before
populating TypeDB.  Incremental startup validates the full graph after
the all-node read and before publishing the server environment.
Manual database rebuild also validates before replacing stores.

Separated graph construction from override-invariant validation in the
init/rebuild paths.  Callers now build an 'InRustGraph' explicitly and
then call 'error_unless_override_invariants_hold', so validate-only
uses no longer look like they are depending on a returned graph.

** stage 5 : source mismatch representation
*** planned
Wrote 'TODO/full-schema/5_source-mismatch-representation.org',
a focused implementation plan for replacing the audit's set-shaped
source mismatch adapter with a scalar source mismatch representation.

The plan keeps relationship role-player-set mismatches as sets, adds
an explicit source mismatch variant, and updates audit formatting so
source names are not represented as node IDs.
*** implemented
Replaced the audit's single set-shaped 'Mismatch' struct with a
variant representation:
- relationship mismatches keep relation, role and role-player ID sets,
- source mismatches carry the node PID, the in-Rust source name, and
  the TypeDB source name,
- missing TypeDB nodes are represented as their own mismatch variant.

Updated audit formatting and the ad-hoc audit probe so source
mismatches are printed as node-level source facts rather than as
'relation=has_source role=node' relationship-set mismatches.

Updated scheduled-audit org output and audit-store recording to handle
the new mismatch variants.  The audit store still uses six CSV fields,
but recorded payloads are now strings, so source names are not forced
through 'ID'.

The first implementation briefly added an audit-specific TypeDB source
cardinality helper, but that duplicated the TypeDB schema constraint
that each node plays 'has_source:node' exactly once.  The implementation
now uses the existing scalar source-name lookup and represents the
TypeDB side of source mismatches as a concrete 'SourceName'.  If the
node+source query returns no row, the audit records a distinct missing
TypeDB node mismatch instead.

** collector membership initially broke subscribee-as-such semantics
Stage 6 changed generated collection members from implicit container
children to children explicitly marked 'parentIs=collector'.  That exposed old
subscribee-as-such code paths that recognized editable subscribees
only through non-ignored/container parent status.

The symptom was hidden-subscription completion either failing to
prefetch child data or failing to infer hide/unhide edits after a user
moved a generated hidden child into visible subscribee content.
*** resolution
Changed the generated-collection membership tests so collection
members must have 'parentIs=collector'.  The older omitted/default
'parentIs=container' shape is no longer accepted as collection
membership.

The two affected meanings were:
- a TrueNode child of 'subscribeeCol' is still a subscribee
  member only when it is marked 'parentIs=collector',
- a TrueNode child of a definitive subscribee-as-such branch is
  still visible subscribee content if it is either an ordinary
  container child or a generated child marked 'parentIs=collector'.

The change was deliberately split by meaning.  Generated
collection membership is strict: 'subscribeeCol', 'overriddenCol' and
the read-only relation Cols count only children marked 'parentIs=collector'.  Ordinary
content completion and ordinary save extraction still use
'parentIs=container' / non-ignored when deciding whether a node is real
containment content.  Visible content inside an expanded
subscribee-as-such branch can still include a child marked 'parentIs=collector', because
moving a generated hidden child into the visible branch is the existing
gesture for inferred unhide.

Local validation now rejects TrueNode children of
'subscribeeCol' or the relation Cols unless they have
'parentIs=collector'.  Test fixtures that meant collection membership
were updated to say so explicitly; test fixtures that meant "extra
display child, not a collection member" now use 'parentIs=independent'.

Updated the affected rendering and integration fixtures because the
new intended output is different: collection scaffold headlines have
empty titles, and generated collection members now print
'(parentIs collector)' instead of relying on omitted/default
'parentIs=container'.
