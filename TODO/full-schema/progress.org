*** constraints this work imposes on future work or use
- The per-connection invariant "diff mode on implies active set
  'all'" now has two guarded doors.  Any future entry point that
  mutates either flag (e.g. a config-reload feature, a future
  per-view mode) must preserve it or re-derive the refusals.
- Refusal texts for the diff-toggle endpoint must never contain the
  substring "\nWarning:" (the Emacs handler's window-pop trigger);
  a comment at the refusal site says so.
- The quiet shape is now load-bearing protocol: anything Emacs
  sends after preemptively locking buffers must be answered either
  by the real rerender stream or by the empty one.
- 'NodeChanges' now diffs all four stored relation lists; a future
  relation must extend 'diff_in_nodechanges' (the compiler forces
  the match arm) and gets attribution, outbound goal lists, the
  inverse scan and col existence for free if it follows the same
  shapes.
- The inverse scan runs once per inbound col per diff render,
  scanning only CHANGED files; cost is proportional to the change.
  A pathological diff (thousands of changed files) pays it per
  inbound col on screen; if that ever bites, the scan's product
  could be computed once per render and cached in the context (the
  plumbing precedent exists).
*** unanticipated future potential costs and benefits
- BENEFIT: diff-mode rendering now performs NO git object reads
  beyond the initial per-source diff collection (the outbound and
  filter goal lists reconstruct everything from the per-stage

- BENEFIT: 'three_snapshots_of_relation_list' is general (any

- COST / AWARENESS: 'newM' on an inbound col member can appear on a

- AWARENESS: under a new owner FILE (Added, not Modified), outbound

  files (node_changes is only read for Modified files).

***** claude: elaboration -- the scenario, the choice, and the alternative
The scenario takes a user gesture to reach; rendering never
produces it.  Suppose P contains N, and R overrides N, so a view
of P draws the marked line

  ** (skg (node (id R) ... (viewStats (overridesHere N)))) R

Now the user KILLS that line and YANKS it somewhere else -- the
metadata, marker included, travels with the text.  Where it lands
determines what the marker means:

- Yanked into CONTENT position (under some container Q): the
  marker is honored.  Q's contains gains N, the original -- the
  line is still "N, shown through R".  This is the round-trip rule
  working as designed.

- Yanked under a WRITABLE PartnerCol -- S's subscribeeCol, say:
  the col's membership list is read from each member's OWN id, not
  its collected id.  So S comes to subscribe to R, the node whose
  title the user was looking at -- NOT to N.

That asymmetry is the decision I made unilaterally (it is cheap to
reverse).  The case for it: what the user SEES on that line is R's
title, and "subscribe to the thing I dragged here" is the least
spooky reading; honoring the marker instead would make the
relationship silently target a node the user may never have heard
of, steered by a metadata tail the heralds render as just "Oh".
The case against: a user who thinks of the line as "really N,
just displayed as R" would expect S to subscribe to N, and gets R
without any warning.

(Read-only cols are unaffected -- their membership is never read
from the buffer.  And within content position there is no choice
to make: collecting R there would rewrite P's contains, the
corruption this whole stage exists to prevent.)

If you prefer the other reading, the change is one line per
writable-col collector ('subscribeeCol_members' /
'partnerCol_members' switching to 'collected_id ()') plus tests;
say the word.  A third option: refuse the save when a marked node
sits under a writable col, forcing the user to strip the marker
and choose explicitly.

** collector membership initially broke subscribee-as-such semantics
Stage 6 changed generated collection members from implicit container

*** resolution
Changed the generated-collection membership tests so collection
members must have 'parentIs=collector'.  The older omitted/default
'parentIs=container' shape is no longer accepted as collection
membership.

The two affected meanings were:
- a TrueNode child of 'subscribeeCol' is still a subscribee
  member only when it is marked 'parentIs=collector',
- a TrueNode child of a definitive subscribee-as-such branch is
  still visible subscribee content if it is either an ordinary

collection membership is strict: 'subscribeeCol', 'overriddenCol' and
the read-only relation Cols count only children marked 'parentIs=collector'.  Ordinary
content completion and ordinary save extraction still use
'parentIs=container' / non-ignored when deciding whether a node is real
containment content.  Visible content inside an expanded
subscribee-as-such branch can still include a child marked 'parentIs=collector', because
moving a generated hidden child into the visible branch is the existing
gesture for inferred unhide.

'subscribeeCol' or the relation Cols unless they have
'parentIs=collector'.  Test fixtures that meant collection membership
were updated to say so explicitly; test fixtures that meant "extra
display child, not a collection member" now use 'parentIs=independent'.
