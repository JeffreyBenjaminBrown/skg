* implementing a UI for the full schema
** status
Currently, the 'contains' and 'textlinks_to' relations
are (probably?) completely implemented.
The 'subscribes' and 'hides' relations are somewhat implemented --
the user can view and modify them.
The 'overrides' relation is barely implemented --
it is represented in the dbs, but there is no way for a user
to read or write override relationships through the client.
** goals
*** a "hides" herald
There is currently metadata sent from the server to indicate
that a node is involved (in either role of the binary relationship)
in subscribes or overrides relationships.
These data correspond to the "S" and "O" heralds.
There should be similar metadata for hides relationships,
and a corresponding "H" herald.
*** read-only org-branches for a node's subscribers, overriders and hiders
Suppose we are interested in viewing the relationships involving node N.
The user can already view what N subscribes to and hides,
via the subscribeeCol scaffold (and the Hides*Col scaffolds
contained in the subscribeeCol's descendents).
We need similar -Col scaffolds, but read-only ones,
for what subscribes *to* N, what hides N, and what overrides N.
These are all in-links, which means the filesystem represents them
not in the .skg file for N, but in the .skg files for the nodes
connecting to N.

The content of these -Col branches will initially be rendered indefinitive.
(The user can request a definitive view for any of them,
just like they can for any other viewnode.)

Users can reorder the children of any of these read-only -Col scaffolds;
saving will respect that order, rather than restore the previous one.
If they insert new children, they will be relabeled parentIs=Independent.
If they try to delete content, it will respawn.
(The "completion" code has established tooling and idioms
for this kind of work.)

User edits to the title or body of one of these -Col scaffolds
will be discarded; the -Col will respawn as before.
In the cases of text edits to the col or deletions of its children,
users should be warned that the edit was discarded,
but it should not stop the save pipeline.
*** 'indef' symbols for read-only collections
Certain collections are always read-only.
These include the new collections of
subscribers, overriders and hiders,
and also (I believe) IDCol and HiddenInSubscribeeCol,
but (I believe) not HiddenOutsideOfSubscribeeCol.
(Please fact-check me on those beliefs.)
The server does not need modification, but the herald rules
should attach the green peace symbol used elsewhere
to the heralds for of these kinds of collection.
*** writeable Overridee branches
Similar to the subscribeeCol, the overridenCol will permit the user
to create relationships in which the overriddenCol parent
overrides each of the overriddenCol's children.
*** visiting overridden nodes
Most ways of rendering an overridden node
will result in a view of the overrider,
not the overridden. There is one exception:
If V is a viewnode representing an overridden node N,
and V is a child (with parentIs=container)
of an 'overriddenCol', then visiting N from V,
or requesting a definitive view from V,
will actually use N, rather than its overrider.
(The overrider is the parent of the overriddenCol,
so we can already see what's there; this mechanism lets a user
also see what's in the original overridden node.)

Where an overridee is displayed without being overridden,
nodes hidden by the overrider
are nonetheless displayed. (The 'hides herald'
described earlier will indicate that there is
hiding action relevant to such a node.)

Note that a viewnode V of node R might contain
both an OverriddenCol child and a SubscribeeCol child,
and the same node E might belong to both collections.
In this case, a definitive view of E requested
from the SubscribeeCol will hide thiings in E that R hides,
but a definitive view of E requested from the OverriddenCol
will not hide them. (This is in keeping with the full name
of the 'hide' relationship: 'hides_from_its_subscriptions'.)
*** more herald rules
Where an overrider has been drawn in place of an overridden,
a red "Og" ("OverridinG") herald should indicate the replacement.
This will require a new kind of "viewnodestats" data:
"overriding".

If an overridden node N is a child of an overriddenCol,
and the corresponding overrider R (N's grandparent)
also subscribes to N, then N should include a red "Se"
("SubscribeE").
This also requires a new viewnodestat: grandparentSubscribes.

If E is a subscribee, and is a child of a subscribeeCol,
and R is the corresponding subscriber (grandparent to E),
and R overrides E, then E should show a red "On" ("overridden")
herald.
