** the part of the save path this vision would replace
'buffer_to_validated_saveplan' in 'server/from_text.rs' runs, in order:
parse, enrich, validate, place, and then the part at issue:

- 'SaveAuthority::from_viewforest' does two whole-tree passes:
  - 'viewforest_with_saveroles' (in 'classify.rs') clones the entire
    viewforest into a 'Tree<ViewNode_in_Role>', computing each node's
    'SaveRole' by looking *up* at its parent, grandparent, and
    sometimes great-grandparent.
  - 'collect_savenode_candidates' walks that tree to find positions
    allowed to own instructions ('Ordinary', 'Subscribee',
    'Overridden').
- 'extract_save_intents' then runs five more passes, each iterating
  over the candidate list against the whole tree:
  'collect_node_edit_basics' (title/body/delete),
  'collect_contains_by_node_id',
  'collect_aliases_by_node_id',
  'collect_subscribees_by_node_id',
  'collect_overridden_by_node_id',
  plus 'subscribee_hiderel_intents_from_candidates'.
- 'assemble_node_edit_intents' zips those five maps into
  one 'NodeIntent' per candidate.
- 'reconcile_nodeIntents' groups intents by PID and errors on
  save+save or save+delete collisions for one PID.
- 'apply_hiderels_from_intents' converts the subscribee-as-such
  visibility signals into hides/unhides deltas on subscriber intents.
  This consults disk, and consults the subscriber's own post-save
  'contains' from the same intent map.
- 'build_disk_supplemented_define_nodes' fills 'MSV::Unspecified'
  fields from disk, canonicalizes IDs, detects source moves.
- 'filter_wouldbe_noop_definenodes' drops saves identical to the
  in-Rust graph.

Separately, 'instructiontriples_from_merge_candidates' (merge
extraction) and 'validate_no_title_or_body_edit_in_subscribeeAsSuch'
also consume the same 'SaveAuthority'.
** how little of the role classification is actually consumed
Evidence for your instinct that this machinery is overgrown:

- 'SaveRole::AliasDisplay', 'IdDisplay', 'HiddenInSubscribeeCol', and
  'HiddenOutsideOfSubscribeeCol' are computed (the latter two with
  nontrivial grandparent/great-grandparent walks that can error) and
  then never matched on anywhere. Only 'Ordinary', 'Subscribee', and
  'Overridden'-vs-rest are ever consumed.
- 'DefinenodeCandidateKind::Overridden' is collected and then filtered
  out by every consumer: naive intents take only 'Ordinary',
  hiderel intents take only 'Subscribee', merge extraction takes only
  'Ordinary'. It is dead weight.
- 'viewforest_with_saveroles' clones every 'ViewNode' in the buffer
  just to attach a role, once per save.

So the replacement is well motivated. The role data that is actually
load-bearing is small and fits naturally in a top-down traversal
context (see the sketch below).
* places where the vision contradicts the code
** SubscribeeCol does generate an instruction today
The vision says

  "If it recurses into a SubscribeeCol, the col itself doesn't
   generate instructions"

but in the current code the *membership* of a SubscribeeCol is exactly
what becomes the parent's 'subscribes_to' field
('collect_subscribees'). The col carries MSV semantics: absent col =
'Unspecified' (no opinion, disk value survives); present-but-empty col
= 'Specified(vec![])' (explicitly unsubscribe from everything).

So SubscribeeCol generates two distinct kinds of information:
- its member list -> the parent's 'subscribes_to' (a field value);
- each definitive subscribee-as-such child's visible content -> a
  hides/unhides signal for the subscriber (not a field value; it
  needs disk data and the subscriber's own post-save contains to
  resolve).

I assume the vision means only that the col generates no instructions
*about itself* (it has no gnode), which is true of AliasCol and
OverriddenCol too. But the spec should state explicitly that
SubscribeeCol emits the 'subscribes_to' field into its parent's map
entry, exactly as AliasCol emits 'aliases' and OverriddenCol emits
'overrides_view_of'. Do you agree?

** the full inventory of instruction-generating positions
You asked whether AliasCol and OverriddenCol are the only

inventory is:

| position                                  | emits                                            | into whose map entry  |
|-------------------------------------------+--------------------------------------------------+-----------------------|
| definitive Active vognode, ordinary       | title, body, contains, or Delete                 | its own               |

| AliasCol (under eligible writer)          | aliases (MSV)                                    | parent's              |
| SubscribeeCol (under eligible writer)     | subscribes_to (MSV)                              | parent's              |

Notably *not* on the list:
- IDCol / ID: display only. 'extra_ids' is never written from the
  buffer; it changes only via merge. (Matches your TODO note that IDs
  should not be writeable in the metadata view.)
- The six read-only RoleCols (Subscriber, Overrider, Hider, Hidden,
  HiddenInSubscribee, HiddenOutsideOfSubscribee): no instructions,
  and today no recursion into them either (see the matrix below).
- 'hides_from_its_subscriptions' is never emitted directly from any
  buffer position; it changes only via the subscribee-as-such signal
  path (and via merge).
** "reads the vnode and its vnode children, and that's all" is one level too shallow
A vognode's aliases live at grandchild depth (vognode -> AliasCol ->
Alias), as do subscribees and overriddens. So a per-node rule that
reads only the node and its children cannot collect those fields.

Two consistent ways to make it local; I believe you intend the
second, but the vision text mixes them:

- Each vognode reads two levels down (children for contains, and
  through any QualCol/PartnerCol children for the rest). The cols
  then emit nothing.
- Each vognode reads one level (itself + children -> title, body,
  contains, delete). Each *col*, when the traversal visits it, emits
  its field into its parent's map entry. This is the "merged into the
  map value keyed by their parent's ID" sentence in the vision.

I recommend the second: it keeps every emission strictly local (one
node and its immediate children), and the traversal context (below)
gives the col its parent's identity for free. The cost is that the
spec must define which parents are eligible (a col under an
indefinitive or subscribee-as-such or read-only-positioned vognode
must not emit; today those cols are simply never consulted because
field collection happens only at 'Ordinary' candidates).

** what is an "instruction"? (granularity)
The vision says the map's values are *sets* of instructions. Today's
'NodeIntent' is a monolith: one struct carrying every field. Sets of
monoliths don't merge well. I see three options:

- per-field instructions (my recommendation) ::
  An enum roughly like:
    SetTitleAndBody { title, body },
    SetContains (Vec<ID>),
    SetAliases (Vec<String>),
    SetSubscribesTo (Vec<ID>),
    SetOverrides (Vec<ID>),
    SubscribeeVisibility { subscribee : ID, visible : Vec<ID> },
    Delete,
    (optionally Merge { acquiree : ID } -- see the merge section)
  Each buffer position emits zero or more of these. Merging into the
  map is trivial; conflict rules are statable per variant (below).
  Lowering map entry -> 'NodeIntent' (with MSV::Unspecified for
  absent variants) is mechanical, so everything downstream survives.

- whole-node intents, merged ::
  Keep 'NodeSaveIntent' but make every field MSV-ish, and define a
  merge operation (Unspecified yields to Specified; two Specified
  that disagree = error). Workable but the merge function is the
  complexity, and 'title' would need a fake MSV wrapping.

- whole-node intents, no merging ::
  Closest to today: the vognode's visit must gather everything,
  which forces the two-levels-down reads rejected above.

QUESTION: which granularity do you want? Everything below assumes
per-field.

** merge/conflict semantics inside the map
If values are sets of per-field instructions, the rules I propose:

- Two instructions of different variants for one ID: both kept.
- Two identical instructions: collapse silently (a set).
- Two same-variant instructions with different payloads for one ID:
  buffer validation error. (Today this is mostly precluded upstream:
  'validate_definitive_view_requests' and friends ensure at most one
  definitive instance per ID, and only definitive instances emit. So
  hitting this would indicate a bug, but the check is cheap and I'd
  keep it as an error rather than an unreachable!().)
- 'Delete' plus any Set* for one ID: error, matching today's
  "Cannot have both Delete and Save for same ID".
- 'Delete' plus 'Delete': fine, collapse.
- 'SubscribeeVisibility' signals: multiple allowed for on
  subscriber (one per subscribee branch). The existing
  per-(subscriber, content) conflict check
  ('validate_no_overlapping_subscribee_hiderel_conflicts') stays
  downstream, since it needs disk contains lists.
- 'SubscribeeVisibility' aimed at a subscriber whose map entry holds
  'Delete': resolution is a no-op, as today
  ('apply_hiderel_delta' ignores Delete intents).

** BFS, preorder, and what must flow down
The vision says "a simple BFS preorder procedure". BFS and preorder
are different traversals; preorder normally means DFS. Since
emissions merge into a map keyed by ID, *visit order cannot matter at
all* -- the only real requirement is that ancestor context flows
downward. So:

- Recommendation: plain recursive DFS preorder, like the rest of the
  codebase (e.g. 'collect_savenode_candidates' itself,
  'update_buffer/complete.rs'). A queue-based BFS buys nothing here
  and the codebase has little precedent for it.

The context that must flow down (this is what *replaces*
'classify.rs', which derived the same facts by walking up):

- the nearest vognode ancestor's ID, and whether that ancestor is an
  eligible writer (definitive + Active + ordinary position). Cols
  need this to emit, or to stay silent.
- whether the current position is inside a SubscribeeCol whose parent
  is an eligible writer, and if so the subscriber's ID. This is the
  subscribee-as-such case, including the at-most-one-writer-per-ID
  guard currently in 'subscriber_instance_is_definitive' (the
  comment in 'subscribee_hiderel_intents.rs' explains why losing it
  would let one subscriber receive contradictory hide edits).
- whether the current position is inside a read-only col (emit
  nothing, and today: don't even recurse).

That's a three-or-four-variant context enum passed as a function
argument. No tree cloning, no upward walks, no roles computed that
nobody reads.

* dangers
** behavior drift in the membership predicates
The highest-risk part of this rewrite is not the traversal; it is the
small predicates deciding membership, which encode a lot of
hard-won policy:

- 'member_counts_for_relation_collection'
  (parentIs == Affected, not a would-be diff phantom, not toDelete);
- the contains filter in 'collect_contents_to_save_from_children'
  (same three conditions, plus role Ordinary, plus Inactive vognodes
  included unless 'inactiveNode_is_phantom');
- 'dedup_vector' applied to aliases/subscribees/overriddens but NOT
  to contains (contains may legitimately repeat? -- QUESTION: is
  that deliberate? If contains is deduped elsewhere I haven't found
  it, and 'reconcile' in update_buffer suggests repeats are legal);
- MSV absent-vs-empty semantics per col.

Recommendation: the new code should *call these same predicates*, not
re-derive them. Move them somewhere both old and new code can reach
during the transition, and port the tests that pin them down.

** mixed map values: field values vs unresolved signals
The map will contain both real field assignments and
'SubscribeeVisibility' signals that *cannot* be resolved during
collection even in principle: resolution reads the subscribee's disk
contains, the subscriber's disk hides, and the subscriber's own

'SaveExtraction' does today); slightly less elegant, slightly more
typed.
