* Problems with the local-view-update vision

I like the goal: replacing global, procedural completion with smaller
local update rules would make this part of Skg easier to reason about.
But the present spec leaves several important semantics undefined, and
some of the proposed local rules appear to conflict with behavior that
the current code relies on.

* New questions arising from your responses
** Should collateral locking use the saved view's whole node set, or only changed nodes?
The current early lock message over-approximates by locking every open
view sharing any PID with the saved view. The later collateral rerender
pass is narrower: it rerenders views containing PIDs in the produced
=DefineNode=s. Your response says "Everything that shares any node with
the saved buffer." That matches the lock behavior, but is broader than
current collateral rerender selection.

If the new design intentionally makes lock and rerender use the same
broad set, that is simpler and safer, but may rerender more buffers.
If not, the plan should distinguish "locked views" from "actually
rerendered collateral views".

**** Resolution (Claude)
This already falls out of how children are built, so the rule to state
is just "replace, don't preserve". =build_child_creation_data=
(complete_preorder/truenode.rs:599-679) decides per goal-list ID: if the

placeholder/real choice on the next render automatically.

stable anchor across source-set changes; when the active set changes, the
updater deletes the stale placeholder (a view-leaf) and the rebuilt goal
list inserts the opposite representative in its place -- inactive→Normal
when the source becomes active, Normal→inactive when it goes inactive.

* The update driver is not specified
** "Traversal order will matter less" is not enough
The document says each ViewNode should update itself, and maybe some
neighbors, in isolation. But many current update steps are only correct
because they run in a specific order:

- content and SubscribeeCol expansion before postorder scaffold repair
- view requests after the first content expansion
- HiddenInSubscribeeCol and HiddenOutsideOfSubscribeeCol after their
  surrounding subscribee/subscriber structure exists
- alias/id repair after the parent node's current disk state is known
- empty scaffold deletion after its children have been reconciled

If the new model is a single pass, a fixed-point loop, or a typed queue
of local update jobs, that needs to be explicit. Otherwise "local" can
silently become either order-dependent again or nonterminating.
*** DONE can you define each of the terms used in this question?
(I suspect ordinary BFS will work perfectly.)
**** Definitions
- A single pass means each node is visited once in some order, e.g. BFS,
  and no node is revisited because of changes made elsewhere.
- A fixed-point loop means repeatedly running update passes until a pass
  makes no changes.
- A typed queue means update actions enqueue more specific actions, e.g.
  "update this Vognode's content", "reconcile this HiddenInCol", or
  "execute this definitive request", and the scheduler drains the queue.
- A local update job means one such unit of work, usually tied to one
  ViewNode plus whatever nearby context that kind is allowed to inspect.

Ordinary BFS might work if each update either only affects already-seen
neighbors in harmless ways or explicitly enqueues later work for newly
created children/scaffolds. The plan should state which of those two
rules it uses.

**** Analysis (Claude)
Bottom line: ordinary top-down BFS works, with one genuine exception and
a short list of explicit scheduler rules. Four of your five cases are
fine; the fifth is fine for the *generated* scaffolds but not for
=DeadScaffold= chains. Nothing in the current completers violates the
"create descendants, not ancestors" rule -- including containerward and
sourceward, which (despite the name) attach as *descendants*, not
ancestors.

The reason BFS works so well is that the current code is *already* mostly
top-down: =expand_true_content_until_stable= (complete.rs:101-115) is a
parent-first recursion that recurses into freshly created children. The
"postorder" passes that follow are postorder for collection convenience,
not because each scaffold needs its descendants populated first -- every
generated collection computes its membership *from the graph* during its
own visit, never from its view-children. That is the key fact that makes
order not matter.

***** The five cases:

****** (a) content / SubscribeeCol expansion before postorder scaffold repair.
   FINE. A Normal node expands its content and creates its SubscribeeCol
   during its own visit (=expand_true_content_at_truenode=); BFS then
   descends into the SubscribeeCol, which reconciles from the subscriber's
   on-disk =subscribes_to= (subscribee_col.rs:69-106) -- not from sibling
   view-nodes. HiddenOutsideOfSubscribeeCol likewise reads the
   subscriber's =hides=/=subscribes_to= from disk
   (hiddenoutsideof_subscribeecol.rs:78-101), never the sibling subscribee
   view-nodes. So there is no "siblings must already exist" hazard;
   parent-before-child is all BFS needs.

****** (b) view requests after the first content expansion.
   FINE, *given a buffer-global definitive map*. A node can run its own
   view requests during its visit, right after expanding content. The
   one-definitive-per-ID invariant is what needs global state, and the
   code already has it: =DefinitiveMap= (=visited=, to_org/util.rs:43-44).
   Whoever BFS reaches first registers as definitive
   (=make_indef_if_repeat_then_extend_defmap=, util.rs:144-161); a later
   duplicate is forced indefinitive. An explicit definitive *request*
   wins over an incidental occurrence by indefinitizing it via the map
   (=execute_definitive_view_request=, definitive.rs:92-97). Two explicit
   requests for one ID can't collide -- validation already rejects that
   (=MultipleDefinitiveRequestsForSameId=). So order is irrelevant *as
   long as the map is threaded across the whole BFS*, not reset per
   subtree.

******** Assessment (Claude): your map works; also consider not making child DVRs at all
Two viable models; both stay bounded by the node limit:

1. Your Tentative/Final map (= SavedAsDefinitive / UpdatedToDefinitive).
   Sound, and a real improvement: it makes DVR execution *idempotent* and
   order-robust. Each ID is "claimed" once -- by a saved-definitive node or
   by the first DVR to reach it -- and marked Final; later DVRs for that ID
   are dropped, leaving their nodes indefinitive. That terminates (each ID
   is indefinitized at most once via a DVR) and keeps exactly one
   definitive per ID. One thing to pin down: make a *user-submitted* DVR
   beat a *completion-created* one for the same ID. Simplest guarantee:
   seed the map Final with the saved-definitive nodes and execute user DVRs
   before completion creates any; validation already permits at most one
   user DVR per ID, so the winner is unambiguous. (Without that, raw BFS
   order decides, and a user DVR on a deep node could lose to a
   completion-created DVR on a shallower occurrence of the same ID.)

2. Don't create per-child DVRs at all. Today's definitive expansion
   (=extendDefinitiveSubtreeFromLeaf=, definitive.rs:225-272) does *not*
   put DVRs on children -- it builds the content subtree directly and marks
   repeats indefinitive via the map (=make_indef_if_repeat=). If the new
   completion keeps that, the only DVRs that exist are user-submitted ones,
   which validation already bounds to one-per-ID, so "completion creates
   conflicting DVRs" never arises and you need no finality flag at all.
   Recursion depth is still bounded by the node limit the same way.

Path 2 is strictly simpler (no finality state, no priority rule). Path 1
is worth it only if you specifically want DVRs to be the *uniform*
expansion mechanism (vision.org leans that way). Your call.

Map: =ID -> Definitiveness=, where =Definitiveness = Tentative(NodeId) | Final(NodeId)=.
- Completing an *ordinary* definitive node (a saved-definitive node, or
  gap-fill content drawn directly) registers *Tentative*.
- Executing a *DVR* registers *Final*.

When a node would be drawn definitive for ID X:
- X already Final    -> newcomer stays indefinitive (this is the "discard
  the DVR" rule; it also covers an ordinary node deferring to a Final one).
- X Tentative, newcomer via DVR  -> indefinitize the old occurrence;
  newcomer becomes Final (clobber).
- X Tentative, newcomer ordinary -> newcomer stays indefinitive (first
  Tentative wins).
- X absent           -> register Tentative or Final per how it was drawn.

Cascade: *only a Final (DVR-made) node* hands DVRs to its Normal children.
That is what makes one user request draw its whole subtree -- each
descendant's DVR clobbers whatever was Tentative there, all the way down,
bounded by the node limit. Ordinary (Tentative) definitive nodes do *not*
cascade DVRs; they draw/reconcile their content directly.

Why binary Tentative|Final suffices (your priority worry, resolved by
construction): the server never invents a DVR on its own -- every DVR is
either user-typed or a cascade descendant of one. So there are no
independent "server DVRs" to out-prioritize. Final beats Tentative (an
explicit request beats incidental definitiveness); Final-vs-Final is
first-in-BFS-wins, and since both are user-originated it doesn't matter
which. No finer priority ranking is needed.

Naming (your confusion point): keep =definitiveView= as the user-facing
request keyword; internally distinguish by *finality* (the map's
Tentative/Final tag = the result), not by who issued the request. The
cascade DVRs the server adds under a Final node are an internal scheduling
detail and need no user-visible name. So "ordinary vs DVR node" stops being
a type distinction and becomes two map states plus the cascade flag --
exactly your "almost identical" observation.

One consequence to state in the plan (correct, but worth flagging so it
doesn't surprise anyone): because a saved-definitive node is Tentative, a
DVR elsewhere -- or its cascade -- can demote it to indefinitive.
One-definitive-per-ID has to give somewhere, and giving the explicit
request precedence is right: the alternative (DVR defers to a saved
definitive) would leave the requested subtree unexpanded, defeating the
request.

   subscribee is the col's parent or grandparent (visited first). Both
   cols read the subscriber's NodeComplete from disk during their own

   parents during the parents' visits* so BFS descends into them. Today
   the HiddenInSubscribeeCol is created in a separate Phase-5 sweep

****** (d) alias/id repair after the parent's disk state is known.

   -- =should_sync_from_disk_even_though_definitive=), then descends and
