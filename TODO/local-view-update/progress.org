* 1. What death-leafward is, in one paragraph
In the single-visit BFS, each col (a non-vognode viewnode: QualCol, PartnerCol)
reconciles at its own visit and reads its ancestor vognode(s). If a required
ancestor died this save its =.skg= is gone, the read returns =Err=, and that
aborts the whole (often *collateral*) rerender -- spuriously rejecting the
user's save. The fix: every col, at its BFS visit, first checks whether it is a
*generalized orphan* (ANY ancestor in its required-ancestry chain -- the full
chain, not just the parent -- is the wrong *viewnode kind*; death is a VIEW
property, no graph read). If orphaned it deadens itself and disposes its
children instead of reconciling, so the failing read never happens.

* 3. The deviation (and why I didn't wake you)
plan.org §5.1.a says a deadened col converts its *nested col* children
(QualCol/PartnerCol) to =DeadScaffold=. That is wrong, and the integration test
caught it: converting a nested col to DeadScaffold *freezes* it before its own
BFS visit, so it never disposes ITS OWN leaf members. Concretely: when a
SubscribeeCol died, its HiddenOutsideOfSubscribeeCol got frozen to DeadScaffold,
so its hidden leaf (=also-hidden=) survived stranded under a dead chain, which
in turn kept the SubscribeeCol (and the whole deleted node) alive.

The fix: a deadened col *leaves nested cols untouched*. A nested col is itself a
generalized orphan under the now-dead parent, so it deadens *itself* -- and
disposes its own leaves -- at its own visit. This is exactly what plan.org's §5
note already intends ("a nested col ... deadens itself at its own visit too");
§5.1.a's literal instruction was the bug. Only =Qual= leaves (which do not
self-dispatch, so nothing would ever deaden them) are still converted to
DeadScaffold in the disposal.

With the fix, a deleted node's cols empty out completely and prune, so it
becomes childless and §6.6 removes it -- "11 gone entirely" (plan.org §9),
matching the *existing* integration-test expectation exactly. I judged this a
clear correctness fix aligned with the plan's intent, fully validated by the
green integration suite, so I logged it and continued rather than stopping.

***** TODO Explanation from first principles (Claude)
Short version: this whole item is about deleting *one if-statement* and then
deciding what to do with the now-orphaned boolean that fed it. The reason it's
"fiddly" is entirely the second part -- the deletion itself is trivial.

****** The vocabulary you need (no plan_v2 required)
- *Definitive node*: a content node the view is sure about -- it was saved as
  definitive, or completed as an ordinary child, or hit by a draw rule. Contrast
  with an *indefinitive* node, which is a placeholder we haven't pinned down.
- *Sync from disk*: copy a node's =title=, =source=, and =body= out of the
  freshly-saved snapshot (the post-extraction =NodeComplete=) into the ViewNode
  in the buffer. I.e. "make the on-screen node match what's now on disk."
- *Saved view vs collateral view*: when you save buffer A, that's the *saved*
  view. But the same nodes may also appear in *other* open buffers (B, C, ...).
  Re-rendering those other buffers off the same save is a *collateral* render.
  =is_saved_view= is the one bool that says "am I redrawing the buffer that was
  just saved (true), or a bystander buffer (false)?"

****** What the code does today
In =server/update_buffer/complete_preorder/truenode.rs= there's a tiny helper:

#+begin_src rust
  fn should_sync_from_disk_even_though_definitive(self) -> bool {
    ! self . saved_view }   // i.e. "sync only for collateral views"
#+end_src

and at the one call site (truenode.rs:143):

#+begin_src rust
  if mode . should_sync_from_disk_even_though_definitive() {
    sync_truenode_from_disk(tree, node, &nodecomplete)?; }
#+end_src

So today the rule is: *a definitive node in a collateral buffer re-syncs its
title/source/body from disk; a definitive node in the saved buffer does NOT.*

The reasoning behind the carve-out: the node in the saved buffer is the node you
just typed. The view already holds exactly that text -- it *defined* it. So
re-reading it off disk felt redundant, and the carve-out skipped it as a
micro-optimization / "don't touch what's already right."

****** Why the plan wants to delete it
After extraction, the snapshot on disk *already equals the saved buffer's text*
(that's what extraction just wrote). So syncing the saved node from disk copies
the same bytes back onto themselves -- a *no-op*, not a risk. Removing the
carve-out therefore loses nothing and buys uniformity: *every* definitive node,
saved or collateral, follows the identical one-line rule "sync from disk."
One code path instead of two, no special case to reason about. That's the entire
upside -- it is a tidiness win, not a bug fix.

So the change itself is: delete the helper, delete the =if=, and always call
=sync_truenode_from_disk=. Five minutes.

****** Why my predecessor called it "fiddly" and skipped it
Here's the snag, and it's the only real content of the whole item. Trace where
=is_saved_view= comes from and what reads it:

  update_buffer.rs  -->  CompletionContext field (complete.rs:39)
                    -->  passed to expand_true_content_at_truenode (complete.rs:218)
                    -->  CompletionMode::new(is_saved_view)
                    -->  should_sync_from_disk_even_though_definitive  <-- ONLY reader

That carve-out is the *only thing in the entire server that reads
is_saved_view.* Delete the carve-out and the bool has no consumer left. Now
you're forced into a choice, and every option is slightly unsatisfying:

1. *Delete is_saved_view too.* Cleanest in isolation -- but plan_v2 §12 explicitly
   lists =is_saved_view= as a field the refined CompletionContext should *keep*
   (the plan anticipates future per-kind rules wanting to know saved-vs-collateral).
   So deleting it now contradicts a written design decision and may have to be
   re-added later.
2. *Keep is_saved_view, just unused.* Then Rust emits a =dead_code= warning, or
   you silence it with =#[allow(dead_code)]= -- i.e. you carry a field that's
   wired through three layers (update_buffer -> context -> truenode) for nobody.
   Ugly, and the plumbing is pure dead weight until some future phase uses it.
3. *Keep it and find it a real reader.* That means doing some *other* phase-12
   work now to justify the field -- scope creep onto an item that was supposed to
   be a one-line cleanup.

None of those is hard; they're just all mildly wrong, and none is clearly right
*in isolation* from the larger phase-12 context-refinement work. That's the
"fiddly": a trivial deletion that drags a design question (does the context keep
=is_saved_view= or not?) which doesn't want to be answered by itself.

****** My recommendation
This is genuinely low-value, and the cleanest resolution is *not to do it alone*.
The field-keep-or-drop question belongs to phase 8 (unify de-novo) / the §12
context refinement, where you're touching CompletionContext anyway and will
either give =is_saved_view= a second reader or decide it has none. Pulling the
carve-out then -- in the same change that resolves the field's fate -- avoids the
dead_code-vs-churn dilemma entirely, because you're already editing those
signatures. Doing it now just creates a wart you have to revisit.

So: I'd leave it deferred exactly as my predecessor did, and *bundle it into
phase 8* rather than treat it as a standalone cleanup. If you'd rather see it

* 9. Discussion
** How do phantoms use parentIs and indef_or_def?
Two real decisions read these off a =Vognode::DiffPhantom=. (When I first called
the §11 reduction "not worth it" I was right that the plan's premise was wrong,
but I should have named the exact decisions -- here they are.)

*** parentIs -- one decision, and it genuinely varies
=remove_branches_that_git_marked_removed= (server/update_buffer.rs ~462), run by
=strip_stale_diff_state= at the very start of every =rerender_view=, walks the
incoming buffer and DETACHES a phantom branch iff:
#+begin_src rust
  ViewNodeKind::Vognode (Vognode::DiffPhantom (t)) =>
    ! is_viewforest_root_child
    && t . parentIs == ParentIs::Affected,
#+end_src
Its own docstring states the rule: "Does not strip: top-level branches (forest
roots); non-content nodes (parentIs != Affected)." The point is to throw away
STALE diff phantoms so the diff overlay can regenerate fresh ones -- but only the
CONTENT ones (parentIs=Affected). A phantom the user dragged out of content
position (parentIs=Independent), or a phantom that is itself a view root
(parentIs=Absent), is preserved. So the decision genuinely varies by parentIs:
=Affected= phantom -> stripped + regenerated; =Independent=/=Absent= phantom ->

** Part 1 -- the vocabulary, from the ground up

*** node, ViewNode, Vognode
The view you see in the editor is a TREE of =ViewNode=. A =ViewNode= is one line
(plus its body) in that tree. Some ViewNodes correspond to an actual graph node
(these are =Vognode=); others are structural decorations (see "scaffold/col").

*** Normal node
A =Vognode::Normal= is the ordinary case: a graph node shown as itself, editable,
present in the worktree. Most lines you see are Normal nodes.

*** phantom (DiffPhantom)
A =Vognode::DiffPhantom= is a DIFF-ONLY placeholder. It marks a node that git
says was there but isn't now: either its =.skg= file is gone ("removed"), or the
file still exists but the node is no longer a member of this parent's contains
list ("removedHere"). A phantom exists ONLY in the git-diff view; it is never
saved. (After this week's §11 work a phantom carries a slim payload and is always
read-only / indefinitive.)

*** "flip" / normal_to_phantom
The act of turning a Normal node into a phantom is a "flip" -- the function is
=normal_to_phantom=. Whether a node should flip is decided from its diff axes
(below): if some stage shows the node removed, it flips.

*** existence axis / membership axis (the "diff axes")
Two per-stage facts git gives us about a node:
- EXISTENCE: did the node's =.skg= FILE change between HEAD↔index ("staged") or
  index↔worktree ("unstaged")?
- MEMBERSHIP: did the node's APPEARANCE in this parent's contains list change, in
  each of those two stages?
These are stored on the node as small +/- markers and are what the client renders
as =(staged removedM)=, =(unstaged removedX removedM)=, etc.

** Part 2 -- how diff is computed TODAY, and the §9 history

Phase 5 (plan_v2 §9) made a deliberate choice: compute the WORKTREE view first
(driver with =source_diffs = None=), then add ALL content/scaffold diff in the
single overlay post-pass. The goal was "one diff implementation instead of two."
To get there, phase 5 DELETED the old inline diff code from the driver -- the
in-BFS scaffold emission and the in-BFS phantom flip are gone (the driver's own
comments now say so). The sharing-col diff is the one exception that still runs
inline.

So today:
- driver (=complete_viewforest=): builds the pure worktree view. No phantoms.
- overlay (=apply_diff_to_viewforest=): runs after, adds all content diff.

*** Why "after" is load-bearing (the hazard)
The overlay flips nodes to phantoms and inserts phantom children. If that happened
DURING the driver's BFS instead of after it, then: the driver visits node N at
level k, flips N to a phantom and gives it phantom children; later (level k+1) the
driver reaches N's cols and tries to reconcile them -- but now their owner N is a
phantom, not a Normal node. The col's ancestor-read and the generalized-orphan
check would be looking at a phantom owner. §9 avoided this entire class of
interaction by running diff only after completion is fully done. That is the
"phantom-during-completion hazard."
