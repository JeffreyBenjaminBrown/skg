#+TITLE: Propagate death leafward: cols check their ancestry before acting
#+STARTUP: content

* 1. The concrete problem this solves
In the new single-visit BFS (commit a88adebb), each col (a non-vognode
viewnode: QualCol, PartnerCol, etc.) is reconciled at its *own* visit. A col's
reconcile reads its ancestor vognode(s) -- e.g. =reconcile_alias_col_children=
reads the parent vognode's =NodeComplete= to list its aliases. If that ancestor
was deleted by this save, its =.skg= file is gone, the read fails, and the col
reconcile returns =Err=, which aborts the whole (often *collateral*) rerender
(=update_buffer.rs= pushes it into the save's =errors=, so the client shows
"inconsistencies" and the save is rejected).

This bit us in =save_collateral_delete_then_edit_under_it=: node 11 is deleted
in one buffer; in a collateral buffer 11 had an =aliasCol=; the rerender visited
11's aliasCol, read deleted-11's NodeComplete, and aborted.

The pre-BFS code dodged this because col reconciliation was a *later, separate
pass*: by then =scaffolds_with_deadOrDeleted_parents_become_dead= (a bracketing
sweep) had already turned scaffolds-under-a-Deleted-parent into =DeadScaffold=,
so they were never reconciled as live cols. The BFS interleaves creation and
reconciliation, so that bracketing sweep no longer runs "in time".

* 2. Current stopgap (to be replaced by this)
=detach_stale_deleted_content= (in =complete_preorder/truenode.rs=): when a
Normal node reconciles its content, any content child whose pid is in
=deleted_by_this_save_pids= and is no longer in the goal is *detached* before
reconcile, so it never grows/keeps cols to trip on.

Its flaw (the chaos-monkey case you raised): it deletes the *whole* deleted
subtree, including any user content the user deliberately moved under a node
that later got deleted elsewhere. The mechanism below should instead *preserve*
such user descendants (demote) while killing only the now-meaningless
col-managed scaffolding.

* 3. The proposed treatment (your idea, restated)
As part of the BFS update -- *before* the diff overlay and *before* the prune
sweep -- when the BFS visits a col, it first checks whether it still has an
*appropriate ancestry chain*. If not:
- convert the col to a =DeadScaffold= (do not reconcile it); and
- delete the *non-definitive leaves it would have been responsible for* with a
  valid ancestry chain.
A definitive (user) descendant is *not* deleted -- it survives (presumably
demoted to Independent), so the user never loses authored content.
Generalize: this is the same shape as =scaffolds_with_deadOrDeleted_parents_become_dead=,
but (a) keyed on the *full* required ancestry, not just the immediate parent,
and (b) done *in the BFS visit*, not as a bracketing sweep.

* 4. What I'm missing / need decided
** 4.1 What is "appropriate ancestry" per col kind?
Each col kind needs a specific live-ancestor chain. My reading:
| col                          | required live vognode ancestry                         |
|------------------------------+--------------------------------------------------------|
| QualCol(Alias), QualCol(ID)  | parent Normal vognode                                  |
| PartnerCol(relation roles)   | parent Normal vognode                                  |
| PartnerCol(Subscribee)       | parent = subscriber (Normal)                           |
| HiddenOutsideOfSubscribee    | parent SubscribeeCol, grandparent subscriber (Normal)  |
| HiddenInSubscribee           | parent = subscribee-as-such (Normal), gp = SubscribeeCol, ggp = subscriber (Normal) |
Q: is "appropriate ancestry" simply "every vognode ancestor the reconcile would
*read* is alive"? If so I can derive it from what each reconcile reads, rather
than hard-coding a table. Do you want it derived or tabulated?

** 4.2 What counts as "dead" for an ancestor vognode?
Candidates, possibly all: (a) kind is =Vognode::Deleted= or =DeadScaffold=;
(b) pid in =deleted_by_this_save_pids=; (c) its =NodeComplete= won't load
(disk/graph miss). (a) is the post-visit marker; (b) is the save's intent
*before* the node's own visit; (c) is the ground truth the reconcile actually
trips on. I lean toward: dead iff (a) OR (c) -- i.e. "the data the col needs
isn't there." Is (b) needed too, or does (a) cover it once the node's visit
runs (which, in BFS order, is *before* the col's visit)? I think (a) suffices
because a Normal node converts to Deleted at its own visit, strictly before any
descendant col is visited. Confirm?

** 4.3 "Non-definitive leaves it would have been responsible for"
A col with broken ancestry *cannot recompute its goal* (it can't read the gone
ancestor). So "would have been responsible for" can only mean its *current*
children. Proposal: delete the col's current *indefinitive* children (its
managed members are created indefinitive), keep *definitive* ones (user
content). Is that the intent, or do you mean something it computes from a
still-readable source?
Sharp edge: an indefinitive node the user *deliberately* placed under a col is
indistinguishable from a col-managed indefinitive member. So "delete
indefinitive leaves" can delete a user-placed indefinitive reference. I think
that's acceptable (an indefinitive leaf has no subtree to lose, and you already
ruled stale leaf -> delete), but flagging it.

** 4.4 Composition with the existing machinery
- Does this *replace* =scaffolds_with_deadOrDeleted_parents_become_dead=, or run
  alongside it? (I'd fold the immediate-parent case into this and drop the
  separate sweep, but it currently also runs as a *pre*-pass to handle scaffolds
  that arrived dead in the incoming view -- keep that pre-pass?)
- =remove_empty_deleted_scaffolds= (prune sweep) still removes the resulting
  childless DeadScaffolds afterward -- unchanged.
- **Dependency on §6.6** (Deleted delete-if-childless, plan phase 7, NOT yet
  done): for the collateral-delete test's "11 gone entirely" to hold under
  *demote* (rather than the stopgap's detach), 11 must end up a *childless*
  Deleted node (its cols died and were pruned) and then be removed. That removal
  is §6.6. So this treatment likely can't fully replace the stopgap until §6.6
  lands too. Do you want them done together?

** 4.5 BFS interaction after a col is killed mid-visit
When a col becomes DeadScaffold at its visit, the BFS will still dequeue and
visit its children. Their visits must no-op (a child under a DeadScaffold).
Today =scaffolds_with_deadOrDeleted_parents_become_dead= would catch them, but
in-BFS we'd want the dispatch to treat "parent is DeadScaffold" as no-op, or to
convert-on-visit. Need to make the child-visit robust to a just-killed parent.

* 5. Pushback (dumb / awkward / dangerous)
** Dangerous: aborting a *collateral* save on one bad col is the real bug.
Even with this treatment, I'd argue a col reconcile that hits a missing
NodeComplete should *degrade* (skip + mark dead), never return =Err= that aborts
an unrelated buffer's save. The ancestry pre-check is the clean version of that,
but I'd also make the reconcile read itself fail-soft as a backstop. A single
malformed/raced node should never be able to reject the user's save.
** Awkward: "before diff, before cleanup" puts a third structural sweep-like
concern *inside* the per-node BFS. The plan (§3) wants per-node rules local and
the only non-local cleanup to be the §3.4 prune sweep. An in-visit ancestry
check is local (a node reading its own ancestors is allowed, §3), so it fits --
but "delete the leaves it managed" is a small downward mutation, which is fine
(descendants-only). Just noting it nudges the col rules from "reconcile" toward
"reconcile or self-euthanize".
** Possibly-dumb simplification: do we even need a *general* ancestry check, or
just "is my nearest required vognode ancestor alive"? The only multi-level cases
are HiddenIn/HiddenOutside (subscribee chains). If a subscribee is Deleted, its
HiddenInSubscribeeCol's *immediate parent* is that Deleted subscribee, so the
immediate-parent check already catches it. When is a *non-immediate* ancestor
dead while the immediate one lives? (e.g. subscriber deleted but subscribee-as-
such not.) If that's unreachable, the immediate-parent rule (today's, moved
in-BFS) suffices and we avoid the per-col ancestry table entirely. I suspect the
immediate-parent check, applied *in-BFS to every scaffold at its visit*, plus
fail-soft reconciles, plus §6.6, covers everything -- which would make this much
smaller than a new subsystem. Worth confirming before building the general form.

* 6. Smallest version I'd actually build first (for your ok)
1. In the BFS col dispatch, before reconciling any col, if its parent is
   =Vognode::Deleted= or =DeadScaffold=, convert it to =DeadScaffold= and skip
   reconcile (today's immediate-parent rule, now in-BFS and timely).
2. Make each col reconcile fail-soft on a missing parent NodeComplete (skip +
   mark dead) as a backstop, so no col can ever abort a save.
3. Land §6.6 (Deleted delete-if-childless) so a node whose cols all died is
   pruned -> "gone entirely".
4. Then drop =detach_stale_deleted_content=; deleted branches now *demote*
   (preserving user descendants) and the dead cols self-clean.
If §4.5/§4.1 show non-immediate-ancestor death is real, extend (1) to the full
chain. Tell me if you want the general form from the start instead.
