#+TITLE: Propagate death leafward: cols check their ancestry before acting
#+STARTUP: content

* 1. The concrete problem this solves
In the new single-visit BFS (commit a88adebb), each col (a non-vognode
viewnode: QualCol, PartnerCol, etc.) is reconciled at its *own* visit. A col's
reconcile reads its ancestor vognode(s) -- e.g. =reconcile_alias_col_children=
reads the parent vognode's =NodeComplete= to list its aliases. If that ancestor
was deleted by this save, its =.skg= file is gone, the read fails, and the col
reconcile returns =Err=, which aborts the whole (often *collateral*) rerender
(=update_buffer.rs= pushes it into the save's =errors=, so the client shows
"inconsistencies" and the save is rejected).

This bit us in =save_collateral_delete_then_edit_under_it=: node 11 is deleted
in one buffer; in a collateral buffer 11 had an =aliasCol=; the rerender visited
11's aliasCol, read deleted-11's NodeComplete, and aborted.

The pre-BFS code dodged this because col reconciliation was a *later, separate
pass*: by then =scaffolds_with_deadOrDeleted_parents_become_dead= (a bracketing
sweep) had already turned scaffolds-under-a-Deleted-parent into =DeadScaffold=,
so they were never reconciled as live cols. The BFS interleaves creation and
reconciliation, so that bracketing sweep no longer runs "in time".

* 2. Current stopgap (to be replaced by this)
=detach_stale_deleted_content= (in =complete_preorder/truenode.rs=): when a
Normal node reconciles its content, any content child whose pid is in
=deleted_by_this_save_pids= and is no longer in the goal is *detached* before
reconcile, so it never grows/keeps cols to trip on.

Its flaw (the chaos-monkey case you raised): it deletes the *whole* deleted
subtree, including any user content the user deliberately moved under a node
that later got deleted elsewhere. The mechanism below should instead *preserve*
such user descendants (demote) while killing only the now-meaningless
col-managed scaffolding.

* 3. The proposed treatment (your idea, restated)
As part of the BFS update -- *before* the diff overlay and *before* the prune
sweep -- when the BFS visits a col, it first checks whether it still has an
*appropriate ancestry chain*. If not:
- convert the col to a =DeadScaffold= (do not reconcile it); and
- delete the *non-definitive leaves it would have been responsible for* with a
  valid ancestry chain.
A definitive (user) descendant is *not* deleted -- it survives (presumably
demoted to Independent), so the user never loses authored content.
Generalize: this is the same shape as =scaffolds_with_deadOrDeleted_parents_become_dead=,
but (a) keyed on the *full* required ancestry, not just the immediate parent,
and (b) done *in the BFS visit*, not as a bracketing sweep.

**** RESPONSE (Claude): you're right, no special no-op is needed.
First, what I meant, concretely. The BFS loop is: pop a node, run its update
(dispatch by kind), then push its *current* children onto the queue. So when we
visit a col, decide it's orphaned, and turn it into a DeadScaffold, we still
push its children, and they get visited later. My worry was only about its
children that are *vognodes* (e.g. a relation col's Normal members): when the

col-parent just died. (Qual/scaffold children are harmless: nothing dispatches
on them.) "No-op them" meant: skip =visit_normal_node= for a vognode whose
view-parent is a DeadScaffold.

But your §4.3 rule already disposes of those children *at the col's death*: an
Affected member leaf is deleted (so never visited), and an Affected member
branch is demoted to Independent. A demoted-Independent vognode then gets
visited normally -- which is *fine*: it's preserved user content drawn as its
own independent subtree. So nothing runs "as if live under a dead parent" in a
harmful way, and no no-op is needed. Skg does not currently special-case this;
the old code only avoided it by doing col-reconcile in a later pass.

One residual I want your call on: a demoted-Independent vognode survivor stays a
*child of* the DeadScaffold (re-parenting it upward to the col's parent would
violate the descendants-only rule, plan §3). So the DeadScaffold is no longer
childless and the prune sweep won't remove it -- you'd see a dead col with an
independent subtree hanging under it. Acceptable? Or should a dead col with a
surviving demoted child be left *as the col it was* (not deadened) so the
subtree reads sensibly? I lean: keep it simple -- deaden the col, let the
survivor hang under it; it's a rare chaos-monkey shape and the alternative
(re-parent upward) breaks the locality rule.

* 6. Agreed design (incorporating Jeff's answers)
** Death is a VIEW property (§4.2). A col is *orphaned* iff, walking its
   required ancestry nearest->farthest, some required ancestor is the wrong
   viewnode kind: a position that must be a vognode is anything but
   =Vognode::Normal=; a position that must be a specific scaffold is anything
   but that scaffold. First mismatch wins (short-circuit). No graph/NodeComplete
   read. Full chain, not just the immediate parent (§5: cheap because we only
   look past the parent when the parent is fine).
*** TODO yes but let's call this 'generalized-orphaned'
Traditionally in CS, orphaning only refers to immediate parents.
In whatever function name the term 'generalized orphan' appears,
we should include a comment defining it.
** Required ancestry is the §4.1 table, read *through* a shared helper so the
   reconciles can only read ancestors the table lists (dedups the structural
   contract; per-col field reads stay per-col).
** On an orphaned col, at its BFS visit:
   1. Dispose of its children by the §6.0-shaped rule (§3/§4.3):
      - Affected leaf            -> delete.
      - Affected branch, vognode -> demote to parentIs=Independent (survives).
      - Affected branch, scaffold-> convert to DeadScaffold.
      - not-Affected child       -> keep untouched.
   2. Convert the col itself to =DeadScaffold=. Skip its reconcile.
   No child-visit no-op is needed (§4.5): the survivors are either gone or
   Independent, and visiting a demoted-Independent node normally is correct.
** This REPLACES =scaffolds_with_deadOrDeleted_parents_become_dead= (§4.4):
   the in-BFS check is strictly more powerful (full ancestry vs immediate
   parent) and the BFS visits every scaffold, so it subsumes the old pre/inter
   passes. =remove_empty_deleted_scaffolds= (prune sweep) is unchanged and
   removes the resulting childless DeadScaffolds.
** §6.6 (Deleted delete-if-childless) is sequenced separately (§4.4); once both
   land, drop the =detach_stale_deleted_content= stopgap so a deleted node's
   subtree demotes/dies leafward instead of being detached wholesale.
