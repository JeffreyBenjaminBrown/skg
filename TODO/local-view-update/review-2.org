** 2.1 LATENT PANIC: an Independent same-id content child aborts the rerender  [verified*]
=server/update_buffer/complete_preorder/truenode.rs= -- in
=complete_content_children=, the "already present" predicate counts a Normal
child only when =t.parentIs == ParentIs::Affected= (truenode.rs:393), but the
prefetch =build_child_creation_data= builds its skip-set =child_sources= from
*all* Normal children with no =parentIs= filter (truenode.rs:519-522) and skips
prefetch for any id already a child (truenode.rs:540). So the skip-set is
strictly larger than the present-set: a goal id whose only in-tree occurrence is
an *Independent* Normal child is neither prefetched nor treated as present, so
=complete_relevant_children_in_viewnodetree= routes it to the create closure,
which calls =child_data.get(id).expect("complete_content_children: child data
not pre-fetched")= (truenode.rs:412-413) and *panics*, aborting the (often
collateral) rerender.

This is reachable, not theoretical: an Independent same-id content child arises
(a) from the still-live non-root containerward integrate path
(=build_and_integrate_containerward=, whose =find_child_by_id= reuses an existing
child as Independent), and (b) from the §6.0 "demote a stale Affected branch to
Independent" path -- if that id later re-enters the content goal list on a
rerender (=cap_goal_list_to_budget= keeps it, since it counts Normal children
regardless of =parentIs= as existing), the panic fires.

*This is already documented*: progress.org §18 (lines 1953-1962) records it as a
VERIFIED latent crash whose fix was *deliberately not applied* (because the
root-containerward redesign routed its own trigger away from the integrate
path). It is worth fixing on its own, because trigger (b) and the non-root
integrate path (a) still exist.

*Fix* (one line, from §18): gate =build_child_creation_data='s Normal-child
collection on =t.parentIs == ParentIs::Affected= (truenode.rs:520) so the
skip-set coincides exactly with the present-set (DiffPhantom/Inactive already
coincide). Add a regression test: a goal id present only as an Independent child.

** 2.2 Removed-member phantom can land out of HEAD position in a two-stage diff  [verified*]
=render/diff.rs:129-142= claims a removed-member phantom "lands at its correct
HEAD position among surviving siblings." That holds for the common single-stage
case but not when the *same parent's* =contains= changed in *both* staged and
unstaged. =net_diff_from_per_stage= takes its order from
=axes_from_per_stage_diffs=, whose baseline is =unstaged.or(staged)= with
items seen only in the *other* stage appended at the tail (=git.rs:222-245=). A
worked case: HEAD=[A,X,B], index=[A,B], worktree=[A,C,B] yields net order
[A,C,B,X], so the removed X is appended *after* B instead of between A and B;
=phantom_insertion_plan= anchors it to =None= and it renders at the bottom.
Purely cosmetic (the net classification is still correctly =Removed=; no crash,
no bad read), and it needs a fairly specific partially-staged state. *Fix*:
either drop the unqualified "correct HEAD position" claim in the comment, or
build =net_contains= by diffing HEAD-contains vs worktree-contains directly (an
interleaved LCS) rather than stitching two per-stage diffs; add a
=types_git.rs= two-stage ordering test either way. =[verified* -- the skeptic
reproduced the misordering end-to-end]=

** 2.3 A content-read failure omits =SaveLock= and wedges the test wait-loop  [verified*]
=serve/handlers/save_buffer.rs= sends the early =SaveLock= only inside the
=Ok(initial_buffer_content)= arm (~l.90-98); the content-read =Err= arm (~l.144)
sends only =SaveResult=. The client registers =save-lock= and =save-result= as
*one-shot* (each bumps =skg-lp--pending-count=), so on a read failure only
=save-result= fires -- the count decrements to 1, never 0, and the next save's
wait blocks. *Caveats the skeptic established*: this is *pre-existing* (the same
asymmetry is on =main=; the branch only added the =SaveLock= body, not the
arm structure), and the blast radius is the integration-test wait-loop, not
interactive saves (which don't block on the count, and whose stale one-shot
handler is harmlessly replaced next save). So: real, low, pre-existing -- worth a
defensive unconditional =SaveLock= (even with an empty lock-views list in the
=Err= arm) for the same reason §8.1 made =save-relax-lock= non-one-shot, but not
urgent. =[verified*]=

** 2.4 The lock handler's error path unlocks the saved buffer too  [verified*]
=elisp/skg-request-save.el:186-188= -- =skg--save-lock-handler= serves both
=save-lock= and =save-relax-lock=, and its =condition-case= error branch calls
=(skg--unlock-all-save-locked)=, which removes the lock overlay from *every*
buffer including the saved one. A malformed lock payload would thus unlock the
saved buffer mid-pipeline, breaking the "saved stays locked until =save-result="
invariant the code's own comment (l.104-105) states; a subsequent
=collateral-view= / =save-result= then erase-inserts the buffer, silently
discarding any edit the user slipped in during the unlocked window. Only
reachable on a malformed message (the server's =format_lock_views_sexp= output
is always well-formed), so low. *Fix*: on error unlock only the non-saved
buffers (=skg--unlock-non-collateral-buffers saved-uri nil=), or document the
fail-open. =[verified*]=

** 2.5 Test-only de-novo shim swallows a file-read error into an empty graph  [verified*]

* 4. Elegance
The recurring shape: a value or subtree is computed twice, or kept alive to feed
something dead. None is a bug; each will drift.
- *IDCol/AliasCol diff membership is derived twice.* =process_truenode_diff=
  fully populates the cols (=prepend_idcol_with_children= /

  / =reconcile_alias_col_children= recompute the *same* merge at the col's own
  visit. Two independent derivations of identical membership from one

  worker create the *empty* col and let the reconciler be the single populator
  (review.org §3.4). =[verified*]=

- *=hidden_ids= is fetched on every Final DVR but consumed only on a dead branch*
  (§3.3).

These are independent and individually small; §3.3 (delete the dead branch) is
the highest elegance-per-line win and also retires review §5.3.
