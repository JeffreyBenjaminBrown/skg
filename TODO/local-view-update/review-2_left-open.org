* Glossary
- *view* / *buffer* :: an org-mode buffer showing a slice of the graph as an
  indented tree. The user reads and edits the graph through these.
- *viewnode* :: one node (one headline) in that tree.
- *vognode* :: a viewnode that stands for a real graph node (it has an ID).
- *scaffold* :: a viewnode that is NOT a vognode -- pure display structure (a
  column header, a marker), with no graph node of its own.
- *col* :: a scaffold that *collects* members and carries "Col" in its name:
  IDCol, AliasCol, SubscribeeCol, the relation cols (Subscriber / Hider /
  Hidden / Overrider / Overridden), and the two Hidden-in/outside-subscribee
  cols. A single alias entry or ID entry is a scaffold but NOT a col. (This is
  the scaffold-vs-col distinction from plan_v2 §19.)
- *completion* / *the driver* :: the server step that takes a view tree -- freshly
  opened, or rebuilt after a save -- and fills in everything derived from the
  graph: content children, the scaffolds/cols and their members, and the
  git-diff marks, repairing anything stale. It is now a single top-down
  breadth-first pass (I call it "the one BFS driver"); "completion" is the older
  name still used in the code.
- *reconcile* :: within completion, bringing one col's members into line with
  what the graph currently says they should be -- adding missing ones, dropping or
  demoting stale ones.
- *de-novo* vs *post-save* (a.k.a. *rerender*) :: de-novo = building a view from
  scratch when you open a node; post-save / rerender = rebuilding already-open
  views after a save changed the graph. Both now run the same driver.
- *definitive* / *indefinitive* :: a definitive viewnode is the editable,
  fully-expanded occurrence of a node (its body shows, its content is drawn); an
  indefinitive one is read-only and collapsed and does not pull in absent
  content. An ID is definitive in at most one place per buffer.
- *definitive view request* (DVR) :: an internal marker meaning "draw this node
  definitively." Opening a node sets one.
- *draw rule* / *cascade* :: the rule deciding, when an ID appears more than once
  in a buffer, which occurrence gets the single definitive slot; the cascade is
  how a definitive node hands "be definitive" down to its own content children.
- *node budget* :: a per-view cap on how many new viewnodes one render may
  create, so a node that (say) thousands of people subscribe to does not try to
  draw all of them at once.
- *git-diff view* :: a mode that overlays git status onto a view, marking what
  changed since the last commit.
- *phantom* (DiffPhantom) :: a diff-only placeholder for something git shows as
  removed -- a node whose file was deleted, or a child removed from a parent's
  list ("removed-here"). It exists only in the diff view.
- *existence* / *membership* axes; *staged* / *unstaged* :: how a node or phantom
  records its git status. Existence = did the node's file appear or disappear;
  membership = did the node appear or disappear from a particular parent's
  child-list. Each is tracked per git stage: staged compares HEAD vs index,
  unstaged compares index vs worktree. (HEAD = last commit; worktree = the files
  on disk right now.)
- *IDCol* / *AliasCol* :: scaffolds that list a node's IDs / aliases; in the diff
  view they mark which were added or removed.
- *idempotence* :: running completion twice on an unchanged graph yields the
  identical view. The refactor depends on this -- it is what makes traversal order
  not matter.

* 5. A cosmetic mis-placement of phantoms in partially-staged diffs
** Context
In the diff view, a child removed from its parent's list shows as a phantom,
ideally sitting in the slot it used to occupy among its surviving siblings. The
code computes that slot by stitching together git's two stages (staged =
HEAD-vs-index, unstaged = index-vs-worktree). When the *same parent's* child-list
was edited in *both* stages at once, the stitching can drop a removed item at the
*bottom* of its siblings instead of in its original slot.

It is purely cosmetic: the item is still correctly labeled "removed"; nothing
breaks, no data is wrong, and it only happens in a fairly specific partially-
staged state. I made the code's comment honest about the limitation today.
