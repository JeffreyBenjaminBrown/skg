#+TITLE: Override chains: permit user-owned chains, redefine monogamy, add a fork-your-own-node gesture
#+TODO: TODO ONGOING | DONE

* Summary
  Today an =overrides_view_of= edge (overrider -> overridden) obeys two
  invariants on /user-owned/ edges, checked at save and at init/rebuild
  (see [[../../server/dbs/in_rust_graph/override_invariants.rs][override_invariants.rs]]):
  - *monogamy*: no node has more than one user-owned overrider;
  - *acyclicity ("no chains")*: if user-owned X overrides Y and
    user-owned Y overrides Z, that is a violation.

  We want to:
  1. *Permit user-owned chains.* A user may override their own overrider.
     Concretely: a public clone =C= overrides foreign =N=; a private clone
     =D= overrides own =C=. The chain =D -> C -> N=, all user-owned, must
     become legal. Motivation: a paranoiac wants a public fork of someone
     else's node /and/ a private fork of their own, so that even the
     /existence/ of private thoughts is not leaked. (A foreign reader of
     your public =C= can see that =C='s =contains= names an ID absent from
     your public data; routing the private material through a separate
     private overrider =D= keeps =C= clean.)
  2. *Keep monogamy*, restated as "no two user-owned nodes override the
     same node (owned or not)". This is /exactly/ the current target-end
     check; nothing changes but the prose.
  3. *Confirm* that one node may override several distinct targets. It
     already can, end to end; no code change.
  4. *Replace* acyclicity with a *no-cycle* check, since the resolver must
     still terminate, and a cycle is now reachable through normal gestures.
  5. Add an explicit *'skg-fork-node'* command (=C-c m f=, new
     =(viewRequests fork)= request) so the user can fork a node they own —
     the gesture that /creates/ =D -> C=.

  The work is small in the resolver and data model (which already tolerate
  chains) and concentrated in three places: the *invariant validators*, the
  *tamper check*, and the *new fork gesture*. One subtlety is load-bearing
  and easy to miss — the tamper check breaks under a restricted source-set
  (see [[*Work item 3: Generalize the overridesHere tamper check (the subtle one)][Work item 3]]).

* Background (terms, so the open questions read standalone)
  - *override substitution*: when content =N= would be drawn, the server
    draws the /effective overrider/ instead — the end of the chain of
    user-owned, currently-/visible/ =overrides_view_of= edges out of =N=.
    The drawn node carries the marker =(overridesHere N)= (herald "Oh").
    Resolver: 'resolve_override' in
    [[../../server/dbs/in_rust_graph/override_resolution.rs][override_resolution.rs]].
  - *collected id*: what a drawn viewnode contributes to its parent's saved
    =contains=. For a substitute it is the marker's original (=N=), not the
    drawn node — so substitution never rewrites a container's =contains=.
    'collected_id' in [[../../server/types/viewnode.rs][viewnode.rs]].
  - *visibility gating*: under a restricted source-set the resolver follows
    an edge only if the overrider's source is /active/. So the effective
    (drawn) node can be a *middle* node of the full chain, when a later
    link is invisible.
  - *the tamper check*: 'overridesHere_marker_errors' in
    [[../../server/from_text/buffer_to_viewnodes/validate_tree.rs][validate_tree.rs]]
    rejects a save whose =(overridesHere N)= marker the server "would not
    have drawn", to stop a hand-edited/yanked/stale marker from silently
    rewriting a =contains= list.
  - *monogamy makes the overrider-walk functional*: with at most one
    user-owned overrider per node, "follow my overrider" is a partial
    /function/, so a user-owned cycle is always a simple cycle and the
    resolver's seen-set catches it.

* The conceptual change, in three clauses
** (a) Monogamy: unchanged
   The current 'MultipleUserOwnedOverriders' check /is/ "no two user-owned
   nodes override the same node". It is a target-end check:
   'user_owned_overriders_of' groups overriders by the node they override
   and flags any group of size > 1. Keep it verbatim. The fork monogamy
   pre-check 'existing_user_owned_overrider_of' (used by 'validate_fork_specs'
   in [[../../server/from_text/fork.rs][fork.rs]]) also stays — it is what makes a
   second fork of the same node fail with =ForkAlreadyExists=.

** (b) One overrider, many targets: already legal (verified, no change)
   =overrides_view_of= is an =MSV<ID>= (a list) from disk through
   'NodeComplete', the inverse index =overriders_of= is keyed per target,
   the resolver imposes monogamy /per target/ (not per overrider), save
   lowering serializes the whole list, stats count every target, and the
   round-trip is per-container via the =overridesHere= marker. Drawing the
   same overrider =R= under two containers (for two different originals)
   round-trips each correctly, because each position carries its own
   =(overridesHere ...)=. *No code change; add a confirming test.*

** (c) Chains: permitted; cycles: forbidden
   Drop the "no chains" prohibition. Add a no-cycle invariant in its place,
   enforced at the same two gates (save, init/rebuild). A user-owned cycle
   is now /reachable/ (see [[*Danger: a cycle is reachable via the writable overriddenCol][the danger note]]), so this is mandatory, not
   decorative.

** Work item 3: Generalize the overridesHere tamper check (the subtle one)
  File: [[../../server/from_text/buffer_to_viewnodes/validate_tree.rs][validate_tree.rs]], 'overridesHere_marker_errors'.

  *The bug the feature introduces.* Today the check demands
  =carrier == resolve_override(N, active=None).effective= — the
  visibility-/ungated/ end of the chain. For a single override that is
  always the drawn node, so it holds. For a chain =D -> C -> N= rendered
  under a source-set where =D='s source is *inactive*:
  - rendering (gated) draws =C= (the visible prefix end), marked
    =(overridesHere N)=;
  - the tamper check (ungated) computes effective = =D=;
  - =carrier C != effective D= -> the save *aborts* with
    =OverridesHere_Mismatch=, even though =C= is an honest carrier.
  This is *confirmed* by tracing 'build_child_creation_data'
  ([[../../server/update_buffer/reconcile/content.rs][content.rs]], the marker's original is always the goal id =N=)
  and the ungated resolve in 'overridesHere_marker_errors'.

  *The fix.* The honest carriers of =(overridesHere N)= are exactly the
  nodes on =N='s user-owned override chain — under some source-set, each is
  the visible prefix end. So replace the equality with membership:
  =carrier ∈ resolve_override(N, active=None).path=. Add a helper
  'carrier_on_user_owned_chain(config, graph, original, carrier) -> bool'
  in [[../../server/dbs/in_rust_graph/override_resolution.rs][override_resolution.rs]] (cohesion with the chain logic) and call
  it from the tamper check. Keep the existing fail-closed behavior when the
  global graph handle is unavailable.

  *Why not just gate the check to the connection's source-set?* That looks
  simpler (=carrier == resolve_override(N, active=connection_set).effective=)
  but it reintroduces the very failure the current ungated design was built
  to avoid: a marker that was honest when rendered must not start failing
  because the source-set changed afterward (a yanked marker line, a stale
  buffer). The ungated-/path/ predicate is source-set-independent and still
  rejects a marker on a node that is /not/ on =N='s chain — so it does not
  weaken the check: it admits exactly the legitimate substitutes (each of
  which legitimately stands for =N=, so collecting =N= for its position is
  correct).

****** Tiny glossary (for this note only):
- /marker/: a hidden tag =(overridesHere N)= the server puts on a node
  it draws, meaning "I'm standing in for node N right here".
- /carrier/: the node the marker is attached to — the node actually
  shown on screen.
- /original/ (=N=): the node being stood-in-for (the one named inside
  the marker).
- /chain/: =D overrides C overrides N=. Each node stands in for the
  next one down.
- /effective overrider/: which node actually gets drawn in =N='s place.
  With every source visible it's the far end of the chain (=D=). If
  =D='s source is hidden, the server can only draw as far as =C=, so
  =C= is effective then.
- /ungated/: "ignore which sources are visible right now" — i.e. walk
  the WHOLE chain.
- /path/: the list of nodes the chain runs through, e.g. =[C, D]=.
- /tamper check/: when you save, the server re-derives every marker it
  would itself have drawn and refuses the save if your buffer carries a
  marker it would NOT have drawn — so a hand-typed or yanked marker
  can't trick the save into rewriting some parent's child list.
****** The actual point, in three steps:
1. OLD rule: "the carrier must equal the one node the server would draw
   for =N=, ignoring visibility" — that's the far end of the chain
   (=D=). This was fine while chains were illegal, because then a chain
   was always ONE hop, so the far end was the only overrider there was.
2. PROBLEM once chains are legal: the node actually drawn can be a
   MIDDLE of the chain — =C=, whenever =D='s source is hidden. So the
   carrier on screen is =C=, but the old rule still demands =D=. They
   differ, and the save is wrongly rejected. (This is exactly your
   public-=C= / private-=D= use case viewed with the private source
   off.)
3. NEW rule: "the carrier must be SOMEWHERE on =N='s chain" — =C= or
   =D=, either is acceptable. Every node on =N='s chain genuinely
   stands in for =N=, so accepting any of them is safe; a marker on a
   node that is NOT on =N='s chain (a faked/stale one) is still
   rejected, so the check is not weakened. "Ungated path membership"
   is just shorthand for: take the whole chain (ungated) and check the
   carrier is one of its nodes (path membership), instead of demanding
   it equal the last node.
