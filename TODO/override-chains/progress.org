** DONE Phase F — verification of the motivating case
   The plan called for MANUAL verification via /run. The environment is
   headless (no interactive Emacs display), and a manual /run drive is
   neither feasible nor reproducible here, so I verified the full
   motivating case with an AUTOMATED integration test instead --
   stronger evidence and a lasting regression guard. New test
   tests/integration/fork-chain/ drives, through the real client+server:
   1. fork foreign N -> public clone C overrides N (implicit);
   2. 'skg-fork-node' on the DRAWN SUBSTITUTE C, rotating the clone to a
      PRIVATE source -> private D overrides C (the fork-of-a-fork, the
      highest-risk path: forking the substitute targets C's own id, not
      N via the marker);
   3. reopening N's container draws the chain end D (a private node) in
      N's place, marked (overridesHere N);
   4. saving that chain-end view succeeds (the tamper check accepts the
      chain-end carrier);
   5. on disk the public clone C overrides the foreign N and names NO
      private id (run-test.sh checks the .skg files).
   The half-visible (source-set hides the chain end -> middle carrier
   drawn, save still keeps N) behavior is covered at the server level by
   Phase B's tests/override_substitution/fixtures-chain. All three fork
   integration tests pass.

* Unanticipated future potential costs and benefits
  - The global validator's cycle scan calls 'resolve_override' once per
    user-owned node without memoizing proven-acyclic pids (the plan
    permitted skipping memoization on the cold init/rebuild path). Cost
    is O(V · chain depth) worst case; chains are short in practice.
    Cheap to add memoization later if init time ever matters.
  - Benefit: 'resolve_override' is now the single home of the
    chain/cycle walk -- both validators, the tamper helper, and content
    substitution share it, so the rules cannot drift apart.

** DONE Dismissing the confirmation buffer (any way) strips the fork atom
   Originally only 'skg-decline-fork' (C-c C-k) stripped the lingering
   (viewRequests fork) atom; killing the confirmation buffer directly
   left it, so the next save re-raised the fork. Fixed: the
   fork-confirmation buffer now carries a buffer-local 'kill-buffer-hook'
   ('skg--fork-confirmation-on-kill') that strips the origin's fork atom
   on ANY dismissal. 'skg-approve-fork' suppresses it (the atom must
   survive to its re-save, which commits the fork; the server then drops
   it on re-render) via the buffer-local 'skg--fork-suppress-strip-on-kill'.
   Stripping is idempotent, so it is a harmless no-op after a decline.
   The fork-owned integration test gained a part that dismisses the
   confirmation by killing it directly and asserts the atom is stripped.
