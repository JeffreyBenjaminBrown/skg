* Changes to the plan's envisioned order

** Mechanical renames done by script, not hand-edit
The plan describes editing each site; for the pure-token renames I used
'sed'/Python instead (verified unique tokens, then 'cargo build'):
- 0a/0b token renames; 'Vognode::Normal' -> 'Vognode::Active' (covers
  'MpVognode::Normal' too); and the regular full-nested 'ViewNodeKind::Vognode
  (Vognode::DiffPhantom..)' -> 'ViewNodeKind::Phantom (Phantom::Diff..)' moves.
The irregular sites (or-patterns, bare constructions, spacing variants, the two
hazards) were hand-edited, driven by the compiler error list.

** lib-green and test-green folded into ONE split commit
The plan sequences a step-4 (lib first-green) commit then a step-5 (test-green)
commit. Both reached green together quickly, so the split is a single commit
(prep commits 0a/0b stay separate). Net commits: 0a, 0b, split.

** Introduced a helper instead of splitting 9 identical or-patterns
'maybe_placed_viewnode.rs' validators repeated
'matches!(.., MpVognode::Active(_) | MpVognode::DiffPhantom(_))' ~9 times. Rather
than split each cross-enum or-pattern inline (plan's literal approach), I added
'MpViewnode::is_active_or_diff_phantom()' and call it. Cleaner + DRY; same
behavior. (The placed-side 'find_children_by_ids' and a few test/typedb sites
were still split inline, as those bind the matched value or mix in other kinds.)

* What's imperfect about what's done

** Stale prose mentions of 'DeletedNode' in DO-NOT-TOUCH oracle files (0b)
After the 0b struct rename, 23 references to the old name 'DeletedNode' remain in
'.el'/'.org'/'.sh' narrative under tests/integration/save_collateral_*/ (e.g.
README/detail.org, test-emacs.el). These are prose descriptions of the concept,
not Rust identifiers and not wire-format strings. The plan lists these exact
files under "Group 8 -- DO NOT TOUCH" (a diff there during the refactor signals a
wire-format break), so I left them untouched. They are now mildly stale-named but
harmless; a future doc pass could update them outside this refactor.

** 0b's blanket sed corrupted a user-facing string literal (found + fixed)
The 0b struct-rename sed ('DeletedNode' -> 'PhantomDeleted') also hit a STRING
LITERAL -- the validation message "View roots must be TrueNodes or DeletedNodes."
became "...PhantomDeleteds." The full test suite caught it
('test_find_buffer_errors_for_saving', whose regex looked for 'deletednode').
Fixed in the polish pass: the message now reads "View roots must be TrueNodes or
deleted nodes." (clean user-facing prose, decoupled from the internal type name),
and the test's regex/label match. Lesson: a type-name sed can silently rewrite
prose inside string literals; only this one user-facing string was affected (wire
atoms are lowercase, so untouched -- verified).

** 'ExpectedAncestor::NormalVognode' left un-renamed (plan deferred)
The enum variant in 'ancestry.rs' still reads 'NormalVognode' (it now means
"Active vognode"). The plan marked this rename optional/stylistic; left as-is to
avoid scope creep. Its doc/uses are internally consistent.

* Choices determining the future

** Three narrow id accessors, no unified 'id()'
Per the settled decision, identity is read through three named accessors:
'Vognode::id' / 'Phantom::id' (each enum), 'ViewNode::id_if_vognode'
(Active/Inactive only), and 'ViewNode::active_or_diff_phantom_id' (Active + Diff).
No catch-all 'ViewNode::id()'. New callers should pick the accessor whose name
matches the id-set they actually want.

** Removed the dead 'Vognode::active_id' the plan specified (1d)
The plan added 'Vognode::active_id', but after promoting the cross-enum reach to
'ViewNode::active_or_diff_phantom_id' nothing called it. I removed it rather than
leave an unused 'pub' method. If a Vognode-only "active id, else None" is wanted
later, re-add it.

** Hazard C resolved toward behavior-preservation
'pid_and_source_from_ancestor' now resolves through a Diff/Deleted phantom
ancestor (added 'Phantom' arm). This was the plan's flagged judgment call; I chose
to preserve the pre-split behavior exactly rather than start erroring on phantom
ancestors. Easy to reverse (drop the arm) if a phantom ancestor should error.

** 'MpViewnode::is_active_or_diff_phantom' is now a shared predicate
Used by ~10 validators. If the set of "TrueNode-ish" Mp kinds ever changes,
that one method is the place to change it.
