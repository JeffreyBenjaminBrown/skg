* Problems found during herald overhaul research
** (mostly fixed) Save validation and extraction use different edit-request surfaces
   The save extraction path uses 'collect_savenode_candidates' to define
   which view positions can produce save/merge instructions. Some
   validation still walks the whole viewforest directly, especially
   merge validation and contradictory instruction collection.

   That can make validation and execution disagree for edit metadata
   placed under display/scaffold positions: validation may see it even
   when extraction ignores it, or vice versa if the candidate policy
   changes. The cleaner fix is probably to validate write-intent
   metadata through the same candidate/role policy used by extraction,
   and explicitly reject edit requests outside that surface.

   Mostly fixed 2026-06-10 by local instruction collection: extraction
   now recurses into every position, so a definitive node's self-edits
   (including edit requests) are honored wherever validation can see
   them. The residual gap is positions where edit requests are
   deliberately ignored rather than rejected: a Delete or nodeMerge
   request on a subscribee-as-such is silently dropped, which validation
   could instead flag.

** Save rerender may not insert missing HiddenInSubscribeeCol
   While experimenting with the "delete as-a-subscribee content
   implies hide" feature, I saw a case where the subscriber's
   'hides_from_its_subscriptions' list had gained the hidden child,
   but the returned saved view still did not show a
   HiddenInSubscribeeCol under the subscribee.

   The likely reason is that the save rerender path completes an
   existing HiddenInSubscribeeCol in postorder, but a buffer that
   initially had no hidden children also has no scaffold to complete.
   The eventual feature implementation probably needs to ensure that
   a HiddenInSubscribeeCol is inserted when the updated hide list
   intersects a subscribee's content.

** 'mk_indefinitive_from_viewnode' silently drops fields
   server/types/viewnode.rs line 538.
   It rebuilds a fresh viewnode via 'mk_indefinitive_viewnode',
   losing 'view_requests', 'diff', 'graphStats', 'viewStats',
   and any future fields added to TrueNode.

   Any new field (like 'parent_relation' if added)
   must be manually preserved here, which is easy to forget.

   Consider refactoring to mutate in place:
   clear body, set indef_or_def to Indefinitive,
   update parentIs — but keep everything else.

** 'path_from_pid_and_source' uses .expect() and can panic
   server/util.rs. Already noted in MEMORY.md.
   If an invalid source reaches this function (e.g. from a search view),
   the server crashes. Should return Result.

** 'head_node' field on NodeCompleteDiff is a misnomer

** 'mk_removed_child_viewnode' hardcodes unstaged=Minus axes
   server/to_org/expand/definitive.rs lines 394, 398.
   When a parent .skg file is deleted, its children are loaded from
   git HEAD and rendered as phantoms. Both existence and membership
   axes are hardcoded as 'staged: None, unstaged: Some(Minus)',
   which misattributes already-staged parent-deletions to the
   unstaged side. Sibling bug to the one just fixed in phantom_axes
   (types/phantom.rs). Fix is analogous: derive per-stage signs from
   the parent's file-level diff status (staged vs unstaged maps in
   SourceDiff), rather than assuming the change is on the unstaged
   side. Additionally, for a deleted-parent scenario the child's
   membership axis should probably mirror the parent's existence
   axis (if parent removed staged, children are 'removed from
   contains' staged too), which is a bit more nuanced than the pure
   contains_diff lookup used in phantom_axes.

* Performance

** 'fetch_all_graphnodestats' scales linearly in pids, ~60ms each
   server/dbs/typedb/search/all_graphnodestats.rs.

   The file header promises "one TypeDB query PER PID, all in parallel"
   and cites ~15ms per query. In practice the TypeDB server (or the
   driver) caps effective concurrency at roughly 4, so the total time
   is (pids * ~250ms) / 4. Small views finish in well under a second;
   medium-big views take multiple seconds.

   Measured on data/public/ (29k .skg files):
   - visit 'science' (root with 6 contents, ~10 pids in view): 0.68s total,
     with fetch_all_graphnodestats ~0.35s.
   - visit 'to watch' (root with 3 contents, ~95 pids in view): 5.5s total,
     with fetch_all_graphnodestats ~5.0s.

   Verified by forcing join_all -> a sequential await loop: per-query
   time was ~250ms, total scaled 1:1 with pid count ('science' 2.5s,
   'to watch' 23.5s). So join_all IS achieving parallelism; there is
   just a ~4x ceiling that eats the gains as views grow.

   Per-pid breakdown (emitted from temporary tracing in
   fetch_one_pid_stats):
   - tx.transaction(..) await  : ~8ms
   - tx.query(..) await returns: ~270ms avg (it already waits some)
   - first stream.next() await : +~4.5s (!) -- this is where to-watch
     spends its time; most pids wait in a TypeDB-side queue before
     their stream yields a document
   - draining the stream       : negligible (single-row result)

   The code comment "1 PID: 15ms, 24 PIDs: 4300ms" compared only the
   old N-way-disjunction approach against the new per-pid approach.
   The new approach is better than the old one but is still O(N), just
   with a smaller constant and an unknown parallelism cap.

   Options to try, in decreasing order of likely payoff:

   1. Batch many pids into a single TypeDB query. Something like

        match
          $node isa node, has id $id;
          { $id == "pid1" } or { $id == "pid2" } or ... ;
        fetch { "id": $id, "containers": [...], ... };

      returning one document per pid. Even if TypeDB evaluates the
      disjunction poorly for joins, a fetch that only *dispatches* on
      id should still be 1 query instead of N. Would remove the per-
      transaction and per-round-trip overhead entirely.

      TODO (Jeff speaking): We tried (1) once long ago. It was slower.

   2. Share one transaction across all pids (driver.transaction once,
      then run N queries on it). Might bypass whatever per-transaction
      bottleneck is capping concurrency at ~4.

   3. Split fetch_one_pid_stats into 6 separate relation-specific
      aggregate queries ("find all containers of nodes in {pids}",
      "find all contents of nodes in {pids}", etc) with the id set
      passed in as a TypeDB collection. Each query does one relation,
      across all pids, and folds results pid-by-pid in Rust.

   4. Cache. The same pid can appear in many views within a session;
      results are stable between writes. A small in-memory cache keyed
      by pid and invalidated on save would eliminate repeats.

   Until this is fixed, any view with more than a few dozen nodes
   (i.e. any real working set above initial_node_limit/2) will have
   multi-second first-render latency.

** 'bash/start-servers.sh' process management is not safe for multiple Skg servers in one container
   Found while writing docker-unification-strategies.org.

** 'skg-first-sexpr-on-line' fabricates a (node ...) child
   elisp/skg-sexpr/skg-sexpr-search.el / skg-strip-heralds-from-sexp.
   The stripper calls 'skg-edit-nested-sexp' with instructions
   '(skg (node (graphStats (DELETE ...)) (viewStats (DELETE ...))))'.

   accidentally dodge the bug by matching the more specific
   '(skg (node (id)))'. Fix candidates: stripper should use
   DELETE-only instructions, or short-circuit when the target lacks
   '(node ...)' to begin with.

** 'compute_interleaved_diff' can collide for large unique-item lists
   WARNING: TODO.org refers to the headline title verbatim.

   server/types/list.rs maps each unique item to a Unicode scalar and
   feeds the resulting strings to 'similar::TextDiff::from_chars'.

   The mapping starts at 0x100 and uses
   'char::from_u32(char_counter).unwrap_or('?')'. Once the counter
   reaches surrogate code points, or eventually passes Unicode's max
   scalar value, multiple distinct items map to '?'. That can make the
   diff incorrect.

   It probably does not matter for small relationship lists, but a
   generic list diff helper should either avoid scalar encoding or
   return an error instead of silently colliding.

** One TypeDB process has a fixed idle CPU tax
   Investigated on 2026-05-14;
   see 'TODO/typedb-performance-research.org'.
   After Skg is ready and no requests are arriving, TypeDB CE 3.10.3

** GraphStats rendering still lacks parent-kind context
   Observed on 2026-05-31 while splitting 'ParentIs' from 'Birth'.

   'server/org_to_text.rs' renders a TrueNode's graphStats from only
   the TrueNode metadata. After 'Container' and 'Collector' collapsed
   to 'ParentIs::Affected', relation collection children and ordinary

** Merge silently discards the acquiree's aliases
   server/merge/mergeInstructionTriple.rs.
   'three_merged_nodecompletes' (line 162) clones the acquirer and unions
   extra_ids, contains, subscribes_to, hides_from_its_subscriptions, and
   overrides_view_of from the acquiree -- but never aliases. The acquiree's
   title is preserved (mangled to "MERGED: ...") in an
   acquiree_text_preserver, whose own aliases are set to
   MSV::Unspecified (create_acquiree_text_preserver, line 229). So after
   merging X into Y, the result is no longer findable by any of X's
   aliases.

   This is asymmetric with extra_ids, which ARE unioned so the node stays
   findable by X's id. Either aliases should be unioned into the acquirer
   the same way, or the behavior should be a documented decision. Decide
   what "merge" should preserve.

** Inactive nodes are never converted to Vognode::Deleted
   server/update_buffer/complete_preorder/truenode.rs.
   When a save in one buffer deletes a node, collateral views convert a
   matching Normal vognode to Vognode::Deleted via
   'mutate_truenode_to_deletednode' (line 137-140). But that check lives
   inside 'expand_true_content_at_truenode', which only runs on
   Vognode::Normal (line 115-118). A node shown as Vognode::Inactive (from
   an inactive source) whose pid is in 'deleted_by_this_save_pids' is never
   converted, so it lingers as an Inactive placeholder for a node that no
   longer exists. Rare (the user must delete an inactive-source node), but
   real. Fix: have the parent's reconcile convert any child -- Normal or
   Inactive -- whose pid is in 'deleted_by_this_save_pids'.

** Diff scaffolds are not added to nodes made definitive by a view request
   server/update_buffer/complete.rs, server/to_org/expand/definitive.rs.
   'ensure_diff_scaffolds' (complete.rs:50) runs once, before
   'execute_truenode_view_requests' (complete.rs:56), and only for nodes
   that are already definitive and non-phantom. A node that becomes

   So interactively expanding a node definitively while git diff mode is on
   yields content with no TextChanged/IDCol/AliasCol scaffolds and no diff

   Needs a test to confirm reachability (diff mode + a definitive view
   request on a previously-indefinitive node with changed text/ids/aliases
   or changed content). The planned "separate diff pass over Vognode::Normal"
   (TODO/local-view-update) would fix it by visiting every Normal vognode

** Collateral graphStats can go stale when only a node's relationships change
   server/update_buffer.rs / find_collateral_view_uris.
   Collateral views are selected by the PIDs in the SavePlan's DefineNodes.
   If saving buffer X changes A.contains to include B (only A is saved),
   then a different view rooted at B is not rerendered, even though B's
   container count (graphStats.containRels) changed. B's herald is stale
   until B's view is next rerendered. Pre-existing and arguably acceptable;
   fixing it would require selecting collateral by the neighbors of changed

** (not a bug, but a simplification) Lock over-approximates the rerender set
   server/serve/handlers/save_buffer.rs, elisp/skg-request-save.el.
   The early lock message ('uris_of_views_to_lock') locks every open view
   sharing any PID with the saved buffer, but only views sharing a
   *changed* PID receive a 'collateral-view' message
   ('find_collateral_view_uris'). I checked the client: a buffer that is
   locked but never rerendered is still unlocked, because
   'skg--save-result-handler' calls 'skg--unlock-all-save-locked' on
   save-result (skg-request-save.el:171, and again in its unwind-protect at
   174). So there is no permanently-locked bug.

   It is therefore only an opportunity to simplify: narrowing the lock to

* Performance: the save-buffer pathway (see profiling-save.org)
  Profiling the save of an insert-a-child edit (node 846207ef, 'doing')
  found four inefficiencies, in impact order. Full evidence, numbers,
  and recommendations are in the top-level 'profiling-save.org'.

** Override validation re-scans the whole graph on every save (~24%)
   'validate_override_invariants_after_save' (server/save.rs), called
   unconditionally from 'update_graph_including_merges', runs
   'validate_override_invariants' (server/dbs/in_rust_graph/
   override_invariants.rs), which iterates 'graph.nodes' twice over all
   ~60k nodes. The cost is O(total graph), independent of the save, so
   every save slows as the user's data grows. Disabling it cut the save
   from 343 ms to 256 ms. Fix: validate only the saved nodes and their
   'overrides_view_of' neighborhood.

** Tantivy is written and committed twice per save (~38%)
   'update_tantivy_from_saveinstructions' (server/save.rs) indexes each
   saved node with 'context_origin_type = ""'; then
   'update_context_origin_types' (server/dbs/tantivy/context_update.rs)
   opens a second 50 MB IndexWriter, reads those docs back, deletes and
   re-adds them with the computed type, and commits again. Skipping the
   second pass cut the context phase from 71 ms to 4 ms. The file's own
   header comment names the fix ("compute origin types before
   indexing"): compute the type first, index once, commit once, and
   reuse one long-lived IndexWriter.

** TypeDB relationship update is delete-all-then-recreate-all (~15%, O(children))
   'update_typedb_from_saveinstructions' (server/save.rs) deletes and
   recreates ALL outbound edges of each saved node every save, so
   saving an N-child node is O(N) and building one up is O(N²).
   'create_all_relationships' grew from 48 ms (~25 children) to 137 ms
   (~100). Fix: diff old vs new edges (the in-Rust snapshot has the old
   set) and apply only the delta.

** Save cost degrades 3-4x within a session of repeated saves
   404 ms -> 1220 ms over 11 consecutive saves; a 'rebuild dbs' resets
   it. Cause: each tiny Tantivy commit spawns a segment and merges fall
   behind. Mostly fixed by the single-commit change above; also consider
   debouncing commits (Tantivy is a rebuildable cache) and dropping the
   forced per-commit 'reader.reload()' in production.

* Problems found while researching local instruction collection (2026-06-09)
  Found while reading the save-extraction code for
  TODO/local-instruction-collection/. These all support that vision's
  premise; if the rewrite happens they die with the old code, but they
  are real today.

  FIXED 2026-06-10: all four died with the old extraction path when
  local instruction collection replaced it

** (fixed) 'DefinenodeCandidateKind::Overridden' is computed and discarded everywhere
   'collect_savenode_candidates' pushes candidates of kind Overridden,
   but every consumer filters them out:
   - 'naive_node_edit_intents_from_candidates' keeps only Ordinary;
   - 'subscribee_hiderel_intents_from_candidates' keeps only Subscribee;
   - 'instructiontriples_from_merge_candidates' keeps only Ordinary.
   So OverriddenCol members are classified, candidate-listed, and then
   ignored as instruction owners. Either that is intended (members of
   an OverriddenCol are inert, like read-only col members), in which
   case the variant should be removed, or their subtrees were meant to
   be searched for instruction owners and are not — a latent bug.

** (fixed) Four 'SaveRole' variants are computed but never consumed
   'classify.rs' computes 'AliasDisplay', 'IdDisplay',
   'HiddenInSubscribeeCol { subscriber, subscribee }', and
   'HiddenOutsideOfSubscribeeCol { subscriber }'. The latter two do
   nontrivial grandparent/great-grandparent walks that can return Err
   and abort a save. No code ever pattern-matches on any of the four
   (they are handled only by catch-all arms). Wasted work on every
   save, and an error surface with no corresponding feature.

** (fixed) 'viewforest_with_saveroles' deep-clones the whole buffer per save
   It clones every ViewNode in the viewforest just to pair it with a
   SaveRole ('ViewNode_in_Role'), and 'SaveAuthority::from_viewforest'
   runs on every save (and again in the legacy test wrappers). For
   large buffers this is a meaningful allocation on the save critical
   path; profiling-save.org may already reflect it.

** (fixed) 'reconcile_same_id_instructions.rs' is dead in production
   'server/from_text/viewnodes_to_instructions/reconcile_same_id_instructions.rs'
   (DefineNode-level same-ID reconciliation) is referenced only by its
   tests and by a doc comment in 'viewnodes_to_instructions.rs' --
   and that comment is misleading, because the reconciliation the

* Problems found during full-schema metaplan 2 research (2026-06-10)
** Sharing-col members from inactive sources make a buffer unsavable
   Under a restricted source-set,
   'apply_source_set_to_viewforest' ('server/source_sets.rs')
   converts EVERY Active viewnode from an inactive source into an
   InactiveNode placeholder -- including members of sharing cols
   (subscribeeCol, overriddenCol, and the read-only relation cols).
   But buffer validation ('validate_inactive_node' in
   'server/from_text/buffer_to_viewnodes/local.rs') requires an
   inactive placeholder's parent to be a gnode. So a buffer whose
   subscribeeCol contains a member from an inactive source cannot be
   saved at all: the save aborts with "Inactive placeholder must
   have a TrueNode parent."

   Worse, the failure mode if that validation were relaxed without
   further work is silent data loss: 'relation_col_members'
   ('server/from_text/local_instruction_collection/traverse.rs')
   collects only Active children, so the inactive member would be
   dropped from the 'SetSubscribesTo' / 'SetOverrides' list, which
   lowers to 'MSV::Specified' and overwrites the disk list
   ('supplement_unspecified_fields_from_disk' fills only Unspecified
   fields). The subscription/override to the invisible node would be
   silently deleted.

   Planned fix: TODO/full-schema/metaplan_2.org, section "preserve
   source-set safety for sharing relations" (omit inactive members
   from col display; merge visible-subset edits with disk's
   inactive members at save; assert no instruction touches an
   inactive source).
** 'complete_relevant_children_in_viewnodetree' panics on non-Active relevant children
   Its orderkey closure panics ("relevant child not a normal graph
   node") if the relevance predicate ever admits a non-Active
   viewnode. Today every caller's predicate requires
   Vognode::Active, so the panic is unreachable, but the invariant
   lives in a pair of closures rather than in a type or a Result.
   A future relevance predicate that admits, say, InactiveNode
   members would turn a save into a server crash. Returning Err
   would cost little.
** Architecture note: "drawn in place of another node" is override-specific data
   Override substitution (TODO/full-schema/
   11_override-rendering-and-navigation.org, executed 2026-06-11)
   lets a viewnode stand in for a different graph node than the one
   whose ID it carries: the buffer draws overrider R where the
   container's contains list says N. The representation is
   'ViewNodeStats.overridesHere : Option<ID>' (and a parallel field
   on 'InactiveNode', for substitutes a source-set switch retains),
   carrying the original's ID -- data, not the earlier-planned
   bool-plus-convention, which was rejected 2026-06-11 because
   inverting the resolution at save time is ambiguous when one
   overrider stands for two members of the same parent. Save
   extraction, the content distinctness validators and the content
   reconciler's orderkey all read it through 'collected_id ()'.

   The name is override-specific. If a second substitution-like
   feature ever appears (redirects, merge previews, etc.), do not
   add a second marker field beside it; generalize this one (e.g.
   rename to 'drawn_in_place_of', adding a reason enum if consumers
   need distinguishing). Deferred until that second consumer exists
   (Jeff, 2026-06-10).
** (fixed) 'rerender_view' ignored its 'create_partnerCols' flag
   Found 2026-06-11 during the heralds stage (the compiler's
   unused-variable warning exposed it). Stage 9's switch-rerender
   gave 'rerender_view' ('server/update_buffer.rs') a
   'create_partnerCols' parameter, documented as "true for the
   source-switch rerender, where pruning removed them and the new
   set decides which return" -- but the function hardcoded
   'create_partnerCols_for_fresh_nodes : false', dropping the flag.
   Consequence: after a source-set switch, PartnerCols that the
   convert-and-prune prepass removed were never re-created, so
   widening the set did NOT reveal cols as intended; the stage-9
   handler test did not assert col re-creation, so it passed.
   Fixed 2026-06-11 (one line: pass the flag through). The heralds
   stage's progress notes record the verification.
** Flaky test: 'ids::save' group in tests/git_diff_view.rs

   'test_delete_id_scaffolds_aborts') fails intermittently when the
