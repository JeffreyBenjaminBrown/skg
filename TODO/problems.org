** (mostly fixed) Save validation and extraction use different edit-request surfaces

* Performance

** 'fetch_all_graphnodestats' scales linearly in pids, ~60ms each
   server/dbs/typedb/search/all_graphnodestats.rs.

   The file header promises "one TypeDB query PER PID, all in parallel"
   and cites ~15ms per query. In practice the TypeDB server (or the
   driver) caps effective concurrency at roughly 4, so the total time
   is (pids * ~250ms) / 4. Small views finish in well under a second;
   medium-big views take multiple seconds.

   Measured on data/public/ (29k .skg files):
   - visit 'science' (root with 6 contents, ~10 pids in view): 0.68s total,
     with fetch_all_graphnodestats ~0.35s.
   - visit 'to watch' (root with 3 contents, ~95 pids in view): 5.5s total,
     with fetch_all_graphnodestats ~5.0s.

   Verified by forcing join_all -> a sequential await loop: per-query
   time was ~250ms, total scaled 1:1 with pid count ('science' 2.5s,
   'to watch' 23.5s). So join_all IS achieving parallelism; there is
   just a ~4x ceiling that eats the gains as views grow.

   Per-pid breakdown (emitted from temporary tracing in
   fetch_one_pid_stats):
   - tx.transaction(..) await  : ~8ms
   - tx.query(..) await returns: ~270ms avg (it already waits some)
   - first stream.next() await : +~4.5s (!) -- this is where to-watch
     spends its time; most pids wait in a TypeDB-side queue before
     their stream yields a document
   - draining the stream       : negligible (single-row result)

   The code comment "1 PID: 15ms, 24 PIDs: 4300ms" compared only the
   old N-way-disjunction approach against the new per-pid approach.
   The new approach is better than the old one but is still O(N), just
   with a smaller constant and an unknown parallelism cap.

   Options to try, in decreasing order of likely payoff:

   1. Batch many pids into a single TypeDB query. Something like

        match
          $node isa node, has id $id;
          { $id == "pid1" } or { $id == "pid2" } or ... ;
        fetch { "id": $id, "containers": [...], ... };

      returning one document per pid. Even if TypeDB evaluates the
      disjunction poorly for joins, a fetch that only *dispatches* on
      id should still be 1 query instead of N. Would remove the per-
      transaction and per-round-trip overhead entirely.

      TODO (Jeff speaking): We tried (1) once long ago. It was slower.

   2. Share one transaction across all pids (driver.transaction once,
      then run N queries on it). Might bypass whatever per-transaction
      bottleneck is capping concurrency at ~4.

   3. Split fetch_one_pid_stats into 6 separate relation-specific
      aggregate queries ("find all containers of nodes in {pids}",
      "find all contents of nodes in {pids}", etc) with the id set
      passed in as a TypeDB collection. Each query does one relation,
      across all pids, and folds results pid-by-pid in Rust.

   4. Cache. The same pid can appear in many views within a session;
      results are stable between writes. A small in-memory cache keyed
      by pid and invalidated on save would eliminate repeats.

   Until this is fixed, any view with more than a few dozen nodes
   (i.e. any real working set above initial_node_limit/2) will have
   multi-second first-render latency.

   WARNING: TODO.org refers to the headline title verbatim.

* Performance: the save-buffer pathway (see profiling-save.org)
  Profiling the save of an insert-a-child edit (node 846207ef, 'doing')
  found four inefficiencies, in impact order. Full evidence, numbers,
  and recommendations are in the top-level 'profiling-save.org'.

** Override validation re-scans the whole graph on every save (~24%)
   'validate_override_invariants_after_save' (server/save.rs), called
   unconditionally from 'update_graph_including_merges', runs
   'validate_override_invariants' (server/dbs/in_rust_graph/
   override_invariants.rs), which iterates 'graph.nodes' twice over all
   ~60k nodes. The cost is O(total graph), independent of the save, so
   every save slows as the user's data grows. Disabling it cut the save
   from 343 ms to 256 ms. Fix: validate only the saved nodes and their
   'overrides_view_of' neighborhood.

** Tantivy is written and committed twice per save (~38%)
   'update_tantivy_from_saveinstructions' (server/save.rs) indexes each
   saved node with 'context_origin_type = ""'; then
   'update_context_origin_types' (server/dbs/tantivy/context_update.rs)
   opens a second 50 MB IndexWriter, reads those docs back, deletes and
   re-adds them with the computed type, and commits again. Skipping the
   second pass cut the context phase from 71 ms to 4 ms. The file's own
   header comment names the fix ("compute origin types before
   indexing"): compute the type first, index once, commit once, and
   reuse one long-lived IndexWriter.

** TypeDB relationship update is delete-all-then-recreate-all (~15%, O(children))
   'update_typedb_from_saveinstructions' (server/save.rs) deletes and

   saving an N-child node is O(N) and building one up is O(N²).
   'create_all_relationships' grew from 48 ms (~25 children) to 137 ms
   (~100). Fix: diff old vs new edges (the in-Rust snapshot has the old
   set) and apply only the delta.

** Save cost degrades 3-4x within a session of repeated saves
   404 ms -> 1220 ms over 11 consecutive saves; a 'rebuild dbs' resets
   it. Cause: each tiny Tantivy commit spawns a segment and merges fall
   behind. Mostly fixed by the single-commit change above; also consider
   debouncing commits (Tantivy is a rebuildable cache) and dropping the
   forced per-commit 'reader.reload()' in production.

   The two ORDER-MEANINGFUL lists do not. 'content_members'
   (traverse.rs:373-389) pushes each non-phantom InactiveNode's
   'collected_id' into the parent's 'SetContains'; 'subscribeeCol_members'
   (traverse.rs:353-366) does the same into the owner's 'SetSubscribesTo'.
   So a RETAINED inactive node (one kept in the view because it has
   visible descendants) becomes a positional member of its
   container's/owner's saved list, drawn from the buffer. The comments
   justify this ("subscribee order matters, so a buffer-present
   InactiveNode counts as a positional member"), and weave.rs's
   "POSITIONED ... includes retained InactiveNodes" was co-designed with
   it -- so this is deliberate drift from the metaplan model, not an
   oversight. But it is both unnecessary and unsafe:

   - Unnecessary: the weave already restores an omitted invisible
     member to its correct position by anchoring it to its nearest
     preceding positioned disk member. With disk [A, X, B] (X inactive)
     and buffer [A, B], the weave yields [A, X, B] whether or not X is
     in the buffer list. So excluding X loses no position in the common
     case.

   - Unsafe: because the buffer list is fed to the weave as authority,
     including X lets a stale or concurrently-edited buffer RESURRECT an
     invisible member that was authoritatively removed. Concretely: disk
     P.contains = [A, B] (another save removed X); a stale buffer still
     shows [A, X, B] with X retained inactive; saving P (no user edit)
     extracts SetContains [A, X, B]; the weave pushes every buffer
     member, so X reappears in P.contains on disk. The noop filter does
     not catch it (buffer != disk), and write-suppression does not catch
     it (P's source is active, so P's instruction stands). Symmetrically,
     reordering a read-only inactive placeholder among visible siblings
     persists as a real write to the container -- editing the position of
     something the user cannot edit.

   This is the inbound twin of the original "silent deletion" bug: that
   one dropped invisible members by omission; this one re-adds them by
   inclusion. The fix is to make 'content_members' and
   'subscribeeCol_members' collect only Active members (mirroring
   'partnerCol_members'), and let the weave own invisible members'
   presence AND position. Tradeoff to confirm with Jeff: a retained
   inactive node then cannot be repositioned by the user (its post-save
   position is weave-determined), which is the correct read-only
   semantics but a visible behavior change. 'inactiveNode_is_phantom'
   gating and the InactiveNode 'collected_id' accessor become dead on the
   save path if this lands.

* Problems found while planning forks (2026-06-13)
  Found while reading the sharing code to write 'TODO/forks/plan.org'.
  The first is a real, fork-independent bug; it is also the prerequisite
  the fork/clone feature depends on, and is discussed at length in that

** The subscribee-as-such display subtracts only hides, not the subscriber's own contains
   'content_goal_list' (server/update_buffer/reconcile/content.rs:398-419)
   computes the content shown under a subscribee-as-such as
   'subscribee.contains MINUS subscriber.hides_from_its_subscriptions'.
   It does NOT subtract the subscriber's own 'contains'.

   Consequence (the move-into-graph integration path, vision.org
   example 2): 'resolve_visibility' (server/from_text/local_instruction_collection/resolve_visibility.rs:61-66)
   deliberately does NOT create a hide when the user moves subscribee
   content into the subscriber's own graph ("hiding one's own content
   would be silly"). So that content is in 'R.contains' but not in
   'R.hides', and the formula above still shows it under the
   subscribee-as-such. It therefore appears twice: once as R's
   integrated content, once as "unintegrated" subscribed content. This
   contradicts vision.org example 2 (which expects it to drop out of the
   subscribee view) and 'docs/sharing-model.md' para 2 ("an implicit
   hides_from_its_subscriptions relationship from the clone to all of
   its normal content ... any branch that shows up as ordinary
   integrated content does not need to also show up in the
   subscriptions").

   Test gap that hid it:
   'tests/hidden_from_subscriptions.rs::test_moving_foreign_subscribee_content_to_subscriber_does_not_hide'
   moves 'e1' into the subscriber and asserts only that (a) no hide is
   created and (b) 'e1' appears as content of 'r'. It never asserts 'e1'
   is removed from the subscribee-as-such branch, and its
   'contains("** (skg (node (id e1) ...")' check would even match a
   deeper '**** ' occurrence, so a duplicate passes unnoticed.

   This is also a hard blocker for forks: a fresh clone C has
   'C.contains = N.contains' and 'C.hides = {}', so the subscribee-as-
   such view of N under C would re-show every child C just integrated.

   Proposed fix (see 'TODO/forks/plan.org', "Prerequisite"): make
   'content_goal_list' for a subscribee-as-such subtract the
   subscriber's own 'contains' too --
   'subscribee.contains - subscriber.hides - subscriber.contains' --
   implementing 'sharing-model.md's "implicit hides to all normal
   content" without writing redundant data. Mirror the change anywhere
   else the subscribee-as-such membership is derived, and audit the
   HiddenIn/HiddenOutside goal lists ('goal_list.rs') so a node that is
   both integrated and explicitly hidden does not double-count.
   (If a filter I missed already excludes the subscriber's content
   downstream, this is moot -- but I could not find one.)

* Problems found during the org-export feature (2026-06-13)
** export marker bodies are not valid TOML
   The instruction node 3d9aa9be-... says a marker child's body "is
   interpreted as .toml", and both it and the one real marker
   (512ea9da, body =target_filepath = org_export_is_partial=) write
   the value UNQUOTED. That is not valid TOML: unlike YAML, TOML
   requires string values to be quoted, so =toml::from_str= would
   reject these bodies.

   Resolution (agreed with Jeff, TODO/export-to-org/discussion.org
   Q3): the exporter (server/export_org.rs 'parse_target_filepath')
   does a tolerant =key = value= parse instead of strict TOML -- the
   value may be bare or quoted, and is forward-compatible with quoted
   TOML. A bare value containing whitespace is refused with an
   explanatory message; absolute paths and '.'/'..' components are
   refused so a target cannot escape org-exports/. The instruction
   node's wording ("interpreted as .toml") is now slightly
   inaccurate; if it ever matters, either quote the values in the
   data or soften the wording to "a tolerant key = value line".
** heading-anchor links rely on heading-text uniqueness
   Deep links use org's =[[./file.org::*Heading][label]]= search,
   which matches the FIRST headline with that text. When two
   DISTINCT nodes in one exported file have the same title, links to
   the second resolve to the first. The exporter emits one warning
   per such collision, but org has no per-node anchor available
   without property drawers (which we omit, since exported files
   must not require org-roam). Headings are rendered as the title's
   plain-label form (links collapsed to labels), so a heading's text
   always equals the anchor used to link to it -- the earlier
   heading-vs-anchor divergence is gone.
** a literal bracket in a title can break a deep link
   Anchors and labels are interpolated verbatim into
   =[[*anchor][label]]=. A title containing a literal =]= (e.g.
   "Notes [draft]") yields a malformed org link. The textlink regex
   keeps labels free of =][=, but a title is arbitrary user text.
   Rare; not escaped. Robust fix would require org link-syntax
   escaping of anchors/labels.
