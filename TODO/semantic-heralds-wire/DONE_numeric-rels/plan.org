* Direction (Claude agrees)
Move relationship-herald PRESENTATION out of the server and into the
clients. The server sends SEMANTIC FACTS about a node's relationships;
each client decides letters, colors, order, layout. This fixes both of
Jeff's concerns:
- the raw =(skg ...)= metadata in the buffer becomes roughly meaningful
  to a human (no pre-styled color spans);
- the server stops dictating appearance, so a client can render these
  completely differently.

It is honestly cleaner than the styled-span wire I just shipped: those
spans existed only because the Emacs "lens" engine can't color a single
token per-character, which forced the server to pre-bake the colors.
Once the CLIENT owns rendering, that constraint disappears.

This is the THIRD wire shape for these heralds (frontloaded strings ->
styled spans -> semantic facts). The good news: the semantic facts are
the underlying truth, so this shape should finally be stable -- future
appearance tweaks won't touch the wire at all, only client code.

* The principle, stated once
- WIRE = facts. What relates to what, how many, which tracked ancestors,
  what is the reason-for-being. No colors, no letters, no order-as-meaning.
- CLIENT = presentation. The C/L/S/O/H letters, the a/b/c ancestor
  letters, the group colors, the birth/multi-contains highlights, the
  fixed order, the count-omission, the a(b,c) link form.
The server already COMPUTES all the facts (server/update_buffer/viewnodestats.rs:
=RelationCounts=, =AncestorFlags=, =birth_relations=, the edge-level
gating). That computation STAYS server-side -- it needs the graph and the
active source-set, which clients don't have. Only the styled-span
ASSEMBLY (server/herald_tokens.rs) and its client rendering change.

* Proposed wire format
Replace the current =(rels (COLOR "text") ...)= payload with per-relation
semantic sub-forms. A relation appears only if it has something to say;
a side (in/out) appears only if its count > 0; a sub-list appears only if
non-empty (Jeff's "empty-except-for-its-key => absent" rule).

** Grammar
#+begin_example
(rels
  (contains     (in  COUNT (ancestors GEN...))
                (out COUNT (ancestors GEN...)))
  (subscribes   (in ...) (out ...))
  (overrides    (in ...) (out ...))
  (hides        (in ...) (out ...))
  (textlinksTo  (in  COUNT (surprising B) (withContent C))   ; no ancestors on in
                (out (ancestors GEN...)))                     ; no count on out
  (aliases  K)      ; the old A token
  (extraIds K)      ; the old I token
  (birth RELNAME...))  ; the reason-for-being relation(s)
#+end_example
- =in= = "N nodes RELATION it" (e.g. contains-in = nodes that contain it).
- =out= = "it RELATIONs N nodes".
- =COUNT= is the whole member count on that side (always present when the
  side is present -- it is a fact; the client omits it when it equals the
  number of ancestors).
- =(ancestors GEN...)= lists the tracked ancestors that are members on
  that side, by generation distance (1 = visible parent, 2 = grandparent,
  ...). The client maps generation -> letter (1->a, 2->b, ...). See Q1.
- =textlinksTo= is the odd one, matching today's =a(b,c)L=: its IN side
  never carries ancestors (only the surprising/with-content sub-split of
  the link total); its OUT side is only the linkSource-birth case and
  carries ancestors but no count.
- =(birth RELNAME...)= names the relation(s) the node was drawn for
  (usually =contains=; =[hides contains]= for a HiddenInSubscribee
  member). See Q2.

** Worked examples (each reproduces a real current render)
- =aC= (parent contains it), birth contains:
  =(rels (contains (in 1 (ancestors 1))) (birth contains))=
  -> client: count 1 == 1 ancestor so omit count; "a" (yellow), "C"
     (white, birth). Renders "aC".
- =2aC1= (2 containers incl. parent; contains 1), birth contains:
  =(rels (contains (in 2 (ancestors 1)) (out 1)) (birth contains))=
  -> "2" orange (multi-contains, in-count>1 on contains), "a" yellow,
     "C1" white (birth). Renders "2aC1".
- =bS= (grandparent subscribes to it), birth subscribes:
  =(rels (subscribes (in 1 (ancestors 2))) (birth subscribes))=
  -> "b" yellow, "S" white. Renders "bS".
- =2(1,1)L= (surprising links), no birth:
  =(rels (textlinksTo (in 2 (surprising 1) (withContent 1))))=
- =La= (links to its parent), birth textlinksTo:
  =(rels (textlinksTo (out (ancestors 1))) (birth textlinksTo))=

* DONE problems I anticipate
** DONE Multi-client duplication is the real cost
The full rendering (order, letters, colors, omission, link form) now
lives in EVERY client and must be kept consistent unless you WANT
divergence. This is exactly what the served rule table was built to
avoid. Two clients today (Emacs, nvim), a vim one planned -- so the logic
gets written ~3 times. Accepting this is the price of the principle; it
is the single biggest downside. (The lens engine is already duplicated
per client, so there's precedent.)
** DONE Scope: just rels, or the whole rule table?
This makes RELS presentation client-owned, but the OTHER heralds -- the
col-description TEXT ("It subscribes to these."), the ☮/⊥ glyphs and
their colors, cycle, sourceHerald, diff axes -- still come from the
SERVER-SERVED rule table. So presentation ends up SPLIT: rels client-side,
everything else server-side. That is inconsistent with "the server is
agnostic about display." Fully realizing the principle means moving the
whole rule table client-side (each client ships its own default; the
server stops serving one) -- a much larger change. Decide: (a) do rels
now, leave the rest server-driven; or (b) go all-in.

** DONE Birth is a per-node fact that doesn't fit "each rel has ins/outs"
You said each rel knows only its ins and outs, but the reason-for-being
is a whole-node fact (which relation the node was drawn FOR), and the
client can't easily re-derive it (it depends on col type, backpath role,
and overridesHere -- server-only context). So I propose a rels-level
=(birth RELNAME...)= sibling, not something inside a rel. See Q2.
** DONE Ancestors as generation NUMBERS vs LETTERS
Your example used =(ancestors a)= (letters), which is friendlier to read
but bakes the a/b/c convention -- a presentation choice -- into the wire,
against goal #2 (a different client might show "P/G/GG" or a count).
Generation NUMBERS =(ancestors 1 2)= are fully agnostic but a hair less
human-friendly in the raw buffer. See Q1.
