* Motivation, and a little history

The save-buffer path is too complex.
Some of that is because I (Jeff) initially tried to minimize tree traversals,
and bundled many operations that are more easily understood separately.
Some of that has recently (since branching from main) been improved.
This plan after this paragraph was written by Codex, <2026-05-14 Thu>,
to describe other opportunities for simplification.

* Simplify the save-buffer path after direct subscribee visibility

The recent role-tree work fixed the direct =AsSubscribee= hazard: role
context now travels with each view-tree occurrence, and a subscribee
shown in subscribee position no longer globally suppresses ordinary
=contains= edits for the same graph ID elsewhere.

That leaves the save path much better, but still transitional. The
remaining opportunities are mostly about making each phase say what it
means directly, instead of briefly encoding one interpretation and then
correcting it later.

Desired shape:

#+begin_src text
checked view tree
  -> role-annotated tree
  -> graph edit intents + subscriber visibility intents
  -> reconciled owned-node intents
  -> DefineNode instructions
  -> disk supplementation / source moves
  -> unchanged filtering
#+end_src

* Ranked opportunities

** a. Split 'viewforest_to_nonmerge_save_instructions' into named phases

This is the highest-priority cleanup. The function currently performs
role annotation, visibility extraction, graph-intent extraction,
same-ID reconciliation, conversion to =DefineNode=, disk
supplementation, source-move detection, direct-subscribee =contains=
restoration, visibility folding, and unchanged filtering.

The behavior is now covered by tests, so the next pass should be a
behavior-preserving extraction into named helpers. The top-level
function should read like a save pipeline, with policy details pushed
behind names that describe the phase.

This is not primarily about shortening a function. It is about making
the order of save policies explicit enough that the next sharing
change does not have to rediscover the whole pipeline.

** b. Simplify 'build_disk_supplemented_define_nodes'

Item a should leave behind a helper shaped roughly like
'build_disk_supplemented_define_nodes'. That helper will be an
improvement over the current top-level tangle, but it should not
become the new place where unrelated save policies hide.

The next cleanup should make that helper boring and locally readable.
It should separate these concerns behind names:

- convert a reconciled 'NodeEditIntent' into a preliminary
  'DefineNode';
- pass delete instructions through without disk supplementation;
- load the disk node for saves;
- canonicalize IDs from disk;
- detect source moves;
- supplement unspecified fields from disk;
- apply the temporary direct-subscribee =contains= restoration
  guardrail;
- collect 'SourceMove' values.

This can still be behavior-preserving. The point is not yet to remove
the direct-subscribee guardrail or move visibility folding; that comes
later. The point is to make the helper's internal sequence explicit
enough that the later policy changes have one obvious edit point.

Acceptance criterion: after this item, both
'viewforest_to_nonmerge_save_instructions' and
'build_disk_supplemented_define_nodes' should read as short named
pipelines, and the only remaining obviously policy-specific wrinkle in
the disk phase should be a clearly named temporary helper for restoring
direct-as-subscribee 'contains' from disk.

** c. PARTLY DONE Move subscribee visibility folding before final =DefineNode= construction

This is tied with item a for importance.

The current representation used to do this:

#+begin_src text
direct AsSubscribee child list
  -> apparent subscribee.contains edit
  -> restore subscribee.contains from disk
  -> separately infer subscriber hide/unhide intent
#+end_src

The cleaner target is:

#+begin_src text
direct AsSubscribee child list
  -> subscriber visibility intent
  -> fold into the subscriber's node intent
  -> never becomes a graph contains edit for the subscribee
#+end_src

That would remove the need for the direct-subscribee =contains=
guardrail during disk supplementation. It would also make visibility
edits less sensitive to the fact that they currently run after graph
=SaveNode= construction.

Status after the first item-c pass: visibility folding now happens at
the =NodeEditIntent= layer, before =DefineNode= construction and before
disk supplementation. That removed the late mutation of completed
=DefineNode= values.

What remains: a direct =AsSubscribee= node still carries its displayed
child list as a temporary =NodeSaveIntent.contains= value. Because that
still looks like a graph =contains= edit until later, disk
supplementation must still call the temporary guardrail that restores
the subscribee's =contains= from disk.

So item c is not complete until direct-as-subscribee child lists never
enter the graph-save intent as =contains= in the first place. That
remaining work should be handled by item d's explicit intent shape.

** d. Replace =children_affect_only_hiding= with an explicit intent shape

The boolean is a useful bridge, but it makes =NodeSaveIntent= carry a
role-position policy as if it were a graph-save property.

Prefer a representation that cannot accidentally treat direct
subscribee children as graph =contains=:

- separate variants such as =NodeEditIntent::GraphSave= and
  =NodeEditIntent::SubscribeeVisibilitySurface=; or
- a small enum such as
  =DirectChildrenMeaning::{GraphContains, SubscriberVisibility}=.

If item c lands first, separate variants will probably be clearer than
an enum field.

After the partial item-c pass, this item is the next required cleanup.
It should split the direct =AsSubscribee= branch out before graph-save
=contains= is built:

- ordinary definitive nodes should produce graph-save intents whose
  direct children mean graph =contains=;
- direct =AsSubscribee= nodes should produce visibility-surface
  intents whose direct children mean subscriber-visible subscribee
  content;
- recursive ordinary descendants below a direct subscribee should still
  be allowed to produce ordinary graph-save intents;
- a direct subscribee occurrence and an ordinary occurrence for the
  same graph ID should no longer reconcile as two ordinary-looking
  saves with a boolean distinction.

Acceptance criterion: after this item, there should be no need for a
direct-subscribee =contains= restoration guardrail during disk
supplementation, because no direct-subscribee visibility surface ever
becomes a subscribee graph =contains= edit.

** e. Reconcile visibility surfaces explicitly

=SubscribeeVisibilityIntent= already exists, but the save path still
needs clearer reconciliation rules once direct =AsSubscribee= branches
are no longer encoded as ordinary-looking saves.

After the partial item-c pass, visibility edits are folded into
=NodeEditIntent= before =DefineNode= construction. That is an
improvement, but visibility surfaces still are not reconciled as their
own intent class.

Once item d introduces an explicit visibility-surface intent, same-ID
reconciliation should consume graph edits and visibility surfaces as
distinct inputs:

- duplicate ordinary graph saves for the same ID still conflict;
- ordinary graph saves and direct-subscribee visibility surfaces for
  the same graph ID can coexist;
- duplicate visibility surfaces conflict only when they express
  incompatible visibility for the same subscriber/content pair;
- subscriber hide/unhide deltas should be folded into the owned
  subscriber's graph-save intent before final =DefineNode=
  construction.

This item should remove the need for visibility-specific special cases
inside ordinary graph-save reconciliation.

It should also replace the current =Vec<NodeEditIntent>=-as-database
pattern with an explicitly addressable reconciled collection. The code
now sometimes scans a vector looking for "the save intent for this
PID"; that is a smell because after same-ID reconciliation the intended
invariant is already at most one graph edit per PID. Prefer a
structure along these lines:

#+begin_src rust
struct ReconciledNodeEditIntents {
  order  : Vec<ID>,
  by_pid : HashMap<ID, NodeEditIntent>,
}
#+end_src

or an =IndexMap= if that dependency already exists or is justified.
The important properties are:

- lookup by PID is direct when folding subscriber hide/unhide deltas;
- output order remains stable enough for tests and review;
- visibility surfaces remain first-class until their deltas are folded
  into owned subscriber graph-save intents.

** f. Keep disk supplementation policy-neutral

Disk supplementation should restore unspecified fields and detect
source moves. It should not need to know that a node appeared as a
direct subscribee or that its =contains= must be restored from disk.

Visibility folding now happens before =DefineNode= construction, but
that alone was not enough to make disk supplementation policy-neutral.
Disk supplementation can become boring only after item d stops direct
=AsSubscribee= child lists from becoming graph-save =contains= values.

After that, the supplementation phase should do only this:

- canonicalize IDs from disk;
- detect source moves;
- supplement unspecified fields;
- pass the result onward.

Acceptance criterion: delete
='restore_subscribee_content_from_disk_if_needed' or its equivalent.
No disk-supplementation helper should inspect direct-subscribee role
policy, =children_affect_only_hiding=, or any replacement field.

That is an important simplification because supplementation is already
cross-cutting enough without also carrying view-position policy.

** g. Tighten same-ID reconciliation around intent variants

The current same-ID reconciliation now happens at the
=NodeEditIntent= layer, which is the right direction. After the intent
shape is split, the rules can be stated more directly:

- ordinary graph save wins over a direct-subscribee visibility surface
  for the same graph ID;
- duplicate ordinary graph saves conflict;
- save/delete conflicts reject;
- duplicate visibility surfaces conflict only when they express
  incompatible visibility for the same subscriber/content pair.

This should preserve the existing regression where the same ID appears
both in a direct subscribee position and as an ordinary definitive
node, and the ordinary node's =contains= edit survives.

** h. Revisit validation once save roles are stable

Keep pre-save validation unchanged for now. That boundary was useful
for making the direct-subscribee fix narrow.

Later, consider whether validation should consume the role-annotated
tree too. That would let foreign-node validation, display-only
traversal, and direct-subscribee visibility policy all use the same
positional facts. This is lower priority because moving validation too
early risks mixing this cleanup with a larger behavior change.

** i. Document the remaining rerender/completion boundary

The save extraction path is now clearer than the rerender path that
feeds it. The remaining completion contract should be written down
before broadening edit behavior inside subscribee branches:

- saved direct =AsSubscribee= branches must preserve user-authored
  child-list signal long enough for save extraction;
- generated display scaffolds must not become graph edits;
- diff-mode phantom or removed children must not accidentally infer
  hide/unhide intent.

This is not a blocker for the save-path cleanup above, but it is the
next place where inappropriate bundling can reappear.

* Test expectations

Each simplification should keep the current direct-subscribee behavior
covered:

- child removal under a direct subscribee infers subscriber hide;
- child restoration under a direct subscribee infers subscriber unhide;
- child-list edits in subscribee position do not become foreign
  subscribee =contains= edits;
- foreign title/body edits still report =ModifiedForeignNode=;
- overlapping subscribee visibility conflicts reject save;
- an ordinary definitive occurrence of the same graph ID still saves
  its own =contains= edit.

Add one focused regression with each structural change rather than
waiting for a large end-to-end rewrite.

* Assumptions

- No TCP API, Emacs protocol, TypeDB schema, or =.skg= format change.
- Start with save extraction only; broader rerender and validation
  role annotation can follow once this pipeline is simpler.
- Prefer extra clear tree traversals over tightly coupled one-pass
  extraction.
- Preserve current user-visible save behavior while changing internal
  representation.
