* What remains after direct as-subscribee hide/unhide

This report reviews all current files in =TODO/sharing/= and the
branch diff from =main= through commit =b3191269=. The recent branch
does the important feature slice: direct definitive AsSubscribee
child-list edits now infer subscriber hide/unhide relations, the
saved-view rerender preserves the edited subscribee branch long
enough for save-intent extraction, and hidden-in columns are inserted
after the new hides land.

The items below are what still looks left on the table.

* Highest priority

** Resolve the diff-mode preservation TODO

Current code:

#+end_src

The key predicate is:

with the comment:

This means SubscribeeCol branch preservation is explicitly disabled
when =source_diffs= is present. That may be correct as a first
boundary, but it is not a settled invariant. The desired behavior
needs a focused decision:

- In normal saved views, direct AsSubscribee children are user-authored
  visibility signal and must not be regenerated before extraction.
- In diff views, phantom/removed children are generated display and
  should not be mistaken for hide/unhide intent.
- The TODO says the code should eventually distinguish those two facts
  directly, instead of using =source_diffs.is_none()= as a coarse
  escape hatch.

Suggested next step: add a diff-mode regression proving that phantoms
do not infer hides/unhides, then see whether preservation can remain
enabled for the non-phantom direct children in a saved diff view.

** Make fork/visibility edits first-class instead of late DefineNode mutation

The branch introduced a small internal =NodeEditIntent= layer, but it
is still converted immediately to =DefineNode=. Visibility edits are
then folded later by mutating completed SaveNodes in:

#+begin_src text
server/from_text/viewnodes_to_instructions.rs:139
apply_hiderels_from_subscribee_visibility
#+end_src

This works, but it preserves the architectural debt called out in
=0p5_DONE_simplify-save.org=: fork edits are still not first-class. The
late helper must load disk nodes, inspect current completed
instructions, synthesize a subscriber SaveNode if missing, and then
update =hides_from_its_subscriptions= directly.

This is the main reason the guardrail above still exists.

Suggested next step: introduce a real save-intermediate shape, e.g.
=GraphNodeIntent= plus =SubscriberVisibilityIntent=, fold visibility
into owned subscriber node intents, and only then construct
=DefineNode= values.

** Define the conflict semantics for overlapping subscribees

The current algorithm applies each
=SubscribeeVisibilityIntent= sequentially. It computes inferred
unhides from the subscriber's disk hide list, and inferred hides from
the current visible list plus projected subscriber content.

This leaves an edge case untested: the same content ID can be direct
content of multiple subscribees for the same subscriber. One edited
AsSubscribee branch might imply "hide X", while another visible branch
might imply "unhide X". Since the hide relation is subscriber-wide,
not per-subscribee, that conflict needs a defined policy.

Possible policies:

- "visible anywhere wins": if any expanded subscribee branch shows X,
  remove X from the subscriber's hides.
- "absence anywhere wins": if any expanded subscribee branch omits X,
  add X to the subscriber's hides.
- "conflicting visibility edits are invalid": report a save error.

The current implementation may be order-sensitive for same-save
hide/unhide conflicts because unhide detection consults disk hides,
not the current accumulated hide list.

Suggested next step: add a fixture with two subscribees sharing one
content node, then test the chosen policy.

* Semantics still intentionally narrow

** Direct only; recursive subscribee content is not a visibility surface

The implemented visibility intent reads only direct children under a
definitive AsSubscribee node. That matches the current plan, but it is
still a user-facing constraint:

- deleting a direct child from a subscribee branch can hide it;
- replacing a direct hidden child can unhide it;
- editing deeper recursive descendants under that subscribee remains
  ordinary graph editing for those descendant nodes, not subscriber
  visibility editing.

This should probably be documented in user-facing terms before the
feature is presented as complete.

** HiddenIn/HiddenOutside columns are still not directly editable

The branch enables an unhide gesture by moving a hidden item out of
=HiddenInSubscribeeCol= into visible direct subscribee content. It
does not make HiddenInSubscribeeCol or HiddenOutsideOfSubscribeeCol
themselves general editable relation columns.

That is consistent with the current plan, but leaves open UX
questions:

- Should deleting a row from HiddenInSubscribeeCol mean unhide?
- Should adding a row to HiddenOutsideOfSubscribeeCol mean hide from
  all subscriptions?
- Should these scaffolds advertise that they are display-derived, not
  normal editable relation columns?

For now, direct AsSubscribee visible content is the only hide/unhide
edit surface.

** Unknown/extra buffer children under AsSubscribee need an explicit policy

=0p5_DONE_simplify-save.org= says buffer children under a direct
AsSubscribee branch that are not disk children of the subscribee
should be rejected, ignored, or normalized by explicit policy, but
never saved as =subscribee.contains=.

The branch satisfies the "never saved as subscribee.contains" part via
the guardrail, but it does not appear to expose an explicit validation
or warning policy for extra direct children. Those extra children may
still be meaningful as ordinary graph nodes depending on their role
and source, but they are not subscribee visibility edits.

Suggested next step: add tests for inserting a new child under a
foreign direct AsSubscribee branch and under an owned direct
AsSubscribee branch. Decide whether the save should reject, ignore,
or normalize that edit.

** Owned AsSubscribee title/body edits remain a policy choice

The plan says real edits to an owned subscribee should happen when the
node is viewed directly, not as a subscribee, and Jeff agreed. The
current tests guard that foreign title edits still reach foreign-node
validation. I did not see an equivalent test that owned direct
AsSubscribee title/body edits are rejected or normalized according to
that policy.

Suggested next step: add one test for owned direct AsSubscribee
title/body edits and either enforce the policy or document that owned
metadata edits are still allowed there.

* Test coverage still worth adding

** Diff-mode phantom regression

The planning docs repeatedly call this out. Phantoms should not infer
hides or unhides. The current implementation filters phantom direct
children out of visible-content collection, but without a diff-mode
test this is easy to regress or misunderstand.

** Persistence assertions for hide/unhide

The end-to-end tests primarily assert returned buffers. The
save-instruction tests assert =hides_from_its_subscriptions= before
the write path, which is useful. It may still be worth adding one
integration-style assertion against the temp fixture's saved =.skg=
file after a hide and after an unhide. That separates "rerender showed
the right thing" from "filesystem persistence wrote the right field."

** Multiple subscriber views / duplicate-definition conflict

The docs note that multiple appearances of the same subscriber in one
saved buffer could express conflicting visibility edits. Existing
duplicate-definition validation probably prevents the worst cases,
but there is no focused sharing test proving the failure mode is
clean.

** Collateral view behavior

The preservation rule is saved-view-only. Collateral views are
expected to refresh from the graph. A regression that opens/saves one
view and verifies a collateral subscriber view updates hidden content
would guard that boundary.

* Broader product work still open

** Starting a subscription still needs a command

If a node has no subscriptions, no =SubscribeeCol= is automatically
present. The user can still type the scaffold by hand, but
=status-before-plan.org= correctly calls this tedious and
error-prone. A command should insert a SubscribeeCol and probably help
choose/insert the first subscribee.

** Overrides remain data-layer-only

=overrides_view_of= is still plumbed through the data model but has no
real view/edit surface. The herald flag is not enough to tell which
side of the relation a node is on, much less edit it. This is outside
the direct subscribee hide/unhide feature, but still a sharing-model
gap.

** Relationship columns for all directions remain aspirational

=vision.org= still lists "cols for all rels in either direction" and
"How to show a subscribee's subscribees?" The current work improves
one relation surface: subscriber visibility over direct subscribee
content. It does not generalize relation display/editing.

** User-facing language is not settled

The code and docs now use several related phrases:

- subscribee;
- shown as a subscribee;
- direct AsSubscribee branch;
- visibility intent;
- hiderel;
- HiddenInSubscribeeCol.

The implementation distinction is important, but a user command or
manual should probably describe the action as something like "hide
this subscribed item from this view" / "show this subscribed item
again", not as "edit AsSubscribee direct child lists."

* Probably done; do not reopen without new evidence

The following older plan items appear to be complete enough for this
feature slice:

- source-move detection is split out of disk supplementation;
- role classification exists and save extraction uses it;
- naive save extraction is split into candidates, minimal node data,
  field collection, and intent conversion;
- direct AsSubscribee child-list removals no longer become
  subscribee graph =contains= edits;
- raw direct AsSubscribee visibility intents are extracted before the
  guardrail restores =contains= from disk;
- direct hide and unhide inference both have save-instruction tests;
- saved-view SubscribeeCol branch preservation is implemented for
  normal non-diff saved views;
- saved-view rerender inserts =HiddenInSubscribeeCol= when newly
  inferred hides require it;
- the TypeDB cleanup problem in =hidden_from_subscriptions= has a
  shared guarded cleanup path.

These may still be refactored as part of the architectural cleanups
above, but they are not missing behavior for the narrow direct
hide/unhide feature.

* Suggested next sequence

1. Replace the stale comments in =server/to_org/complete/sharing/mod.rs=.
2. Add the overlapping-subscribees conflict test and decide the policy.
3. Add the diff-mode phantom regression and resolve the
   =source_diffs.is_none()= preservation TODO.
4. Add explicit tests for extra direct children under AsSubscribee and
   owned AsSubscribee title/body edits.
5. Refactor the late =DefineNode= mutation/contains-restoration
   guardrail into a first-class visibility/fork intent stage.
6. Only after that, move on to UI commands: start subscribing,
   user-facing hide/unhide gestures, and eventually overrides.
