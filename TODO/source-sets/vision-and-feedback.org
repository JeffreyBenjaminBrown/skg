* vision : source sets
I would like to implement a new language element for skgconfig.toml,
'source-set'. A user can have any number of source sets,
name each set however they want,
and change the 'active source-set' to restrict search and display.

This should not chnage Skg's dbs -- e.g. typedb does not reload big graph chunks
just because someone switched the active source-set.

Making search results consult only the relevant sources is straightforward.
Changing display is less obvious.

For this first cut, heralds can still reflect the whole graph --
e.g. if a private node still shows in the count of link sources
targeting a public node, it doesn't matter --
but wherever we encode the adjacent machinery we should leave a
"PITFALL | TODO" comment explaining that they lie.
But then again it does seem easy to do so:
We would just restrict the sources used by the relevant TypeDB queries.
The SkgConfig probably already reaches them so we wouldn't even
need to thread more arguments through the call chain.

Beyond heralds, the bigger questions are which viewnodes to show,
and how to handle spaces where censored viewnodes would have been.
A lot of special cases will arise, but the basic principle should be:
- don't show information from inactive sources
- don't even represent absent nodes (e.g. no "phantoms" for them,
  the way deleted nodes show phantoms)
- yet (somehow?) permit new nodes inserted by a user
  to cause appropriate modifications to the node's full
  list of content

** Unclear contract for "active source-set"
*** Is it process-global, per Emacs connection, per buffer, or per request?
    Today =ViewsState= is per TCP connection and stores things like
    diff mode and open views. =SkgConfig= is static process state. If
    the active source-set is a mutable field in =SkgConfig=, changing it

    per-buffer may be necessary if users can keep a public view and a

** "Restrict TypeDB queries by source" is not quite the right model
   Source is an attribute of nodes/files. Relationships are ID-based and
   source-independent. There is no source on the =contains= relation, on
   =textlinks_to=, etc.

   That means display filtering has to define which endpoint controls
   visibility:

   - For content: probably show child C of parent P only if C's node
     source is active. But then active children of inactive parents have
     no visible containerward context.
   - For textlinks-in: probably show link source L only if L's source is
     active, regardless of the target's source.
   - For graph stats: counts can be filtered by source of the related
     node, but that is not the same thing as filtering the relation.

   The TypeDB queries can implement those endpoint-source rules, but the
   rule itself should be explicit before coding.

    source-set display should probably be read-only.
