  I traced the whole path, client to server. The relevant facts:

  - The three 'C-c r' commands are thin aliases
    ('elisp/skg-keymaps-and-aliases.el:24-29, 123-126') over
    'skg-request-*-view' ('elisp/skg-request-views.el'). Each inserts a
    '(viewRequests TOKEN)' atom into the node's metadata; the server
    fulfills the request during view completion.
    - 'skg-reveal-aliases' inserts '(viewRequests aliases)' and does
      NOT save. The server builds an 'AliasCol' scaffold, populated.
    - 'skg-reveal-containerward' / 'skg-reveal-sourceward' insert
      'containerwardView' / 'sourcewardView' AND auto-save.
  - 'ViewRequest' ('server/types/viewnode.rs:433-441') has exactly four
    variants: Aliases, Containerward, Sourceward, Definitive. There is
    NO request for any PartnerCol.
  - "Collections" in your sense = scaffold cols: the 8 PartnerCols plus
    AliasCol. "Paths" = the backpath engine
    ('server/to_org/expand/backpath.rs'), which climbs a relation
    transitively (until a fork/cycle/root) and grafts the partners as
    inverted, indefinitive, 'parentIs=independent' content children with
    a 'birth' provenance marker.
  - The backpath engine is ALREADY relation-generic: the relation is a
    3-string parameter '(relation, input_role, output_role)' and the
    in-Rust lookup ('server/dbs/typedb/search.rs:103-164') already
    dispatches all five relations in BOTH directions. So the path
    algorithm needs no new logic — only new call sites + plumbing
    (birth markers, request variants, commands).
  - 'C-c c' and 'C-c p' are both completely free prefixes today. Good.

  Two things in the spec are factually off (details in Problems):
  - Your 'C-c c o' is assigned to overrides AND hides AND subscribes.
  - "containment and link heralds are already defined in both
    directions" — links are NOT; only the node->parent direction
    ('←') exists.

** Q2. DANGEROUS: how a collection gets "put there" — request atom, NOT literal scaffold
   You wrote: "If it's missing, put it there -- even if it will be
   empty ... the view update subroutine of the save will complete it."
   Taken literally — insert an empty col headline, then save — this is
   unsafe for the two WRITABLE cols (overriddenCol, subscribeeCol):

   - Save EXTRACTION reads the buffer before completion runs. A present
     but empty 'overriddenCol' emits 'SetOverrides(vec![])'
     ('from_text/local_instruction_collection/traverse.rs:307-323'),
     which lowers to 'MSV::Specified(vec![])' ('.../lower.rs:268-273')
     — i.e. "set this node's overrides to the empty set". If the node
     HAD overrides on disk, saving the empty col WIPES them. The
     reconcile that would have repopulated the col runs in the rerender
     step, which is too late.
   - (Same hazard for subscribeeCol -> 'subscribes_to', and even
     aliasCol -> aliases.)

   The existing 'skg-reveal-aliases' is safe precisely because it does
   NOT insert a literal empty col — it inserts a '(viewRequests
   aliases)' atom, and the server builds the col already POPULATED from
   the graph. An empty result is then a true "this node has none",
   never a destructive overwrite.

   So I want to implement all collection commands the same way: insert a
   request atom; the server builds each col populated from the graph.
   That requires a new request mechanism for cols (none exists today).

   Recommend: implement collections as requests, mirroring aliases —
   never as hand-inserted empty scaffolds. (Mechanism in Q2a.)
   ANSWER: That's perfect.

*** Q2a. One parameterized request type, or many variants?
    Collections need ~4 new request kinds (or 6 col-targets), and Paths
    need ~9. Enumerating each as its own 'ViewRequest' variant means
    ~13 new variants, each with a 'REPRS_IN_CLIENT' entry and a herald
    'req:' rule. Since the backpath engine and the col reconcilers are
    already relation-generic, I'd rather add TWO parameterized requests,
    e.g. (sketch)
    : ViewRequest::Col  ( PartnerCol-or-QualCol )

    '(viewRequests (path overrides overrider))'. The elisp side still
    gets 13 nicely-named commands for discoverability/keybindings; only
    the wire atom is parameterized. This keeps 'ViewRequest', the herald
    table, and the parser from sprouting 13 near-identical entries.

    Recommend: two parameterized requests (Col, Path), with named elisp
    commands on top.
    ANSWER: That seems good. But rather than direction, let's use role name.

** Q3. BLOCKER: "even if empty" cannot hold for the read-only (inbound) cols
   Of the cols your commands create, only THREE survive a completion
   pass while empty: 'overriddenCol', 'subscribeeCol', and 'aliasCol'.
   The other five — 'overriderCol', 'subscriberCol', 'hiderCol',
   'hiddenCol', and the two filter cols — are pruned when empty by
   'is_self_deletable_when_empty' ('server/update_buffer/complete.rs:325-337').

   Concretely, of your two-col commands:
   - overrides: overriddenCol persists empty; overriderCol VANISHES empty.
   - subscribes: subscribeeCol persists empty; subscriberCol VANISHES empty.
   - hides: BOTH hiderCol and hiddenCol VANISH empty (hides has no
     writable side at all).

   So "put it there even if empty" is currently impossible for the
   read-only halves. There's also a real question of whether you'd even
   want it: a read-only col is informational and not user-extendable
   (inserting a node into it just relabels it 'independent'), so an
   empty 'overriderCol' only ever says "nobody overrides this" — which
   is arguably better conveyed by its absence.

   Options:
   - (A) Accept the asymmetry: empty read-only cols simply don't appear;
     the command shows them only when populated. Empty WRITABLE cols
     (overriddenCol/subscribeeCol) and aliasCol do persist, giving you
     the editable "add here" surface. Zero code change to the prune rule.
   - (B) Make a *requested* col sticky: carry a flag so a col the user
     explicitly asked for survives empty even when read-only. Costs a
     new bit on the col + a prune-rule exception, and an empty read-only
     col that does nothing.

   Recommend: (A). It's safe, needs no prune change, and matches what
   the cols mean.
   ANSWER: Sure, (A).

* The paths family

** Q5. Transitive chains vs. a single step
   Today's containerward/sourceward paths climb transitively — N's
   container, its container, and so on until a fork/cycle/root
   ('server/dbs/typedb/paths.rs:203-247'). Pointed at the sharing
   relations, the same engine yields transitive chains. For some of
   these that's degenerate or odd:
   - overrides: user-owned overrides are monogamous + acyclic, so an
     owned override chain is one step anyway (foreign chains can be
     longer).
   - hides: chaining "hiders of hiders of N" is semantically thin —
     hiding is really about subscriptions, not a chainable ancestry.

   Recommend: keep transitive (it's free — same engine — and harmless,
   since one-step relations just produce one-step paths). I'll only
   special-case if you disagree.
   ANSWER: Let's keep it uniform, just to reduce the user's cognitive load.
   (Maybe noteworthy: I will soon implement user-owned overrides chains.)

** Q7. Provenance markers for the seven new path directions
   Each grafted path node currently carries a 'birth' marker recording
   how it was born ('Birth::ContainsParent' -> "}",
   'Birth::LinksToParent' -> "←"). 'Birth' has only those two
   relation-bearing variants ('server/types/viewnode.rs:36-40'), and the
   save-side 'validate_parentIs_relationships' checks birth-vs-relation
   consistency. So each new path direction needs *some* provenance, and
   there's a design fork:
   - (A) Add a 'Birth' variant + herald per new direction (~7 of them):
     verbose but uniform with how containers/links work today.
   - (B) Add one generic 'Birth::Backpath' (provenance = "grafted
     ancestry") and let the per-direction *relationship* be carried by
     the parent-relationship heralds from Q8 (Op/pO/Sp/pS/Hp/pH +
     reverse-link). Less herald sprawl; needs the validation to accept a
     generic backpath birth.

   off the Q8 heralds. It also nudges us toward unifying the three
   different ways "relationship to org-parent" is encoded today (see

** Q8. BLOCKER-ish: the parent-relationship herald matrix is incomplete
   You want Op/pO, Sp/pS, Hp/pH, and you suspected not all exist.
   Correct — here's the true state across all 5 relations x 2
   directions (node->org-parent vs org-parent->node):

   | relation   | node -> parent          | parent -> node                      |
   | contains   | "}" containsParent  OK  | "{" parentIsContainer  OK           |
   | textlinks  | "←" linksToParent   OK  | MISSING (no "parent links to node") |
   | overrides  | "Op" overridesParent OK | "pO" parentOverrides — see caveat   |
   | subscribes | "Sp" MISSING            | "pS" MISSING                        |
   | hides      | "Hp" MISSING            | "pH" MISSING                        |

   Caveat on pO: the atom + herald rule exist, but it is produced ONLY
   by the fork-confirmation buffer ('server/from_text/fork.rs:135-137'),
   is NOT computed in the general viewStats pass, and is NOT even
   parseable ('parse_metadata_sexp.rs' has no arm for it — it'd hit the
   error branch on round-trip). So for the 'paths-through-overridden'
   view it effectively does not work yet.

   Each grafted path node lands the matching herald: e.g.
   'paths-through-overriders' children get "Op", 'paths-through-overridden'
   children get "pO", subscribers->"Sp", subscribees->"pS",
   hiders->"Hp", hidden->"pH", and 'paths-through-link-targets' children
   need the MISSING reverse-link herald.

   So to make the new path family readable I need to add five new
   heralds (Sp, pS, Hp, pH, reverse-link) and properly wire pO
   (general computation + parser arm). The graph layer already supports
   every direction ('relation_membership_is_real' +
   'NodeRelation::{Subscribes,HidesFromItsSubscriptions}'), so each is a
   mechanical 6-7 site change (field, computation, serializer, herald
   rule, atom list, conformance fixture, and — unlike today's pO —
   a parser arm).

   Recommend: add Sp, pS, Hp, pH, and reverse-link as red, parent-
   relative viewStats heralds (same shape as Op); fix pO to be computed
   generally AND parseable. Glyph for reverse-link: "→" (mirror of "←").
   ANSWER (confirm the set, and the "→" glyph): Yes.

** Q9. Hp/pH semantics — hiding is ternary
   'hides_from_its_subscriptions' is really "subscriber X hides node Y
   *from X's subscriptions*". A bare parent-relative "this node hides its
   org-parent" (Hp) drops the "from which subscription" context. The
   edge IS stored binary, so the herald is computable, but it asserts
   less than the col view ('hiderCol'/'hiddenCol') does.

   Recommend: emit Hp/pH as plain binary heralds (node has an outbound/
   inbound hide edge to/from its org-parent), matching how the other
   parent-heralds work, and rely on the existing hide cols for the full
   subscription context. Flag if you want hide heralds suppressed
   outside subscription contexts instead.
   ANSWER: Yes.

** DANGEROUS — "put an empty col there, save completes it" can wipe data
   The literal reading of the collections spec wipes 'overrides_view_of'
   / 'subscribes_to' / aliases when an empty writable col is saved

   request-atom design avoids it. This is the one thing in the spec I'd

** INCORRECT — three statements in the spec don't hold
   - 'C-c c o' is triple-booked (overrides/hides/subscribes) (Q1).
   - "containment and link heralds are already defined in both
     directions" — the reverse-LINK direction is missing; only "←"
     exists (Q8). (Containment really is both-directional.)
   - Not your spec, but adjacent: 'docs/COMMANDS.org:254-256' describes
     'sourceward' backwards (says "nodes the node links to"; the code
     surfaces nodes that link to it).

** AWKWARD — "aliases" is a collection but not a PartnerCol
   AliasCol is a 'QualCol', not one of the 8 PartnerCols, and aliases
   alone behaves differently (no auto-save; QualCol vs PartnerCol prune
   rules). Bundling it under 'skg-show-collection-*' is fine for the
   user, but the implementation can't treat all "collections"
   uniformly — the Col request (Q2a) must span QualCol + PartnerCol.

** AWKWARD — one command, two cols of opposite editability
   See Q4. Bundling a writable col with a read-only one under a single
   verb ("show collection") risks implying symmetry that isn't there,
   especially for 'hides' (no writable side).

** MISSING ABSTRACTION — "relationship to visible org-parent" is encoded three ways
   Provenance/relationship to the org-parent is currently expressed as:
   (a) a 'birth' marker (containment, links), (b) a viewStats parent-
   herald (overrides: Op/pO), and (c) for containment, BOTH at once
   ("}" appears as birth and as viewStats). Extending to subscribes/
   hides invites picking one. A single concept — "this node's relation
   to its visible org-parent, with role/direction" — computed in one
   place ('set_relation_relative_stats') and rendered by one herald
   family, would cover all 5 relations x 2 directions uniformly and
   retire the birth/viewStats split. Recommended by Q7(B) + Q8.

** MISSING ABSTRACTION — no parameterized Col/Path request
   Without Q2a's parameterized requests, this feature adds ~13 ViewRequest
   variants, 13 'REPRS_IN_CLIENT' rows, 13 herald 'req:' rules, and 13
   parser arms — all near-identical, when the engines beneath are already
   relation-generic. Two typed requests carrying '(relation, direction)'
   and '(col)' collapse that to ~2.

** LATENT BUG — pO is emit-only and not parseable
   'parentOverrides' is emitted ('fork.rs') but has no parser arm
   ('parse_metadata_sexp.rs:690-698') — it survives only because it lives
   in a read-only buffer. Any path/col feature that puts pO (or a new
   pS/pH) into a saveable buffer will hit the parser's error branch on
   round-trip. Must be fixed as part of Q8. (Logged in 'TODO/problems.org'.)
