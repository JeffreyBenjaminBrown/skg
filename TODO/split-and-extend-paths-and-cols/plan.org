* Plan — separate & extend the collection / path command families
  Splits today's three 'C-c r' reveal commands into two families --
  COLLECTIONS ('C-c c', scaffold cols) and PATHS ('C-c p', backpath
  ancestries) -- and extends both across all five graph relations in
  both directions.

  Rationale, alternatives, and every settled decision live in
  'discussion.org' (this folder); this file is the build plan and does
  not re-argue them. Where a decision needs a one-line reminder it is in
  "Settled design" below.

  Formatting: per 'coding-advice/', code refs use single apostrophes;
  no "phase N" labels -- the commits below are simply done in order.

* Settled design (what this plan implements)

** Two command families + keybindings
   Collections ('C-c c') -- each builds BOTH cols of its relation:
   | C-c c a | skg-show-collection-aliases    | aliasCol                      |
   | C-c c o | skg-show-collection-overrides  | overriddenCol + overriderCol  |
   | C-c c h | skg-show-collection-hides      | hiderCol + hiddenCol          |
   | C-c c s | skg-show-collection-subscribes | subscribeeCol + subscriberCol |

   Paths ('C-c p') -- UPPER = the partner's "first" (active) role,
   lower = its "second" (passive) role:
   | C-c p C | skg-show-paths-through-containers   | containers of N        | exists |
   | C-c p L | skg-show-paths-through-link-sources | nodes linking to N     | exists |
   | C-c p l | skg-show-paths-through-link-dests   | nodes N links to       | new    |
   | C-c p O | skg-show-paths-through-overriders   | nodes overriding N     | new    |
   | C-c p o | skg-show-paths-through-overridden   | nodes N overrides      | new    |
   | C-c p H | skg-show-paths-through-hiders       | nodes hiding N         | new    |
   | C-c p h | skg-show-paths-through-hidden       | nodes N hides          | new    |
   | C-c p S | skg-show-paths-through-subscribers  | nodes subscribing to N | new    |
   | C-c p s | skg-show-paths-through-subscribees  | nodes N subscribes to  | new    |

   ('C-c r' and the 'skg-reveal-*' aliases, incl. 'skg-reveal-link-sourceward',
   are removed. No legacy syntax is kept -- single user, single dev.)

** Wire grammar
   Requests, inside the existing '(viewRequests ...)' container:
   : (viewRequests (col RELNAME))    ;; builds BOTH cols of the relation
   : (viewRequests (path ROLENAME))  ;; builds the backpath for that one role
   : (viewRequests definitiveView)   ;; unchanged (neither col nor path)
   Birth (provenance on a grafted path node):
   : (birth backpath ROLENAME)

   RELNAME in: 'aliases', 'overrides', 'hides', 'subscribes'.
   ROLENAME in the nine below. A col is named by its RELATION (it spans
   both roles); a path and a birth are named by the single ROLE the
   partner plays (= the role the grafted node plays toward its
   org-parent). One ROLENAME determines its relation, so a single token
   suffices for both '(path ...)' and '(birth backpath ...)' -- a
   deliberate tightening of the discussion's '(birth backpath RELATION
   ROLE)' sketch (the relation is implied by the role; one vocabulary
   serves requests and births alike). Flag on review if you want the
   relation spelled out too.

** Types (server/types/viewnode.rs unless noted)
Reuse the existing 'RelationRole { relation: NodeRelation, position:
BinaryRolePosition }' ('server/dbs/in_rust_graph/relation_accessors.rs') --
it already enumerates all five relations and both role positions, and
the in-Rust lookup already dispatches all ten role-pairs
('server/dbs/typedb/search.rs:103-164').

Replace the flat four-variant 'ViewRequest' with:
: pub enum ViewRequest { Col (ColRelation), Path (RelationRole), Definitive }
: pub enum ColRelation { Aliases, Overrides, Hides, Subscribes }
For a Path/Birth, the 'RelationRole' names the PARTNER's (output)
role. Replace the flat 'Birth' with:
: pub enum Birth { Unremarkable, Backpath (RelationRole) }

One ROLENAME <-> RelationRole <-> backpath-triple <-> glyph table is
the single source of truth (put it next to 'RelationRole' or in a new
small module; both the request layer and the backpath engine read it):
| ROLENAME   | RelationRole                      | (relation, input_role, output_role)         | glyph |
| container  | Contains, First                   | contains, contained, container              | }     |
| linkSource | TextlinksTo, First                | textlinks_to, dest, source                  | ←     |
| linkDest   | TextlinksTo, Second               | textlinks_to, source, dest                  | →     |
| overrider  | OverridesViewOf, First            | overrides_view_of, overridden, overrider    | Op    |
| overridden | OverridesViewOf, Second           | overrides_view_of, overrider, overridden    | pO    |
| hider      | HidesFromItsSubscriptions, First  | hides_from_its_subscriptions, hidden, hider | Hp    |
| hidden     | HidesFromItsSubscriptions, Second | hides_from_its_subscriptions, hider, hidden | pH    |
| subscriber | Subscribes, First                 | subscribes, subscribee, subscriber          | Sp    |
| subscribee | Subscribes, Second                | subscribes, subscriber, subscribee          | pS    |
('contained' / "content" is intentionally absent -- the recursive
content view already serves it.)

** Safety invariants (do not regress)
   - Collections are REQUESTS, never hand-inserted empty scaffolds: the
     server builds each col POPULATED from the graph. An empty literal
     writable col ('overriddenCol'/'subscribeeCol') saved before
     completion lowers to 'MSV::Specified(vec![])' and WIPES the
     relation ('traverse.rs:307-323' -> 'lower.rs:268-273'); the request
     path avoids this because the col is populated before any save reads
     it.
   - Empty-col policy (decision A): an empty READ-ONLY col simply does
     not appear (existing prune, 'complete.rs:325-337'); an empty
     WRITABLE col DOES appear (the editable "add here" surface). So
     'skg-show-collection-overrides' on a node with no overrides shows
     an empty 'overriddenCol' and no 'overriderCol'; '...-hides' on a
     node with no hides shows nothing (both sides read-only).
   - All collection and path commands AUTO-SAVE (incl. aliases, which
     currently does not).
