**** the question
    plan-11 put an 'overridesHere' marker on InactiveNode for exactly
    one case, which step 7 removed. THE CASE: override substitution
    draws overrider R in place of original N as recursive content of a
    container C (glossary "override substitution"). If a source-set
    switch then makes R's source inactive while N stays active, the
    OLD code converted R to an InactiveNode that still carried the
    marker, so it kept standing for N (collected_id = N): C showed one
    node, R-standing-for-N, cleanly.

    NOW R becomes an ANONYMOUS inactive placeholder (no marker). On the
    post-switch rerender, the content reconciler no longer sees R as
    satisfying goal member N (R is irrelevant, provides no orderkey),
    so it CREATES N alongside the retained anonymous R. Net effect: C
    shows BOTH the anonymous placeholder AND N (and, if N's content is
    active, that subtree is drawn under both). It self-heals on the
    next de-novo render (an inactive overrider does not substitute, so
    N is drawn directly, once).

    SEVERITY: cosmetic. NO data loss -- I traced it: on save,
    'content_members(C)' excludes the irrelevant inactive node, so
    C.contains extracts to [N] and the weave keeps N. The anonymous
    placeholder is view-only; it never reaches disk. (I did NOT build
    an integration test for this -- the scenario needs an
    override-edge fixture drawn in content position, which the
    source_sets fixtures lack. The reasoning is the evidence.)

**** Claude: I don't think parentIs=Independent alone clears it -- here's why
I think your instinct is right that R is "really" still the user's
node and only the route to it went inactive, but as literally stated
the proposal has two snags. Let me lay them out, because the fix is
close.

WHO IS WHAT (so we're agreed on terms): R is the overrider, drawn in
place of the original N as content of container C. The node whose
SOURCE went inactive on the switch is R (that is why the prepass
touches it); N's source is still active.

Snag 1 -- privacy. If we keep R an ACTIVE node (which we must, to give
it a parentIs to set), then R's title/body still render. But R's
source is now inactive -- it is exactly the private content the user
restricted the source-set to HIDE (e.g. they switched to "public"
while screen-sharing, and R is their private fork). Showing R would
leak it. So R has to be hidden, not shown-as-Independent.

Snag 2 -- the double persists anyway. Whether R is Independent-active
or anonymous-inactive, it is IRRELEVANT to the content reconciler
(only parentIs=Affected Active children are relevant). So it is
preserved as-is AND, because N is still in C's goal list, N is
created beside it. C ends up showing both. Independent doesn't remove
that; it only relabels R.

Also a mechanical note: an inactive node is now fieldless, so it has
no parentIs to set -- "set parentIs=Independent" can only apply to a
node we keep ACTIVE, which is snag 1.

WHAT I THINK YOU'RE PROTECTING, and that it's safe: I read your
"better yet" as "don't just delete R; the user might have parked work
under it." Good news: there is NO data loss in any option. A node the
user parked under R with parentIs=Independent is saved as a standalone
graph node (Independent means it is not a member of R or C), so it
keeps existing in the graph regardless of what this view does with R.
Detaching R only drops R (and the parked node's VIEW position under
R), never graph data. The substitute's "real" children are N's
content and are redrawn under N on the rerender.

So the clean, correct, privacy-respecting result is: R should
DISAPPEAR from this position and N should show, once. R was never a
containment member of C here -- it was only N's display stand-in, and
an inactive overrider does not substitute -- so dropping it is right,
not lossy.

PROPOSAL (option 2, refined): in the source-switch prepass, when the
now-inactive Active node carries viewStats.overridesHere = Some(_),
DETACH it instead of converting it to an inactive placeholder. The
rerender then draws N directly, once; nothing leaks; no double; no
graph data lost. (The marker is read off the still-Active node at
switch time, so no field returns to InactiveNode.)
