  Goal (Jeff, 2026-06-12): remove every data field from InactiveNode

  is an anonymous "something inactive is/was here" marker that exists
  ONLY to host already-drawn active descendants after a source-set
  reduction.

** Two facts I verified (they make the removal sound)
   - DIFF MODE AND SOURCE RESTRICTION ARE MUTUALLY EXCLUSIVE, enforced:
     'diff_analysis.rs' refuses unless the active set is "all"
     ("Diff analysis requires active source-set all"), and
     'source_sets.rs' refuses a restricted switch while diff mode is on
     ("git diff mode is on, and it requires active source-set all").
     Under "all" there are no inactive nodes. So every InactiveNode arm
     in 'to_org/render/diff.rs' is unreachable dead code.
   - COMPLETION NEVER EXPANDS INACTIVE CONTENT. At an inactive node's
     BFS visit ('complete.rs') the only action is the deleted-check;
     it never reconciles the node's own content. So completion never
     needs the inactive id to fetch contents. (This is exactly Jeff's
     "don't complete the content of an inactive node -- that's a leak"
     principle, already honored.)

** Where inactive placeholders come from (so retention is well-scoped)
   - DE NOVO render under a restricted set: an inactive content member
     is OMITTED from the goal list ('omit_inactive_members') and never
     created. Its active descendants, reachable only through it, do not
     appear -- correct, and already true today.
   - SOURCE-SET REDUCTION of an already-drawn buffer
     ('update_buffer/source_switch.rs'): active nodes from now-inactive
     sources are converted in place to InactiveNodes; their
     already-drawn subtrees are kept. THIS is the only way a "retained
     inactive node with active descendants" exists. The children
     pre-exist in the tree; nothing is fetched by id.

** The one real refactor: id-free retention
   Today a retained inactive node survives reconciliation because its
   id is injected into the parent's goal list
   ('retained_inactive_children' -> 'omit_inactive_members' retained
   set), so the reconciler treats it as a RELEVANT child and matches it
   to that goal id. That is the only structural use of the id.

   Replacement: make inactive nodes IRRELEVANT in the reconcilers'
   'relevant' predicates ('reconcile/content.rs', and the partner-col
   path via 'complete/partner_col/child_data.rs'). 'complete_relevant_
   children' preserves irrelevant children as-is (it only stale-checks
   relevant ones), so a retained inactive node survives WITHOUT a goal
   id. Then:
   - 'omit_inactive_members' drops its 'retained' parameter and simply
     omits every inactive goal id (it resolves each id's source from
     the graph/disk, never from an InactiveNode field).
   - 'retained_inactive_children' is deleted.

   CONSEQUENCE (open question 1): irrelevant children are reordered to
   the FRONT of their siblings. So on rerender a retained inactive
   subtree jumps above its active siblings. Acceptable under the
   read-only-position rule we already adopted, but visible.
