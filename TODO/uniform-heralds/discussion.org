#+title: Uniform heralds: plan and problems

* overview

This responds to [[file:vision.org][vision.org]]. It proposes a plan, then
collects the tricky bits, the architectural choices the vision leaves
open, and the things I want to push back on. A final section lists
problems I found that are out of scope.

Throughout, "herald" means one of the short colored tokens the Emacs
client paints beside an org headline. The data is computed in Rust,
serialized as =(skg ...)= metadata atoms, and rendered by a generic
Emacs lens engine driven by a rule table. The relevant code:

* Tentative plan

** The target grammar, stated precisely

The vision describes a single positional mini-language. Let me pin it
down, because several details are implied rather than stated.

*** One token per relationship

There are exactly five relationship letters:

| letter | relation     | inbound = "X it"        | outbound = "it X" |
|--------+--------------+-------------------------+-------------------|
| C      | contains     | its containers          | its contents      |
| L      | textlinks_to | nodes that link to it   | nodes it links to |
| H      | hides        | nodes that hide it      | nodes it hides    |
| S      | subscribes   | nodes that subscribe to it | its subscribees |
| O      | overrides    | nodes that override it   | nodes it overrides |

A token reads: =[inNum][inLetters] X [outNum][outLetters]=. The
letters hug the relationship letter; the numbers sit outside them.
Worked from the vision's own examples:

- =pC=    : its parent contains it (count 1 = just the parent, so the
            number is omitted)
- =3pC=   : 3 nodes contain it, among them its parent
- =3C=    : 3 nodes contain it (none of them a tracked ancestor)
- =C4p=   : it contains 4 nodes, among them its parent
- =pC5=   : its parent contains it, and it contains 5 nodes
- =Cp=    : it contains its parent (today's "}" / 'containsParent')

The ancestor letters are =p= (parent), =g= (grandparent), =b=
(bizabuelo / great-grandparent), =t= (tatarabuelo /
great-great-grandparent). A letter says "this specific ancestor is
among the members on this side."

*** Number-omission rule

On each side: render the numeric count, then the ancestor letters,
EXCEPT omit the number when the count equals the number of flagged
ancestors present (so =pC= not =1pC=, but =3pC= because 3 ≠ 1). If a
side has no members, that side is empty. See [[*Problems][Problems]] for the one
genuinely ambiguous case (count 1, member is not a tracked ancestor).

*** Omission of whole tokens

A relationship token is omitted iff both its sides are empty (zero
inbound, zero outbound) — even for contains, even for roots. This is
the vision's [[file:vision.org::*omission][omission]] rule. Note the consequence: an ordinary
content node always has at least its container (the parent), so its C
token is always present — but it is present as the orange /birth/
herald (below), not as a separate blue token, so the screen does not
get noisier.

*** Surprising inbound links: =a(b,c)L=

The L token's inbound side gets a refinement. With =a= inbound links:

- =b= = of those =a=, how many differ from it, are bodyless AND
  contentless, and whose normalized title is something other than the
  link label;
- =c= = of those =a=, how many have their own content;
- the remainder (=a−b−c=) are "plain" links whose source is just a
  link with one of this node's aliases as label and nothing else.

Rendered =a(b,c)L=, with the parenthetical collapsing per the vision:
omit a zero, omit the comma when c=0, keep the comma when b=0, omit the
whole parenthetical when both are zero.

*** Birth herald: orange, first

Every drawn node has a /birth relationship/ — the single relationship
to a visible ancestor that explains why it was drawn:

- ordinary content: =contains=, inbound from its parent → its C token
  (=pC=, =3pC=, =pC5=, ...);
- a backpath-grafted partner: its 'Birth::Backpath(role)' relation;
- a member of a writable/read-only partner col: the col's relation
  toward the col's owning ancestor (e.g. a subscribee-as-such →
  =gS=, its grandparent subscribes to it; an overrider-as-such →
  =Og=, it overrides its grandparent).

That ONE relationship's token is moved to the front, placed right
after the ☮ peace symbol if present, and colored orange. Every other
relationship token follows it, colored blue.

So the birth relation is shown once (orange), not duplicated. The
current separate edge heralds (=Op=, =pO=, =Sp=, =pS=, =Hp=, =pH=,
=gO=, =gS=) cease to be their own tokens; they become ancestor-letter
annotations inside the five unified tokens.

*** Action / alias heralds

- =A= (aliasing, boolean) → =Ak=, k = number of aliases.
- =O=, =S= (overriding / subscribing booleans) disappear: the counted
  O and S tokens are strictly more informative.
- =H= (hiding boolean) likewise folds into the H token.
- =I= (extraIDs boolean): see the TODO in [[*Problems][Problems]] — drop, or mirror
  aliases as =Ik=?

*** Which ancestors to check (efficiency)

The expensive part is not the counts (they are single set-size reads
off the in-Rust inverse indexes) but deciding which ancestor letters
to flag. The rule:

- For every node, check its relationship to its immediate visible
  parent — one hop. This yields the =p= flags. (This is exactly what
  'set_relation_relative_stats' does today for six edges.)
- For a col member, additionally check the relationship to its /owning
  ancestor/ and every intervening vognode, via the owning-ancestor
  table. This yields the =g=/=b=/=t= flags. The depth is small and
  fixed per col kind.

** What stays out of scope (unchanged)

Per the vision, only the relationship heralds, the alias/action
heralds, and the surprising-links refinement change. These keep their
current behavior: 'cycle' (⟳), 'overridesHere' (=Oh=, which is
LOAD-BEARING for save extraction, see [[../../server/types/viewnode.rs][viewnode.rs]] 'collected_id'),
'sourceHerald' (⌂:label), 'parentIs' independent (⊥) / absent,
'textChanged', the staged/unstaged diff axes, 'notInGit', 'editRequest'
(delete / merge), 'viewRequests', the deleted / unknown / inactive /
deadScaffold placeholders, the read-only-col ☮ scaffold markers, and
the col labels themselves ("aliases", "subscribeeCol", ...). The node
'indef' ☮ stays but is repositioned to lead, so the birth herald can
sit right after it.

* Problems

** ONGOING Ancestor-letter distance: the hiddeninsubscribee example is off by one

The herald letters =p/g/b/t= must measure SOME distance. The existing
code already uses =gO=/=gS= for a subscribee-as-such whose owner is the
/grandparent/ — i.e. the letter counts TREE ancestors (member → col
scaffold → owner = 2 levels = "g"), which also matches what a user sees
(two headline levels up). Good.

But the vision says a hiddeninsubscribeecol member's owning ancestor is
its "great-grandparent". The actual view tree is

: subscriber → SubscribeeCol → subscribee → HiddenInSubscribeeCol → member

so the subscriber (the owner) is FOUR headline levels up =
great-/great/-grandparent = =t=, not =b=. The code reconcilers read it
at col-index 2 (member-index 3), confirming four hops.

Question for you: are the letters tree distance (matching =gS=/=gO=
and the visible headline count), making the hiddeninsubscribee owner
=t=? If so the vision's "great-grandparent" is just a slip and the
plan should use =t=. (If instead you intend the letters to count only
gnode ancestors, skipping col scaffolds, then a subscribee-as-such's
owner would be =p=, which contradicts the existing =gS=/=gO= — so I
don't think that's what you mean. Please confirm.)

** ONGOING extraIDs (=I=): drop it, or show =Ik=?

The vision lumps "I, O and S" together as removable. O and S clearly go
because the counted O/S tokens replace them. But =I= (extraIDs) is not
a graph relationship and has no token to fold into. Two readable
options: drop the bare boolean entirely (the idCol surfaces the extra
IDs on demand), or mirror aliases and show =Ik= (k = number of extra
IDs). I lean toward =Ik= for symmetry with =Ak=, but it is your call.

** ONGOING Non-birth relationship heralds: uniformly blue loses the red "alarm"

The vision says non-birth relationship heralds are blue. Today the

** ONGOING Surprising-links =b=: how do we recover each link's label?

Computing =a= and =c= is cheap and already essentially done
('sources_with_content'). Computing =b= needs, per inbound edge, the
LABEL the source used in =[[id:thisnode][label]]= — and that label is
discarded everywhere except in the source node's raw title/body text.
'textlinks_in' is a 'HashSet<ID>' of source pids only (no labels, no
multiplicity); 'NodeRust.textlinks_to' drops the label at construction.
Two options:

- (A) Re-parse each source's title+body on demand (the existing
  'textlinks_from_node' regex), in memory, no disk/TypeDB. Cheap per
  node; pays a regex pass per inbound source.
- (B) Add the label to the 'textlinks_in' index. More invasive (touches
  add/remove/migrate of the inverse indexes) but O(1) per edge, and it
  also fixes the current multiplicity loss (a source that links twice
  collapses to one pid today).

I lean toward (A) first (no data-model churn), unless you already want
(B) for other reasons. Which do you prefer?

  body? ('NodeRust.body' is 'Option<String>'.)
- If one source links to this node twice with two different labels,

** (likely solved): The number-omission edge case (count 1, no flagged ancestor)

When a side has count 1 and that single member is NOT a tracked
ancestor, my rule renders =1C= (bare number). The vision never shows
this shape because in practice the lone container of a content node IS
its parent (=pC=). The only way to hit =1C= is viewing a node where its
single container is off-screen / not an ancestor (e.g. a backpath). I
think =1C= is the correct, unambiguous rendering there; flagging it

** (likely solved): Token strings are assembled in Rust, not in the rule table

The compact grammar (conditional numbers + variable ancestor letters on
both sides + the =a(b,c)= form) is more positional than the Emacs lens
engine's INTERC directive can express. Rather than extend the engine, I
plan to assemble each token string in Rust and emit it as a value the
rule table echoes (via ANY/IT), with the table still owning color
(orange/blue) and order (birth first). This keeps Emacs a dumb renderer
and is consistent with [[../../server/heralds.rs][heralds.rs]] already being the Rust-side
presentation authority. The cost: the grammar lives in Rust code, not
in the declarative table, so the table no longer self-documents the
token shape. I think that is the right trade; tell me if you'd rather
push the grammar into the lens engine.

** Pushback: the vision under-specifies, but is not over-complex

I do not think any part of the vision is unnecessary or gratuitously
complex — the unification genuinely removes a mishmash (=}=, ={=, =→=,
=←=, plus eight separate edge heralds) and replaces it with ≤5 learnable
tokens. The real risk is the opposite: the compact grammar is dense.
=3pC4p= is precise but cryptic until learned. That is a deliberate
density trade you are clearly choosing; I just want it on record that
the win is "fewer tokens, more to learn per token." The blast radius is
also large (≈400 golden test strings), so this is a sizable change even
though the runtime logic is contained.
