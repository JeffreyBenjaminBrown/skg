* Summary
The accordion is a better idea than the chain. Its central move —
one node, one ID, several per-source files, no connecting
relationship — deletes the worst problems of the autofork design
rather than solving them:

- No carrier IDs exist, so nothing must be hidden, translated, or
  kept stable. The chain's largest implementation risk (carrier
  transparency across every inbound query) vanishes outright.
- Renormalization stability is free: a part's identity is
  (ID, source), fully deterministic, so saves cannot churn
  generated IDs.
- No sixth relationship, no perceived-node/carrier distinction, no
  interaction with override monogamy, menus, or markers.

And yes — [[*Does the accordion cover everything 1_vision wanted?][it covers everything 1_vision wanted]], plus one thing the
chain could not express (per-relationship privacy falls out
uniformly).

What it costs: it breaks the foundational invariant "one ID, one
file, one source", whose assumptions are scattered (init
validation, TypeDB cardinality, stage-moves' move heuristic, diff
analysis, the audit) — a bounded but grep-not-compiler kind of
sweep. And it introduces one genuinely new problem the chain did
not have: [[*Foreign accordions fold in an order the receiver must guess][the fold order of a foreign accordion lives in the
receiver's config, not in the data]].

The one outright error I see: [[*The proposed default for col relationships re-creates the ID leak][the proposed default for col-defined
relationships]] ("most public representation of N") re-creates the
private-ID leak for private targets. The fix is a uniform rule —
and the doc's own "incomplete thought" about ternary relationships
is, I think, [[*The ternary model is the answer to its own incomplete thought][the place that rule comes from]].

* Incorrect things
** DONE The proposed default for col relationships re-creates the ID leak
"By default ... keep all the relationships V defines in the same
file as the most public representation of N itself." If N is
public and subscribes to a *private* S, this default writes S's ID
into a public file — exactly the leak ("the public one need not
even mention private node IDs") this whole design exists to
prevent. Contains avoids it only because its default is stated
per-child.

The uniform rule that fixes it (and subsumes the contains rule):

  a relationship instance defaults to the level of the more
  private of its two endpoints' homes; obfuscation may RAISE that
  level, never lower it.

For contains with a private child, that is the child's level (the
vision's rule); for a subscription to a private S, it is S's level
(leak avoided); for a public child of a public node, it is the
plain public membership of today. The validator should enforce the
"never lower" direction: a membership more public than its target
is precisely an ID leak.

* Dangerous things
** Foreign accordions fold in an order the receiver must guess
The chain carried its own order in its edges; the accordion's fold
order comes from the *config's* privacy ordering of sources. For
your own sources, fine. For a colleague's shared source-pair, the
order is whatever you guessed when you added them to your config —
guess wrong and the fold runs backward: plain-vs-relative shapes
land in the "wrong" direction and anchors dangle en masse.
Mitigations, cheapest first: (a) the anchor direction is
detectable — a part whose anchors resolve only in the *other*
direction indicates an inverted order; warn and suggest the flip.
(b) A convention for publishers to state the order (a README, or a
tiny manifest in the source). (c) Accept degradation with loud
warnings. Some choice is needed, because this is the accordion's
one structural regression versus the chain.

** The one-ID-one-file assumption is scattered, and the compiler will not find it
Unlike adding a =NodeRelation= variant (exhaustive matches), the
"an ID has exactly one file/source" assumption hides in ordinary
code: init's duplicate check, the daily audit, diff analysis's
duplicate bucket, 'source_from_disk' and 'pid_and_source'
(which source do they answer?), get-file-path, delete-propagation,
and — sneakiest — *stage-moves*: its move heuristic is "an ID
vanished from exactly one source and appeared in exactly one
other", which is indistinguishable from an accordion
re-levelling a membership... and both can be true at once. A
written audit list of these sites, checked off during
implementation, is the only protection; budget for it explicitly.

saved buffer, which FILE records it. Suppose a bug (or an
ambiguous rule) computes an edge's level as the more PUBLIC of the
two endpoints instead of the more private. Saving public N with
private member C then writes C's UUID into N's public file.
Nothing crashes; every view still renders exactly right (the fold
shows C under N either way); tests that only inspect rendered
buffers pass. You discover it, if ever, by reading the public
.skg file or its git diff — likely after pushing, at which point
the leak is in public git history permanently. That is what
"silent" means here: rendering gives no signal at all.

The sharpest variant needs no bug, only a lossy round-trip: you
privatize an edge above its default with
'skg-privatize-relationship'; later you save the same node from a
different buffer that never carried the privatized level; the
renormalizer re-derives the DEFAULT level and silently
de-privatizes the edge. Hence two guards:
- the ~SOURCE herald's metadata is load-bearing at save, so
  buffers carry edge levels through round-trips; and
- the STICKY RULE: renormalization may raise an edge's level (per
  the default rule, or the hide rule) but never lowers an existing
  edge's level; lowering happens only through the explicit
  gesture's cycle.
Property tests pin both, precisely because no user-visible
behavior changes when they break.

** Dangling anchors: the fold must be total and deterministic

run, or at START), a completion warning, self-heal on next save,
and anchor resolution through 'pid_of' (nodeMerges rename IDs).
'weave' in [[../../server/from_text/weave.rs][weave.rs]] remains the precedent — and see

**** DONE (Claude) The scenario, concretely
Machine A holds your public and private sources; your work
machine B holds only the public one. On B you delete public
member c1 from node N — an ordinary, correct edit; B cannot know
c1 anchors anything, because the private part is not on that
machine. You push; machine A pulls. N's private part still says
(anchor c1, [c2]), but c1 is no longer in the public fold: the
anchor DANGLES, and two perfectly correct saves produced it. So
dangling anchors are a normal runtime condition, not invalid
data, and the fold must handle them:
- total: c2 still renders — it is real content — appended at the
  end of N's content per the order-independent fallback above,
  with a warning;
- deterministic: every machine folds the same broken state the
  same way, so views do not depend on which box you sit at;
- self-healing: the next save of N from a view where c2 is
  visible rewrites the private part with a live anchor;
- and a merged (rather than deleted) anchor is a non-event: if c1
  was nodeMerged into c9, 'pid_of' resolves the anchor and
  nothing dangles.

* Awkward, inelegant, or overly complex things
** The per-relationship bookkeeping dissolves under the ternary reframing
The doc apologizes twice for sloppiness (which file is plain, per
relationship; where relationship instances live). Both
awkwardnesses are artifacts of thinking in *files*. Thinking in
*leveled edges* ([[*The ternary model is the answer to its own incomplete thought][below]]) makes the file layout a derived normal
form, and the sloppiness disappears rather than needing apology.

** The weave's job shrinks
A pleasant simplification to note: under prefix source-sets and
the accordion invariants, a *visible* file can only list members
at its own level or more public — i.e. also visible. So the
save-time preservation of invisible members ('weave',
'set_difference_merge') stops being load-bearing for
accordion-shaped data; it remains as a backstop for junk and for
any non-prefix sets that survive. Worth stating in the plan so the
weave is not extended when it could be retired toward backstop
duty.

** Two representations of "no opinion" now coexist
A part can express "no opinion" about a relationship by omitting
the field; MSV expresses the same thing in memory. Fine — but the
doc comment on 'NodeComplete' explicitly says the on-disk
distinction is unnecessary, and after this change that sentence is
false for accordion parts. Small, but exactly the kind of stale
comment that misleads later work; sweep it with the format change.

* Missing concepts and abstractions (especially types)
** The ternary model is the answer to its own incomplete thought
The doc's final section ("The relationships could be explicitly
ternary ... I don't know where to go with that") is, I think, the
keystone rather than an aside. Take it as the *semantics*:

  every relationship instance carries a level:
  (level, node) --relationship--> target

Then, uniformly:
- an accordion part at level k is nothing but "the level-k slice
  of N's edges", plus order info relative to more-public slices;
- the default-level rule becomes one sentence (an edge defaults to
  the more private of its endpoints' homes — [[*The proposed default for col relationships re-creates the ID leak][the leak fix]]);
- obfuscation is "raise this edge's level";
- the validator's job is "no edge more public than its target,
  no two plain slices, anchors resolve more-publicly";
  DONE jbb : no two plain slice for the same rel (repeating myself)
- the fold is "concatenate slices in level order".
Where to go with it concretely: adopt it in memory (edges
annotated with a level/SourceName) and in the docs' vocabulary,
while keeping .skg files exactly as proposed — the accordion IS
the ternary model's on-disk normal form. Nothing about storage
changes; what changes is that every rule above becomes derivable
instead of case-by-case.

** Fold and unfold as a load/save adjoint pair
One module owning both directions: load groups same-ID files,
orders parts by the config's privacy order, folds to an effective
'NodeComplete' (plus retained per-level decomposition for save);
save decomposes leveled edges back to parts, normalizing. The
property tests live here. Precedents to imitate: 'weave' (anchored
runs), 'resolve_override' (guarded walk totality), and the
=source=-never-serialized pattern in
[[../../server/types/nodes/complete.rs][complete.rs]] / [[../../server/types/nodes/fs.rs][fs.rs]] (the type system already enforces one
in-memory-only field; the accordion extends that boundary).

With every list a Vec<(Source, ID)> in fold order, the placement

  the strongest protection available for the scattered-assumption
  danger, and a reason to make this type change FIRST in the plan.
(MSV keeps wrapping the outer lists exactly as today.)

** An =Accordion= value at the FS boundary
Group-by-ID currently ends in an error report
('check_for_duplicate_ids_across_sources'); it should end in a
value — =Accordion { home : NodeFS, parts : Vec<(SourceName,
NodePart)> }= or similar — that the shape validator takes and the
fold consumes. Making "several files, one node" a type, not a
condition, is what keeps the scattered-assumption sweep
([[*The one-ID-one-file assumption is scattered, and the compiler will not find it][above]]) from regressing later: new code meets =Accordion=, not
a surprising second file.

** The privacy order as a first-class config value
Unchanged from the chain discussion, but now a hard prerequisite:
an ordered list (later possibly a tree) of owned sources in
skgconfig.toml, foreign sources interspersed at user-chosen
positions for visibility purposes; prefix selection replaces named
source-sets. The fold, the defaults, the validator, and the
selection UI all read this one value.

* Efficiency notes (minor)
- Folds are per-accordion and tiny; cache per ID, invalidate when
  any part is written. Init's group-by-ID replaces today's
  duplicate scan at the same cost.
- File count grows by one per (node, level-with-parts) — the same
  as the chain's carriers, without their IDs.
- Collateral view selection must treat "part written" as "node
  changed" — trivial now, since the part shares the node's ID
  (contrast the chain, where this needed carrier translation).
- Private git diffs stay small and quiet, as with the chain: a
  public reorder touches no private file.
