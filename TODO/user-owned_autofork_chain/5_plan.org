* The design in one page (digest, so this file stands alone)
*** One node = one ID = one *privacy accordion*: a set of same-ID
  .skg files (each is a "*section*" in code; in prose,
  accordion section), at most one per source.
  The most public section is the *home*; it
  alone carries title and body. No relationship connects sections;
  grouping is by ID. (The short names — accordion, section, level —
  are for code; comments and docs use the long forms, per the
  terminology note below.)
*** Sources are totally ordered by publicness in the config, most
  public first. A session's active set is a *prefix* of that order
  ("select the most private source to make available"); named
  source-sets are retired.
*** Every relationship instance ("edge") carries a *privacy level*
  ("level" in code and hereafter): the
  source of the file that records it. Default level = the more
  private of the two endpoints' homes; 'skg-privatize-relationship'
  may raise it; nothing ever lowers it silently (the *sticky
  rule*). Hides take max(hider, hidden, min over explaining
  subscription edges).
*** On disk, per relation: every section stores ONE YAML sequence,
  one entry per line — a bare ID (a member) or =anchor: ID= (a
  one-key map). The most public section mentioning the relation
  carries no anchors: its sequence IS the base list, in order. In
  a more private section, entries before the first anchor are the
  prepend, and each anchor starts a run inserted after that member
  of the strictly-more-public fold (anchors resolve via 'pid_of').
  Members render identically in every section, so a membership
  moving between sections diffs as a clean one-line delete/add
  pair (the readability requirement from the stage-moves thread).
  The semantic type is still
  =(prepend : List ID, runs : List (anchor : ID, List ID))=.
  Unordered relations (hides, overrides, aliases) never carry
  anchor entries; for them, fold = union.
*** In memory, placements do not exist: every list is a single
  ordered *leveled list* (=Vec<(SourceName, ID)>= in spirit). The
  *fold* (load) and *unfold* (save) convert; unfold DERIVES runs
  and anchors canonically (a level-s run anchors to the nearest
  preceding strictly-more-public member, else joins the prepend).
*** Two folds exist and must never be confused: the *full fold*
  (every section; what the save path and TypeDB see) and the
  *visible fold* (sections within the active prefix; what
  rendering sees). They are distinct Rust types, so confusing them
  cannot compile.
*** Files are rewritten only when their semantics change; a section
  whose content is unchanged stays byte-identical.
*** Cross-file invariant violations load as WARNINGS with repair
  offers; only single-file malformations hard-error.
*** Explicitly out of scope: same-ID overlays on foreign homes
  (validator rejects), node-level accordion heralds (none), private
  textlinks (body is home-only), source trees (later), any change
  to the manual fork system.

** section-format-and-fold: the .skg section format, the Accordion value, fold/unfold
The heart. One module owning both directions plus the grouping.

Work:
*** 'NodeFS' becomes the SECTION format: =title : Option<String>=
  (home only), =body= likewise. Every list field is ONE YAML

  untagged two-variant ITEM enum in serde) — the diff-readable
  serialization your stage-moves note asked for: members render as
  identical "- ID" lines in every section, anchors look visibly
  different, entries before the first anchor are the prepend, and
  the base list is simply the anchor-free sequence in the most
  public mentioning section. Unordered relations never carry
  anchor entries. Today's files parse unchanged as single-section
  accordions (no anchors, title present).
  DONE jbb : rename part -> section (repeating myself).
  (Claude: applied throughout; this item and its cross-references
  are now =section-format-and-fold=.)
*** =Accordion=: the group of same-ID sections, sorted by
  =source_order=. Init's group-by-ID produces Accordions;
  'check_for_duplicate_ids_across_sources' is subsumed by the
  shape validator (next item) — same-ID-different-source stops
  being an error and starts being the feature.
*** 'fold_accordion': Accordion -> leveled 'NodeComplete'. Ordered
  relations: start from the base list (the most public mentioning
  section's); apply each more-private section's placement in level
  order — runs insert after their anchors
  (anchor resolution through 'pid_of'; duplicate anchors:
  concatenate runs in file order; dangling anchor: the run
  attaches after the run that precedes it in the file, or after
  the prepend if first, and emits a warning). Unordered: union,
  level-tagged, deterministic order (level, then file order).
  Title/body from home; a non-home title is a warning and the
  most public one wins. TOTAL: no input panics; junk degrades to
  warnings.
*** 'unfold_node': leveled 'NodeComplete' -> sections. Per
  relation: members grouped by level; the most public mentioning
  level gets the anchor-free base in its order; each more private
  level gets derived runs (maximal same-level streaks anchored to
  the nearest preceding strictly-more-public member, else
  prepend). Sections with no fields left are deleted (file
  removal); the home file persists while the node exists.
*** The two folds as two TYPES: 'fold_accordion' produces
  =FullFold= (every section) and, given the active prefix,
  =VisibleFold= — distinct wrapper types. Render-side functions
  accept only =VisibleFold=; the save path, TypeDB writer, and
  audit accept only =FullFold=. Confusing the folds is then a
  compile error, not a discipline. (Added while answering the
  two-folds question at the bottom.)
*** Byte-stability: the writer diffs serialized section content
  against disk and skips identical files (precedent:
  strip-body-whitespace).
*** Property tests, in this module: fold(unfold(x)) == x for
  arbitrary leveled lists; unfold(fold(files)) is idempotent from
  the second application; every member's level survives both
  directions; dangling/duplicate-anchor inputs fold totally and
  deterministically; byte-stability on no-op saves.

** accordion-validators: the shape and leak table, at both gates
Work:
*** =AccordionInvariantViolation=, produced by ONE shared primitive

** save-leveling: defaults, sticky rule, hide levels, precise re-anchoring
The pipeline learns levels; this is where the silent-leak dangers
are closed.

Work:
*** Extraction (local instruction collection): intents carry
  =Option<SourceName>= per member — =Some= from the load-bearing
  metadata atom (see =render-and-gating=), =None= meaning derive.
*** Lowering resolves =None=: existing edge (same relation, same
  endpoints, from the graph snapshot) keeps its level (STICKY);
  new edge gets the default, max(home(N), home(target)).
  A =Some= below the sticky/default floor is a save error naming
  the rule; a =Some= at-or-above is honored.
*** Hide inference ('resolve_visibility',
  'infer_hides_from_contains_removals'): new hides level at
  max(hider home, hidden home, level of the triggering
  subscription edge); the renormalizer re-raises existing hides
  when their most public explanation vanishes or rises
  (min-over-explanations rule; raising only).
*** Level-ungated maintenance: the save computes with FULL folds
  regardless of the connection's prefix. Members above the active
  level pass through from disk untouched — structurally free,
  since they live in untouched sections — EXCEPT precise
  re-anchoring: when this save deletes or moves a member that a
  more-private run anchors to, the server (holding the old fold)
  rewrites that run's anchor to the nearest surviving more-public
  predecessor. Only semantically-changed sections are rewritten.
*** Deletion refusal: deleting a node whose accordion has sections
  above the active level is refused with "N has accordion sections
  in inactive sources" (the agreed small leak). 'apply_delete_propagation_cleanup'
  learns to scrub a deleted ID from leveled lists (which unfolds
  into scrubbing base lists, runs, prepends, and anchors — an
  anchor scrub re-anchors the run as above).
*** The weave: not extended. Under accordion invariants a visible
  file cannot list an invisible member, so 'weave' /
  'set_difference_merge' remain only as backstops for junk data;
  comment them as such.

***** DONE (Claude) Explanation
"Gating" = filtering by the active source-set: X is gated on Y
when X may render only if Y is inside the active prefix. Today
all gating is per NODE ("is this node's source active?"). The
accordion adds per-EDGE levels, and every edge is reachable from
BOTH ends. The outbound end is safe by construction: rendering
N's content uses the visible fold, so a privatized membership
stays out of N's child list at public level. The INBOUND end is
not: containerward ancestry, backpaths, inbound PartnerCols, and
the container-count heralds all read the inverse indexes
('contained_by', 'subscribers_of', ...), which today record only
"N contains C" — no level. Concrete leak: you privatize public
C's membership under public N (the reading-list case); at public
level, visiting C would still show N in its ancestry and count N
among its containers, revealing exactly the relationship you
privatized. Is it solved? In the PLAN, yes — =render-and-gating=
stores each edge's level in the inverse indexes and filters every
inbound surface by it, and the leak battery pins each surface
with a test. In CODE, nothing exists yet. I flagged it because
the discussion's design had only covered the outbound direction;
without this item the feature would have shipped leaking.

***** DONE (Claude) Explanation
Same sense of gating as above: text search must not display
matches the active prefix forbids. Tantivy stores one document
per title/alias, mapping its text to an ID, each document
carrying a source field; search drops results whose source field
is not active. Today that field is always THE NODE's source —
correct while a node was one file. With per-alias levels it goes
