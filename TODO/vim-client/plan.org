does, to the extent possible; where not possible, a final report describes
every necessary deviation. Zero server changes are anticipated (one open
question below confirms this constraint).

The Emacs client is ~7,850 lines of elisp across ~40 files, plus 31 unit
test files and ~35 integration test directories. The port is a full
rewrite of the client tier against the same wire protocol.

* What the client actually is (summary; details in the inventory)
A view buffer's text comes entirely from the server and returns entirely
to the server; no file backs it. The client's five real jobs:
1. A persistent TCP connection with length-prefixed
   ('Content-Length: N\r\n\r\n' + N bytes) s-expression messages, and a
   response-type → handler dispatch registry that supports multi-message
   streams (save-lock → save-relax-lock → collateral-view* → save-result;
   rerender-lock → rerender-view* → rerender-done; search-results →
   request-snapshot → search-enrichment).
2. Org-style display of server text: outline folding, TODO cycling, and
   the herald transform (conceal each headline's '(skg ...)' metadata,
   render short colored glyphs in its place, driven by a rule table
   fetched from the server).
3. Metadata surgery: parse a headline into (stars, metadata sexp, title),
   apply declarative edits (the DELETE/REPLACE/ENSURE merge DSL), write
   the line back. All ~30 graph commands reduce to this plus auto-save.
4. Round-trip state preservation: 'focused', 'folded', 'bodyFolded'
   markers added before save, acted on and stripped after redraw; point
   and scroll restoration from server-echoed offsets; buffer locking
   while a save/rerender stream is in flight.
5. Odds and ends: text search UI, the linkstack, git helpers (git-add
   script generation, goto-in-magit, readable-ids annotations), the
   metadata edit buffer (sexp ↔ org bijection), config/TOML reading,
   source pickers, logging, reload.

* Confirmed groundwork (verified in this container, 2026-07-02)
- nvim 0.12.3 with the baked-in plugins from docker.nix: nvim-orgmode
  (whose org tree-sitter parser loads and parses a scratch buffer),
  neogit + plenary + diffview.
- Extmarks with conceal + inline virt_text work — the herald mechanism.
  virt_text chunk lists carry per-chunk highlight groups, which matches
  the lens engine's colored-token output exactly.
- 'vim.uv' TCP handles are available (networking).
- 'plenary.busted' and 'plenary.test_harness' load — the Lua unit-test
  harness, already in the image as neogit's dependency.

One byte-vs-character note (cf. coding-advice/pitfall-bytes-vs-characters.org):
Lua strings and the nvim API are byte-oriented (extmark columns are byte
columns), while elisp columns are character-oriented. The LP framing is
byte-based, so it gets *simpler* in Lua; but 'point-column' math and
extmark placement around herald glyphs (⌂, →, ⤊ are 3-byte chars) must
consistently use byte columns internally and convert only at the wire
boundary where the server expects what Emacs sent. The save-request
'point-column' field's unit (chars, from Emacs 'current-column') will be
pinned by a test against the live server.

* Expected deviations (known before starting; final report will confirm)
- Keybindings: vim is modal and terminal C-c is unusable, so the Emacs
  chords cannot be literal. Proposed scheme in Q4.
- Lock messages: a locked buffer rejects edits with vim's generic
  "E21: Cannot make changes, 'modifiable' is off" rather than the
  custom "wait for the save to finish" message. (Mitigation: also set a
  winbar/notify hint while locked.)
- The fork-confirmation buffer's protected title region and the metadata
  edit buffer's read-only display group become commit-time validation
  instead of type-time rejection (nvim has no per-range read-only text).
- Conceal behavior on the cursor line differs from an Emacs display
  overlay (vim reveals concealed text depending on 'concealcursor'; we
  will pick the closest setting, likely 'nc', so insert-mode on a
  metadata line shows the raw sexp — arguably clearer, but different).
- M-x cosmetics (alias hiding via completion-predicate) have no analog;
  the Ex-command surface simply exposes the public names.
- Shell-map linkstack bindings: Emacs binds paste/pop in shell-mode;
  nvim terminal buffers get a terminal-mode paste mapping (feeds keys),
  pop-variants likely dropped there.
- skg-reload: 'package.loaded' clearing reloads Lua modules, but
  autocmds/extmark namespaces are re-created rather than preserved by
  identity; behavior equivalent, mechanism different.

* Risks (candidate stumpers; each has a planned fallback)
- nvim-orgmode interference: it may bring mappings or edits (indent,
  todo behavior) that fight server-owned text. Fallback (Q2): a
  self-rolled minimal org layer (foldexpr on stars + TAB cycle +
  tree-sitter highlighting only).
- bodyFolded fidelity: representing "body hidden, children visible" in
  vim's line-range folds needs the foldexpr to give a body its own fold
  one level deeper than its headline. If a corner case can't be
  represented, the deviation is recorded (marker still round-trips).
- goto-in-magit-parent's "point on the first hunk line containing this
  id" requires programmatic section/hunk navigation; neogit's API may
  not expose it. Fallback: open the repo's diff in diffview or a plain
  git-diff buffer and search for the id — same information, different
  chrome.
- The response-handler registry is keyed by response-type (same as
  Emacs), so the same in-flight-collision PITFALL applies; the port
  reproduces the Emacs behavior rather than redesigning it.

** Q4: Keybinding scheme
Terminal vim cannot use the Emacs chords (C-c is taken; C-x C-s is
untypeable). Proposed scheme, normal-mode, in skg buffers only:
- ':w' saves (BufWriteCmd), replacing C-x C-s.
- '<localleader>' replaces the C-c prefix, then the SAME mnemonic
  letters as Emacs: 'C-c f i' → '<localleader>fi', 'C-c g b' →
  '<localleader>gb', 'C-c s d' → '<localleader>sd', 'C-c v h' →
  '<localleader>vh', 'C-c p C' → '<localleader>pC', etc. Where Emacs
  uses RET as the final key, the letter doubles: skg-search
  ('C-c f RET') → '<localleader>ff', skg-goto ('C-c g RET') →
  '<localleader>gg'. Capital variants keep their capitals
  ('C-c G m' → '<localleader>Gm', 'C-c O i' → '<localleader>Oi').
  Delete ('C-c <backspace>') → '<localleader>dd', recursive
  '<localleader>dR' (no prefix-arg idiom worth keeping).
- Motions get vim idioms too: id-next/id-prev also on ']i' / '[i';
  the C-a title-toggle becomes remapped '^' (first press: title start;
  again: column 0).
- Everything is also an Ex command (':SkgGoto', ':SkgSearch', ...), the
  discoverable/mappable surface, so users can rebind freely.
- Inside prompts (our command-line inputs), linkstack paste/pop get
  c-mode mappings, mirroring the Emacs minibuffer bindings.
Alternative if you prefer: a which-key-style menu under one key, or
literal-as-possible Ctrl chords (I advise against — collisions with
core vim bindings are unavoidable). Also: default '<localleader>'
suggestion? (vim convention is often ',' or '\'; users can override.)
