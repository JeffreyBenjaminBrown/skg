* Goal
Make Skg easy to install and run on Windows 10 or 11 for a user who has:
- Emacs
- Docker
- a DockerHub account

The desired deliverable sounds like a downloadable package containing:
- a =docs/= folder
- an install script that fetches the Docker image
- a run script that:
  - asks where to mount the Skg data folder
  - launches the Docker image
  - runs =bash/start-servers.sh= inside the container
  - launches Emacs if it is not already running
  - loads Skg into Emacs the way =docs/setup.org= currently does

* Current Linux shape
The current setup flow is:
- build or pull =jeffreybbrown/hode:latest=
- run a Linux container with the project source bind-mounted at =/home/ubuntu/host=
- run =bash/start-servers.sh= inside the container
- initialize Emacs with:

#+begin_src elisp
  (progn (load-file "../elisp/skg-reload.el")
         (skg-reload)
         (skg-client-init "../data/skgconfig.toml"))
#+end_src

=bash/start-servers.sh= starts TypeDB, then starts Skg through =cargo watch -w server/ -x "run --bin skg -- $SKG_CONFIG"=.
That means the container currently needs the Skg source tree, not only the user's data.

The current =bash/docker.sh= is host-specific:
- container name =skg2=
- host source path =/home/rust/skg2=
- container source path =/home/ubuntu/host=
- Linux audio mount =/run/user/1000/pipewire-0=
- Linux user =1000:1000=
- platform =linux/amd64=
- port mapping derived from =data/skgconfig.toml=

* Desired starter data
The starter =data/= folder should resemble the current real data folder, but only include:
- =public/=
- =private/=
- =list-of-repositories.txt=
- =bash/pull-all.sh=
- =bash/push-all.sh=
- =bash/remote-all.sh=
- =bash/status-all.sh=

=public/= and =private/= should be empty git repositories.
=list-of-repositories.txt= should list only those two repos.

A likely starter =skgconfig.toml= would be:

#+begin_src toml
  db_name = "skg"
  tantivy_folder = ".index.tantivy"
  port = 1731
  beep_when_server_becomes_available = false

  [[sources]]
  name = "public"
  path = "public"
  user_owns_it = true

  [[sources]]
  name = "private"
  path = "private"
  user_owns_it = true
#+end_src

I have not yet confirmed whether =beep_when_server_becomes_available = false= is desired for Windows, but it avoids depending on Linux audio plumbing.

* Packaging uncertainties
The major open design choice is whether the Windows package should include the Skg source tree.

The current Docker image supplies Rust, TypeDB, Emacs, Magit, cargo-watch, and related tools, but =docker.nix= does not appear to copy this Skg repository into the image.
The Linux run path gets the source by bind-mounting it from the host.

For Windows, that implies one of these directions:
- include the Skg source tree in the downloadable package and mount it into the container
- build a new image that contains the Skg source and either runs =cargo run= there or includes a prebuilt Skg binary
- split "developer image" from "user image", with the user image containing only runtime dependencies and the built Skg server/client files

* Implemented first pass
The first pass lives in =windows/=.

It includes:
- =Dockerfile=, which builds =jeffreybbrown/skg:latest= from this repo
  by copying the source into =/opt/skg= and installing a prebuilt
  =/usr/local/bin/skg= binary.
- =container/start-servers.sh=, which starts TypeDB and the prebuilt Skg
  server binary inside the container.
- =install.ps1=, which pulls the Docker image.
- =run.ps1=, which:
  - asks for and remembers the data folder
  - validates an existing =skgconfig.toml=
  - initializes =public/= and =private/= as empty git repositories when no
    config exists yet
  - copies the helper bash scripts into the data folder
  - reuses an existing same-name container only when it uses the requested
    image and data mount
  - exposes only the Skg server port to Windows
  - searches for =emacsclientw.exe= and lets the user choose or type a path
  - starts native Windows Emacs if necessary
  - loads Skg into native Windows Emacs with the same =skg-reload= /
    =skg-client-init= pattern as =docs/setup.org=
- =starter-data/bash/*.sh=, the four repository helper scripts.
- =docs/=, with user and image-build notes.
- =build-package.sh=, which assembles a downloadable package folder with
  the two PowerShell scripts, docs, starter data scripts, and the Elisp
  client files.

This execution environment does not have Docker installed, so the image
build has not been tested here.
