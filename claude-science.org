* Claude's reference notes, discoveries and methods
This file is a reference repository for patterns, benchmarks and
techniques discovered during development. Bug investigations live
in their own files (e.g. save-is-slow.org); only brief summaries
appear here.

** TypeQL v3 reference

*** Value types
TypeDB v3 supports: =string=, =integer=, =double=, =boolean=, =datetime=.
NOT =long= (use =integer=). NOT =float= (use =double=).

*** Attribute definition
#+begin_example
define
  attribute depth, value integer;
  entity node,
    owns depth;
#+end_example

*** Cardinality
Default =owns= is =@card(0..1)= — one value per attribute type.
Adding a second value causes commit error [CNT5].
For multi-valued: =owns nickname @card(0..);=.

*** Reserved words
=label= is reserved in TypeQL v3. Cannot use as identifier. ([DEX31])

*** Querying by property value
Exact match: =has tag "leaf"=
Bind and compare: =has depth $d; $d > 1;=
Boolean: =has is_root true=

*** Bulk container lookup
Single query returning (contained, container) pairs for N nodes:
#+begin_example
match
  $child isa node, has id $child_id;
  {$child_id == "c";} or {$child_id == "d";} or {$child_id == "f";};
  $parent isa node, has id $parent_id;
  $rel isa contains (container: $parent, contained: $child);
select $child_id, $parent_id;
#+end_example
Nodes with no container simply don't appear in results (root detection).
Performance: 5 nodes ~143ms, 20 nodes ~156ms (nearly flat scaling).

*** Fetch syntax (clean JSON output)
#+begin_example
fetch {
  "contained": $child_id,
  "container": $parent_id
};
#+end_example
Returns ={ "contained": "c", "container": "a" }= per row.
Nested subqueries return arrays:
#+begin_example
fetch {
  "contents": [
    match $child isa node, has id $cid;
          $rel isa contains (container: $n, contained: $child);
    fetch { "child_id": $cid };
  ]
};
#+end_example

*** Disjunction patterns and performance
=$child_id == "x"= style (bind then compare) ~143ms for 5 nodes.
=$contained has id "x"= style (attribute lookup) ~181ms for 5 nodes.
The == style is faster because it compares already-bound values.

PITFALL from all_graphnodestats.rs: complex nested fetch + large
disjunction scales nonlinearly (1 PID: 15ms, 24 PIDs: 4300ms).
Simple select + disjunction scales nearly linearly IN CONSOLE TESTS,
but in-process the =or= pattern for extra_id fallback dominates:
=find_containers_of_bulk= (N-way disjunction + =or= for extra_id):
  10 nodes: ~4.2 seconds per query.
=find_containers_of= (single node, =or= for extra_id):
  10 nodes via join_all: ~450ms per level (~200ms per query).
=find_container_ids_of_pid= (single node, no =or=, direct =has id=):
  10 nodes via join_all: total path stats in ~500ms for 5 levels.
Lesson: the extra_id =or= clause is very expensive.
When you know inputs are primary IDs, skip it.

*** Reduce (aggregation)
=reduce $count = count;= — global count.
=reduce $count = count groupby $child_id;= — grouped (one word: groupby).

*** Update syntax (v3)
Delete uses =has $var of $entity=:
#+begin_example
match $n isa node, has id "e", has tag $old_tag;
delete has $old_tag of $n;
insert $n has tag "leaf";
#+end_example

*** Computed values with let
=let $doubled = $d * 2;= works after match clause.

*** Optional match (try)
#+begin_example
try {
  $parent isa node, has id $pid;
  $rel isa contains (container: $parent, contained: $n);
};
#+end_example
Returns empty $pid for nodes with no container. Both matched
and unmatched rows are included in results.

*** Shared variable joins
Two entities sharing a variable are joined:
=has depth $d= on both =$target= and =$sibling= finds same-depth nodes.

*** Relation properties
Relations can own attributes:
=relation contains, owns weight;=
Query: =$rel isa contains (...), has weight $w;=

*** Identity vs equality
=not { $c1 is $c2; }= tests identity (different instances).
=$child_id == "c"= tests value equality.
=$r1 != $r2= does NOT work in v3 — throws [REP1].

*** Entity inheritance
=entity special_node sub node;= — subtypes inherit all owns/plays.

*** Recursive functions (=fun=)

TypeDB v3.7.0 has NO rules (removed from v2). Instead it has =fun=.
No =function= keyword — it's =fun=.

**** Syntax
#+begin_example
define
  fun ancestors($n: node) -> { node }:
    match
      { $rel isa contains (container: $anc, contained: $n); }
      or {
        let $mid in ancestors($n);
        $rel2 isa contains (container: $anc, contained: $mid);
      };
    return { $anc };
#+end_example

**** Return types
- Stream (multiple results): =-> { node }:= with =return { $var };=
- Single value: =-> integer:= with =return first $var;=
- Cannot return literals (=return { true };= fails — must return a variable)

**** Recursive tabling
TypeDB automatically tables recursive functions to break cycles.
For a cycle h→i→h, =ancestors(h)= returns ={i, h}= — finite.

**** Variable scoping in disjunctions
Variables inside =not { }= or =try { }= or disjunction branches are locally scoped.
Cannot reuse the same variable name across different =try= or =not= blocks.
Must use unique names per branch ($r0, $r1a, $r2a, etc.).

**** root, fork, end-of-containerward-path: all expressible

**root** (uncontained):
#+begin_example
fun roots() -> { node }:
  match
    $n isa node;
    not { $rel isa contains (contained: $n); };
  return { $n };
#+end_example

**fork** (multiply contained):
#+begin_example
fun forks() -> { node }:
  match
    $n isa node;
    $r1 isa contains (container: $c1, contained: $n);
    $r2 isa contains (container: $c2, contained: $n);
    not { $c1 is $c2; };
  return { $n };
#+end_example

**end-of-containerward-path** (climb until root or fork):
#+begin_example
fun end_of_containerward_path_v2($n: node) -> { node }:
  match
    { not { $r0 isa contains (contained: $n); };
      $n is $result;
    } or
    { $r1a isa contains (container: $c1a, contained: $n);
      $r2a isa contains (container: $c2a, contained: $n);
      not { $c1a is $c2a; };
      $n is $result;
    } or
    { $r3 isa contains (container: $parent, contained: $n);
      not {
        $r4 isa contains (container: $c1b, contained: $n);
        $r5 isa contains (container: $c2b, contained: $n);
        not { $c1b is $c2b; };
      };
      let $result in end_of_containerward_path_v2($parent);
    };
  return { $result };
#+end_example
Exclusivity: the recursive branch fires only when the node is
neither root nor fork. Without this, TypeQL's non-exclusive =or=
causes ALL matching branches to contribute.

**distinct-end-of-containerward-path** (also stops at cycles):
Pass the origin node as an extra parameter. Add a fourth branch:
"if container equals origin, return self (cycle detected)."
#+begin_example
fun distinct_end($origin: node, $n: node) -> { node }:
  match
    { not { $r0 isa contains (contained: $n); };
      $n is $result;
    } or
    { ... fork detection ...; $n is $result;
    } or
    { $r3 isa contains (container: $parent, contained: $n);
      $parent is $origin;
      $n is $result;
    } or
    { $r4 isa contains (container: $parent2, contained: $n);
      not { $parent2 is $origin; };
      ... not fork guard ...;
      let $result in distinct_end($origin, $parent2);
    };
  return { $result };
#+end_example
Called as =distinct_end($n, $n)=.
LIMITATION: Only detects cycles back to the *origin node*.

** Performance methods and patterns

*** When per-ID parallel queries help vs. hurt

Per-ID parallel queries (via ~join_all~) work well for:
- *Write* transactions (~create_all_relationships~, ~delete_out_links~)
  where sequential write-buffer buildup is the bottleneck.

They do NOT help for:
- *Read* queries that already use efficient bulk disjunctions
  (~which_ids_exist~, ~pids_from_ids~).
- Complex nested queries where per-query overhead is high.

TypeDB server shows significant run-to-run variance
(~fetch_all_graphnodestats~ varies 0.7s–4.4s without code changes).
Concurrency sweet spot: ~4-16 for per-node TypeDB queries.

*** Adding data to an existing pipeline

Pattern for enriching an existing pipeline:
1. Extract new data source's IDs from existing intermediate results
2. Fetch new data in bulk (parallel per-PID queries for TypeDB,
   disk reads for .skg files)
3. Build a lookup map (HashMap<ID, NewData>)
4. Pass the map to the formatter, which looks up each ID

*** Regex compilation: always use static LazyLock in hot paths

~Regex::new()~ per-call cost 20s for 28k nodes (textlinks.rs).
Fix: ~static LazyLock<Regex>~. Impact: 20s → 0.18s (112x).
Also sped up Tantivy indexing (~21s → ~0.7s for 29k docs).

** Bug investigation summaries

For full investigation narratives, see the dedicated files.

*** "Saving takes forever" (RESOLVED — see save-is-slow.org)

Root cause: when a search view (~search:TERM~) is open and a save
modifies a PID in the search results, the search view becomes a
collateral view. ~rerender_view~ calls ~path_from_pid_and_source~
for nodes with fake ~source="search"~. The ~.expect()~ panics,
killing the save handler, so Emacs never receives ~save-result~
and hangs forever.

Fix: filter search views from collateral lists in
~find_collateral_view_uris~ and ~uris_of_views_to_lock~.
Result: 120+ seconds → 0.245 seconds.

Remaining: ~path_from_pid_and_source~ uses ~.expect()~ (latent
crash risk for any invalid source). Deferred.

*** "Cannot visit link" (self-contained, no separate file)

When requesting a content view for a node that's already the ROOT
of the current view, the server returns =((switch-to-view URI))=.
Emacs calls switch-to-buffer on the same buffer — nothing happens.

Fix: in ~skg-handle-content-view-sexp~, detect same-buffer case
and show "Already viewing this node (scroll to top to see it)".
