* Key Project Context

This is the **skg** project - a knowledge graph manipulation system with:
- **Rust server** (`server/`) handling core logic with TypeDB and Tantivy
  - TypeDB is now on version 3, which might be after your training data cutoff date, so bear in mind that your instincts might conflict with the latest syntax.
- **Emacs client** (`elisp/`) providing org-mode interface
- **YAML data storage** in `.skg` files
- **TCP communication** between components:
  - Port 1730 (default): Rust-Emacs API communication (configurable)
  - Port 1729: Rust-TypeDB database connection (TypeDB server)

* Formatting Rules

When writing or modifying code:
- **Type signatures**: Name, each argument, and return type on separate lines
- **Variable definitions**: Type signature required, newline after equals sign
- **Function order**: Callers before callees within same module
- **Whitespace**: Honestly, hard to describe, but try to follow the precedents you find in the code. My (this is Jeff speaking) goal is readability.
- **Bracket grouping**: Adjacent punctuation grouped `(( ))` not `(())`
- **Comments**: Inside code blocks they apply to, preserve existing comments
- **No empty lines** in function definitions unless necessary for comment clarity.
- **Don't reformat existing code** - only apply rules to new/modified code

* Testing Dependencies

- `typedb server` must be running for Rust tests to pass
- `cargo run` must be running for some Emacs Lisp tests, and for all integration tests, to pass.

The script `run-servers.sh` persists those things. Only one TypeDB server should be running at a time, but many Skg servers (`cargo run --bash skg`) can run at the same time. Config files like the one at `data/skgconfig.toml` can be used to change TCP ports, database names and data folders,, so that different instances of Skg don't collide.

* Running Tests

There are three ways to test the code:

1. **Emacs Lisp tests**: `bash/emacs-tests.sh`
   - Runs all automated Emacs tests in `tests/elisp/`
   - Uses Emacs batch mode with ERT (Emacs Lisp Regression Testing)
   - Some tests require `cargo run` to be running

2. **Rust unit/integration tests**: `cargo nextest run`
   - Runs Rust tests in parallel
   - Requires `typedb server` to be running
   - Tests use separate database names to avoid race conditions

3. **Integration tests**: `bash/integration-tests.sh`
   - Runs end-to-end tests that exercise both Rust and Emacs components
   - Tests run in parallel with isolated configurations
   - Requires both `typedb server` and `cargo run` to be running
   - Individual tests located in `tests/integration/*/run-test.sh`

* The save-buffer pipeline

  The central pipeline. When Emacs saves a buffer, the server:

  1. Parses buffer text -> ViewNode tree ('org_to_uninterpreted_nodes')
  2. Enriches with missing IDs/sources ('add_missing_info_to_forest')
     - New nodes get UUIDs; sourceless nodes inherit parent's source
  3. Validates tree structure ('find_buffer_errors_for_saving')
  4. Converts to checked types ('unchecked_to_checked_tree')
  5. Generates save instructions ('viewnode_forest_to_nonmerge_save_instructions')
     - Sub-steps: naive -> reconcile same-ID -> filter unchanged -> supplement from disk
  6. Validates foreign nodes ('validate_and_filter_foreign_instructions')
  7. Extracts merge instructions if any
  8. Updates graph ('update_graph_minus_merges'):
     FS first (source of truth), then TypeDB + Tantivy in parallel
  9. Re-renders saved view and collateral views ('update_views_after_save')

  Key entry point: 'handle_save_buffer_request' in server/serve/handlers/save_buffer.rs.
  Key data flow: buffer text -> UncheckedViewNode -> ViewNode -> DefineNode (Save|Delete) -> FS/TypeDB/Tantivy.

* Source (SourceName)

  Source is a SourceName (String wrapper) identifying which configured directory holds a node's .skg file. It is:
  - NOT serialized in .skg files — inferred from file location at load time
  - Stored in TypeDB as a node attribute and in Tantivy as a search field
  - Used by 'path_from_pid_and_source' to compute file paths: source.path / pid.skg
  - Governed by user_owns_it flag: foreign sources are read-only

  All graph relationships (contains, subscribes_to, etc.) reference nodes by ID, not by source. Source is purely about where the .skg file lives on disk.

* Relationships and the view system

  Graph relationships (all ID-based, source-independent):
  - contains: parent -> child ordering (Vec<ID>, order matters)
  - subscribes: subscriber -> subscribee (lensing into foreign data)
  - hides_from_its_subscriptions: filtering within subscriptions
  - overrides_view_of: replacing a node in views (not yet fully implemented)
  - textlinks_to: org-roam style [[id:...]] hyperlinks

  The view system is ephemeral scaffolding rebuilt every save:
  - TrueNode: corresponds to an SkgNode, can be Definitive (editable) or Indefinitive (read-only)
  - Scaffold: display-only nodes (AliasCol, IDCol, SubscribeeCol, HiddenInSubscribeeCol, etc.)
  - 'complete_viewtree' does two DFS passes:
    preorder (parent-first): reconcile children, add subscription scaffolds
    postorder (child-first): expand aliases, IDs, hidden columns

* Recovery and source of truth

  The filesystem (.skg files) is the sole source of truth. TypeDB and Tantivy are derived caches. If either fails during a save, the server rebuilds them from disk ('rebuild_typedb_from_disk', 'rebuild_tantivy_from_disk'). This means any operation that correctly updates the filesystem is recoverable even if the DB updates crash.

* TypeDB sub-fetch deduplication (v3.4)

  When a fetch sub-query joins across a one-to-many relationship (e.g., matching a node and all its contents), the result list can contain duplicate entries for the "one" side.

  Example: finding link sources that are nonempty containers requires joining =textlinks_to= with =contains=. If source A has 3 contents, the raw sub-fetch returns A three times.

  Approaches tested (on skg-real, TypeDB driver 3.4.4):
  - =select $sid= alone: does NOT deduplicate (still returns 2 rows for a 2-content source)
  - =select $sid; distinct;= before =fetch=: DOES deduplicate (returns 1 row). This is the best available approach.
  - =reduce $cnt = count; return first $cnt;= as a scalar sub-fetch: works, returns a =Node::Leaf(integer)= instead of a =Node::List=. Useful for pure counts but we have not used this pattern elsewhere yet.
  - =limit 1= in a sub-fetch: works but truncates the entire list, not per-source. Not useful for per-row existence checks.
  - =limit 1= in a nested sub-sub-fetch: parse error. TypeDB 3.4 does not support =limit= inside a fetch-within-a-fetch.
  - True per-source early-exit (stop after first content hit) is not possible in TypeDB 3.4. =distinct= still does the full join internally.
