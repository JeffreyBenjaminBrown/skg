* Show containerward paths from search results

** What we're building

When the user searches for titles, the results buffer currently shows
a flat list of matches. We enrich each result with its
containerward path — the chain of containers leading up from that node,
rendered as nested children. This reuses the existing
'paths_to_first_nonlinearities' function
(in [[./server/dbs/typedb/paths.rs][paths.rs]]),
which already powers 'skg-branch-containerward'.

The results buffer is immutable (the user cannot edit it).
The server sends an initial response with placeholder text
under each result, then a second response with the actual paths.
The user can interact with Emacs (including clicking search result links)
while the paths are being computed.

** Making the search results buffer immutable

The search buffer currently gets 'org-mode' and 'heralds-minor-mode'
but no read-only protection. We add 'buffer-read-only'.
The Emacs code already uses '(let ((inhibit-read-only t)) ...)'
when writing into the buffer, so read-only won't interfere
with programmatic updates.

** The two-phase protocol

We change search from a simple request/response to a two-phase
length-prefixed protocol, following the precedent set by the save handler.

*** Phase 1: immediate results with placeholders

The server sends a length-prefixed response containing the search results
as they exist today, but with a placeholder body under each level-2 headline:

  #+begin_src org
    * (skg type:searchResult) second
    ** (skg (node (id 5a) (source search) indefinitive)) score: 1.29, [[id:5a][imperfect test second]]
    containerward paths in progress, should appear soon
    ** (skg (node (id 2a) (source search) indefinitive)) score: 0.98, [[id:2a][This is a second test file.]]
    containerward paths in progress, should appear soon
  #+end_src

The client displays this immediately and makes the buffer read-only.

*** Phase 2: enriched results with paths

The server computes containerward paths from each search result node
and sends a second length-prefixed response.
This replaces the buffer contents entirely
(like save phase 2 replaces the saved buffer).

  #+begin_src org
    * (skg type:searchResult) second
    ** (skg (node (id 5a) (source search) indefinitive)) score: 1.29, [[id:5a][imperfect test second]]
    *** (skg (node (id container-of-5a) (source search) indefinitive parentIgnores)) container-of-5a
    **** (skg (node (id root) (source search) indefinitive parentIgnores)) root
    ** (skg (node (id 2a) (source search) indefinitive)) score: 0.98, [[id:2a][This is a second test file.]]
    *** (skg (node (id container-of-2a) (source search) indefinitive parentIgnores)) container-of-2a
  #+end_src

If a search result node is already a root (no containers),
no children appear under it — the placeholder simply disappears.

** Backgrounding and delivery

After sending phase 1, the server spawns a background
'std::thread' (with 'block_on' for async calls)
for path computation. The connection thread returns immediately
to reading requests from Emacs.

*** Shared result slot

The background thread builds the enriched string
and stores it in a shared slot: 'Arc<Mutex<Option<String>>>'.
Only the connection thread ever writes to the 'TcpStream'.

*** Server-side polling with 'set_read_timeout'

'TcpStream::set_read_timeout' with ~100ms makes the connection
thread's 'read_line' wake periodically. After each timeout
(or real request), the thread checks the slot.
If 'Some', it sends the value as a tagged LP message
and clears the slot.

The ~100ms latency after the background thread finishes
is negligible compared to path computation (seconds).
Emacs is unaffected — it receives data asynchronously
via its process filter.

*** Tagged LP messages

The server wraps phase 2 in a tagged LP message:
'((search-enrichment . "SEARCH_TERMS") (content . "..."))'

'skg-handle-rust-response' checks the tag
(it already does content-based routing for busy signals)
and routes to the corresponding search buffer,
bypassing the current 'skg-doc--response-handler'.

This means the background task survives new requests.
The user opens a content view, saves, does whatever —
and the search results silently update when paths are ready.

** Title lookup for path nodes

The existing containerward-view code
(in [[./server/to_org/expand/backpath.rs][backpath.rs]])
already solves this: 'prepend_indefinitive_child_with_parent_ignores'
calls 'skgnode_and_viewnode_from_id', which resolves the ID
via TypeDB, reads the SkgNode from disk, and extracts the title.

The background thread has access to the TypeDB driver
and the SkgConfig, so it reuses this same approach.

** Rendering the containerward paths

Each search result is a level-2 headline. Its containerward path
becomes a chain of level-3, level-4, ... headlines below it.

For a result node with ID 'x' whose containerward path is
'[a, b, c]' (where 'a' is x's container, 'b' is a's container, etc.)
and the path ends in branches '{d, e}':

  #+begin_src org
    ** (skg (node (id x) (source search) indefinitive)) score: 1.29, [[id:x][title of x]]
    *** (skg (node (id a) (source search) indefinitive parentIgnores)) title of a
    **** (skg (node (id b) (source search) indefinitive parentIgnores)) title of b
    ***** (skg (node (id c) (source search) indefinitive parentIgnores)) title of c
    ****** (skg (node (id e) (source search) indefinitive parentIgnores)) title of e
    ****** (skg (node (id d) (source search) indefinitive parentIgnores)) title of d
  #+end_src

Branch ordering follows the same convention as 'integrate_branches_in_node':
sorted then prepended (so visual order is reversed from sorted order).

For fork expansion (origin immediately forks): each branch is expanded
to its own first nonlinearity, producing multiple path subtrees.

Alias headlines (AliasCol scaffolds) are visually distinct
from path nodes ('TrueNode' with 'parentIgnores').
Phase 2 adds paths alongside the AliasCol, not replacing it.

** Phasing the implementation

1. Make search buffer read-only (small, self-contained).
2. Add containerward path computation for search results
   (server-side, synchronous — phase 2 blocks the connection thread).
3. Background the path computation with tagged delivery.

Step 2 is useful on its own even without backgrounding:
the user gets enriched search results, just with a delay.
Step 3 is the polish that makes it non-blocking.

** Testing

The core testable units are:
- 'generate_enriched_search_results' (pure: match data + paths → org string)
- 'compute_paths_for_search_results' (async but deterministic given a graph)

The two-phase protocol can be tested via integration tests
(start server, send search, verify two LP responses).

* DONE : reasoning

** Why a shared slot rather than a stream mutex

Wrapping 'TcpStream' in 'Arc<Mutex<TcpStream>>' would require
every handler that writes to the stream to lock the mutex —
'handle_save_buffer_request', 'handle_single_root_view_request',
'handle_title_matches_request', 'handle_close_view_request',
'handle_get_file_path_request', 'handle_verify_connection_request',
'handle_shutdown_request', 'handle_git_diff_mode_request',
and 'send_response' / 'send_response_with_length_prefix' themselves.

The shared-slot approach keeps the stream single-writer
(the connection thread only). Only 'handle_title_matches_request'
gains a new parameter.

** Why the server polls, not the client

The 100ms timeout is on the server-side connection thread's 'read_line'
call. Emacs is not polling. Emacs installs a process filter
('skg-handle-rust-response') and its event loop calls it
whenever data arrives on the socket — no polling, no thread needed.

When the user sends a request, the server's 'read_line' returns
immediately. The timeout only fires when idle.

** Why tagged LP messages rather than cancellation

Considered alternatives:
- 'AtomicBool' cancellation: any new request cancels the background task.
  This loses the enrichment if the user navigates away and back.
- Stream mutex: the background thread writes directly to the stream.
  Rejected (see above).
- Dedicated per-search slots keyed by search terms: more complex
  than necessary.

Tagged LP messages are simplest. 'skg-handle-rust-response' already
does content-based routing for busy signals. Adding one more check
('((search-enrichment ...') is minimal.

** Why reuse TypeDB+disk for title lookup rather than Tantivy

Tantivy could do a lighter-weight lookup by exact ID.
But 'backpath.rs' already uses 'skgnode_and_viewnode_from_id'
(TypeDB PID resolution + disk read), and the background thread
won't block the connection. Reusing the existing approach
avoids introducing a new lookup path.

** Why 'std::thread::spawn' rather than tokio

The server uses 'futures::executor::block_on' throughout.
Migrating to tokio would mean adding a dependency,
converting the main server loop to 'tokio::main',
converting synchronous TCP reads to async, etc.
That's a large cross-cutting refactor with no functional benefit
for one background thread per search.

** Why 'buffer-read-only' rather than overlay hooks for search

The save pipeline uses overlay modification hooks
(in [[./elisp/skg-request-save/skg-lock-buffers.el][skg-lock-buffers.el]])
for transient locking during saves. That mechanism is needed because:
- It fires a custom error message.
- 'inhibit-read-only' does NOT suppress overlay hooks,
  forcing conscious unlock before writes.

Search results need permanent immutability, not transient locking.
'buffer-read-only' suffices and is simpler.
