* user-facing skg commands
** C-x C-s : skg-save
Each time you save, your view is rebuilt.
This is because you can put 'requests' into your view,
in which you request more data,
such as a node's ancestors, or the nodes that link to it.
Since views can overlap,
saving one open view can change another.
For this reason the client will warn you
if you have unsaved edits in one view
and beegin to edit another one.
It won't stop you, but the results could be bad.
** C-c <backspace> : deletion
***     C-c <backspace> : skg-delete
Marks the node at point for deletion from the graph.
Upon saving the view, it will vanish not just from this view,
but from the graph itself.
*** C-u C-c <backspace> : skg-delete-recursive
Same as skg-delete, but also marks every truenode org-descendent
of the node at point. Descendent headlines that are not truenodes
(phantoms, aliascol, id-col, etc.) are skipped.

Prefix-arg variant of skg-delete; callable directly via M-x
as skg-delete-recursive.
** C-c f : text search
The 'f' is for 'find'.
*** C-c f RET : skg-search
Text search across titles and aliases, returning every
matching node. "Rooty" nodes (roots, cycle members, link
targets, and nodes that had an org-roam ID before import)
are bumped in the ranking via their context-origin multiplier.

Prompts for search terms. All search axes take their conservative
defaults: no per-token regex, titles only (no body), no Tantivy
phrase and operator syntax.

Run 'skg-goto' (below) from one of the links in the search
results to open the view from that node.
*** C-c f i   : skg-search-interactive
Same as skg-search but asks for three axis flags before
sending:

- *Use per-token regex (not phrase search)?* — yes switches
  to per-token regex. Bypasses the query parser; each
  whitespace-separated piece is handed to Tantivy's
  'RegexQuery'.
- *Body?* — yes adds the body field to the search (titles
  are always searched).
- *Use Tantivy phrase and operator syntax (AND/OR/NOT/+/-)?*
  — yes enables phrase quotes plus AND / OR / NOT / +foo /
  -bar semantics. In non-regex mode this syntax passes through
  Tantivy's QueryParser (phrase quotes, grouping, field: all
  work; intra-word operator chars get escaped so =foo:bar=
  remains findable). In regex mode, AND / OR / NOT / +foo /
  -bar combine the per-piece RegexQueries at the document
  level; phrase syntax does not apply there. See the regex=true
  section below.
**** Query syntax with default axes (phrase/operators=false, regex=false)
Queries are literal. The query string is split on whitespace,
and any word containing a Tantivy operator character
(=+ - & | ! ( ) { } [ ] ^ " ~ * ? : \=) or the bare keyword
=AND= / =OR= / =NOT= is wrapped as a phrase so its contents
are searched literally. Words without specials pass through.
- *OR between words*. "dog cat" matches titles containing
  either word; a title with both outranks one with only one.
- *Literal special characters*. A title like "C++ tips",
  "[draft] plan", or "cat:dog" is findable by typing its
  literal text.
**** Query syntax with phrase/operators=true
The query is handed to Tantivy's QueryParser, with one
heuristic adjustment: any operator character sitting strictly
inside a word (with word chars on both sides, scanning
through consecutive operators within the same token) is
backslash-escaped. So =foo:bar= stays findable as a literal,
but =AND= / =OR= / =+foo= / =-bar= / ="fox in socks"= /
=(dog OR cat)= keep their operator meaning.

Known imperfection: the heuristic can't distinguish a
configured field name from a user word, so
=title_or_alias:dog= gets its =:= escaped too. Field-
qualified queries need manual =\:= in operators mode.
**** Query syntax with regex=true
The query is split on whitespace and each piece is handed to
'tantivy::query::RegexQuery' as its own pattern. Matches are
against *tokens*, not raw text: the indexer lowercases and
splits on whitespace, so no stored token ever contains a
space. That's why splitting is the only way multi-word regex
queries return anything. Patterns like =histor.*= work for
prefix search ('history', 'historical'), and =polit.* scien.*=
combines two such prefix searches. Combine with body=true to
also search body fields.

The *=*= gotcha*: in regex =*= means "zero or more of the
previous character", NOT "any characters" (that's =.*=). So
=sci*= matches the tokens =sc=, =sci=, =scii=, … — not
"anything starting with sci". Use =sci.*= for that.

With operators=false, pieces are OR'd (SHOULD-combined)
like the non-regex default.

With phrase/operators=true (the wire flag is still
=operators=true=), the keywords =AND= / =OR= / =NOT= and the
prefixes =+piece= / =-piece= combine the per-piece RegexQueries
at the document level:
- =poli.* AND sci.*= — documents whose tokens include one
  matching =poli.*= AND one matching =sci.*=.
- =poli.* NOT bio.*= — documents matching =poli.*= and NOT
  any token matching =bio.*=.
- =+poli.* -bio.*= — same as above, without the keywords.
- =poli.* OR sci.*= — explicit OR, same as the default.
Keywords are consumed rather than matched as patterns. An
=AND= adjacent to a piece on either side promotes that piece
to MUST.
**** Caveats
- Each alias is indexed as its own document, so a query can
  match on an alias even when the title doesn't mention the
  searched word. The returned node is still the aliased one.
- Results are ranked by BM25 times a context-origin factor
  (Root 100×, CycleMember ~32×, Target 10×, HadID 10×,
  MultiContained ~3.2×, everything else 1×). A weaker text
  match on a root can outrank a stronger match on a leaf,
  but every match is returned — search is never filtered by
  graph role.
- Regex distance is tokenwise: a pattern never spans
  whitespace, because we split the query on whitespace before
  handing the pieces to RegexQuery. A true whole-title regex
  (one pattern that matches across spaces) would need a
  separate non-tokenized =raw_title= field. Not done yet.
*** C-c f l   : skg-search-make-link
Search-then-link: pick a result, return here, get a link.

Like =skg-search= but tailored for inserting a link to a node
you don't yet know the ID of. The current buffer is saved
first (so detours through the search results' links don't
collide with unsaved edits), and the position of point is
remembered. The same conservative search defaults as
=skg-search= apply.

In the resulting search buffer, find the result you want and
press =C-c C-c=. The search buffer closes; you return to the
buffer you started in, at the position you were at; and a
=[[id:X][LABEL]]= link is inserted at point.

The label is the result's title with any inline links inside
it reduced to their labels (so the new link's label is plain
text, never another link inside a link).
** C-c g : going to a node
*** C-c g RET : skg-goto
Calling this when the cursor is on a link, a node's metadata,
a bare UUID, or a .skg filename
will open a new view from the ID it finds there,
or go to the view of it that was already open if that exists.

If you're not sure whether the cursor is on something visitable
(a link or a node's metadata),
use skg-id-next and skg-id-prev
to identify where those are.
*** C-c G RET : skg-goto-and-close-this
Like skg-goto, but also closes the buffer it was called from.
*** C-c g b   : skg-goto-biggest-branch
Go to the branch near point with the most recursive content,
estimated by counting org-descendents in the current buffer.

If the node at point has siblings, this considers that sibling
set. If it has no siblings, it considers the node's immediate
children instead.

This can help when a view is loading slowly:
replacing the content relationship to its biggest branch
with a link relationship (C-c m l)
reduces the view's demands on the server.
*** C-c g i   : skg-goto-by-id
'skg-goto' is usually more convenient.

This requires a literal v4 UUID string, like
3861db2c-7f16-4814-b403-52b5e05d5e0a
It can be useful going from git to skg.
*** C-c G i   : skg-goto-by-id-and-close-this
*** C-c g m   : skg-goto-in-magit
Opens a magit view of the changes to the file containing the
node at point. If the file appears in multiple sections
(unstaged, staged, untracked), point lands on the first and
the user is warned.
*** C-c G m   : skg-goto-in-magit-and-close-this
*** C-c g M   : skg-goto-in-magit-parent
Opens a magit view of the *parent*'s file, with point on
the first line in that section containing this node's ID.
Warns if the ID appears multiple times, or if the parent
appears in multiple sections.
*** C-c G M   : skg-goto-in-magit-parent-and-close-this
** C-c t : git operations
*** C-c t a         : skg-git-add-if-new-recursive-preview
Brings up a buffer with the text of a command which,
if executed (C-x C-e), runs 'git add' on some nodes.
The nodes should consist only of nodes new to git,
and only of nodes in the recursive content (inclusive)
of the node from which skg-git-add-if-new-recursive-preview
was called. The buffer uses skg-readable-ids-mode.
*** C-c t A         : skg-git-add-if-new-recursive
Like skg-git-add-if-new-recursive-preview,
but it doesn't show you the command;
it instead runs that command for you.
*** C-c (g|G) (m|M) : skg-goto-in-magit-*
These four commands are described elsewhere in this document.
*** C-c t r         : skg-diff-report
Asks whether to include staged changes and, if so,
whether to include unstaged changes.
Then opens an org report summarizing semantic graph changes
across all configured sources.

The report is based on git/index/worktree state,
so the command refuses to run while any Skg view buffer
has unsaved edits.
** C-c r : showing more information about a node
These commands don't edit the graph. Each inserts a request
into the metadata of the node at point that, on save, causes
the server to render additional information into the view.
*** C-c r a : skg-reveal-aliases
See what other text you could search for to find this node.
You can always search for its title.
Most nodes have a title and no aliases.
*** C-c r c : skg-reveal-containerward
Show the nodes that contain the node at point (its parents
in the containment graph, one step out).
*** C-c r s : skg-reveal-sourceward
Show the nodes that the node at point links to (its textlink
targets).
** C-c s : setting node properties
*** first some vocab: 'definitive' and 'indefinitive'
A view of a node can be definitive or indefinitive.
Typically the first place in the buffer that depicts a node
is the definitive view of it. So that other views of that node
cannot contradict it, they are 'indefinitive': Edits to them
cannot affect the node they depict. They are also truncated --
that is, shown without content --
in order to prevent the view from becoming redundant and huge.
*** PITFALL: Don't make something 'definitive' by directly editing its metadata.
Instead, use skg-set-definitive, below.
That's because Skg has to make other instances of it
(in this same buffer -- and maybe in the future other buffers too)
indefinitive. That's because a buffer can only be saved
if it has at most one definitive instance of a node.
Otherwise, two definitive nodes could contradict each other.

This same caveat does not apply to making a node indefinitive.
While there does exist a command for that (skg-set-indefinitive)
you can also 'do it yourself'
by visiting a metadata buffer for the node (skg-view-metadata).
*** C-c s d : skg-set-definitive
If you want to see the definitive view of that node
from a place where it is currently shown in an indefinitive form,
call this function. It will make the current definitive view
(if it exists) indefinitive, and then draw a definitive view here.
*** C-c s i : skg-set-indefinitive
This marks a node as 'indefinitive',
as described above.
In brief, it means edits to its title, body or
  (more importantly) children
will not affect it
  (although edits to its children will affect those children,
  unless they are also indefinitive).
*** C-c s s : skg-set-source
Changes the source of the node at point.
The prompt starts with the current source as editable minibuffer text.
You can type the new source name directly,
or use shift-left / shift-right to cycle through owned sources.
Press C-? from the prompt to open an org buffer listing every
configured source name as a headline,
with that source's path as the headline body.

The command only edits the node's metadata in the current buffer.
Save the view afterward to apply the move.
*** C-c s S : skg-set-source-recursive
Like 'skg-set-source', but it also
changes the source of every content descendent of the node at point
whose source matches the original source of the node at point.
Only descendents for which ParentIs=Affected are traversed.
*** C-c s m : skg-set-merge-request
(See [[./merging.org][the docs on merging]] for why would you merge two nodes,
and what it means to do that.)

Run this from the acquirer node.
It prompts for the acquiree, accepting either a bare ID
or an org link like =[[id:ID][label]]=,
and edits the acquirer's metadata to request a merge.

The command only edits the node's metadata in the current buffer.
Save the view afterward to apply the merge.
** C-c m : other graph modifications
*** C-c m c : skg-replace-link-with-content
Replace the leaf at point with a content relationship to the
node it links to, then save the buffer.

Point can be on the headline or in the node body. The node at
point can be new or already in the graph. If it already has an
ID, the command warns that the old node might become an orphan.

The headline title plus body must contain exactly one org bracket
link, and that link must be an =id:= link. The node must have no
org-descendents. Its org-parent must be a definitive truenode
whose source is owned in =skgconfig.toml=.

The command replaces the whole leaf with a same-level
indefinitive truenode for the link target. The parent is
definitive, so saving records the target as content of the
parent; the replacement child is indefinitive, so its own text is
not rewritten from the link label.
*** C-c m l : skg-replace-content-with-link
Replace the branch at point with a link to the node that used to
be there, then save the buffer.

Point can be on the headline or in the node body.
The command uses the headline either way.

The node being replaced must be an existing truenode with an ID,
and an org-child of a definitive truenode whose source is
owned in =skgconfig.toml=. The command acts locally,
replacing only this occurrence in the current buffer:
the headline at point and all of its org-descendents,
including scaffolds and phantoms,
are removed from the view and replaced by a same-level headline
whose title is an =[[id:...][...]]= link to the original node.

If the original headline title already contains a link, the
command asks "Are you sure?" before proceeding.
** C-c v : view-level commands
*** C-c v d : skg-view-diff-mode
If you use git to manage your data,
enable this view to see what you've changed since your last commit,
i.e. how your worktree differs from HEAD.
*** C-c v e : skg-view-new-empty
Asks you to pick a data source,
and then provides a root node with that source.
**** why
Every node needs a source.
Children can inherit from their parents,
but to make a root (which has no parents)
you need to choose its source explicitly.
*** C-c v h : skg-view-heralds-mode
This command toggles it.
This toggles whether, at the start of every node,
metadata is shown in its full plain-text glory,
or is instead summarized via brief 'heralds'.
It is turned on at every view by default.
*** C-c v l : skg-limit-source-set
Limits display and search to one configured source-set for this
Emacs connection.

The command asks which source-set to activate, with tab completion
and shift-left / shift-right cycling. It then closes open Skg
buffers after confirmation, and tells the server to use that
source-set for subsequent requests. Choose =all= to remove the
limit.
*** C-c v m : skg-view-metadata
When called from a node in a view of the graph,
this switches to a view of the metadata of that node,
with one item per line.
It's easy to edit this way.
Note that some properties don't make sense to edit,
like the count of how many contents it has.
(If you want to change how much content a node has,
don't change the node's metadata; change its children in the view.)
*** C-c v o : skg-view-org-ancestry
If you're at a headline so deep in the tree
that you can't remember where it came from,
this brings you to a new view showing only that ancestry.
This actually works in org-mode outside of skg.
(skg uses org-mode, but org-mode can be used without skg.)
*** C-c v r : skg-readable-ids-mode
This changes how node IDs are displayed.
The ID itself is a kind of special random gibberish called UUID v4.
When skg-readable-ids-mode is on,
only the first 8 characters of the ID are shown,
and they are followed by the title of the node with that ID,
if Skg can figure it out.
(It doesn't yet know how to read the git history
to find titles for deleted files.)
*** C-c v s : skg-view-id-stack
View and maybe edit the set of links you've stored.
(See "the linkstack" below.)
*** C-c v w : skg-view-without-metadata
If a region is active, opens a new org buffer containing that region,
but with the '(skg ...)' metadata stripped from every headline.
If no region is active, it does nothing.
This is useful for copying a read-only projection of some data
into another application.
** C-c {i|u|o} : IDs, links and the linkstack
*** some background
**** IDs are mostly invisible, but good to know about.
Every node has an ID,
which you can see if you disable skg-view-heralds-mode.
Every link also has an ID and a label,
but all you see when you look at the ID is the label,
unless you disable org-mode (say, by switching to text-mode).
You should never need to see IDs explicitly.
**** The linkstack
is, like it sounds, a stack of links.
A stack is just a list that you interact with,
mostly from one side, the 'top':
You can 'push' something onto the top of the stack,
and you can 'pop' something off of the top of the stack.
But you can also jump to a view of your entire stack,
using (copying, pasting, deleting) it
without any restrictions on order.
*** C-c i n : skg-id-next
Makes the cursor hop among two kinds of IDs:
The ID associated with a node in the view,
and the ID associated with a link in a node's title.
Every node has an ID,
but a title can include multiple or no links.
*** C-c i p : skg-id-prev
See skg-id-next, above.
*** the stack of links/ids
C-c u for push, C-c o _ for paste, C-c O _ for pop
**** C-c u : skg-id-push
Stores, in the id stack,
a link to the node the cursor is on.
**** C-c o i : skg-paste-id
Inserts the ID at the top of the id stack at the cursor.
Does not modify the stack.
**** C-c O i : skg-pop-id
Like skg-paste-id,
but also removes the node from the stack.
**** C-c o l : skg-paste-link
Inserts, at the cursor, a link to the node
whose ID is at the top of the id stack.
Does not modify the stack.
**** C-c O l : skg-pop-link
Like skg-paste-link,
but also removes the node from the stack.
**** C-c o n : skg-paste-node
This is the easiest way to give a node multiple containers --
push it to the stack, then paste it as a node somewhere else.

Inserts, at the cursor, an indefinitive view
of the node whose ID is at the top of the id stack.
Its metadata includes only the ID and =indef=,
and its title comes from the stack.
Does not modify the stack or save.
Saving will provide the rest of its metadata.
**** C-c O n : skg-pop-node
Like skg-paste-node,
but also removes the node from the stack.
**** C-c v s : skg-view-id-stack
repeated here; described earlier in this doc
** connection, reload, rebuild dbs
Unless you're coding or troubleshooting,
you probably won't need these.
*** skg-connection-verify
Sends a ping to the Rust server
to check if the connection is working.
Displays the result in the minibuffer.
*** skg-connection-end
Closes the TCP connection to the Rust server.
*** skg-rebuild-dbs
Rebuilds TypeDB and Tantivy databases,
using the .skg files on disk.
*** skg-close-all-skg-buffers
This may be warranted after a database rebuild.
*** skg-reload
Unloads all skg elisp modules and reloads them from disk.
Useful when you've changed elisp code
and want to pick up the changes without restarting Emacs.
