* Programmatic metadata edits must be performed ignoring fragility checks

Naïvely replacing an org heading line via =delete-region= + =insert=
crashes (or silently clobbers text) when the line's body or subtree
is folded right beneath it.

When =delete-region= touches a fold boundary, org-fold's fragility
check (=org-fold-core--fix-folded-region= →
=org-fold--reveal-outline-maybe=) fires and does one of two things:

- signals =beginning-of-buffer=, which aborts the caller partway
  through its work, or
- expands the deletion to cover the entire hidden subtree,
  clobbering its contents.

Either way, the visible behavior is "the heading vanishes and the
fold's text is destroyed." This is extra dangerous because the
error can fire during the OUTBOUND save (=skg-add-focused-marker=
editing the root before any TCP round-trip), leaving the buffer
in a corrupted state that the user cannot recover from.

** What to use instead

Route every per-line heading rewrite through
=skg-replace-current-line= in =skg-metadata.el=. It wraps the
=delete-region= + =insert= in =org-fold-core-ignore-fragility-checks=,
which tells org-fold "the caller is aware of the fold and is
rewriting the line verbatim; skip the reveal-on-touch heuristic."
Suppressing the check is safe there because the stars and title
round-trip unchanged, only the metadata s-expr is updated, and the
fold overlays around the line remain intact.

Do NOT call =delete-region= + =insert= directly on a heading line
whose fold state you do not control.

** Where this matters

Any code that rewrites a heading line in place while keeping the
surrounding fold state. Currently, every such call site in the
skg client goes through =skg-replace-current-line=:

- =skg-edit-metadata-at-point= — merge or create metadata
- =skg-delete-kv-pair-from-metadata-by-key= — remove a kv-pair
- =skg-delete-value-from-metadata= — remove a bare value

If you add another metadata-editing function, route it through
=skg-replace-current-line= for the same reason.

* Can't use forward-sexp on skg metadata

Emacs's =forward-sexp= (and by extension =backward-up-list=,
=up-list=, etc.) chokes on skg metadata because herald fields
contain characters that Emacs treats as unmatched delimiters.

For example, =containsHerald= uses ={= without a closing =}=:

: (graphStats (containers 0) (contents 9) (containsHerald 0{9))

=forward-sexp= sees the ={= and either errors or jumps to the
wrong position, because Emacs's syntax tables treat braces as
paired delimiters.

Similarly, =sourceHerald= uses =⌂:= and =linksHerald= uses =→=,
which can confuse sexp navigation depending on syntax table
settings.

** What to use instead

=skg-find-sexp-end= in =skg-sexpr/skg-sexpr-search.el= counts
parentheses manually, ignoring all other characters. Use it (or
=skg--read-sexp-at-point-on-line= which wraps it) instead of
=forward-sexp= when navigating skg metadata.

** Where this matters

Any Emacs Lisp code that needs to find the extent of an =(skg ...)=
sexp in a buffer or string. Currently:

- =skg-first-sexpr-on-line= — returns the first sexp on the line
- =skg-sexp-at-or-after-point= — returns the sexp at/after point
- =skg-metadata.el= — splits headline into stars, metadata, title
- =heralds-minor-mode.el= — finds metadata extent for overlays

All of these use =skg-find-sexp-end= rather than =forward-sexp=.

* skg-magit-titles overlays break home/end navigation when lines are truncated

=after-string= overlays (used to show titles after UUIDs) add
invisible-to-git text that makes lines visually longer. When the
total visual length exceeds the window width and =truncate-lines=
is =t= (lines cut off at the window edge), =home= can fail to
return point to column 0.

** Mitigation in place

- =skg-magit-titles-mode= sets =truncate-lines= to =nil=
  (lines wrap to the next screen line). With wrapping enabled,
  =home= works correctly.

- The overlay uses =after-string= (not =display=), so it does
  not alter the buffer text. Staging, committing, and other
  git operations are unaffected.

** What does NOT cause this

- =field= text properties: overlays' =after-string= does not
  create field boundaries.
- =cursor-intangible=: magit only uses this on collapsed
  (invisible) sections.
- =front-advance= on the overlay: irrelevant to point movement.
