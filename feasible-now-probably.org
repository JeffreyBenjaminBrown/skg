*** an easier algorithm
**** a 'context' is an (ID, Set (ID)) pair.
     The single ID is the context 'origin'.
     (Think 'root', but it really can be other things too.)
     The Set of IDs is its recursive content,
       truncated without inclusion at other origins.
**** a 'treelike context'
     is a pair (O, S)
     such that S is a connected set
       including O and all of its recursive content
       truncated without inclusion at other context origins.

     So for instance, suppose our graph contains two containment chains
       A  { B  { C { D
       A' { B' { C { E
     and no other relationships. Since C is multiply contained,
     it is an origin. Since A and A' are roots,
     they are origins, too.
     The context from origin C
       includes C, D and E (C's recursive content).
     The one from A includes A and B, but not C,
     because C is an origin.
     Similarly, context A' includes A' and B'.
**** a "treelike context origin" is
     any link target
     any root
     anything multiply contained
***** Sidebar: Why 'treelike'?
     Because Any of these qualities
     indicates there is something sufficiently focal about the node
     that it deserves to stand out in the index.
     It is not just the continuation of a containment tree
       started somewhere else -- in that case,
       the somewhere else desreves index priority.
     That's what the 'is a context' property on nodes does;
     it singles out some of them as particularly interesting
     in search results.
     We will index the text of every node,
     but in search results we will give priority to contexts.
**** to construct all the treelike contexts
     First find all the treelike context origins,
       via three TypeDB queries:
       all roots, all link targets, all multiply contained

     Then "grow" each origin in the contentward direction.
     That is, the origin (left member of pair)
       will be paired with  subset of
         its recursive content including itself,
       where the subset is defined via truncation
       wherever we run into one of the other context origins.

     Procedurally: Collect all its content nodes.
       Discard those that are origins,
       and any that were already in this context.
       Whichever survive that filter, add to this context.
       Then repeat the procedure
         by recursing on the immediate content of what we just added.
       DFS, BFS, shouldn't matter for correctness
       (although maybe it does for speed).
**** a cycles-extended context is
***** Any normal context.
***** Any connected set distinct from the normal contexts.
      These are all, I believe, cycles.
      For now we will highlight each node from the cycles
      with the same 'context' property.
      To find them, I think we just subtract the union of all contexts
      from the entire graph,
      and then run a connected-set algorithm.

**** goal
     We index only the text in the titles of contexts.
     A context is a nonlinearity in the
       containerward or link-sourceward direction.

     One kind of context is a root:
       the countainerward boundary of the tree it defines.
       Every root is a context.

     Anything linked to is also context.
     (Maybe this will be unwanted for links made in skg,
     but it is a desirable property of links from org-roam.)

     If containerward traversal hits a cycle
       -- which perhaps should be discouraged --
       index every node in it.
**** algorithm
     Start a 'to_context' bimap from ID to ID.
       IDs on the right are contexts.
       Each context will map to itself,
       and every ID in the bimap will appear on the left.
       A value on either side might be associated with many in the other.
       (If our BiMap implementation only supports
         set-values in one direction, have it support both.)
       The set of values (not keys) in this map
       is the set of 'recognized contexts'.
       We will grow it.

     First mark all link targets as contexts.
       Each maps to itself.

     We're done processing links.
     Next and last, we process the containment relationship.

     Define a 'discovery' as a type with two fields:
     - a set of nodes that starts empty
     - and an optional context node that start None
       and when Some must be a member of the discovery's set of nodes.

     Subtract the set of contexts (right values in the bimap)
     from the set of all IDs,
     and call the result not_recognized_contexts.
     Some of them are in fact contexts, and others aren't.
     We will consume this set.

     Pick some (64? I don't have feng shui for parallelism)
     nodes at random, running in parallel.
     We consume each of these 64 nodes from not_recognized_contexts.

     To start, we create a discovery with
       context = None and
       nodes = the singleton set with that node in it

     Now we repeatedly process the latest node added to each discovery.

     To process the first node,
     which is treated a bit differently from all the subsequent nodes,
     first check if it is a containerward nonlinearity --
     that is, if it plays the role of content 0 or more than 1 time.
     If so, context!
     Map it to itself, end this one and start a new discovery.

     If not, then it has a container.
     Make that the newest member of the discovery's nodes.

     The way we treat that second
     and every subsequent node in the discovery is this:
     Ccheck whether it appears in the left side of the bimap.
     - If it does, let S be the set of IDs that it maps to on the right.
       Each node in this discovery must map to each node in S.
       End this one and start a new discovery.
     - If it doesn't, check whether it is a nonlinearity.
       That can happen in eiither of two ways:
       - If it was already in this discoveries nodes
         before we climbed containerward to it, that's a nonlinearity.
       - If it has 0 or more than 1 container, that's a nonlinearity.
       In either case, it is a context.
       Add it to nodes, and map every node in nodes to it
       (that is, it maps to itself, and they all do to).
       End this one and start a new discovery.

     If this discovery has not ended,
     start again with the container of
     the newest member of this discovery's nodes.

** recursive properties in TypeQL
   Does TypeQL have a good way to express these functions?
   I'll use Haskell notation
   - root :: node -> bool, true if node plays not the content role
   - fork :: node -> bool, true if it plays that role more than once
   - cycle :: node -> bool, true if it is its own contains-ancestor
   - self-contained :: node -> bool, true if it contains itself
   - end-of-containerward-path :: node -> node. For any n, if root n, self-contained n, or fork n, then end-of-containerward-path n = n. Otherwise, it is equal to distinct-end-of-containerward-path n
   - distinct-end-of-containerward-path :: node -> node. For any n, if root n or fork n or cycle m n, then distinct-end-of-containerward-path m = n. Otherwise it equals distinct-end-of-containerward-path m (container n), which we know exists because n is not a root or a fork.
