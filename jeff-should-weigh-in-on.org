#+title: Choices you might reasonably differ on (fork-fixes)
#+startup: overview

Companion to the execution of TODO/fork-fixes.org. Each entry is a
choice I made where your request was open to interpretation, or where
I picked among defensible designs. None blocks you; everything here is
reversible. Ordered roughly by how likely I think you are to care.

* How I read "It should not ask me if I have already specified a source in the saved metadata"
Your Case 2 buffer specifies a source only on the NEW node ('(source
else-public)' on "Can I add to this?"). The clone-to-be (of the
foreign root) is not itself present in the origin buffer, so the only
way you can have "specified a source in the saved metadata" is via
such new children. I therefore implemented:

- If the fork-forcing edit includes NEW immediate children of the
  forked node N whose metadata EXPLICITLY names an owned source, and
  those explicit sources all agree on one source S, the clone's source
  is S, treated as user-specified: the confirmation buffer shows S
  directly (no PICK-A-SOURCE placeholder) and approving does not ask
  for a source.
- Only explicit sources on NEW nodes count. Two things that do NOT
  count as "specified":
  - the inferred nearest-owned-ancestor source (you did not type it;
    it stays a suggestion, offered as the default in the new prompt);
  - the source riding along on an EXISTING node you dragged under N
    (that metadata came with the headline; you did not write it as an
    instruction about the clone).
  Either of these could be flipped to count as "specified" if you
  prefer; each is a small change (one predicate in
  'explicit_fork_sources_for_foreign_nodes').
- If several new children explicitly name DIFFERENT owned sources,
  that is ambiguous, so the flow falls back to asking.

* The y/n confirmation itself still happens in Case 2
I read your "It should not ask me" as being about the SOURCE question,
not about fork confirmation as such: your same sentence continues
"...it ask me to choose each before allowing me to say 'do it'",
which implies the 'do it' step survives in both cases. The
confirmation gate was also a deliberate safety (an accidental edit of
a foreign node should not silently fork). So in Case 2 you now get
ONLY "Fork the listed node(s)? (y or n)" -- no source dance. If you
want a fully silent fork when the source was specified, say so; it is
a small change in 'skg--fork-confirmation-handler' (and arguably in
the server, to skip the confirmation round-trip entirely).

* In Case 1, the new node lands in the clone's source
A metadata-less new node under a foreign parent now joins the fork
instead of erroring. Its own source: I put it in THE SAME source the
clone ends up in (the one you pick/confirm in the confirmation flow),
on the reasoning that the new node will live in the clone's contains,
and a new node created under any owned node inherits that node's
source. So the flow asks you for ONE source (the clone's) and the new
node follows it. Alternatives I did not build: asking separately for
the new node's source, or defaulting it to your config-first owned
source independent of the clone.

* Where the "choose each source" ask happens: minibuffer prompts at confirmation time
When a clone's source is unspecified, the client now prompts you in
the minibuffer (completing-read over the sources you own, suggestion
pre-filled as the default, S-left/S-right cycling) for EACH unresolved
clone, right when the confirmation buffer appears -- before the "Fork
the listed node(s)?" question. C-c s s in the buffer still works if
you decline the prompts (C-g) and want to do it by hand; C-c C-c then
re-asks only for what is still unresolved. I judged minibuffer
prompting to match "it ask me to choose each before allowing me to
say 'do it'" better than the old silent expectation that you go find
C-c s s.

* The silent-nothing bug's cause, for the record
Answering "yes" while a clone still carried PICK-A-SOURCE called
'skg-approve-fork', whose 'user-error' ("Pick a source for each clone
first...") was swallowed by the surrounding 'condition-case' in
'skg--fork-confirmation-handler' and written only to the skg log --
hence "nothing happens". The interactive flow now runs outside that
'condition-case' (which continues to guard response parsing), so any
refusal is loud.

* Hide-on-removal: scope of the new rule
Your third item ("If my fork F contains C and so does the original O,
and I remove C from F, F should now hide C") I implemented as a
general subscriber rule, not a fork-specific one: on save, for any
OWNED node F, any id removed from F's contains that is (per disk)
content of some node F subscribes to becomes a hide by F. Rationale:
"fork" has no distinct identity in the graph -- it is exactly
subscribe + override -- and docs/sharing-model.md already promised
this behavior for clones ("Branches you then delete from the clone,
or move elsewhere, become branches the clone
hides_from_its_subscriptions"). Consequences you might weigh in on:

- MOVING C elsewhere in the same buffer also records the hide (the
  doc says it should; the rule sees only "C left F.contains").
- The symmetric direction: RE-ADDING C to F.contains removes a stale
  hide of C by F (otherwise C's hiderCol would keep listing F, though
  the subscription view would be unaffected either way).
- If the same save both removes C from F.contains and shows C as a
  visible child of the subscribee-as-such, the explicit
  subscription-view gesture wins: no hide.
- The rule also fires at FORK CREATION: if your fork-forcing edit
  deleted some of N's children, the clone is born hiding them
  (previously it was born with no hides, so deleted-at-fork-time
  children would have reappeared as unintegrated subscribed content).

* Confirmation-buffer header: an org headline with body text
Per your complaint, the long '#' comment block is gone. The buffer now
opens with one top-level headline ("Fork confirmation -- what this
buffer is") whose BODY holds a shortened explanation, so you can fold
it away; each fork follows as before (clone-to-be over original). The
per-clone "Suggested source" line remains a '#' comment directly above
each unresolved clone-to-be (one line; the client parses it for the
prompt default). Unresolved clones still display PICK-A-SOURCE in
their metadata -- with the minibuffer prompting, you should rarely see
it, and I kept it because showing the suggested source there would
make an unconfirmed choice look confirmed.
