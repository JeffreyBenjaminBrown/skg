* background
Before even reading this report, maybe you should first take a detour
to investigate how 'merge' works in Skg.
The basic idea is this: Given nodes A and B with content A1, A2 ... An
and B1, B2 ... Bn, we can 'merge A into B'.
The result is that A vanishes, all of A's content becomes B's content instead
(but B retains its initial content, too),
A's IDs (it probably only has one, but there could be multiple)
become extra IDs of B, and B gains a new node as content
to preserve the title and (if present) body of A.
* how I found the bug
I opened the node titled 'honor'. One of the nodes it contained had the title "alias, was 'duty'". (Those data come from org-roam, which does not permit merging, so it was my DIY version of merging. I 'merged' them by moving all of the content of 'duty' to be content of 'honor', and then retitled 'duty' to "'alias, was 'duty'", and then made that also a child of 'honor'.)

* problems revealed
** The merge request should be removed once executed.
Instead, it is still visible in the metadata of the node that carried it.
I could swear that for other requests there already exists code to remove them,
once they have been executed. Maybe that's just view requests,
and we need to extend it to cover edit requests too.
** The new content does not recognize its new container.
This outcome I don't understand.
Consider the first appearance of node cd4a089d-eb00-460d-b018-d858c2b83298
in 'after-merge.org'. In its metadata, it is marked parentIgnores.
Why is that? The diff reveals that node cd4a089d-eb00-460d-b018-d858c2b83298
is indeed now content of node 24da1aaf-341e-4eb4-b615-1f206bb2ed40.
```
modified   24da1aaf-341e-4eb4-b615-1f206bb2ed40.skg
@@ -2,3 +2,20 @@ title: alias, was 'duty'

+- 24da1aaf-341e-4eb4-b615-1f206bb2ed40

```
I suspect what's happening is either:
- the routine that determines whether a node is its parent's content is not recognizing the connection due to the distinction between primary ID and extra ID
- the map(s) between content and containers are not being appropriately updated during the merge
** The node that got absorbed persists unchanged in the buffer.
If we merge A into B -- that is, A is removed from disk and B persists --
A is still in the buffer. That's okay. But nothing indicates to the user
that it no longer exists, which is not okay.

The lowest-hanging solution would, I think, be to use the 'removed'
herald (already implemented for git-diff mode)
to indicate that the node no longer exists.
But using that outside of the git diff view could be confusing,
both for users and for programmers.
