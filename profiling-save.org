#+TITLE: Profiling the save-buffer pathway
#+AUTHOR: Claude (with Jeff)
#+DATE: 2026-06-05

* TL;DR

The save-buffer pathway is dominated by four costs. For the test case
(insert one child under 'doing', id 846207ef-…, source pers-p, 21
children), a *fresh* save costs ~300 ms and, under sustained editing,
degrades to *over 1 s*. Roughly:

| Cost                                                       | Share | Nature                                                     |
|------------------------------------------------------------+-------+------------------------------------------------------------|
| 'validate_override_invariants_after_save'                  | ~24%  | scans the *whole* 60k-node graph each save; unnecessary    |
| Tantivy commit #2 ('update_context_origin_types')          | ~20%  | a second full IndexWriter + commit, re-doing work          |
| Tantivy commit #1 ('update_tantivy_from_saveinstructions') | ~18%  | first IndexWriter + commit (runs concurrently)             |
| TypeDB relationship rewrite (delete-all + create-all)      | ~15%  | deletes & recreates *all* of the node's edges; O(children) |
| parse + rerender + misc                                    | ~8%   | not pathological                                           |

Two of these are pure waste and are the highest-value fixes:

1. *Override-invariant validation re-scans the entire graph on every
   save* (two passes over all 60,129 nodes), even though a save can
   only affect the nodes it touches. ~140 ms (debug build), and it
   grows as the user's data grows. → Scope it to the touched nodes.

2. *Tantivy is written and committed twice per save.* The first pass
   indexes each saved node with an empty 'context_origin_type'; the
   second pass reads those documents back, deletes them, and re-adds
   them with the computed type — a second 50 MB IndexWriter and a
   second commit. The code already carries a TODO for the fix
   ("compute origin types before indexing"). → Compute the type first,
   index once, commit once. Removes ~20% and halves segment churn.

A third, scaling, problem: *saving a node deletes and recreates ALL of
its TypeDB relationships* (every 'contains' edge), so saving a node
with N children is O(N) and building such a node up is O(N²). Measured
'create_all_relationships' growing from 48 ms (~25 children) to 137 ms
(~100 children). → Apply only the edge delta.

A fourth, related problem: *save cost degrades 3–4× over a session of
repeated saves* (404 ms → 1220 ms across 11 consecutive saves) because
each tiny commit spawns a new Tantivy segment and merges fall behind.
Tied to fix #2.

Full detail, evidence, and ranked recommendations below.

* Test case and environment

- *Operation*: open a single-root content view of node
  =846207ef-11d6-49e4-89b4-4558b2989a60= ('doing', source pers-p, 21
  children, buffer = 103 lines / 10 830 bytes), insert one new bare
  child headline as the root's first child, and save.
- *Build*: debug ('cargo run --bin skg', =target/debug/skg=). This
  matters — see Caveats.
- *Runtime*: the whole save runs on one thread via
  'futures::executor::block_on'. Concurrency is 'buffer_unordered(64)'
  over TypeDB transactions (overlapping network I/O on the one thread)
  plus one 'std::thread::spawn' for Tantivy.
- *Data*: 60 129 '.skg' nodes across 8 sources (public alone is
  29 436). The in-Rust graph holds all of them.
- *Stores*: filesystem (source of truth) + TypeDB (port 1729) +
  Tantivy index, all updated on save.

* Method

The server already has ~39 'tracing::info_span!' probes and, with
'timing_log = true', writes every span's CLOSE event (with
'time.busy' / 'time.idle') to =data/logs/server.jsonl=. On the
single-threaded 'block_on' executor, an '.entered()' span guard held
across an '.await' measures *true wall-clock* for that scope — so the
existing top-level spans already give clean numbers.

To follow the "profile top-level branches, then recurse into anything
>20%, stop at <5%" strategy I:

1. Added '.entered()' span probes (same idiom as the codebase) at the
   *un-instrumented* branches — chiefly 'validate_override_invariants_
   after_save', the graph-update call, the context sub-calls, the
   rerender preprocessing — closing the ~37% "gap" the initial
   top-level spans left unexplained.
2. Drove the save over TCP with a small Python harness (single-root
   content view → insert child → save buffer), chaining each save's
   re-rendered result into the next, and parsed =server.jsonl= into a
   per-branch tree.
3. *Confirmed magnitudes with toggle experiments*: gated each suspect
   branch behind a =/tmp/saveprof/skip_*= flag file (checked at save
   time, so togglable without restarting) and measured the delta with
   the branch on vs. off.
4. *Controlled Tantivy drift*: because per-save cost grows within a
   session (finding F4), I issued a 'rebuild dbs' (which rebuilds a
   compact index) before each A/B config so configs start from the
   same state.

All instrumentation was temporary and has been fully reverted; the
test children created during the run were deleted and the databases
rebuilt from the restored on-disk state (see Reproduction & cleanup).

* Findings (each branch ≥ 5% of the total)

** F1 — Override-invariant validation re-scans the entire graph (~24%, ~140 ms)

*Where*: 'validate_override_invariants_after_save' (server/save.rs),
called *unconditionally* at the top of 'update_graph_including_merges',
which calls 'validate_override_invariants' (server/dbs/in_rust_graph/
override_invariants.rs).

*Why it's slow*: 'validate_override_invariants' iterates
'graph.nodes.iter()' *twice* — once for the "monogamy" constraint
(line 49) and once for the "no chains" constraint (line 75) — over
*all 60 129 nodes*, calling 'user_owns_node' (a config lookup) and
'graph.pid_of' per node. The cost is O(total graph size) and is
*independent of how much the save changed*. As the user's knowledge
graph grows, every save gets slower — matching "it has gotten slow."
(The 'graph_snap.clone()' it does first is cheap: 'InRustGraph' uses
'im::HashMap', so the clone is O(1) structural sharing.)

*Evidence (toggle experiment, rebuild-controlled)*: disabling the call
dropped the total from *343 ms → 256 ms* and
'update_graph_including_merges' from *234 ms → 135 ms*. The branch's
own span measured 130–141 ms across runs.

*This is work that does not need to be done.* The two invariants
("no node is overridden by more than one user-owned node"; "no
user-owned override chains") can only be newly violated by nodes the
save touched, or by the nodes those touch via 'overrides_view_of' in
either direction. The rest of the graph was valid before the save and
is unchanged. → *Scope the validation to the saved nodes plus their
override-neighborhood* instead of the whole graph. (Secondary: the two
full passes could be a single pass; but scoping is the real fix.)

Expected win: ~140 ms in debug (~24% of a fresh save), and it removes
the term that grows with the user's total data.

** F2 — Tantivy is written and committed twice per save (~38% combined)

Each save creates *two* fresh 'IndexWriter's (each with a
'TANTIVY_WRITER_BUFFER_BYTES = 50 MB' buffer) and does *two* full
'commit()' + 'reader.reload()' cycles:

*** F2a — first commit: 'update_tantivy_from_saveinstructions' (~18%, the ~104 ms join gap)

Runs on a 'std::thread::spawn'ed thread *concurrently* with the TypeDB
write, so it is mostly hidden — but it takes *longer* than the TypeDB
write, so the main thread blocks at 'tantivy_handle.join()'. That wait
is the ~104 ms "gap" inside 'update_graph_minus_merges'. The Tantivy
thread's own logged duration ranged 77–364 ms over the run. It deletes
each saved id and re-adds it — crucially with 'context_origin_type =
""' (server/dbs/tantivy/write.rs:124), a placeholder that F2b
immediately overwrites.

*** F2b — second commit: 'update_context_origin_types' (~20%, ~118 ms)

'update_context_types_for_saved_nodes' computes each saved node's
context-origin type (its TypeDB queries are negligible:
'map_to_containers' 0.1 ms, 'link_targets' 5 ms, 'path_stats' 0.1 ms),
then 'update_context_origin_types' (server/dbs/tantivy/context_update.rs)
opens a *second* 50 MB IndexWriter, *searches back* the very documents
F2a just wrote, deletes them, re-adds them with the now-known
'context_origin_type', and commits again.

*Evidence (toggle experiment)*: skipping only 'update_context_origin_
types' dropped 'update_context_types_for_saved_nodes' from *71 ms →
4.2 ms* — i.e. essentially the *entire* context phase is this second
Tantivy commit, not the graph queries.

The file's own header comment already names the fix: "the 'compute
origin types before indexing' TODO". → *Compute 'context_origin_type'
before the indexing pass and write it directly in F2a, deleting F2b
entirely.* This needs a small reordering (origin types depend on the
TypeDB write having landed, so compute them after the TypeDB write but
before/within the Tantivy write rather than concurrently). If that
reordering is undesirable, the cheaper interim fix is to *merge the
two passes into one writer and one commit*, and to *keep a single
long-lived 'IndexWriter' on 'TantivyIndex'* rather than allocating a
50 MB writer (and spawning merge threads) twice per save.

Expected win: ~118 ms (~20%) from dropping F2b, plus less segment churn
(helps F4).

** F3 — TypeDB relationship update is delete-all-then-recreate-all (~15%, grows with node size)

*Where*: 'update_typedb_from_saveinstructions' (server/save.rs) calls
'delete_all_outbound_relationships_to_nodes' then
'create_all_relationships' (server/dbs/typedb/relationships.rs).

*Why it's slow / why it grows*: for every saved node these delete *all
five* outbound relation types and then recreate *all* of them.
'create_all_relationships' opens one Write transaction per node and
re-inserts every edge; for the 'doing' node that is its entire
'contains' list. So saving a node with N children is O(N) *even when
one child changed*, and appending children one at a time is O(N²) over
the session. The delete-all is a deliberate correctness guard (the
comment notes that re-saving otherwise duplicates every non-'contains'
relation).

*Evidence*: within this study 'create_all_relationships' rose from
48 ms (node at ~25 children) to *137 ms* (node at ~100 children); the
client-side per-save wall time rose monotonically as the node grew.

→ *Apply only the edge delta.* The old edge set is in the in-Rust
graph snapshot the pipeline already holds; diff it against the new node
and issue just the adds/removes. Effort is higher (careful TypeDB
work) and the win is proportional to node size — large for hub nodes
like 'doing', small for leaves.

** F4 — Save cost degrades 3–4× over a session of repeated saves

*Symptom*: the *same* operation slows down the more you save. Measured:

- 11 consecutive saves: 404 → 1220 ms (client wall).
- ~35 consecutive saves: 308 → 1425 ms (server total).

*Cause*: each save commits a tiny change to Tantivy, creating a new
segment; segments accumulate and the merge policy falls behind, so each
subsequent 'commit()' costs more. Confirmed by 'rebuild dbs' (which
rebuilds a compact index) resetting the cost back to ~300 ms.

→ Largely subsumed by F2 (one writer, one commit per save → far fewer
segments). Additionally: keep a long-lived writer with a tuned merge
policy; consider that Tantivy is a *derived cache* (rebuildable from
disk, and the pipeline already recovers it on failure), so commits
could be debounced/batched rather than forced every save. The
'reader.reload()' after every commit (commit_with_status,
write.rs:150) forces a synchronous refresh that exists for test
determinism; production could rely on the default
'ReloadPolicy::OnCommitWithDelay'.

** Minor branches (5% ≥ each, noted for completeness)

- 'buffer_to_validated_saveplan' (~4–7%, ~22 ms): split between
  'org_to_uninterpreted_viewforest' (~7 ms parsing) and
  'extract_nonmergeSavePlan_from_authority' (~7.5 ms). Not pathological.
- 'update_views_after_save' / 'rerender_view (saved)' (~2–4%, ~11 ms):
  'complete_viewforest' is fast for this node; fine.
- 'coherence_debug_assert' / 'define_nodes_build': ~0.03 ms. Negligible
  even in debug.

* Opportunities for parallelism

- F2b's Tantivy commit (~118 ms) currently runs *after* the whole
  graph-update phase, sequentially. Folding the context-origin type
  into the single Tantivy pass (F2) removes it; failing that, its cheap
  TypeDB reads could overlap the main update.
- The per-save concurrency that *does* exist — TypeDB write ∥ Tantivy
  write — already hides most of the smaller of the two. The win is not
  "add more threads" but "do less work" (F1, F2, F3).
- 'source_moves' and collateral-view rerenders are sequential
  ('TODO ? parallelize' in the code), but were ~0 for this case (no
  source moves, no collateral views).

* Recommendations, ranked

1. *F1 — scope override validation to touched nodes.* Biggest, and the
   one that grows with the user's data. ~24%. Moderate effort, low/mod
   risk.
2. *F2 — index Tantivy once: compute 'context_origin_type' before the
   indexing pass and drop 'update_context_origin_types'; reuse a single
   long-lived 'IndexWriter'.* ~20% plus it fixes F4's degradation.
   Moderate effort (a pipeline reordering), moderate risk.
3. *F4 — fewer/cheaper Tantivy commits* (largely falls out of F2;
   otherwise debounce commits / tune the merge policy / drop the
   forced 'reader.reload()' in production). Stops the within-session
   slowdown.
4. *F3 — incremental TypeDB edge updates* (diff instead of
   delete-all/create-all). Scales the save to the *change*, not the
   node size. Higher effort, biggest payoff for hub nodes.

Doing F1 + F2 alone would, in this debug build, take the fresh save
from ~300 ms to ~150 ms and (via F2/F4) stop the runaway degradation —
the change a user would feel most.

* Caveats

- *Debug build.* The server runs unoptimized ('cargo run'). Release
  would shrink the CPU-bound costs (F1's scan, parsing, in-Rust ops)
  more than the I/O-bound ones (TypeDB round-trips, Tantivy
  commit/fsync). So in release the Tantivy/TypeDB share would likely
  *grow* relative to F1, but every structural finding (whole-graph
  scan, double commit, O(children) edge rewrite, segment drift) is
  independent of optimization level. Re-profiling a release build is
  worthwhile before committing to exact percentages.
- *One node, one operation.* Insert-a-child. Delete- or merge-heavy
  saves exercise other paths ('apply_delete_propagation_cleanup' was
  ~0 here because there were no deletes).
- *Numbers are medians* over 6–13 saves; absolute ms drift with session
  state (F4), which is why magnitudes were confirmed by toggling each
  branch on/off rather than by cross-session absolute comparison.
