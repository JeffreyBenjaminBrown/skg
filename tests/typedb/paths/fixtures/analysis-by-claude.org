* Test plan for multi-cycle fork

** Graph

Contains edges (X contains Y):
| container | content |
|-----------+---------|
| b         | a       |
| b         | e       |
| c         | b       |
| d         | e       |
| d         | c       |
| e         | d       |
| f         | e       |

Fixtures are at [[./tests/rebuild/fixtures-multi-cycle/][fixtures-multi-cycle/]].

** What the test does

Two assertions, both using the same TypeDB graph:

*** Part 1: assert on 'PathToFirstNonlinearity' struct

Call 'path_containerward_to_first_nonlinearity' on node a.
Assert:
- path = [b, c, d, e]
- cycle_nodes = {b, d}
- branches = {b, d, f}

This verifies that multiple cycle nodes are detected
within a single fork. It's the core thing the spec
is designed to exercise.

*** Part 2: assert on the rendered tree

Call 'build_and_integrate_containerward_path' on a.
Assert the tree matches:

  * a
  ** b
  *** c
  **** d
  ***** e
  ****** f
  ****** d
  ****** b

Branch ordering: 'integrate_branches_in_node' sorts [b, d, f]
then prepends in order, so the visual order is reversed: f, d, b.

Note: cycle_nodes is discarded when branches is non-empty
(line 141 of [[./server/to_org/expand/backpath.rs][backpath.rs]]).
The rendered tree doesn't distinguish cycle-branches
from non-cycle-branches. That's why part 1 exists.

** What code paths this exercises

'path_to_first_nonlinearity' walks a → b → c → d → e,
each step finding exactly one container that isn't in path_set.
At e, it finds containers {b, d, f}: three related nodes,
two of which (b, d) are in path_set. Returns with non-empty path,
non-empty branches, and non-empty cycle_nodes.

'paths_to_first_nonlinearities' receives a result
where path is non-empty, so it returns a singleton list.
This is distinct from 'test_fork_expansion_at_origin',
where path is empty and sub-expansion triggers.

Together the two tests cover:
- immediate fork at origin → sub-expansion (existing test)
- linear path then fork with multiple cycles → singleton (this test)
