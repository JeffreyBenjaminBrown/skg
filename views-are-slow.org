** Diagnosis

*** What is slow

'set_graphnodestats_in_forest' runs 6 TypeDB queries sequentially,
each taking ~3.8-4.0s, totalling ~23.3s. Each query uses a
24-way disjunction against a ~28,698-node database with nested
subqueries (fetch within match).

*** Why each query is slow

Each of the 6 queries:
1. Opens a new read transaction
2. Sends a match/fetch with 24-branch OR disjunction
3. Each branch matches a node by ID, then runs a nested
   subquery to find/count relationships
4. The nested subquery patterns are complex: disjunctions
   within disjunctions (for extra_id support)

With ~28.7K nodes and complex nested patterns,
TypeDB v3 spends ~3.8s per query — likely dominated by
query planning and/or scanning.

*** Why the total is so high

The 5 queries in 'fetch_relationship_data' run SEQUENTIALLY:

  count_containers       3.983s  ─┐
  count_contents         3.930s   │
  count_link_sources     3.769s   ├── 19.502s total (sequential)
  has_subscribes         3.981s   │
  has_overrides          3.839s  ─┘

  contains_from_pids     3.795s  ── 6th query, also sequential

These could run in parallel (they are independent read queries).

** Proposed optimizations (by expected impact)

*** 1. Parallelize the 6 independent TypeDB queries (~4× speedup)

All 6 queries in 'set_graphnodestats_in_forest' are independent
read-only queries. Using 'tokio::join!' or 'tokio::try_join!',
they could all run concurrently.

Expected: 23.3s → ~4.0s (limited by the slowest single query).

Implementation: in 'set_graphnodestats_in_forest', use
'tokio::try_join!' for 'fetch_relationship_data' and
'contains_from_pids'; within 'fetch_relationship_data',
use 'tokio::try_join!' for its 5 sub-queries.

*** 2. Merge queries to reduce round-trips (~2× on top of #1)

Instead of 5 separate fetch_relationship_data queries, combine
into 1 or 2 queries that fetch all relationship types at once.
The 'count_relations' function already parameterises the
relation/role; a combined query could fetch all 5 in one
nested structure.

Similarly, 'contains_from_pids' could be folded into the
combined query.

Expected: 1 round-trip × ~4s = ~4s, vs 6 × ~4s = ~24s.
But combined with #1 the marginal gain is smaller
(~4s → ~4s), so this is most valuable if #1 is insufficient.

*** 3. Index tuning / query simplification

The extra_id disjunction pattern (supporting lookup by
non-primary IDs) doubles the query complexity. If extra_id
lookups are rare in the graphnodestats path, they could be
dropped or handled separately.

Also: the 'has_subscribes' and 'has_overrides' queries use
'participates_in_relation', which checks BOTH roles (subscriber
and subscribee, replacement and replaced) via a 4-branch OR.
For the graphnodestats use case, we might only need to check
one role direction.

*** 4. Caching relationship data

If the same node is viewed repeatedly, caching graphnodestats
per-PID would avoid redundant queries. A simple HashMap<ID,
GraphNodeStats> with invalidation on save would suffice.

*** 5. Lazy loading of relationship stats

Instead of computing all stats before rendering, render the
view immediately with placeholder stats and fill them in
asynchronously. This would give instant visual response.

*** Remaining bottleneck

A single TypeDB query against ~28.7K nodes with a 24-way
disjunction and nested subqueries takes ~4-6 seconds. Further
improvement requires either faster queries (simplification,
indexing) or fewer round-trips (query merging).

** Further optimizations to consider

*** Merge into fewer queries (may help with contention)

Combining the 5 'count_relations' / 'participates_in' queries
into a single query that fetches all relationship types would:
- Reduce transactions from 6 to 2 (or 1)
- Eliminate contention between concurrent reads
- Each individual query might be faster without competition

*** Simplify the extra_id disjunction

The graphnodestats queries match by PID. PIDs are always the
primary ID. The extra_id OR branch adds complexity for a case
that cannot occur in this code path. Removing it would halve
the disjunction size inside the nested subqueries.

*** Simplify has_subscribes / has_overrides

'participates_in_relation' checks BOTH roles (4-branch OR).
For graphnodestats we only need to know "does this node
subscribe?" or "does this node override?", not whether it's
the subscribee/replaced side. Checking one role halves the
query complexity.

*** Cache graphnodestats results

For repeated views of the same node (common during editing),
cache per-PID stats and invalidate on save.

** Experiment 3: single combined query (fetch_all_graphnodestats)

*** Changes made

Replaced 6 separate TypeDB queries with a single combined
query in 'all_graphnodestats.rs'. One match/fetch that returns
all 5 relationship types per node, plus derives the
contains_from_pids data from the same results.

Simplifications over the old queries:
- No extra_id disjunction (PIDs are always primary IDs here)
- Uses $node directly in relationship match (no re-match by id)
- One transaction instead of 6
- No contention between concurrent reads

** Experiment 4: per-PID parallel queries (the breakthrough)

*** Insight

Single-node query: 15ms. 24-node disjunction query: 4300ms.
TypeDB's disjunction handling scales very non-linearly.
Instead of one big disjunction, run 24 tiny queries in parallel.

*** Changes made

Rewrote 'fetch_all_graphnodestats' to spawn one future per PID
using 'futures::future::join_all'. Each future runs a single
TypeDB query matching exactly one node (no disjunction at all).
