* Windows support for Skg
This folder is an unfinished Windows packaging attempt.

The intended Windows setup is:
- Docker Desktop runs the Skg server and TypeDB in a Linux container.
- Native Windows Emacs runs the Skg Emacs Lisp client.
- A generated package folder provides =run.ps1=, docs, starter data
  scripts, and the Elisp files.
- The Docker image provides the prebuilt =skg= server binary and TypeDB.

** Folder map
- =Dockerfile= builds the user image =jeffreybbrown/skg:latest=.
- =build/= contains host-side build and publish scripts.
- =testing/= contains host-side Docker smoke-test scripts.
- =container/start-servers.sh= runs inside the container and starts TypeDB
  plus the prebuilt Skg server binary.
- =install.ps1= is the Windows-side DockerHub pull script.
- =run.ps1= is the Windows-side launcher.
- =build-package.sh= assembles the downloadable Windows package folder.
- =starter-data/= contains scripts copied into a new user's data folder.
- =docs/= contains Windows-specific docs that are included in the package.
- =docker-diagnostics.org= contains a PowerShell diagnostic block for a
  Windows machine whose Docker Desktop is not starting.

Generated files are not part of the source package:
- =out/= is generated by =build/02-build-package.sh=.
- =testing/tmp/= is generated by the smoke tests.

** Important artifacts
There are two separate things a Windows user eventually needs:

- Docker image: =jeffreybbrown/skg:latest=
- Windows package folder: =windows/out/skg-windows-package/=

The image contains the server binary and container runtime files.
The package folder contains PowerShell launch scripts, docs, starter data
scripts, and the Emacs Lisp client.

If the image is available from DockerHub, the Windows user can run
=install.ps1= to pull it. If not, the image must be transferred another
way with =docker save= and =docker load=.

** Build on the NixOS host
Run these from the repository root.

Build the Docker image:

#+begin_src bash
  ./windows/build/01-build-image.sh
#+end_src

Build the downloadable Windows package folder:

#+begin_src bash
  ./windows/build/02-build-package.sh
#+end_src

The package is written to:

#+begin_src bash
  windows/out/skg-windows-package
#+end_src

Push the Docker image after smoke tests pass:

#+begin_src bash
  ./windows/build/03-push-image.sh
#+end_src

Override the image tag for any build/test/push script with =IMAGE_NAME=:

#+begin_src bash
  IMAGE_NAME=jeffreybbrown/skg:test ./windows/build/01-build-image.sh
#+end_src

** Test on the NixOS host
Run these from the repository root.

Check that the image is self-contained:

#+begin_src bash
  ./windows/testing/01-check-self-contained.sh
#+end_src

Run the one-line smoke test:

#+begin_src bash
  ./windows/testing/03-smoke-test.sh
#+end_src

The smoke-test defaults are:
- image: =jeffreybbrown/skg:latest=
- container: =skg-win-test=
- host data root: =windows/testing/tmp/skg-win-test=
- host/container Skg port: =1739=

Override the port if it collides with local Skg instances:

#+begin_src bash
  SKG_TEST_PORT=1742 ./windows/testing/03-smoke-test.sh
#+end_src

Other test helpers:

#+begin_src bash
  ./windows/testing/02-prepare-data.sh
  ./windows/testing/04-logs.sh
  ./windows/testing/05-cleanup.sh
#+end_src

If test artifacts are owned by Docker-created users and cannot be removed
from inside a dev container, remove them from the host with appropriate
permissions. They are ignored by =.dockerignore=.

** What the package builder includes
=build-package.sh= writes a package containing:
- =install.ps1=
- =run.ps1=
- =WINDOWS.org=
- project =README.org= at the package root
- root =glossary.md=
- selected project docs under =docs/=
- =elisp/=
- =starter-data/=

The root =README.org= is intentionally one directory above =docs/= so
relative links like =[[./docs/setup.org]]= keep working.

Selected docs currently include:
- =docs/setup.org=
- =docs/COMMANDS.org=
- =docs/merging.org=
- =docs/moving-nodes-across-repos.org=
- =docs/hyperlinks.md=
- =docs/forks.md=
- =glossary.md= at package root

** Windows-side intended flow
On the Windows machine:

1. Install Docker Desktop and make sure it can run WSL2 containers.
2. Install GNU Emacs for Windows.
3. Make sure the Docker image is available, either by DockerHub:

#+begin_src powershell
  docker pull jeffreybbrown/skg:latest
#+end_src

or by loading an image archive:

#+begin_src powershell
  docker load -i C:\path\to\skg-docker-image.tar
#+end_src

4. Copy =windows/out/skg-windows-package/= to a normal Windows folder.
5. In regular PowerShell:

#+begin_src powershell
  cd C:\path\to\skg-windows-package
  .\run.ps1
#+end_src

The run script asks for a data folder, remembers the answer, initializes
=public/= and =private/= as empty git repos when needed, starts/reuses the
Docker container, waits for Skg's TCP port, and loads Skg into native
Windows Emacs.

** Docker Desktop diagnostics
If Docker Desktop hangs at "Starting Docker Engine", copy or read
=docker-diagnostics.org= on the Windows machine. It contains a PowerShell
diagnostic block that writes a report to the user's Desktop.

Useful manual checks on Windows:

#+begin_src powershell
  wsl -l -v
  docker info
  docker version
#+end_src

For Docker Desktop with WSL2, =wsl -l -v= should show WSL2 distributions,
and Docker should eventually create/use Docker-related WSL distributions.
